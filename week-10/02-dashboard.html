Week 10 · บทที่ 2

Dashboard — รวมสถิติ + การ์ดสรุป

หน้าภาพรวมที่เปลี่ยนข้อมูลดิบเป็นสิ่งที่ผู้ใช้ตัดสินใจได้

🔑 คีย์เวิร์ดภาพจำประจำบท

📊 สรุปที่ server = aggregate (COUNT/SUM) 🃏 ตัวเลขสำคัญ = stat card 🚀 ยิงพร้อมกัน = Promise.all 📉 ข้อมูลน้อยก็ต้องสวย = empty state

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "สรุปที่ server ไม่ใช่ที่ client"

อยากนับ "หนังสือที่ยืมอยู่" — อย่าดึงทั้งหมดมานับที่ frontend (ช้า, โหลดข้อมูลเยอะ) · ให้ DB นับด้วย COUNT แล้วส่งแค่ตัวเลข · aggregate ที่ใกล้ข้อมูล (DB) เร็วกว่าและประหยัด bandwidth มาก · นี่คือหลักการ dashboard ที่ scale ได้

หลักการ 2: "Dashboard ต้องตอบคำถาม ไม่ใช่โชว์ข้อมูล"

การ์ดสถิติที่ดีตอบ "อะไรสำคัญตอนนี้" (หนังสือเหลือกี่เล่ม, ค้างคืนกี่เล่ม) ไม่ใช่ dump ทุกตัวเลข · ออกแบบจากคำถามที่ผู้ใช้อยากรู้ → เลือกแสดงเฉพาะนั้น · dashboard ที่รก = ไม่มีใครดู

หลักการ 3: "ดึงหลายแหล่งขนานกัน (Promise.all จาก Week 1)"

dashboard มักรวมข้อมูลหลายอย่าง (จำนวนหนังสือ + สมาชิก + ที่ยืม) · ยิงพร้อมกันด้วย Promise.all เร็วกว่ารอทีละอัน · ความรู้ async จาก Week 1 ใช้จริงตรงนี้

ส่วนที่ 1: Aggregate ที่ Backend

// backend: endpoint สรุป — ให้ DB นับ ไม่ดึงมานับเอง
// GET /api/dashboard
const stats = await pool.query(`
  SELECT
    (SELECT COUNT(*) FROM books) AS total_books,
    (SELECT COUNT(*) FROM books WHERE available = false) AS borrowed,
    (SELECT COUNT(*) FROM members) AS total_members
`);
res.json({ ok: true, data: stats.rows[0] });

DB นับให้ → ส่งแค่ตัวเลขไม่กี่ค่า · ต่างจากดึงหนังสือพันเล่มมานับที่ frontend (ช้า เปลือง) — aggregate ใกล้ข้อมูลเสมอ

ส่วนที่ 2: Stat Cards (Tailwind)

// การ์ดสถิติ — reusable component
function StatCard({ label, value, icon }: {
  label: string; value: number; icon: string;
}) {
  return (
    <div className="p-5 rounded-xl bg-white shadow flex items-center gap-4">
      <div className="text-3xl">{icon}</div>
      <div>
        <p className="text-sm text-gray-500">{label}</p>
        <p className="text-2xl font-bold">{value.toLocaleString()}</p>
      </div>
    </div>
  );
}

// dashboard grid (responsive)
<div className="grid grid-cols-1 md:grid-cols-3 gap-4">
  <StatCard label="หนังสือทั้งหมด" value={stats.total_books} icon="📚" />
  <StatCard label="ถูกยืมอยู่" value={stats.borrowed} icon="📖" />
  <StatCard label="สมาชิก" value={stats.total_members} icon="👥" />
</div>

toLocaleString() = ใส่ comma (1,234) อ่านง่าย · grid responsive (1 คอลัมน์มือถือ, 3 จอใหญ่)

ส่วนที่ 3: ดึงหลายแหล่งขนานกัน

// ถ้า dashboard ต้องรวมจากหลาย endpoint — ยิงพร้อมกัน (Week 1)
async function loadDashboard() {
  const [stats, recent, overdue] = await Promise.all([
    fetch("/api/dashboard").then(r => r.json()),
    fetch("/api/borrows/recent").then(r => r.json()),
    fetch("/api/borrows/overdue").then(r => r.json()),
  ]);
  // เร็วกว่ารอทีละอัน ~3 เท่า
}
production: ถ้าบางแหล่งอาจล่ม ใช้ Promise.allSettled (Week 1) — dashboard ยังแสดงส่วนที่ได้ ไม่พังทั้งหน้า

npm install recharts

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: นี่คือหัวใจงานคุณเลย — dashboard การตลาดที่รวม Meta/LINE/IG/GA4 · aggregate ที่ server (สรุป engagement, reach), ยิงหลายแหล่งด้วย Promise.allSettled (บางแพลตฟอร์มล่มก็ยังแสดงที่เหลือ), stat cards + กราฟแนวโน้ม · Week 10 = ซ้อมใหญ่ของงานจริงคุณ

🌏 อุตสาหกรรม: ทุก SaaS/B2B มี dashboard · aggregate ที่ DB, ดึงขนาน, stat card + chart เป็น pattern มาตรฐาน · dashboard ที่ออกแบบดี (ตอบคำถาม ไม่รก) คือสิ่งที่ลูกค้าจ่ายเงิน

📝 Quiz ท้ายบท

ข้อ 1: นับ "หนังสือที่ยืมอยู่" ควรทำที่ server หรือ client เพราะอะไร?

Server — ให้ DB นับด้วย COUNT แล้วส่งแค่ตัวเลข · ดึงทั้งหมดมานับที่ client = โหลดข้อมูลเยอะ ช้า เปลือง bandwidth · aggregate ใกล้ข้อมูล (DB) เร็วและ scale ได้

ข้อ 2: dashboard ที่ดีต่างจาก "dump ทุกตัวเลข" ยังไง?

ดี = ตอบคำถามที่ผู้ใช้อยากรู้ (อะไรสำคัญตอนนี้) แสดงเฉพาะนั้น · dump = โชว์ทุกอย่างจนรก ไม่มีใครดู · ออกแบบจากคำถาม → เลือกแสดง

ข้อ 3: dashboard ดึง 3 แหล่ง ทำยังไงให้เร็ว? ถ้าบางแหล่งอาจล่ม?

ยิงพร้อมกันด้วย Promise.all (เร็วกว่ารอทีละอัน) · ถ้าบางแหล่งอาจล่ม ใช้ Promise.allSettled — แสดงส่วนที่ได้ ไม่พังทั้ง dashboard (ความรู้ Week 1)

ข้อ 4: อยากมีกราฟใน dashboard ควรวาดเองไหม?

ไม่ — ใช้ library (Recharts/Chart.js) ใส่ข้อมูลแล้วมันวาดให้ · วาดเองเสียเวลามากและทำได้ไม่ดีเท่า · เลือกกราฟที่ตอบคำถาม (แนวโน้ม) ไม่ใช่เพื่อความสวย

✅ Checkpoint

  • aggregate endpoint (DB COUNT/SUM) ไม่นับที่ client
  • stat cards (reusable, responsive grid)
  • ดึงหลายแหล่งด้วย Promise.all/allSettled
  • (optional) chart ด้วย library
