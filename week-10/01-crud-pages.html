CRUD Pages — List, Add, Edit, Delete

4 หน้าจัดการข้อมูลที่เป็นหัวใจของ business app ทุกตัว

🔑 คีย์เวิร์ดภาพจำประจำบท

🎛 input ผูก state = controlled component 📝 form เดียว = add + edit (รับ initial) 🔄 หลัง mutate ต้องอัปเดต UI ⚡ เปลี่ยน UI ก่อนรอ server = optimistic

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Controlled component — state เป็นเจ้าของค่า input"

ใน React ค่าของ input ควรอยู่ใน state (single source of truth) ไม่ใช่ใน DOM · input แสดงค่าจาก state, พิมพ์แล้วอัปเดต state · ทำให้ควบคุม validate, reset, prefill ได้ง่าย · นี่คือหลัก Single Source of Truth (Week 1) ในบริบท form

หลักการ 2: "UI ต้องตรงกับ server เสมอ — mutate แล้วต้อง sync"

เพิ่ม/แก้/ลบ ข้อมูลแล้ว UI ต้องอัปเดตให้ตรงกับ server · 2 ทาง: refetch (ดึงใหม่ — ง่าย ชัวร์) หรือ optimistic (เปลี่ยน UI ทันที แล้ว rollback ถ้าพัง — เร็วแต่ซับซ้อน) · ลืม sync = ผู้ใช้เห็นข้อมูลเก่า งง

หลักการ 3: "Form add/edit คือของเดียวกัน — ต่างแค่ค่าเริ่มต้น"

add = form ว่าง · edit = form ที่ prefill ค่าเดิม · logic เหมือนกัน 90% → ใช้ component เดียวรับ prop initial · เขียน 2 form แยก = ซ้ำ แก้ field ต้องแก้ 2 ที่ · DRY

ส่วนที่ 1: Controlled Form

"use client";
function BookForm({ initial, onSubmit }: {
  initial?: Book; onSubmit: (data: BookInput) => Promise<void>;
}) {
  const [title, setTitle] = useState(initial?.title ?? "");   // prefill ถ้า edit
  const [author, setAuthor] = useState(initial?.author ?? "");
  const [error, setError] = useState("");
  const [saving, setSaving] = useState(false);

  async function handleSubmit(e: React.FormEvent) {
    e.preventDefault();
    if (!title.trim()) return setError("กรุณากรอกชื่อหนังสือ");   // validate (UX)
    setSaving(true); setError("");
    try {
      await onSubmit({ title, author });          // ส่งให้ parent จัดการ
    } catch {
      setError("บันทึกไม่สำเร็จ ลองใหม่");          // จัดการ error ตอน save
    } finally {
      setSaving(false);
    }
  }
  return (
    <form onSubmit={handleSubmit}>
      <input value={title} onChange={e => setTitle(e.target.value)} />
      <input value={author} onChange={e => setAuthor(e.target.value)} />
      {error && <p className="text-red-600">{error}</p>}
      <button disabled={saving}>{saving ? "กำลังบันทึก..." : "บันทึก"}</button>
    </form>
  );
}
ใช้ทั้ง add และ edit: <BookForm onSubmit={create} /> (ว่าง) · <BookForm initial={book} onSubmit={update} /> (prefill) — form เดียว ต่างแค่ initial + handler

ส่วนที่ 2: อัปเดต UI หลัง Mutate — 2 วิธี

RefetchOptimistic
ทำยังไงmutate เสร็จ → ดึงข้อมูลใหม่เปลี่ยน UI ทันที → mutate → rollback ถ้าพัง
ข้อดีง่าย ชัวร์ตรง serverเร็ว (ผู้ใช้ไม่รอ)
ข้อเสียรอ network (ช้านิด)ซับซ้อน ต้องจัดการ rollback
ใช้เมื่อ⭐ default — เริ่มจากนี่action ที่ต้องรู้สึกเร็ว (like, toggle)
// Refetch (เริ่มจากนี่ — ง่ายและถูกต้อง)
async function handleDelete(id: number) {
  if (!confirm("ลบหนังสือเล่มนี้?")) return;        // confirm ก่อนลบ
  await fetch(`/api/books/${id}`, { method: "DELETE" });
  await refetchBooks();                            // ดึงใหม่ → UI ตรง server
}

เริ่มจาก refetch เสมอ (ถูกต้องชัวร์) · optimistic เป็น optimization ทำทีหลังเมื่อจำเป็น — อย่า over-engineer ตั้งแต่แรก

ส่วนที่ 4: ตัวอย่างประยุกต์

🏢 งานวี: ระบบจัดการ content/ลูกค้าใน dashboard — CRUD pattern เดียวกันเป๊ะ (list ลูกค้า, เพิ่ม, แก้ config, ลบ) · form เดียวสำหรับ add/edit config + refetch หลังบันทึก · นี่คือ 80% ของ admin tool ทุกตัว

🌏 อุตสาหกรรม: CRUD + form handling เป็นงานหลักของ frontend dev ในทุกบริษัท · controlled component + การ sync หลัง mutate เป็นพื้นฐานที่ interviewer ดูว่า "ทำแอปจริงเป็นไหม" · ทำเป็นแล้วต่อยอด React Query/SWR (จัดการ refetch อัตโนมัติ) ได้ง่าย

📝 Quiz ท้ายบท

ข้อ 1: controlled component คืออะไร ดียังไง?

ค่า input อยู่ใน state (ไม่ใช่ DOM) — input แสดงค่าจาก state, พิมพ์อัปเดต state · ดี: ควบคุม validate/reset/prefill ง่าย, เป็น single source of truth · ต่างจาก uncontrolled (ค่าอยู่ใน DOM อ่านตอน submit)

ข้อ 2: ทำไม add form กับ edit form ควรเป็น component เดียว?

logic เหมือนกัน 90% (field, validate, submit) ต่างแค่ค่าเริ่มต้น (add=ว่าง, edit=prefill) · component เดียวรับ prop initial → DRY, แก้ field ที่เดียว · เขียน 2 form = ซ้ำ ลืม sync

ข้อ 3: ลบข้อมูลแล้วไม่อัปเดต UI เกิดอะไร แก้ยังไง 2 วิธี?

ผู้ใช้ยังเห็นข้อมูลที่ลบไปแล้ว (UI ไม่ตรง server) งง · แก้: ① refetch (ดึงใหม่หลังลบ — ง่าย ชัวร์) ② optimistic (ลบจาก UI ทันที rollback ถ้าพัง — เร็วแต่ซับซ้อน) · เริ่มจาก refetch

ข้อ 4: empty state ที่ดีต่างจากแย่ยังไง?

แย่: "ไม่มีข้อมูล" เปล่า ๆ (ผู้ใช้งง ทำไงต่อ) · ดี: บอกบริบท + ทางออก เช่น "ยังไม่มีหนังสือ — [+ เพิ่มเล่มแรก]" · empty state คือโอกาส guide ผู้ใช้ ไม่ใช่หน้าตาย
