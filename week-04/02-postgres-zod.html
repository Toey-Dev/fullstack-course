🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Relational = ข้อมูลมีโครงและความสัมพันธ์"

JSON file เก็บอะไรก็ได้มั่ว ๆ — database แบบ relational บังคับ "โครง" (schema): book ต้องมี title เป็นข้อความ, price เป็นเลข ความเข้มงวดนี้คือข้อดี: ข้อมูลสะอาด, query เร็ว (index), เชื่อมตารางได้ (book → member ที่ยืม) library-system จะมี books, members, borrows ที่เชื่อมกัน — นี่คืองานของ relational DB

หลักการ 2: "Never trust input — รวมถึงตอนคุยกับ DB"

SQL Injection คือช่องโหว่อันดับ 1 ของ OWASP มานานหลายปี — เกิดจากเอา input ผู้ใช้ไป "ต่อ string" เป็น SQL ตรง ๆ คุณในฐานะ pentester เคยใช้มันเจาะระบบมาแล้ว ตอนนี้เรียนวิธี "ปิด" ด้วย parameterized query (DB แยกคำสั่งกับข้อมูลออกจากกัน)

หลักการ 3: "Validate ที่ขอบระบบ — ก่อนข้อมูลเข้าไปข้างใน"

validation ควรเกิดที่ "ประตู" (รับ request) ก่อนข้อมูลไหลเข้า business logic — เหมือนด่านตรวจที่สนามบิน Zod ทำให้ validation เป็น "schema เดียว" ที่อธิบายว่าข้อมูลที่ถูกต้องหน้าตายังไง ใช้ซ้ำได้ แปลง type ให้ และ error สวยกว่า if มือ

ส่วนที่ 1: SQL พื้นฐาน — CRUD 4 คำสั่ง

สร้างตารางก่อน (รันใน DBeaver):

CREATE TABLE books (
  id        SERIAL PRIMARY KEY,        -- auto เพิ่มเลขเอง + เป็นบัตรประชาชนของแถว
  title     VARCHAR(200) NOT NULL,     -- ข้อความ ห้ามว่าง
  author    VARCHAR(200),
  available BOOLEAN DEFAULT true,
  created_at TIMESTAMP DEFAULT now()
);
งาน (CRUD)SQL
CreateINSERT INTO books (title, author) VALUES ('Clean Code', 'Martin');
Read ทั้งหมดSELECT * FROM books;
Read เฉพาะSELECT * FROM books WHERE id = 1;
UpdateUPDATE books SET available = false WHERE id = 1;
DeleteDELETE FROM books WHERE id = 1;

CRUD = Create/Read/Update/Delete — ตรงกับ POST/GET/PUT/DELETE ของ REST ที่เรียน Week 2 พอดี (API คือหน้ากากของ SQL)

ส่วนที่ 2: เชื่อม Express กับ PostgreSQL ด้วย pg

npm install pg                    # PostgreSQL client สำหรับ Node

// src/db.js — สร้าง connection pool ใช้ร่วมทั้งแอป
const { Pool } = require("pg");
const pool = new Pool({ connectionString: process.env.DATABASE_URL });
module.exports = pool;

ใช้ใน repository (ชั้นที่คุย DB):

// src/repositories/books.repo.js
const pool = require("../db");

exports.findAll = async () => {
  const result = await pool.query("SELECT * FROM books ORDER BY id");
  return result.rows;          // pg คืน { rows, rowCount, ... }
};

exports.findById = async (id) => {
  const result = await pool.query("SELECT * FROM books WHERE id = $1", [id]);
  return result.rows[0] ?? null;   // ไม่เจอ = undefined → คืน null
};
ทำไมใช้ Pool ไม่ใช่ connect ทุกครั้ง: เปิด connection ใหม่ทุก request ช้ามากและเปลือง — Pool = "ถังคอนเนกชันสำรอง" reuse ได้ เร็วกว่าหลายเท่า (production มาตรฐาน)

ส่วนที่ 3: 🔐 SQL Injection — ช่องโหว่ที่คุณต้องปิดเป็น

กฎเหล็ก: ห้ามเอา input ผู้ใช้ไปต่อใน SQL string เด็ดขาด — ใช้ $1, $2 (placeholder) แล้วส่งค่าเป็น array เสมอ pg จะ escape ให้อัตโนมัติ ทำให้ input เป็นได้แค่ "ข้อมูล" ไม่มีทางกลายเป็น "คำสั่ง"

มุม pentester ของคุณ: นี่คือช่องโหว่ที่คุณเคยล่าตาม OWASP — ตอนนี้คุณเป็นคนสร้างกำแพง AI Review Drill สัปดาห์นี้ให้หา SQL Injection ในโค้ด AI เพราะ AI มักต่อ string ตรง ๆ (จุดที่คุณมองออกก่อนคนอื่น)

ส่วนที่ 4: Zod — validation ที่เป็นมาตรฐาน

npm install zod

// src/schemas/book.schema.js
const { z } = require("zod");

const BookSchema = z.object({
  title:  z.string().min(1, "ต้องมีชื่อหนังสือ").max(200),
  author: z.string().optional(),
  available: z.boolean().default(true),
});

module.exports = { BookSchema };

ใช้เป็น middleware validate ก่อนเข้า controller:

// validate middleware
const validate = (schema) => (req, res, next) => {
  const result = schema.safeParse(req.body);
  if (!result.success) {
    return res.status(400).json({ ok: false, error: result.error.issues });
  }
  req.body = result.data;     // ⭐ ใช้ข้อมูลที่ผ่าน+แปลง type แล้ว
  next();
};

router.post("/", validate(BookSchema), booksController.create);
Zod ทำให้ต่างจาก if มือยังไง
เช็คครบในที่เดียวif กระจายหลายบรรทัด ลืมง่าย
แปลง type ให้ ("5" → 5)if ต้องแปลงเอง
error message มีโครงif ต้องเขียน error เอง
schema ใช้ซ้ำได้ (สร้าง/แก้ ใช้ตัวเดียว)if ต้อง copy
safeParse vs parse: safeParse คืน {success, data/error} ไม่ throw (เหมาะกับ middleware) · parse throw error ถ้าผิด (เหมาะกับที่มี try/catch) — เลือกตามว่าอยากจัดการ error แบบไหน

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: automation report ที่รับ config ลูกค้า (ช่วงเวลา, แพลตฟอร์ม) — Zod schema validate ว่า config ถูกต้องก่อนยิง API กัน config พังทำให้ report เพี้ยน · และถ้าเก็บ data ลง PostgreSQL จะ query/สรุปได้เร็วกว่าอ่านไฟล์มาก

🌏 อุตสาหกรรม: Zod เป็นมาตรฐานในทีม TypeScript ทั่วโลก (จะเข้าคู่กับ TS ใน Week 6 ได้สวยมาก — infer type จาก schema) · parameterized query เป็นข้อบังคับใน code review ทุกที่ที่ใส่ใจ security — เขียนผิดครั้งเดียวอาจเป็นข่าวข้อมูลรั่ว

📝 Quiz ท้ายบท

ข้อ 1: ทำไม relational DB (PostgreSQL) ดีกว่าเก็บ JSON file สำหรับ library-system?

① บังคับโครง (schema) ข้อมูลสะอาด ② query เร็ว (index) ③ หลายคนเข้าพร้อมกันได้ (JSON file ชนกัน) ④ เชื่อมตารางได้ (book ↔ member ที่ยืม) ⑤ transaction (ยืม-คืนต้องสำเร็จทั้งคู่ — Week 5)

ข้อ 2 (security): `SELECT * FROM books WHERE id = ${id}` อันตรายยังไง แก้ยังไง?

SQL Injection — ถ้า id = "1; DROP TABLE books" DB จะอ่านเป็นคำสั่งและลบตาราง · แก้ด้วย parameterized: query("...WHERE id = $1", [id]) — DB แยก "คำสั่ง" (มี $1) ออกจาก "ข้อมูล" (array) input เป็นได้แค่ข้อมูล

ข้อ 3: Zod ดีกว่า if เช็คมือยังไง (3 ข้อ)?

① เช็คครบในที่เดียว (schema) ไม่กระจาย ② แปลง type ให้ ("5"→5, default value) ③ error มีโครงพร้อมใช้ · บวก: schema ใช้ซ้ำได้ทุก route และ infer TypeScript type ได้ (Week 6)

ข้อ 4: ทำไมใช้ connection Pool ไม่ใช่ pool.connect() ใหม่ทุก request?

เปิด connection ใหม่ทุก request ช้ามาก (handshake กับ DB) และเปลือง resource จนเต็มได้ · Pool = ถังคอนเนกชันสำรองที่ reuse ได้ request มาก็หยิบจากถัง ใช้เสร็จคืน — เร็วกว่าหลายเท่า เป็น production มาตรฐาน
