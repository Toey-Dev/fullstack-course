Week 4 · บทที่ 1

Docker — แก้ปัญหา "แต่มันรันในเครื่องผมได้นะ!"

เครื่องมือที่ทุกบริษัทยุคใหม่ใช้ — และทำให้คุณรัน PostgreSQL ได้โดยไม่ลงเลอะเครื่อง

🔑 คีย์เวิร์ดภาพจำประจำบท

📦 กล่องพร้อมของครบ = Container 📜 พิมพ์เขียวสร้างกล่อง = Image 🍱 สั่งหลายกล่องทีเดียว = docker-compose 🏠 "รันในเครื่องผมได้" = ปัญหาที่ Docker แก้ 🔌 ต่อท่อออกมา = port mapping

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Works on my machine" คือปัญหาที่ Docker เกิดมาเพื่อฆ่า

คุณเขียนโค้ดในเครื่อง Windows + Node 20 + PostgreSQL 15 → ส่งให้เพื่อน Mac + Node 18 → พัง เพราะเวอร์ชันไม่ตรง, library ไม่ครบ, config ต่าง Docker แก้ด้วยการ "แพ็คทุกอย่าง (โค้ด+runtime+config) ลงกล่องเดียว" กล่องนี้รันเหมือนกันเป๊ะทุกเครื่อง — เครื่อง dev, เครื่องเพื่อน, server production

หลักการ 2: "Isolation" — แยกของแต่ละโปรเจกต์ไม่ให้ชนกัน

โปรเจกต์ A ต้องการ PostgreSQL 14, โปรเจกต์ B ต้องการ 16 — ลงลงเครื่องตรง ๆ ชนกันแน่ Docker ให้แต่ละโปรเจกต์มี container ของตัวเอง แยกกันสนิท ลบทิ้งก็ไม่เลอะเครื่อง (นี่คือเหตุผลที่เราใช้ Docker รัน PostgreSQL แทนลงลงเครื่อง — สะอาด ลบง่าย)

หลักการ 3: "Infrastructure as Code" — โครงสร้างเป็นไฟล์ที่อ่านได้

docker-compose.yml 1 ไฟล์บอกครบว่าระบบต้องมี service อะไรบ้าง (app, db, redis) เวอร์ชันไหน ต่อกันยังไง — เพื่อนร่วมทีม clone แล้วพิมพ์ docker compose up ได้ระบบเหมือนคุณเป๊ะ ไม่ต้องไล่ลงทีละอย่าง config ที่เป็น "โค้ด" = version control ได้ review ได้ ทำซ้ำได้

ส่วนที่ 1: Image vs Container — เข้าใจ 2 คำนี้ = เข้าใจ Docker ครึ่งหนึ่ง

Image 📜Container 📦
คือtemplate อ่านอย่างเดียวinstance ที่กำลังรันจริง
เทียบกับโค้ดclassobject (new จาก class)
เทียบกับครัวสูตรอาหารจานที่ทำเสร็จ

ส่วนที่ 2: รัน PostgreSQL ด้วย docker-compose

สร้างไฟล์ docker-compose.yml ที่ root ของ library-system:

services:
  db:
    image: postgres:16              # ใช้ image สำเร็จรูปจาก Docker Hub
    container_name: library-db
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: mysecret   # ⚠️ จริง ๆ ควรมาจาก .env
      POSTGRES_DB: library
    ports:
      - "5432:5432"                 # ต่อท่อ: port เครื่องเรา : port ใน container
    volumes:
      - db_data:/var/lib/postgresql/data   # เก็บข้อมูลถาวร (ไม่หายตอนลบ container)

volumes:
  db_data:

แล้วรัน:

docker compose up -d      # -d = รันเบื้องหลัง
docker compose ps         # ดูว่า container รันอยู่ไหม
docker compose down       # หยุด (ข้อมูลยังอยู่ใน volume)
docker compose logs db    # ดู log ถ้ามีปัญหา
2 จุดที่มือใหม่งง:
① port "5432:5432" — ซ้าย = port เครื่องเรา (Node เชื่อมผ่านอันนี้), ขวา = port ใน container · ถ้าเครื่องมี PostgreSQL อยู่แล้วชน ให้เปลี่ยนซ้ายเป็น "5433:5432"
② volume — ถ้าไม่ใส่ ข้อมูลหายตอน down · ใส่แล้วข้อมูลอยู่ถาวรแม้ลบ container

ส่วนที่ 3: ดูข้อมูลด้วย DBeaver (GUI)

PostgreSQL ใน container มองไม่เห็นด้วยตา — ใช้ DBeaver (ฟรี) ต่อเข้าไปดู/แก้ตารางได้:

  • Host: localhost · Port: 5432 · Database: library
  • User: postgres · Password: mysecret (ตามที่ตั้งใน compose)

DBeaver = เครื่องมือดู database แบบ GUI ที่ทุก dev มีติดเครื่อง — เห็นตาราง รัน SQL ทดสอบ ดูข้อมูลจริง ก่อนเขียนโค้ด

ส่วนที่ 4: .env — เก็บความลับให้ถูกที่

# .env (ห้าม commit! ใส่ใน .gitignore)
DATABASE_URL=postgresql://postgres:mysecret@localhost:5432/library
PORT=3000
// โหลด .env ใน Node
require("dotenv").config();      // npm i dotenv
const url = process.env.DATABASE_URL;
🔐 มุม pentester: password, connection string, API key ห้ามอยู่ในโค้ดที่ commit — เพราะ push ขึ้น GitHub public แล้ว bot จะสแกนเจอใน < 1 นาที (มี bot ไล่หา secret ใน repo ตลอดเวลา) ใส่ใน .env + .gitignore เสมอ และทำ .env.example (ไม่มีค่าจริง) ไว้บอกเพื่อนว่าต้องตั้งอะไรบ้าง

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: ถ้า rebuild automation report เป็น service (Week 16) — Docker ทำให้คุณรัน + ส่งให้เพื่อนทดสอบได้โดยไม่ต้องไล่ตั้งค่า "compose up" จบ ลูกค้า/ทีมได้ระบบเหมือนกันเป๊ะ

🌏 อุตสาหกรรม: ทุกบริษัท 100k+ deploy ด้วย container — โค้ดที่ test ในเครื่อง dev (ใน Docker) จะรันเหมือนกันบน production กำจัดปัญหา "เครื่อง dev ได้ แต่ production พัง" ที่เคยกินเวลา debug เป็นวัน · Docker คือทักษะที่ JD ของ DevSecOps (สาย B ของคุณ) ขอแทบทุกตำแหน่ง

📝 Quiz ท้ายบท

ข้อ 1: Image กับ Container ต่างกันยังไง เทียบกับโค้ด?

Image = class (template อ่านอย่างเดียว) · Container = object (instance ที่รันจริงจาก image) — 1 image สร้างได้หลาย container เช่น postgres:16 (image) → รันได้หลาย container แยกกันสำหรับแต่ละโปรเจกต์

ข้อ 2: ใน ports: "5433:5432" เลขซ้ายกับขวาคืออะไร เมื่อไหร่ต้องเปลี่ยน?

ซ้าย (5433) = port บนเครื่องเรา (Node เชื่อมผ่านอันนี้) · ขวา (5432) = port ใน container (PostgreSQL ฟังอันนี้) · เปลี่ยนซ้ายเมื่อ port เครื่องชน (เช่นมี PostgreSQL ลงเครื่องอยู่แล้วใช้ 5432) — ขวาไม่ต้องเปลี่ยน

ข้อ 3: ถ้าไม่ใส่ volume ใน compose แล้ว docker compose down เกิดอะไรกับข้อมูล?

ข้อมูลหายหมด — เพราะ container ถูกลบ และข้อมูลที่อยู่ในตัว container ก็หายไปด้วย · volume คือ "ที่เก็บถาวรนอก container" ทำให้ข้อมูลรอด down/up · production ต้องมี volume เสมอ

ข้อ 4 (security): commit ไฟล์ที่มี DB password ขึ้น GitHub public อันตรายแค่ไหน?

อันตรายมาก — มี bot สแกน secret ใน public repo ตลอดเวลา เจอใน < 1 นาที · แม้ลบ commit ทีหลังก็ยังอยู่ใน git history ป้องกัน: .env + .gitignore เสมอ, ใช้ .env.example (ไม่มีค่าจริง) บอกโครง, ถ้าหลุดแล้วต้อง rotate (เปลี่ยน) password ทันที

✅ Checkpoint

  • docker-compose.yml รัน PostgreSQL ขึ้น (docker compose ps เห็น running)
  • ต่อ DBeaver เข้า database ได้
  • .env + .gitignore (password ไม่หลุดเข้า git)
  • Quiz + โน้ต notes/docker-basics.md
