Week 16 · Mission ใหญ่

📊 สร้าง Automation Report Service

เอางานจริงมาทำเป็น service ที่ใช้ทุกอย่างจาก 15 สัปดาห์ — portfolio ที่แก้ปัญหาจริง

ทำไม Mission นี้คือ portfolio ที่ดีที่สุดของเฟส 2

service นี้รวมทุกอย่าง: TypeScript (W6-7) · resilience/async (W1) · Redis cache (W11) · Docker (W11-12) · CI/CD (W13-14) · logging/Sentry (W15) · security (ทั้งคอร์ส) — และที่สำคัญ แก้ปัญหาจริงในงานคุณ · เล่า interview ได้ว่า "automate งานที่ทำมือทุกเดือน ลดเวลาจาก X เหลือ Y" — impact ที่วัดได้

⚠️ ทำเวอร์ชัน portfolio (generic): ใช้ mock API / sample data — ไม่ใส่ key จริง, ข้อมูลลูกค้าจริง, business logic ลับ · ของจริงในงานเก็บ private repo แยก

🪜 แบ่งเป็น 4 Mission ย่อย

Mission 16.1 — Typed Clients (mock) 75 นาที

  1. MetricsClient interface + clients 2-3 ตัว (mock API ที่ return sample data)
  2. retry + backoff + timeout ใน client
  3. normalize: แต่ละ mock คืน format ต่างกัน → แปลงเป็น Metrics รูปเดียว
✅ เทสต์: client ดึง mock ได้ · ทำให้ mock fail → retry ทำงาน · เพิ่ม client ใหม่ไม่แตะของเดิม

Mission 16.2 — Aggregate + Cache 60 นาที

  1. buildReport ด้วย Promise.allSettled (ทนแหล่งล่ม + บันทึก failedPlatforms)
  2. cache external (Redis) — ไม่ยิง mock ซ้ำ
  3. transformer แยก: รวม/สรุป metrics เป็น report object
เทสต์ resilience: ทำให้ 1 client fail → report ยังออก (มี failedPlatforms) ไม่พังหมด

Mission 16.3 — Scheduled + Export + วัดผล 75 นาที

  1. scheduled job (node-cron) — รันตามเวลา + log เวลาที่ใช้
  2. exporter แยก: JSON + CSV (อย่างน้อย 2)
  3. log metric: "ทำ N report ใน X วินาที" (เทียบทำมือ)
✅ เทสต์: trigger job → ได้ไฟล์ JSON+CSV · log บอกเวลาที่ใช้

Mission 16.4 — Dockerize + Document + push 45 นาที

  1. Dockerfile multi-stage (W11-12) + non-root + .env (key ไม่ commit)
  2. logging (pino) + Sentry-ready (W15) · ตรวจไม่ log key
  3. README: ปัญหาที่แก้ + architecture + metric (เวลาที่ประหยัด) + AI Usage · push + pin
✅ เทสต์: README เล่าได้ว่า "แก้ปัญหาอะไร ประหยัดเวลาเท่าไหร่" — เหมือน case study
