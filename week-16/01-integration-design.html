Week 16 · บทที่ 1

ออกแบบ Integration Service + API Client

ดึงข้อมูลจากโลกภายนอกอย่างเป็นระบบ — typed, resilient, ปลอดภัย

🔑 คีย์เวิร์ดภาพจำประจำบท

🔌 ตัวห่อ API = client (interface เดียว) 🔁 ลองใหม่+รอนานขึ้น = retry + backoff ⏱ ไม่รอตลอดกาล = timeout 🚦 อย่ายิงเกินโควต้า = rate limit 💾 จำคำตอบ = cache external

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "ห่อ API ภายนอกด้วย client — อย่าให้ทั่วทั้งระบบรู้จัก vendor"

ถ้าเรียก fetch ไป Meta API ตรง ๆ ทั่วโค้ด — วันที่ Meta เปลี่ยน API ต้องแก้ทุกที่ · ห่อด้วย "client" (interface เดียว fetch(): Promise<Metrics>) → ข้างในเปลี่ยนได้ ข้างนอกไม่กระทบ · เพิ่มแพลตฟอร์มใหม่ = client ใหม่ที่ implement interface เดิม · นี่คือ adapter pattern — แยก "ระบบเรา" จาก "vendor ภายนอก"

หลักการ 2: "External call จะพังเสมอ — ออกแบบเผื่อ"

API ภายนอกควบคุมไม่ได้: ล่ม, ช้า, rate limit, เปลี่ยน format · ต้องมี timeout (ไม่ค้างตลอดกาล), retry + backoff (ลองใหม่ แต่รอนานขึ้นแต่ละครั้ง ไม่กระหน่ำ), และยอมรับว่าบางครั้งล้มเหลว · "never trust the network" (Week 1) ที่ scale ขึ้น

หลักการ 3: "เคารพ rate limit + cache — เป็น API citizen ที่ดี"

API ภายนอกมี rate limit (เรียกได้กี่ครั้ง/นาที) · ยิงเกิน = โดน block · cache คำตอบที่ไม่ค่อยเปลี่ยน (Redis Week 11) → ลดจำนวน call, ไม่ชน limit, เร็วขึ้น · dashboard ที่ลูกค้าเปิดบ่อยไม่ควรยิง Meta API ทุกครั้ง

ส่วนที่ 1: Typed API Client (Adapter Pattern)

// interface เดียวสำหรับทุกแพลตฟอร์ม (TS จาก Week 6-7)
interface MetricsClient {
  platform: string;
  fetchMetrics(clientId: string, period: DateRange): Promise<Metrics>;
}

// แต่ละแพลตฟอร์ม implement interface เดียวกัน
class MetaClient implements MetricsClient {
  platform = "meta";
  async fetchMetrics(clientId: string, period: DateRange): Promise<Metrics> {
    const res = await this.http.get(`/insights`, { params: {...} });
    return this.normalize(res.data);     // แปลงเป็น Metrics รูปเดียวกัน
  }
  private normalize(raw: unknown): Metrics { ... }   // vendor format → ของเรา
}

// ใช้ — ระบบรู้จักแค่ interface ไม่รู้จัก vendor
const clients: MetricsClient[] = [new MetaClient(), new GA4Client(), ...];
พลังของ pattern นี้: เพิ่ม TikTok? เขียน TikTokClient implements MetricsClient ตัวเดียว — โค้ดสรุป report ไม่ต้องแก้ (มันวน clients ทั้งหมดด้วย interface เดียว) · และ normalize แปลง format vendor ที่ต่างกันเป็นรูปเดียว → โค้ดสรุปจัดการ Metrics รูปเดียว ไม่ต้องรู้ว่ามาจากไหน

ส่วนที่ 2: Retry + Backoff + Timeout

// retry with exponential backoff — ลองใหม่ รอนานขึ้นแต่ละครั้ง
async function withRetry<T>(fn: () => Promise<T>, max = 3): Promise<T> {
  for (let attempt = 1; attempt <= max; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (attempt === max) throw err;              // ครบแล้วยอมแพ้
      const delay = 2 ** attempt * 1000;           // 2s, 4s, 8s (backoff)
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw new Error("unreachable");
}

// timeout — ไม่รอตลอดกาล (AbortController)
const controller = new AbortController();
const timer = setTimeout(() => controller.abort(), 10_000);  // 10 วิ
const res = await fetch(url, { signal: controller.signal });
clearTimeout(timer);
ทำไม backoff ไม่ใช่ retry รัว ๆ: ถ้า API ล่มชั่วคราว retry ทันทีรัว ๆ = ยิ่งกระหน่ำ (ทำให้แย่ลง + อาจโดน rate limit) · backoff (รอนานขึ้น) ให้ API เวลาฟื้น · timeout กัน request ค้าง (ถ้าไม่มี API ที่ไม่ตอบจะทำให้ทั้ง job ค้าง)

ส่วนที่ 3: Rate Limiting + Cache

// cache external response (Redis จาก Week 11) — ลด call
async function getMetrics(clientId: string): Promise<Metrics> {
  const key = `metrics:meta:${clientId}:${today}`;
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);            // ไม่ยิง API ซ้ำวันเดียวกัน

  const data = await withRetry(() => metaClient.fetchMetrics(clientId, period));
  await redis.set(key, JSON.stringify(data), "EX", 3600);  // cache 1 ชม.
  return data;
}
rate limit: ถ้าต้องดึงหลายลูกค้า อย่ายิงพร้อมกันหมด — จำกัด concurrency (เช่น p-limit ทีละ 5 ตัว) หรือใส่ delay ระหว่าง call · เคารพ Retry-After header ถ้า API ส่งมา (บอกว่าให้รอกี่วิ) · เป็น API citizen ที่ดี

ส่วนที่ 4: 🔐 จัดการ API Key

// API key อยู่ใน env เท่านั้น (Week 4) — ไม่ commit, ไม่ log
const metaToken = process.env.META_ACCESS_TOKEN;
// ❌ logger.info({ token: metaToken })  — รั่ว! (Week 15)
// ❌ commit .env  — bot สแกนเจอ (Week 4)
// ✅ .env + .gitignore + log แค่ "fetching meta for client X" (ไม่มี token)
มุม pentester ของคุณ: API key ของ Meta/GA4 มีมูลค่าสูง (เข้าถึงข้อมูลลูกค้า, อาจมีค่าใช้จ่าย) · คุณเคยเจอ key หลุดใน repo/log ตอน pentest มาแล้ว · ระบบนี้ต้อง: key ใน env/secret manager, ไม่ log, ไม่ส่งเข้า Sentry (beforeSend), rotate ได้ถ้าหลุด · นี่คือจุดที่ทำให้ service ของคุณ "ปลอดภัยพอใช้กับลูกค้าจริง"

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี (โดยตรง): นี่คืองานคุณเป๊ะ — MetaClient/GA4Client/LineClient/IGClient ที่ implement interface เดียว, retry เมื่อ API ล่มชั่วคราว, cache ไม่ให้ยิงซ้ำ, key ปลอดภัย · เปลี่ยนจาก "นั่งดึงมือทีละแพลตฟอร์ม" เป็น "service ดึงให้อัตโนมัติ"

🌏 อุตสาหกรรม: integration service (ต่อ API ภายนอกหลายเจ้า) เป็นงานพบบ่อยมาก (payment, shipping, social, analytics) · adapter pattern + retry/backoff + rate limit + cache เป็น pattern มาตรฐาน · "ต่อ third-party API อย่าง resilient เป็น" คือทักษะ backend ที่มีค่า

📝 Quiz ท้ายบท

ข้อ 1: ทำไมห่อ external API ด้วย client (interface) ดีกว่า fetch ตรง ๆ ทั่วโค้ด?

vendor เปลี่ยน API → แก้แค่ใน client ที่เดียว (ข้างนอกไม่กระทบ) · เพิ่มแพลตฟอร์มใหม่ = client ใหม่ที่ implement interface เดิม · normalize แปลง format vendor ต่างกันเป็นรูปเดียว → โค้ดสรุปจัดการรูปเดียว · adapter pattern แยกระบบเราจาก vendor

ข้อ 2: ทำไม retry ต้องมี backoff ไม่ใช่ลองใหม่รัว ๆ?

API ล่มแล้ว retry รัว ๆ = กระหน่ำซ้ำ (ทำให้แย่ลง + โดน rate limit) · backoff (รอนานขึ้นแต่ละครั้ง: 2s,4s,8s) ให้ API เวลาฟื้น · + timeout กัน request ค้าง (API ไม่ตอบ = job ค้างทั้งตัวถ้าไม่มี timeout)

ข้อ 3: ทำไมต้อง cache external API response?

ลดจำนวน call → ไม่ชน rate limit + เร็วขึ้น + ลดภาระ API ภายนอก · ข้อมูลที่ไม่ค่อยเปลี่ยน (สถิติเมื่อวาน) ไม่ต้องยิงซ้ำทุกครั้ง · dashboard ที่เปิดบ่อยไม่ควรยิง Meta ทุกครั้ง · เป็น API citizen ที่ดี

ข้อ 4 (security): API key ของ Meta/GA4 ต้องจัดการยังไง ทำไมสำคัญ?

key มีมูลค่าสูง (เข้าถึงข้อมูลลูกค้า, มีค่าใช้จ่าย) — เก็บใน env/secret manager (ไม่ commit), ไม่ log, ไม่ส่งเข้า Sentry (beforeSend), rotate ได้ถ้าหลุด · key หลุด = ข้อมูลลูกค้ารั่ว/ค่าใช้จ่ายบาน · จุดที่ pentester ต้องระวังสุด

✅ Checkpoint

  • typed API client (interface เดียว + normalize) — adapter pattern
  • retry + backoff + timeout (resilient)
  • cache external (Redis) + เคารพ rate limit
  • API key ปลอดภัย (env, ไม่ log/commit)
