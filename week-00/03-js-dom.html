📦 const ก่อนเสมอ let เมื่อจำเป็น var ห้ามใช้ ⚖️ === เป๊ะทั้งค่าและ type 🏭 function = โรงงาน รับเข้า-คืนออก 🚂 map แปลงทุกตู้ · filter คัดตู้ · find เจอตู้แรก 🌳 DOM = HTML ที่กลายเป็น object ให้ JS จับ 🛑 preventDefault = ห้าม refresh ก่อน

🧠 แนวคิด & หลักการ

หลักการ 1: "Immutability mindset" — ทำไม map/filter ชนะ for loop

map/filter ไม่แก้ array เดิม แต่สร้างอันใหม่ — ฟังดูสิ้นเปลือง แต่คือเหตุผลที่ระบบใหญ่บั๊กน้อย: ข้อมูลเดิมไม่ถูกใครแอบแก้ ใครเปลี่ยนอะไรเห็นชัดเป็นบรรทัด ๆ React (Week 8) สร้างทั้ง framework บนหลักนี้: state เดิมห้ามแก้ ต้องสร้างใหม่เสมอ — ฝึกนิสัยตั้งแต่วันนี้ = Week 8 ง่ายขึ้นมาก

หลักการ 2: "DOM คือผลลัพธ์ ไม่ใช่ฐานข้อมูล"

มือใหม่เก็บ "ความจริง" ไว้บนหน้าจอ (อ่านค่าจาก DOM มาคิดต่อ) — พังเมื่อหน้าจอซับซ้อน มืออาชีพเก็บความจริงใน ข้อมูล (array/object/localStorage) แล้วให้ DOM เป็นแค่ "ภาพฉาย": ข้อมูลเปลี่ยน → วาดใหม่ นี่คือ Single Source of Truth จาก Week 1 และคือแก่นของ React — pattern render(data) ที่ฝึกใน Mission ร้านค้า/Guestbook คือซ้อมใหญ่

หลักการ 3: "Event-driven thinking" — โปรแกรมที่รอจังหวะ ไม่ใช่เส้นตรง

โค้ดหน้าเว็บไม่ได้วิ่งบนลงล่างจบ — มันตั้ง "เครื่องดักฟัง" (addEventListener) แล้วรอผู้ใช้ทำอะไรสักอย่าง วิธีคิดนี้ตัวเดียวกับ backend: Express ก็คือ "ดักฟัง request" (Week 2) — เรียนหน้าบ้านเข้าใจ = หลังบ้านเข้าใจฟรี

กับดักที่พบบ่อยสุดจากโค้ดจริงของคุณ (Project2): โค้ดทดลองค้างในไฟล์ (ตัวแปรไม่ใช้, comment block ใหญ่) — กฎใหม่: ทดลองเสร็จ ลบทันทีก่อน commit เสมอ reviewer เห็นโค้ดตายแล้วความเชื่อมั่นตก

ส่วนที่ 1: ตัวแปร + Data Types

const pi = 3.14;     // ค่าคงที่ — เปลี่ยนไม่ได้ (ใช้เป็น default)
let score = 0;       // เปลี่ยนได้ — ใช้เมื่อจำเป็นจริง
score = 10;          // OK
// var ❌ ห้ามใช้ — มีปัญหา scope รั่ว ทำให้บั๊กหายาก

ทำไม const-first: เริ่มด้วย const เสมอ เปลี่ยนเป็น let ก็ต่อเมื่อมั่นใจว่าค่าต้องเปลี่ยนจริง — โค้ดที่ใช้ const เยอะ = อ่านแล้วมั่นใจว่าค่าไม่ถูกแอบเปลี่ยน = บั๊กน้อย

ส่วนที่ 2: Logic + Control Flow

console.log(5 === "5");  // false — === เช็คทั้งค่าและ type (ใช้ตัวนี้เสมอ)
console.log(5 == "5");   // true  — == หยวน แปลง type ให้ (เลี่ยง!)

// ternary — if/else บรรทัดเดียว
const type = age >= 18 ? "ผู้ใหญ่" : "เด็ก";

// logic operators รวมเงื่อนไข
const canBorrow = isMember && !hasFines;  // เป็นสมาชิก และ ไม่มีค่าปรับ

=== vs ==: ใช้ === เสมอ เพราะ == แปลง type อัตโนมัติทำให้เกิดบั๊กแปลก ๆ (0 == "" เป็น true!) — interviewer ถามเรื่องนี้บ่อยมาก
ternary ใช้เมื่อ: กำหนดค่าง่าย ๆ บรรทัดเดียว — ถ้า logic ซับซ้อนให้ใช้ if ปกติ อย่าฝืนย่อจนอ่านไม่ออก

ส่วนที่ 3: Loop + Function

for (let i = 0; i < 5; i++) { }   // รู้จำนวนรอบแน่นอน
while (energy > 0) { energy -= 1; } // ไม่รู้รอบ รู้แค่เงื่อนไขหยุด
// break = ออกจาก loop เลย / continue = ข้ามรอบนี้ ไปรอบถัดไป

// Function — โรงงาน รับเข้า (parameter) คืนออก (return)
function add(a, b) { return a + b; }
const addArrow = (a, b) => a + b;       // arrow แบบย่อ (บรรทัดเดียว ตัด return)
const double = n => n * 2;              // 1 param ตัดวงเล็บได้

parameter vs return: parameter = วัตถุดิบที่ส่งเข้า / return = ผลลัพธ์ที่ส่งกลับให้เอาไปใช้ต่อ ฟังก์ชันไม่มี return = ทำงานแล้วจบ (เช่น แค่ console.log) เอาผลไปคำนวณต่อไม่ได้
Scope: ตัวแปรที่สร้างในฟังก์ชัน ใช้ได้แค่ในนั้น — ออกมาข้างนอก = หาไม่เจอ (Error)

ส่วนที่ 4: Array Methods — เลือกตามเจตนา

const prices = [100, 200, 300];

prices.map(p => p * 0.5);        // [50,100,150] — แปลงทุกตัว → array ใหม่
prices.filter(p => p > 150);     // [200,300]   — คัดที่ผ่านเงื่อนไข
prices.find(p => p > 150);       // 200         — เจอตัวแรกหยุดเลย
prices.forEach(p => console.log(p)); // ทำเฉย ๆ ไม่คืน array
เจตนาใช้
เปลี่ยนทุกตัวได้ชุดใหม่map
คัดเอาบางตัวfilter
หาตัวเดียวfind
ทำ side effect (log, ส่ง)forEach

map/filter ไม่แก้ array เดิม = immutability ที่พูดในส่วนแนวคิด → รากของ React state

ส่วนที่ 5: Object + JSON

const user = { name: "Somchai", age: 25 };
user.name;            // "Somchai" (dot notation)

// JSON — รูปแบบส่งข้อมูลระหว่าง server/web
const str = JSON.stringify(user);   // Object → String (เตรียมส่ง/เก็บ)
const obj = JSON.parse(str);        // String → Object (รับมาใช้)

// Destructuring — ดึงค่าออกหลายตัวพร้อมกัน
const { name, age } = user;

// Spread (...) — copy + รวม
const admin = { ...user, role: "Admin" };  // copy user + เพิ่ม role

stringify/parse คู่ไปกลับ: localStorage และ API รับส่งได้แค่ String — ทุกครั้งที่เก็บ Object ต้อง stringify ตอนเก็บ, parse ตอนอ่าน (ตรงกับ Week 1 เป๊ะ)
destructuring + spread เป็น syntax ที่ React ใช้ตลอด — คุ้นตั้งแต่ตอนนี้ได้เปรียบ Week 8

ส่วนที่ 6: DOM + Events

// เลือก element
const title = document.querySelector("#title");   // ตัวแรกที่เจอ
const items = document.querySelectorAll(".item");  // ทุกตัว (NodeList)

// แก้ไข
title.innerText = "ข้อความใหม่";           // ข้อความล้วน (ปลอดภัย)
box.classList.toggle("active");           // สลับ class (ดี/ออก)

// ดักฟัง event
form.addEventListener("submit", (e) => {
  e.preventDefault();        // ⭐ กันหน้า refresh
  const value = input.value;
  // ...ใช้ค่า
  input.value = "";          // เคลียร์ช่อง
});

addEventListener ดีกว่า onclick: ผูกได้หลายตัวบน element เดียว, แยก JS ออกจาก HTML, ถอดออกได้ — onclick ใน HTML เป็นวิธีเก่า
preventDefault ใน submit: ฟอร์มมีพฤติกรรมดั้งเดิม = refresh หน้าเมื่อกดส่ง (ยุคก่อน JS) — ไม่กัน = state หายหมด หน้ากระพริบ ลืมบรรทัดนี้ = บั๊กอันดับ 1 ของฟอร์ม

🔐 มุม pentester — innerHTML คือประตู XSS: el.innerHTML = userInput = ให้คนแปลกหน้าเขียน HTML ลงหน้าเว็บเรา ถ้า input มี script/event handler ฝังมา = โดน กฎ: ข้อความล้วนใช้ innerText / จำเป็นต้องใช้ innerHTML กับข้อมูลผู้ใช้ → ต้อง escape ก่อนเสมอ (โยงตรงไป Quiz localStorage ของ Week 1 และ OWASP ใน Week 17)

📝 Quiz

ข้อ 1: เลือก method ให้ถูก — (ก) เอาเฉพาะสินค้าราคา < 1000 (ข) แปลงทุกราคาเป็น USD (ค) หาสินค้าชิ้นแรกที่ชื่อขึ้นต้น "i" (ง) log ทุกชิ้น

(ก) filter — คัดตามเงื่อนไข (ข) map — แปลงทุกตัวได้ array ใหม่ (ค) find — เจอตัวแรกหยุดเลย (ง) forEach — ทำ side effect เฉย ๆ ไม่คืน array เลือกตามเจตนา ไม่ใช่ใช้ forEach ทำทุกอย่าง — โค้ดบอกเจตนาได้ = อ่านง่าย = ผ่าน review

ข้อ 2: ลืม e.preventDefault() ใน submit handler — เกิดอะไร และทำไม?

หน้า refresh ทันทีที่กดส่ง — เพราะพฤติกรรมดั้งเดิมของ form คือส่งข้อมูลไป action แล้วโหลดหน้าใหม่ (ยุคก่อน JS) ผล: state ใน JS หายหมด, ข้อมูลที่เพิ่ง render หาย "ฟอร์มกดแล้วหน้ากระพริบแล้วไม่มีอะไรเกิด" = อาการคลาสสิกของบั๊กนี้ จำไว้จะ debug ได้ใน 10 วินาที

ข้อ 3 (security): โค้ด list.innerHTML += `<li>${comment}</li>` ใน Guestbook อันตรายยังไง สาธิตการโจมตี 1 แบบ

comment มาจากผู้ใช้ → พิมพ์ <img src=x onerror="alert(document.cookie)"> ลงช่องคอมเมนต์ → ถูกฉีดเป็น HTML จริง → script รันในหน้าของทุกคนที่เปิดดู = stored XSS แก้: ใช้ innerText/textContent สำหรับข้อความผู้ใช้ หรือ escape ก่อน — นี่คือช่องโหว่อันดับต้นที่คุณเคยล่าในฐานะ pentester ตอนนี้อยู่ฝั่งกัน
