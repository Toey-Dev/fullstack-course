หลักการ 1: "Immutability mindset" — ทำไม map/filter ชนะ for loop

map/filter ไม่แก้ array เดิม แต่สร้างอันใหม่ — ฟังดูสิ้นเปลือง แต่คือเหตุผลที่ระบบใหญ่บั๊กน้อย: ข้อมูลเดิมไม่ถูกใครแอบแก้ ใครเปลี่ยนอะไรเห็นชัดเป็นบรรทัด ๆ React (Week 8) สร้างทั้ง framework บนหลักนี้: state เดิมห้ามแก้ ต้องสร้างใหม่เสมอ — ฝึกนิสัยตั้งแต่วันนี้ = Week 8 ง่ายขึ้นมาก

หลักการ 2: "DOM คือผลลัพธ์ ไม่ใช่ฐานข้อมูล"

มือใหม่เก็บ "ความจริง" ไว้บนหน้าจอ (อ่านค่าจาก DOM มาคิดต่อ) — พังเมื่อหน้าจอซับซ้อน มืออาชีพเก็บความจริงใน ข้อมูล (array/object/localStorage) แล้วให้ DOM เป็นแค่ "ภาพฉาย": ข้อมูลเปลี่ยน → วาดใหม่ นี่คือ Single Source of Truth จาก Week 1 และคือแก่นของ React — pattern render(data) ที่ฝึกใน Mission ร้านค้า/Guestbook คือซ้อมใหญ่

โค้ดหน้าเว็บไม่ได้วิ่งบนลงล่างจบ — มันตั้ง "เครื่องดักฟัง" (addEventListener) แล้วรอผู้ใช้ทำอะไรสักอย่าง วิธีคิดนี้ตัวเดียวกับ backend: Express ก็คือ "ดักฟัง request" (Week 2) — เรียนหน้าบ้านเข้าใจ = หลังบ้านเข้าใจฟรี

ส่วนที่ 2: Logic + Control Flow

console.log(5 === "5");  // false — === เช็คทั้งค่าและ type (ใช้ตัวนี้เสมอ)
console.log(5 == "5");   // true  — == หยวน แปลง type ให้ (เลี่ยง!)

// ternary — if/else บรรทัดเดียว
const type = age >= 18 ? "ผู้ใหญ่" : "เด็ก";

// logic operators รวมเงื่อนไข
const canBorrow = isMember && !hasFines;  // เป็นสมาชิก และ ไม่มีค่าปรับ

=== vs ==: ใช้ === เสมอ เพราะ == แปลง type อัตโนมัติทำให้เกิดบั๊กแปลก ๆ (0 == "" เป็น true!) — interviewer ถามเรื่องนี้บ่อยมาก
ternary ใช้เมื่อ: กำหนดค่าง่าย ๆ บรรทัดเดียว — ถ้า logic ซับซ้อนให้ใช้ if ปกติ อย่าฝืนย่อจนอ่านไม่ออก

Truthy / Falsy — หัวใจที่มือใหม่มองข้าม

ใน if ค่าทุกชนิดถูกแปลงเป็น true/false เรียกว่า truthy/falsy — จำ 6 ตัว falsy ให้ขึ้นใจ ที่เหลือ truthy หมด:

// 6 ตัว FALSY (ที่เหลือ truthy ทั้งหมด)
false, 0, "", null, undefined, NaN

if (user.name) { }     // "" (ชื่อว่าง) = falsy → ไม่เข้า
if (cart.length) { }   // 0 ชิ้น = falsy → ไม่เข้า (เช็คว่ามีของไหมแบบสั้น)
if (!data) return;     // null/undefined = falsy → guard clause
กับดักจริง: if (count) เมื่อ count = 0 จะไม่เข้า if ทั้งที่ 0 อาจเป็นค่าที่ถูกต้อง — ถ้าต้องการเช็คแค่ "มีค่าไหม" (รวม 0) ให้ใช้ if (count !== undefined) หรือ != null ชัดเจนกว่า

Optional chaining (?.) + Nullish (??) — เครื่องมือยุคใหม่กัน error

// ❌ แบบเก่า: เช็คมือทีละชั้น (ยาว เผลอลืม = crash)
const city = user && user.address && user.address.city;

// ✅ optional chaining — ถ้าชั้นไหน null/undefined หยุดให้ ไม่ crash
const city = user?.address?.city;     // ได้ค่า หรือ undefined

// ✅ nullish — ใส่ค่า default เฉพาะตอน null/undefined
const name = user?.name ?? "ไม่ระบุ";

?? ต่างจาก || ยังไง: || ใช้ค่าขวาเมื่อซ้าย falsy ทุกตัว (รวม 0, "") — ?? ใช้ค่าขวาเฉพาะ null/undefined เท่านั้น เช่น price ?? 0 ปลอดภัยกว่า price || 0 เพราะถ้า price = 0 จริง ๆ || จะทับเป็น 0 ใหม่ (บั๊กเงียบ) แต่ ?? เก็บ 0 ไว้

switch — เมื่อ if/else if ยาวเกินไป

switch (role) {
  case "admin":  showAdminPanel(); break;   // ⭐ ลืม break = ไหลลงเคสถัดไป!
  case "user":   showDashboard();  break;
  default:       showLogin();
}

ใช้ switch เมื่อเทียบ "ค่าเดียว" กับหลายความเป็นไปได้ — ถ้าเงื่อนไขซับซ้อน (มี > < &&) ใช้ if/else

ส่วนที่ 3: Loop + Function

for (let i = 0; i < 5; i++) { }   // รู้จำนวนรอบแน่นอน
while (energy > 0) { energy -= 1; } // ไม่รู้รอบ รู้แค่เงื่อนไขหยุด
// break = ออกจาก loop เลย / continue = ข้ามรอบนี้ ไปรอบถัดไป

// for...of — วนค่าใน array (ยุคใหม่ อ่านง่ายกว่า for เลขดัชนี)
for (const book of books) { console.log(book.title); }

// for...in — วน key ของ object
for (const key in user) { console.log(key, user[key]); }

เลือก loop ยังไง: รู้จำนวนรอบ/ต้องใช้ index → for · วน array เอาค่า → for...of หรือ .forEach() · วน key object → for...in · ไม่รู้รอบ → while

// Function — โรงงาน รับเข้า (parameter) คืนออก (return)
function add(a, b) { return a + b; }
const addArrow = (a, b) => a + b;       // arrow แบบย่อ (บรรทัดเดียว ตัด return)
const double = n => n * 2;              // 1 param ตัดวงเล็บได้

// Default parameter — ค่าเริ่มต้นถ้าไม่ส่งมา
const greet = (name = "Guest") => `Hi ${name}`;   // greet() → "Hi Guest"

// Rest parameter — รับ argument ไม่จำกัดเป็น array
const sum = (...nums) => nums.reduce((a, b) => a + b, 0);  // sum(1,2,3) → 6

parameter vs return: parameter = วัตถุดิบที่ส่งเข้า / return = ผลลัพธ์ที่ส่งกลับให้เอาไปใช้ต่อ ฟังก์ชันไม่มี return = ทำงานแล้วจบ (เช่น แค่ console.log) เอาผลไปคำนวณต่อไม่ได้
Scope: ตัวแปรที่สร้างในฟังก์ชัน ใช้ได้แค่ในนั้น — ออกมาข้างนอก = หาไม่เจอ (Error)

หลักคิด production — ฟังก์ชันทำหน้าที่เดียว (Single Responsibility): ฟังก์ชันชื่อ calculateTotal ควรแค่คำนวณ ไม่ควรไป render หน้าจอด้วย — แยกหน้าที่ทำให้ test ง่าย แก้ง่าย เอาไปใช้ซ้ำได้ ตั้งชื่อฟังก์ชันเป็น "กริยา" เสมอ (loadBooks, renderList, validateEmail)

ส่วนที่ 4: Array Methods — เลือกตามเจตนา

const prices = [100, 200, 300];

prices.map(p => p * 0.5);        // [50,100,150] — แปลงทุกตัว → array ใหม่
prices.filter(p => p > 150);     // [200,300]   — คัดที่ผ่านเงื่อนไข
prices.find(p => p > 150);       // 200         — เจอตัวแรกหยุดเลย
prices.forEach(p => console.log(p)); // ทำเฉย ๆ ไม่คืน array

reduce — ตัวที่ทรงพลังสุด (และคนกลัวสุด)

const prices = [100, 200, 300];

// หาผลรวม: acc = ค่าสะสม, cur = ตัวปัจจุบัน, 0 = ค่าเริ่มต้น
const total = prices.reduce((acc, cur) => acc + cur, 0);   // 600

// reduce ทำได้มากกว่าบวกเลข — สร้าง object สรุปก็ได้
const byCategory = books.reduce((acc, book) => {
  acc[book.category] = (acc[book.category] ?? 0) + 1;
  return acc;
}, {});   // { "Programming": 3, "History": 2 }

วิธีคิด reduce: "เริ่มจากค่าตั้งต้น แล้วค่อย ๆ สะสมทีละตัวจนจบ" — ใช้ทำผลรวม, ค่าเฉลี่ย, จัดกลุ่ม, แปลง array เป็น object

some / every — เช็คเงื่อนไขทั้งกอง

const cart = [{ inStock: true }, { inStock: false }];
cart.some(item => !item.inStock);    // true  — มีของหมดสต็อกอย่างน้อย 1
cart.every(item => item.inStock);    // false — ไม่ใช่ทุกตัวที่มีของ

⚠️ sort — กับดักที่ทำคนงงทั้งโลก

[10, 2, 1].sort();              // [1, 10, 2] ❌ ผิด! sort เรียงแบบ "ข้อความ"
[10, 2, 1].sort((a, b) => a - b);  // [1, 2, 10] ✅ ต้องบอกวิธีเทียบเอง
// a - b = น้อยไปมาก / b - a = มากไปน้อย
sort แก้ array เดิม (mutate)! ต่างจาก map/filter — ถ้าไม่อยากให้เดิมเปลี่ยน ให้ copy ก่อน: [...arr].sort(...) (เชื่อมหลัก immutability)

Method Chaining — ต่อกันเป็นสายการผลิต (production จริง)

// คัดของในสต็อก → เอาเฉพาะชื่อ → เรียง ก-ฮ  (อ่านจากบนลงล่างเป็นเรื่องราว)
const names = products
  .filter(p => p.inStock)
  .map(p => p.name)
  .sort();

map/filter ไม่แก้ array เดิม = immutability ที่พูดในส่วนแนวคิด → รากของ React state · การ chain ได้เพราะแต่ละตัวคืน array ใหม่

ส่วนที่ 5: Object + JSON

const user = { name: "Somchai", age: 25 };
user.name;            // "Somchai" (dot notation)

// JSON — รูปแบบส่งข้อมูลระหว่าง server/web
const str = JSON.stringify(user);   // Object → String (เตรียมส่ง/เก็บ)
const obj = JSON.parse(str);        // String → Object (รับมาใช้)

ส่วนที่ 6: DOM + Events

// เลือก element
const title = document.querySelector("#title");   // ตัวแรกที่เจอ
const items = document.querySelectorAll(".item");  // ทุกตัว (NodeList)

// แก้ไข
title.innerText = "ข้อความใหม่";           // ข้อความล้วน (ปลอดภัย)
box.classList.toggle("active");           // สลับ class (ดี/ออก)

// ดักฟัง event
form.addEventListener("submit", (e) => {
  e.preventDefault();        // ⭐ กันหน้า refresh
  const value = input.value;
  // ...ใช้ค่า
  input.value = "";          // เคลียร์ช่อง
});

addEventListener ดีกว่า onclick: ผูกได้หลายตัวบน element เดียว, แยก JS ออกจาก HTML, ถอดออกได้ — onclick ใน HTML เป็นวิธีเก่า
preventDefault ใน submit: ฟอร์มมีพฤติกรรมดั้งเดิม = refresh หน้าเมื่อกดส่ง (ยุคก่อน JS) — ไม่กัน = state หายหมด หน้ากระพริบ ลืมบรรทัดนี้ = บั๊กอันดับ 1 ของฟอร์ม

สร้าง element 2 วิธี — innerHTML vs createElement

// วิธี A: innerHTML — เร็ว เขียนง่าย แต่เสี่ยง XSS ถ้าใส่ข้อมูลผู้ใช้
list.innerHTML = books.map(b => `<li>${b.title}</li>`).join("");

// วิธี B: createElement — ปลอดภัยกว่า (textContent ไม่ตีความ HTML)
const li = document.createElement("li");
li.textContent = book.title;     // ใส่ข้อความล้วน ปลอดภัยจาก XSS
list.appendChild(li);

production: ข้อมูลเราคุมเอง (เช่น label คงที่) ใช้ innerHTML ได้ / ข้อมูลจากผู้ใช้หรือ API → createElement + textContent ปลอดภัยกว่า

Event Bubbling & Delegation — เทคนิคที่แยกมือใหม่กับมือโปร

เมื่อคลิกปุ่มใน element ลูก event จะ "ฟองขึ้น" (bubble) ไปหา parent ด้วย — ใช้ประโยชน์ได้:

// ❌ มือใหม่: ผูก listener ทุกปุ่ม (100 ปุ่ม = 100 listener = เปลือง + ปุ่มใหม่ไม่ทำงาน)
buttons.forEach(btn => btn.addEventListener("click", handleClick));

// ✅ Event Delegation: ผูกที่ parent ตัวเดียว ใช้ e.target รู้ว่ากดปุ่มไหน
list.addEventListener("click", (e) => {
  if (e.target.matches(".delete-btn")) {
    const id = e.target.dataset.id;   // อ่าน data-id="..."
    removeItem(id);
  }
});
ทำไมสำคัญในงานจริง: รายการที่ render จาก API (เพิ่ม/ลบได้ตลอด) ถ้าผูก listener ทีละปุ่ม ปุ่มที่เพิ่งสร้างใหม่จะไม่มี listener (ไม่ทำงาน) — delegation แก้ปัญหานี้เพราะ parent มีอยู่ก่อนแล้ว Mission Reading List (Week 1) ตอนลบรายการใช้เทคนิคนี้ได้เลย
🔐 มุม pentester — innerHTML คือประตู XSS: el.innerHTML = userInput = ให้คนแปลกหน้าเขียน HTML ลงหน้าเว็บเรา ถ้า input มี script/event handler ฝังมา = โดน กฎ: ข้อความล้วนใช้ innerText/textContent / จำเป็นต้องใช้ innerHTML กับข้อมูลผู้ใช้ → ต้อง escape ก่อนเสมอ (โยงตรงไป Quiz localStorage ของ Week 1 และ OWASP ใน Week 17)

🧹 Clean Code: เขียน JavaScript แบบ production

1. Guard clause — ดักออกก่อน เลี่ยง if ซ้อนลึก

❌ ซ้อนลึก (อ่านยาก)

function process(user) {
  if (user) {
    if (user.active) {
      if (user.role === "admin") {
        doAdmin();
      }
    }
  }
}

✅ ดักออกก่อน เนื้อหลักไม่ย่อหน้า

function process(user) {
  if (!user) return;
  if (!user.active) return;
  if (user.role !== "admin") return;
  doAdmin();   // เนื้อหลักอยู่ระดับบนสุด
}

2. ใช้ array methods แทน for เมื่อสื่อเจตนาได้ชัดกว่า

❌ for + push (เยอะ ตั้งตัวแปรเปล่า)

const names = [];
for (let i = 0; i < users.length; i++) {
  if (users[i].active) {
    names.push(users[i].name);
  }
}

✅ chain — อ่านเป็นประโยค

const names = users
  .filter(u => u.active)

ข้อ 1: เลือก method ให้ถูก — (ก) เอาเฉพาะสินค้าราคา < 1000 (ข) แปลงทุกราคาเป็น USD (ค) หาสินค้าชิ้นแรกที่ชื่อขึ้นต้น "i" (ง) log ทุกชิ้น

(ก) filter — คัดตามเงื่อนไข (ข) map — แปลงทุกตัวได้ array ใหม่ (ค) find — เจอตัวแรกหยุดเลย (ง) forEach — ทำ side effect เฉย ๆ ไม่คืน array เลือกตามเจตนา ไม่ใช่ใช้ forEach ทำทุกอย่าง — โค้ดบอกเจตนาได้ = อ่านง่าย = ผ่าน review

ข้อ 2: ลืม e.preventDefault() ใน submit handler — เกิดอะไร และทำไม?

หน้า refresh ทันทีที่กดส่ง — เพราะพฤติกรรมดั้งเดิมของ form คือส่งข้อมูลไป action แล้วโหลดหน้าใหม่ (ยุคก่อน JS) ผล: state ใน JS หายหมด, ข้อมูลที่เพิ่ง render หาย "ฟอร์มกดแล้วหน้ากระพริบแล้วไม่มีอะไรเกิด" = อาการคลาสสิกของบั๊กนี้ จำไว้จะ debug ได้ใน 10 วินาที

ข้อ 3 (security): โค้ด list.innerHTML += `<li>${comment}</li>` ใน Guestbook อันตรายยังไง สาธิตการโจมตี 1 แบบ

comment มาจากผู้ใช้ → พิมพ์ <img src=x onerror="alert(document.cookie)"> ลงช่องคอมเมนต์ → ถูกฉีดเป็น HTML จริง → script รันในหน้าของทุกคนที่เปิดดู = stored XSS แก้: ใช้ innerText/textContent สำหรับข้อความผู้ใช้ หรือ escape ก่อน — นี่คือช่องโหว่อันดับต้นที่คุณเคยล่าในฐานะ pentester ตอนนี้อยู่ฝั่งกัน

ข้อ 4: price || 0 กับ price ?? 0 — ถ้า price = 0 จริง ๆ ได้ผลต่างกันยังไง?

price || 0: 0 เป็น falsy → ใช้ค่าขวา = 0 (บังเอิญได้ 0 เหมือนกัน แต่เพราะเหตุผลผิด — ถ้า default เป็นเลขอื่นจะพัง) price ?? 0: 0 ไม่ใช่ null/undefined → เก็บ 0 เดิมไว้ถูกต้อง เคสจริง: discount || 100 เมื่อ discount = 0 จะกลายเป็น 100 (ส่วนลดหายเป็นเต็มราคา = บั๊กเงียบ) → ใช้ ?? ปลอดภัยกว่า

ข้อ 5: [10, 2, 1].sort() ได้ [1, 10, 2] ทำไม และแก้ยังไง?

เพราะ sort เริ่มต้นแปลงเป็น "ข้อความ" แล้วเรียงตามตัวอักษร ("10" มาก่อน "2" เพราะ "1" < "2") — แก้ด้วยใส่ compare function: .sort((a,b) => a - b) (น้อยไปมาก) และอย่าลืม sort แก้ array เดิม ถ้าไม่อยากให้เปลี่ยนใช้ [...arr].sort()

ข้อ 6: มี 100 ปุ่มลบในรายการที่ render จาก API — ทำไมไม่ควรผูก addEventListener ทีละปุ่ม?

① เปลือง (100 listener) ② ปุ่มที่เพิ่ง render ใหม่หลังผูก listener จะไม่ทำงาน — ใช้ event delegation แทน: ผูก listener ที่ parent ตัวเดียว แล้วเช็ค e.target.matches(".delete-btn") — parent มีอยู่ก่อนเสมอ ปุ่มใหม่กี่ตัวก็ครอบคลุม
