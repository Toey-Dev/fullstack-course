Week 0 · บทที่ 1

HTML — โครงกระดูกของเว็บ

สรุปแก่นจาก 021 บท 01-04 + เหตุผลเบื้องหลังที่คอร์สพื้นฐานมักไม่ได้เล่า

🔑 คีย์เวิร์ดภาพจำ

🦴 โครงกระดูก = HTML · 👗 เสื้อผ้า = CSS · ⚡ ระบบไฟ = JS 🧱 Block กินทั้งแถว · 🪡 Inline แทรกในบรรทัด 🗺 ./ ที่นี่ · ../ ถอยหนึ่งชั้น = Relative Path 🏷 ป้ายบอกความหมาย = Semantic 📮 จ่าหน้าซอง = name ใน form

🧠 แนวคิด & หลักการ

หลักการ 1: "Structure before style" — โครงถูกก่อน สวยทีหลัง

HTML ที่ดีต้องอ่านรู้เรื่องแม้ไม่มี CSS เลย — เพราะผู้อ่านตัวจริงไม่ได้มีแค่มนุษย์: Google bot, screen reader, AI search (GEO/AEO ที่คุณทำที่ media-presto!) ล้วนอ่าน "โครง" ไม่ได้ดู "ความสวย" โครงผิด = เครื่องอ่านไม่เข้าใจ = SEO/AEO พัง ต่อให้สวยแค่ไหน

หลักการ 2: "Semantic = ความหมาย ไม่ใช่หน้าตา"

<div> 100 อันทำเว็บได้ แต่บอกความหมายอะไรไม่ได้เลย — <header> <nav> <main> <article> <footer> หน้าตาเหมือน div ทุกประการ แต่ "บอกบทบาท" ให้เครื่องเข้าใจ ทำไมต้องแคร์: SEO ranking, accessibility (กฎหมายหลายประเทศบังคับ), และ AI search ยุคนี้ให้คะแนนเว็บที่โครงชัด — ทักษะ GEO/AEO ของคุณยืนอยู่บนเรื่องนี้ตรง ๆ

หลักการ 3: "Forms คือประตูรับข้อมูล — และประตูแรกที่ถูกโจมตี"

ทุก input ที่เปิดรับ = ช่องทางที่ผู้ใช้ (และผู้ไม่หวังดี) ส่งของเข้าระบบ attribute name คือ "จ่าหน้าซอง" — ไม่มี = ข้อมูลไปไม่ถึง server / required, type="email" คือ validation ฝั่ง browser ซึ่งกันคนพลั้งเผลอได้ แต่กันคนตั้งใจไม่ได้ (ปิด JS หรือยิงตรงด้วย curl ก็ข้ามหมด) — นี่คือเหตุผลที่ Week 2 ต้อง validate ซ้ำฝั่ง server เสมอ

ส่วนที่ 1: โครงสร้าง HTML5 — เข้าใจทุกบรรทัด

<!DOCTYPE html>            <!-- 1 -->
<html lang="th">           <!-- 2 -->
<head>                     <!-- 3 -->
  <meta charset="UTF-8">             <!-- 4 -->
  <meta name="viewport" content="width=device-width, initial-scale=1.0">  <!-- 5 -->
  <title>ชื่อบน tab</title>          <!-- 6 -->
</head>
<body>                     <!-- 7 -->
  <h1>เนื้อหาที่คนเห็น</h1>
</body>
</html>
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
1DOCTYPE บอก browser ว่า "ใช้มาตรฐาน HTML5" — ถ้าไม่มี browser เข้า "quirks mode" จำลองพฤติกรรมเก่ายุค 90 ทำให้ layout เพี้ยนแบบหาสาเหตุยาก
2lang="th" ไม่ใช่ของประดับ — screen reader ใช้เลือกเสียงอ่านภาษาไทย, Google ใช้จัดกลุ่มภาษา, ปุ่ม "แปลหน้านี้" ใช้ตัดสินใจ
3-6head = ข้อมูลสำหรับ "เครื่อง" (ไม่แสดงบนหน้า): charset กันภาษาไทยเพี้ยนเป็น "à¸ªà¸§", viewport ทำให้ responsive ทำงาน (ขาดบรรทัดนี้ = มือถือซูมออกจิ๋ว)
7body = ทุกอย่างที่ "มนุษย์" เห็น
กับดักอันดับ 1 ของมือใหม่: ลืม <meta charset="UTF-8"> → ภาษาไทยกลายเป็นตัวยึกยือ และลืม viewport → เว็บสวยบนคอมแต่จิ๋วบนมือถือ ทั้งคู่แก้ด้วยพิมพ์ ! + Tab ใน VS Code (Emmet สร้างโครงครบให้)

ส่วนที่ 2: Block vs Inline — ทำไม span ปรับขนาดไม่ได้

ทุก element มี "นิสัยการกินที่" 2 แบบ — เข้าใจอันนี้ = เข้าใจ layout ครึ่งหนึ่ง:

🧱 Block — จอมบงการ

  • กินเต็มแถว 100% เสมอ
  • ขึ้นบรรทัดใหม่เสมอ
  • ตั้ง width/height/margin ได้ครบ

<div> <p> <h1> <section>

🪡 Inline — เจียมตัว

  • กว้างเท่าเนื้อหาข้างใน
  • อยู่ต่อกันในบรรทัดเดียวได้
  • ตั้ง width/height ไม่ได้!

<span> <a> <strong>

ทำไม span ใส่ width ไม่ได้: เพราะนิสัยมัน "กว้างเท่าข้อความ" การสั่ง width จึงถูกเพิกเฉย — ทางแก้คือ display: inline-block (อยู่ในบรรทัดเดียวได้ แต่ตั้งขนาดได้) ซึ่งเป็นเหตุผลที่ปุ่มและ <img> เป็น inline-block โดยธรรมชาติ (เรื่องนี้ต่อยอดเต็มในบทที่ 2 Box Model)

ส่วนที่ 3: File Path — ./ กับ ../ ที่ทำคนงงทั้งโลก

สมมติโครงไฟล์:

my-site/
├── index.html      📍 (กำลังเขียนไฟล์นี้)
├── about.html
├── images/logo.png
└── pages/contact.html
จาก index.html อยากลิงก์ไปเขียนอ่านว่า
logo.pngimages/logo.pngเดินเข้าห้อง images
about.html (ข้าง ๆ)./about.html หรือ about.html"ที่นี่"
contact.htmlpages/contact.htmlเข้าห้อง pages

แต่ถ้าย้ายไปเขียนใน pages/contact.html แล้วอยากกลับไป index: ต้อง ../index.html — ../ = "ถอยออกจากห้องปัจจุบัน 1 ชั้น"

วิธีจำ: ./ = ที่นี่ · folder/ = เดินเข้าห้อง · ../ = เดินออกจากห้อง
ทำไม Absolute path (https://...) พังเมื่อย้ายเครื่อง: มันชี้ตำแหน่งตายตัวบนเน็ต — ใช้กับลิงก์ภายในเว็บเรา = ต้องมีเน็ต + URL ห้ามเปลี่ยน ใช้ Relative ปลอดภัยกว่าสำหรับไฟล์ในโปรเจกต์เดียวกัน

ส่วนที่ 4: รูปภาพ — ทำไม alt ไม่ใช่ของแถม

<img src="cat.jpg" alt="แมวส้มนอนบนโซฟา" width="300">

alt ทำงาน 3 หน้าที่ที่มองไม่เห็นแต่สำคัญ:

  1. ภาพโหลดไม่ขึ้น (เน็ตช้า/ไฟล์หาย) → แสดงข้อความแทน ผู้ใช้ยังรู้ว่าตรงนี้คือรูปอะไร
  2. Screen reader → อ่าน alt ให้คนตาบอดฟัง ไม่มี alt = คนตาบอดเจอ "รูปภาพ" เปล่า ๆ
  3. SEO/Google Images → Google อ่านรูปไม่ออก ใช้ alt เข้าใจว่ารูปคืออะไร = ติดอันดับค้นรูป

เคล็ดลับ: ใส่ width หรือ height แค่ด้านเดียว อีกด้านจะ auto รักษาสัดส่วน — ใส่ทั้งคู่เสี่ยงภาพเบี้ยว

ส่วนที่ 5: ตาราง — โครงและการผสานเซลล์

<table>
  <thead><tr><th>สินค้า</th><th>ราคา</th></tr></thead>
  <tbody>
    <tr><td>ส้มตำ</td><td>50</td></tr>
  </tbody>
</table>

ทำไมแยก thead/tbody/tfoot: ไม่ใช่แค่ความสวย — screen reader ใช้บอกว่าแถวไหนคือ "หัวข้อ" เวลา scroll ตารางยาว browser/CSS รู้ว่าควรตรึง header, และ Google เข้าใจโครงข้อมูลดีขึ้น

ผสานเซลล์: colspan="2" = กินกว้าง 2 คอลัมน์ (แนวนอน) · rowspan="2" = กินสูง 2 แถว (แนวตั้ง) เช่น "พักเที่ยง" ที่กิน 2 ช่วงเวลา

ในงานจริงไม่ใช้ table จัด layout หน้าเว็บ (ยุค 90 ทำกัน เลิกแล้ว) — table ใช้แสดง "ข้อมูลตาราง" จริง ๆ เท่านั้น จัด layout ใช้ CSS/Grid (บทที่ 2)

ส่วนที่ 6: ฟอร์ม — ประตูรับข้อมูล (และด่านแรกที่ถูกโจมตี)

<form action="/api/register" method="POST">
  <label for="email">อีเมล:</label>
  <input type="email" id="email" name="email" required>
  <button type="submit">สมัคร</button>
</form>
ส่วนหน้าที่ + ทำไมสำคัญ
actionส่งข้อมูลไป URL ไหน
methodGET = ค้นหา (ข้อมูลโผล่ใน URL) / POST = บันทึก (ข้อมูลอยู่ใน body) — password ต้อง POST เสมอ ไม่งั้นรหัสโผล่ใน URL + ประวัติ browser
name⭐ สำคัญสุด — คือ "จ่าหน้าซอง" ไม่มี name = ข้อมูลไปไม่ถึง server (บั๊กเงียบที่เจอบ่อยมาก)
label forผูกกับ id ของ input → กดที่ข้อความแล้ว cursor เด้งเข้าช่อง + screen reader อ่านคู่กัน
requiredbrowser กันส่งฟอร์มว่าง — แต่กันได้แค่คนพลั้งเผลอ
🔐 มุม pentester: required, type="email" ทำงานใน browser ที่ผู้ใช้ควบคุม 100% — เปิด DevTools ลบ attribute ทิ้ง หรือยิง POST ตรงด้วย curl ก็ข้ามหมด ดังนั้น validation ฝั่ง browser = UX ช่วยคนทั่วไป / security ต้อง validate ซ้ำฝั่ง server เสมอ (Week 2 จะทำจริง)

📝 Quiz

ข้อ 1: ฟอร์มที่ input ไม่มี name — กดส่งแล้วเกิดอะไร และทำไม?

ฟอร์มส่งได้ปกติ แต่ข้อมูลช่องนั้นไม่ถูกแนบไป — เพราะ name คือ "ชื่อตัวแปร" ที่ browser ใช้จ่าหน้าข้อมูล ไม่มีชื่อ = ไม่รู้จะเรียกว่าอะไร = ทิ้ง (บั๊กเงียบที่เจอบ่อยมากในงานจริง: ฟอร์มดูทำงาน แต่ข้อมูลหายบางช่อง)

ข้อ 2: ทำไม required + type="email" ฝั่ง browser ถึง "กันคนตั้งใจ" ไม่ได้?

เพราะมันทำงานใน browser ของผู้ใช้ ซึ่งผู้ใช้ควบคุมได้ 100% — เปิด DevTools ลบ attribute ทิ้ง, หรือข้าม browser ไปยิง request ตรงด้วย curl/Postman ก็ส่งอะไรก็ได้เข้า server ดังนั้น browser validation = UX (ช่วยคนพลั้งเผลอ) ส่วน security ต้องอยู่ฝั่ง server เสมอ

ข้อ 3: เว็บสองเว็บหน้าตาเหมือนกันเป๊ะ เว็บแรกใช้ div ล้วน เว็บสองใช้ semantic — ต่างกันที่ไหนบ้าง?

ต่างที่ "เครื่องอ่าน": SEO (Google เข้าใจโครงเนื้อหา → อันดับดีกว่า), accessibility (screen reader นำทางได้), AI search/GEO (สกัดเนื้อหาไปตอบได้ถูก), และ maintainability (dev คนใหม่อ่านโครงรู้เรื่องทันที) — มนุษย์ตาดีมองไม่เห็นความต่าง แต่ระบบทั้งหมดข้างบนเห็น
