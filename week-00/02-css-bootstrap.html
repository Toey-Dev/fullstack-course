CSS + Bootstrap — แต่งตัวและจัดวาง

สรุปแก่นจาก 021 บท 05-08 — โดยเฉพาะ Box Model ที่เป็นรากของทุก layout บนโลก

🔑 คีย์เวิร์ดภาพจำ

🧅 หัวหอม 4 ชั้น = Box Model 🛋 ดันของข้างใน = padding · 🚶 ผลักเพื่อนบ้าน = margin 🎯 ชี้เป้า = Selector (.class ใช้บ่อยสุด) 🍱 เค้ก 12 ชิ้น = Bootstrap Grid 📱 col-md-6 = "จอใหญ่พอค่อยแบ่งครึ่ง"

🧠 แนวคิด & หลักการ

หลักการ 1: "Everything is a box" — เข้าใจกล่อง = เข้าใจ layout ทั้งหมด

ปัญหา layout 90% (ของไม่ตรง, ช่องว่างประหลาด, ล้นจอ) มีคำตอบอยู่ใน 4 ชั้นของกล่อง: content → padding → border → margin จำง่าย: อยากให้ข้างในหลวม = padding / อยากห่างจากคนอื่น = margin เวลา debug layout ให้เปิด DevTools ดูกล่องสีฟ้า-เขียว-ส้มก่อนเสมอ ไม่ใช่เดาสุ่มแก้เลข

หลักการ 2: "Specificity ก่อนเวทมนตร์" — ทำไม CSS บางทีไม่ทำงาน

เมื่อหลายกฎชี้เป้าเดียวกัน CSS ตัดสินด้วยลำดับความเจาะจง: inline > id > class > tag "CSS ไม่ทำงาน" ส่วนใหญ่คือมีกฎอื่นเจาะจงกว่าทับอยู่ ไม่ใช่ browser เพี้ยน — DevTools แท็บ Styles จะขีดฆ่ากฎที่แพ้ให้ดู อ่านตรงนั้นก่อน อย่าเพิ่งใส่ !important (นั่นคือการก่อหนี้)

หลักการ 3: "Mobile-first ไม่ใช่แฟชั่น แต่คือลำดับการเขียน"

Bootstrap ออกแบบ class แบบ "เล็กก่อน ใหญ่ทับ": col-12 col-md-6 = มือถือเต็มจอ, จอ ≥768px ค่อยแบ่งครึ่ง เหตุผล: เริ่มจากจอเล็ก (พื้นที่จำกัด บังคับเรียงความสำคัญ) แล้วขยาย ง่ายกว่าเริ่มจากใหญ่แล้วพยายามยัดลงเล็ก — ผู้ใช้ไทยส่วนใหญ่เปิดเว็บจากมือถือ ลูกค้า media-presto ของคุณก็เช่นกัน

เมื่อไหร่ใช้ Bootstrap / ไม่ใช้: ใช้เมื่อต้องเร็วและทีมรู้จักมัน (งาน agency, admin panel) / โปรเจกต์ที่ design เฉพาะตัวจัด ๆ มักเขียนเอง หรือใช้ Tailwind (จะเจอใน Week 8 — หลักคิด utility เดียวกัน คุ้น Bootstrap แล้วต่อยอดไว)

ส่วนที่ 1: 3 วิธีใส่ CSS — และทำไม External ชนะ

<!-- 1. Inline — ฝังใน tag (เลี่ยง) -->
<h1 style="color:blue">Hi</h1>

<!-- 2. Internal — ใน <head> (เฉพาะหน้าเดียว) -->
<style> h1 { color:blue; } </style>

<!-- 3. External — แยกไฟล์ (มาตรฐาน) -->
<link rel="stylesheet" href="style.css">

ทำไม External ชนะ 3 เหตุผล: ① Cache — browser โหลด style.css ครั้งเดียวใช้ได้ทุกหน้า เว็บเร็วขึ้น ② แยกหน้าที่ — HTML=โครง, CSS=หน้าตา แก้สีไม่ต้องยุ่งกับโครง ③ ทีมทำงานร่วม — คนทำ style กับคนทำโครงไม่ชนกัน
Inline ใช้เมื่อ: ทดสอบเร็ว ๆ หรือ CSS ที่ JS สร้าง dynamic เท่านั้น

ส่วนที่ 2: Selectors — ชี้เป้าให้ถูกตัว (เจาะลึก)

2.1 Selector พื้นฐาน 3 แบบ

Selectorเลือกอะไรใช้เมื่อ
p { }ทุก <p> ในหน้า (Type/Tag)ตั้ง default ทั้งหน้า (ระวังกระทบหมด)
.box { }ทุก element ที่มี class="box"⭐ ใช้บ่อยสุด — ซ้ำได้ ยืดหยุ่น
#logo { }element เดียวที่ id="logo"ของชิ้นเดียวห้ามซ้ำ (เหมือนเลขบัตรประชาชน)

ทำไมยุคนี้ใช้ class เป็นหลัก: class ออกแบบมาให้ "ใช้ซ้ำ" — ปุ่ม 10 ปุ่มใช้ .btn เดียวกันได้ ส่วน id ออกแบบให้ "มีตัวเดียว" ใช้ใน CSS แล้วเจาะจงเกินไป (ดู 2.4) แก้ทับยาก id เหมาะกับ JavaScript (getElementById) และ anchor link (#section) มากกว่า CSS

2.2 Combinator — เลือกตาม "ความสัมพันธ์" (จุดที่คนงงเยอะสุด)

Selectorชื่อเลือกอะไร
.card pDescendant (เว้นวรรค)p ทุกตัว"ข้างใน" .card — ลูก หลาน เหลน ทั้งหมด
.card > pChild (>)p ที่เป็นลูกตรงชั้นเดียวของ .card เท่านั้น (หลานไม่โดน)
h2 + pAdjacent sibling (+)p ตัวแรกที่อยู่ติดหลัง h2 (พี่น้องชิดกัน)
h2 ~ pGeneral sibling (~)p ทุกตัวที่อยู่หลัง h2 (พี่น้องระดับเดียวกัน)
⚠️ ภาพลวงตาที่ต้องระวัง (เจอจริงตอน debug): ถ้าใช้ .box > div ใส่ background:red แล้วเห็น "div หลานก็แดงด้วย" — อย่าเพิ่งคิดว่า > พัง ความจริง: div หลานไม่มีสีพื้นหลังของตัวเอง (โปร่งใส) เลยมองทะลุเห็นสีแดงของ div พ่อที่อยู่ข้างหลัง background-color ไม่ inherit แต่ความโปร่งใส "โชว์สีพ่อทะลุ" → เวลา debug สี ให้ถามเสมอ "นี่สีของมันเอง หรือสีพ่อทะลุมา" (เช็คใน DevTools → Computed → background-color)

2.3 Pseudo-class & Pseudo-element — เลือกตาม "สถานะ" และ "ส่วนเสมือน"

ของจริงในงาน production ใช้ตัวนี้เยอะมาก — ทำให้ UI โต้ตอบได้โดยไม่ต้องเขียน JS:

Selectorเลือกเมื่อใช้จริงกับ
a:hoverเมาส์ชี้เปลี่ยนสีลิงก์/ปุ่มตอน hover
input:focusช่องถูกคลิก/พิมพ์อยู่ไฮไลต์ช่องฟอร์มที่กำลังกรอก
button:disabledปุ่มถูกปิดทำปุ่มจาง ๆ ตอนกดไม่ได้ (เชื่อม Week 1!)
li:first-child / :last-childลูกตัวแรก/สุดท้ายตัดเส้นขอบล่างของรายการสุดท้าย
tr:nth-child(even)แถวคู่ทำตารางสลับสี (zebra stripe)
p::first-line / ::beforeส่วนเสมือน (pseudo-element ใช้ ::)ใส่ไอคอน/เครื่องหมายหน้าข้อความ
/* ตัวอย่าง production: ปุ่มที่มี state ครบ */
.btn { background: #2563eb; color: #fff; transition: 0.2s; }
.btn:hover { background: #1e40af; }      /* ชี้แล้วเข้มขึ้น */
.btn:disabled { opacity: 0.5; cursor: not-allowed; }  /* ปิดแล้วจาง */

/* ตารางสลับสี — โค้ดที่ทุก dashboard ใช้ */
tr:nth-child(even) { background: #f7f8fa; }

เกร็ด: : (โคลอนเดียว) = pseudo-class (สถานะ) · :: (โคลอนคู่) = pseudo-element (ส่วนเสมือน)

2.4 Specificity — คำนวณจริงว่ากฎไหนชนะ

เมื่อหลายกฎชี้ element เดียวกัน CSS ตัดสินด้วย "คะแนนความเจาะจง" นับเป็น 3 หลัก (id, class, tag):

Selectoridclasstagคะแนน
p0010,0,1
.box0100,1,0
.box p0110,1,1
#main .box p1111,1,1 (ชนะทุกตัวข้างบน)

วิธีอ่าน: เทียบหลักซ้ายสุดก่อน (id) — มากกว่าชนะทันที / เท่ากันค่อยดูหลักถัดไป เหมือนเทียบเลข 3 หลัก 110 > 011 เพราะหลักร้อยมากกว่า

ทำไมเรื่องนี้สำคัญในงานจริง: "CSS เขียนแล้วไม่ทำงาน" 90% คือมีกฎอื่นเจาะจงกว่าทับอยู่ inline style (style="...") เจาะจงสุด (นับเป็นหลักพิเศษเหนือ id) — นี่คือเหตุผลที่ inline ใน HTML ทับ external CSS เสมอ และ !important อยู่เหนือทุกอย่าง (จึงเป็นหนี้ — ครั้งหน้าต้อง !important ทับ !important ไม่จบ)

2.5 🔗 เชื่อมกับ HTML: ทำไม selector บางทีหา element ไม่เจอ

บางครั้ง selector ถูกแต่ไม่ทำงาน เพราะ DOM ที่ browser สร้าง ≠ HTML ที่เราเขียน:

<!-- เราเขียนแบบนี้ -->
<p>
  <h1>หัวข้อ</h1>   <!-- ❌ p ใส่ block element ไม่ได้! -->
</p>

<p> ใส่ได้แค่ inline content — พอ browser เจอ <h1> (block) ข้างใน มันจะ ปิด p อัตโนมัติ แล้วดัน h1 ออกมาเป็นพี่น้อง ผลคือ .container > p > h1 หา h1 ไม่เจอ (เพราะ h1 ไม่ได้อยู่ใน p อีกต่อไป) → เปิด DevTools → Elements เช็ค DOM จริงก่อนเสมอเวลา selector ไม่ทำงาน

Block ที่ห้ามใส่ใน p: h1-h6, div, section, article, ul/ol, p อีกตัว — ถ้าต้องครอบหลายอย่างใช้ div/section

หลักปฏิบัติรวม (production): ① ใช้ .class เป็นหลัก ② เลี่ยง #id ใน CSS ③ เลี่ยง selector ลึกเกิน 3 ชั้น (.a .b .c .d = เปราะ แก้ยาก) ④ ตั้งชื่อ class ตามหน้าที่ ไม่ใช่หน้าตา (.btn-primary ดีกว่า .btn-blue — วันหน้าเปลี่ยนสีไม่ต้องแก้ชื่อ)

ส่วนที่ 3: Box Model — หัวใจของทุก layout (เจาะลึก)

.box {
  width: 200px;            /* เนื้อใน */
  padding: 20px;           /* เบาะดันข้างใน (รอบตัว) */
  border: 5px solid black; /* ขนาด + รูปแบบ + สี (ครบ 3) */
  margin: 30px;            /* ผลักเพื่อนบ้าน */
}

Shorthand — วนตามเข็มนาฬิกา (บน→ขวา→ล่าง→ซ้าย):

margin: 10px 20px 30px 40px;  /* บน ขวา ล่าง ซ้าย */
margin: 10px 20px;            /* (บน-ล่าง) (ซ้าย-ขวา) */
margin: 10px;                 /* ทุกด้านเท่ากัน */
กับดักคลาสสิก: ตั้ง width:200px แต่กล่องกว้าง 250px จริง! เพราะ default box-sizing:content-box นับ width แค่เนื้อใน (200 + padding 20×2 + border 5×2 = 250) ทางแก้มาตรฐานยุคนี้: box-sizing:border-box ทำให้ width = ตัวเลขที่ตั้งจริง (Bootstrap ตั้งให้ทั้งระบบแล้ว)

ส่วนที่ 4: Display — ควบคุมการวางตัว

ค่าพฤติกรรมตั้ง width/height
blockเต็มแถว + ขึ้นบรรทัดใหม่✅ ได้
inlineอยู่ในบรรทัดเดียวกัน กว้างเท่าเนื้อหา❌ ไม่ได้
inline-blockอยู่บรรทัดเดียวกัน + ตั้งขนาดได้ (ลูกผสม)✅ ได้
noneหายไปเลย ไม่กินพื้นที่—

display:none vs visibility:hidden: none = หายสนิทเหมือนไม่เคยมี (พื้นที่หายด้วย) / hidden = มองไม่เห็นแต่ยังกินพื้นที่ (เหมือนใส่ผ้าคลุมล่องหน) — เลือกตามว่าอยากให้ที่ว่างยุบหรือไม่

ส่วนที่ 5: Bootstrap Grid — เค้ก 12 ชิ้น

<div class="container">        <!-- คุมความกว้างไม่ให้ล้นจอ -->
  <div class="row">            <!-- แถวแนวนอน -->
    <div class="col-12 col-md-6">ซ้าย</div>   <!-- มือถือเต็ม, จอใหญ่ครึ่ง -->
    <div class="col-12 col-md-6">ขวา</div>
  </div>
</div>

กฎ: หน้าจอแบ่ง 12 คอลัมน์เสมอ · col ในแถวรวมกันได้ 12 · container→row→col ต้องซ้อนตามลำดับนี้

เขียนหมายความว่า
colหารเท่ากันเองอัตโนมัติ (ขี้เกียจคิดเลข)
col-6ครึ่งจอ ตลอดกาล ทุกขนาดจอ
col-md-6จอ <768px เต็มจอ (ซ้อนแนวตั้ง), ≥768px ค่อยแบ่งครึ่ง ⭐ ใช้บ่อยสุด

Breakpoints: sm≥576 · md≥768 (แท็บเล็ต) · lg≥992 (แล็ปท็อป) · xl≥1200

ส่วนที่ 6: Spacing Utilities + Components

สูตร spacing: {property}{side}-{size}

  • property: m=margin · p=padding
  • side: tบน bล่าง sซ้าย eขวา xซ้ายขวา yบนล่าง (ว่าง)ทุกด้าน
  • size: 0=0 ถึง 5=3rem · auto

ตัวอย่าง: mt-5=margin บนเยอะ · px-4=padding ซ้ายขวา · mx-auto=จัดกึ่งกลาง (เหมือน margin:0 auto)

Components ที่ควรใช้เป็น (อ่าน docs เองต่อได้): Navbar (ย่อเป็น hamburger บนมือถืออัตโนมัติ), Card, Carousel (สไลด์), Accordion (FAQ ยืดหด) — เปิด getbootstrap.com copy มาปรับ

🧹 Clean Code: เขียน CSS แบบ production

1. ตั้งชื่อ class ตามหน้าที่ ไม่ใช่หน้าตา

❌ ผูกกับหน้าตา

.red-text { color: red; }
.big-box { ... }
.mt-20px { margin-top: 20px; }

✅ ผูกกับความหมาย

.text-danger { color: red; }
.product-card { ... }
.section-spacing { margin-top: 20px; }

เหตุผล: วันหน้าเปลี่ยน danger เป็นส้ม แค่แก้ค่าในที่เดียว — ถ้าชื่อ .red-text จะกลายเป็นโกหก (ชื่อแดงแต่สีส้ม)

2. ใช้ CSS Variables — แก้ที่เดียว เปลี่ยนทั้งเว็บ (production มาตรฐาน)

❌ ฮาร์ดโค้ดสีซ้ำ 50 ที่

.btn { background: #2563eb; }
.link { color: #2563eb; }
.badge { background: #2563eb; }
/* เปลี่ยนสีแบรนด์ = แก้ 50 จุด */

✅ ตัวแปรกลาง

:root { --brand: #2563eb; }
.btn { background: var(--brand); }
.link { color: var(--brand); }
/* เปลี่ยนสีแบรนด์ = แก้ 1 จุด */

นี่คือเทคนิคที่ style.css ของคอร์สนี้ใช้ (ดู :root ด้านบนไฟล์) และเป็นรากของ "design token" ในทีมใหญ่

3. จัดกลุ่ม property ตามตรรกะ (อ่านง่าย review ง่าย)

.card {
  /* 1. Layout/ตำแหน่ง */    display: flex;  position: relative;
  /* 2. Box model */          width: 300px;  padding: 16px;  margin: 8px;
  /* 3. Typography */         font-size: 14px;  line-height: 1.6;
  /* 4. Visual */             background: #fff;  border-radius: 8px;
}
กฎทอง: เขียน CSS ให้ "คนอื่นแก้ต่อได้" — หลีกเลี่ยง magic number (เลขลอย ๆ ไม่มีที่มา), เลี่ยง !important, และอย่า nest selector ลึกเกิน 3 ชั้น เพราะยิ่งลึก specificity ยิ่งสูง = แก้ทับยากในอนาคต

ตัวอย่างประยุกต์ในโปรเจกต์จริง

🏢 งานวี — welovefukushima: เว็บคุณมี CSS bundler ที่รวม partials เป็นไฟล์เดียว — หลักการเดียวกับ "External CSS ชนะเพราะ cache + แยกหน้าที่" ที่ scale ใหญ่ขึ้น ถ้าเปิดดูจะเห็นว่าใช้ class ตามหน้าที่ (utility-style) คล้าย Bootstrap — คุณทำถูกอยู่แล้วโดยสัญชาตญาณ บทนี้ทำให้อธิบาย "ทำไม" ได้

🏢 งานวี — automation dashboard: ถ้าทำหน้า dashboard การตลาด ตารางสถิติจะใช้ tr:nth-child(even) สลับสี + :hover ไฮไลต์แถว — UX ดีขึ้นโดยไม่เขียน JS เลย (จาก 2.3)

🌏 อุตสาหกรรม: ระบบ admin/internal tool ของบริษัทไทยจำนวนมากคือ Bootstrap ล้วน เพราะเร็วและทุกคนอ่านออก — ส่วน product หน้าบ้านยุคใหม่ (LINE MAN, Agoda) เทไป Tailwind (Week 8) ซึ่งคุณจะเรียนไวเพราะหลัก utility-first เดียวกัน และ design token (CSS variables ใน 2 ข้อบน) คือสิ่งที่ทุก design system ระดับองค์กรใช้

ข้อ 1: กล่อง width:200px + padding:20px + border:5px — กว้างจริงบนจอกี่ px และทำไมมือใหม่งง?

250px (200+20×2+5×2) เพราะ default box-sizing: content-box นับ width แค่เนื้อใน — มือใหม่งงเพราะตั้ง 200 แต่ได้ 250 ทางแก้มาตรฐานยุคนี้: box-sizing: border-box (นับรวมหมดให้ width = ตัวเลขที่ตั้ง) ซึ่ง Bootstrap ตั้งให้ทั้งระบบอยู่แล้ว

ข้อ 2: col-6 กับ col-md-6 ต่างกันยังไงบนมือถือ?

col-6 = ครึ่งจอตลอดกาลทุกขนาดจอ / col-md-6 = จอ <768px เต็มจอ (ซ้อนแนวตั้ง), ≥768px ค่อยแบ่งครึ่ง — เกือบทุกเคสจริงต้องการแบบหลัง เพราะครึ่งจอบนมือถือ = เนื้อหาบี้จนอ่านไม่ได้

ข้อ 3: CSS ที่เขียนไม่ทำงาน — ลำดับการ debug ที่ถูกคืออะไร (ไม่ใช่ใส่ !important)?

① เปิด DevTools เลือก element ดูแท็บ Styles ② ดูว่ากฎเราโดนขีดฆ่าไหม → แพ้ specificity ให้กฎไหน ③ เช็ค selector พิมพ์ถูก/ไฟล์ CSS โหลดจริง (แท็บ Network) ④ เช็ค DOM จริง (element อยู่ตรงที่คิดไหม — เช่น p ถูกปิดอัตโนมัติ) ⑤ แก้ด้วยการเพิ่มความเจาะจงอย่างตั้งใจ — !important คือทางออกสุดท้ายที่สร้างหนี้

ข้อ 4: .box > div ใส่ background แดง แต่ div หลานก็ดูแดง — > พังหรือเปล่า? อธิบาย

ไม่พัง — > เลือกแค่ลูกตรง div หลานไม่ได้โดน CSS แต่หลานพื้นหลังโปร่งใส (default) เลยมองทะลุเห็นสีแดงของพ่อ background-color ไม่ inherit แต่ความโปร่งใสโชว์สีพ่อ — พิสูจน์ด้วยการใส่ .box div div { background: yellow } หลานจะเหลือง ไม่แดง

ข้อ 5: เทียบ specificity — #nav .item กับ .menu .item a อันไหนชนะ?

#nav .item = (1id, 1class, 0tag) = 1,1,0 / .menu .item a = (0id, 2class, 1tag) = 0,2,1 เทียบหลักซ้ายสุด (id) ก่อน: 1 > 0 → #nav .item ชนะ แม้อีกฝั่งมี class+tag เยอะกว่า (เพราะ id 1 ตัวมีน้ำหนักเหนือ class กี่ตัวก็ตาม)
