Week 18 · บทที่ 2

Token Storage + Refresh + Rotation

ปิดประเด็น "token เก็บที่ไหน" ให้จบ — ตั้งแต่ Week 1 จนถึงตอนนี้

🔑 คีย์เวิร์ดภาพจำประจำบท

🎫 access token = อายุสั้น ใช้บ่อย 🔄 refresh token = อายุยาว ขอ access ใหม่ ♻️ ใช้แล้วเปลี่ยน = rotation 🔒 refresh ใน HttpOnly cookie

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Access สั้น + Refresh ยาว = สมดุล security กับ UX"

access token อายุสั้น (15 นาที) — ถ้าหลุด ผู้โจมตีใช้ได้แค่สั้น ๆ · แต่ถ้าให้ user login ใหม่ทุก 15 นาที = UX แย่ · refresh token (อายุยาว) ขอ access token ใหม่ได้เงียบ ๆ · สมดุล: เสียหายจำกัด (access สั้น) + ไม่ต้อง login บ่อย (refresh)

หลักการ 2: "Refresh rotation — ใช้แล้วเปลี่ยน ตรวจการขโมย"

ทุกครั้งที่ใช้ refresh token ขอ access ใหม่ → ออก refresh token ใหม่ด้วย (ตัวเก่าใช้ไม่ได้) · ถ้าผู้โจมตีขโมย refresh ไปใช้ เจ้าของใช้ตัวเดิม (ที่ถูก rotate ไปแล้ว) → ระบบจับได้ว่า "มีคนใช้ token เก่า" → revoke ทั้งหมด · rotation = ตรวจจับการขโมย

หลักการ 3: "ไม่มี storage ที่สมบูรณ์แบบ — เลือกตาม threat (ปิดประเด็น W1)"

localStorage (XSS เสี่ยง) · HttpOnly cookie (XSS ปลอดภัยกว่า แต่ CSRF เสี่ยง — แก้ด้วย SameSite + CSRF token) · คำตอบสุดท้าย: refresh token ใน HttpOnly cookie (SameSite=Strict), access token ใน memory (ไม่เก็บถาวร) — สมดุลที่ดีสุดสำหรับเว็บส่วนใหญ่

ส่วนที่ 2: Refresh Token Rotation

เก็บ refresh token ใน DB (hashed) พร้อม "family" — ใช้ตัวเก่าที่ rotate ไปแล้ว = จับได้ว่ามีการขโมย → revoke ทั้ง family · นี่คือ best practice ปัจจุบัน

ส่วนที่ 3: Token Storage — คำตอบสุดท้าย (ปิดประเด็น W1)

เก็บที่XSSCSRFคำแนะนำ
localStorage🔴 เสี่ยง (JS อ่านได้)🟢 ปลอดภัยเลี่ยงสำหรับ token สำคัญ
HttpOnly Cookie🟢 ปลอดภัย (JS อ่านไม่ได้)🟡 เสี่ยง (แก้ด้วย SameSite+CSRF token)⭐ refresh token
Memory (JS variable)🟡 (หายเมื่อ refresh หน้า)🟢⭐ access token
สูตรที่แนะนำสำหรับเว็บส่วนใหญ่ (สรุปจบ):
• access token → memory (ตัวแปร JS — หายเมื่อ refresh หน้า, ขอใหม่จาก refresh) — โดน XSS ก็ได้แค่อายุสั้น
• refresh token → HttpOnly cookie + SameSite=Strict — JS อ่านไม่ได้ (กัน XSS) + SameSite กัน CSRF
นี่คือคำตอบของ "token เก็บที่ไหน" ที่เราคุยมาตั้งแต่ Week 1 — ขึ้นกับ threat แต่สูตรนี้สมดุลที่สุด

ส่วนที่ 4: Session vs JWT — เลือกเมื่อไหร่

JWT (stateless)Session (stateful)
เก็บ stateใน token (server ไม่จำ)server จำ (DB/Redis)
revoke ทันทียาก (token valid จนหมดอายุ)ง่าย (ลบ session)
scaleง่าย (ไม่ต้องแชร์ state)ต้องแชร์ session store
เหมาะกับAPI, microserviceapp ที่ต้อง revoke ทันที (banking)

ไม่มีถูกผิดเด็ดขาด — fintech/banking มัก session (revoke ทันทีสำคัญกว่า scale) · API/SPA มัก JWT + refresh · เข้าใจ trade-off = ตอบ interview ได้

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: automation service เก็บ OAuth refresh token ของลูกค้า (เพื่อดึงข้อมูลต่อเนื่องโดยไม่ต้องให้ลูกค้า login ทุกครั้ง) — ต้องเก็บ encrypted ใน DB, rotate, revoke ได้ถ้าลูกค้ายกเลิก · นี่คือ token management ระดับ production จริง

🌏 อุตสาหกรรม: access+refresh+rotation เป็นมาตรฐาน auth ของ API สมัยใหม่ · token storage strategy เป็นคำถาม interview security ที่พบบ่อยสุด · "ตอบได้ว่าเก็บที่ไหนเพราะอะไร (threat model)" = แสดงความเข้าใจลึก ไม่ใช่ท่องจำ · session vs JWT trade-off เป็นหัวข้อ system design ด้วย

📝 Quiz ท้ายบท

ข้อ 1: ทำไมแยก access (สั้น) กับ refresh (ยาว) token?

access สั้น = หลุดแล้วเสียหายจำกัด · แต่ถ้า user ต้อง login ใหม่ทุก 15 นาที = UX แย่ · refresh (ยาว) ขอ access ใหม่เงียบ ๆ · สมดุล: security (access สั้น) + UX (ไม่ login บ่อย)

ข้อ 2: refresh token rotation ตรวจจับการขโมยยังไง?

ใช้ refresh แล้วออกตัวใหม่ (ตัวเก่าใช้ไม่ได้) · ถ้าผู้โจมตีขโมยตัวเก่าไปใช้ — เจ้าของใช้ตัวที่ rotate แล้ว → ระบบเห็น "token เก่าถูกใช้ซ้ำ" = สัญญาณขโมย → revoke ทั้ง family · rotation เปลี่ยน refresh จาก "static" เป็น "ตรวจสอบได้"

ข้อ 3: สูตร token storage ที่แนะนำสำหรับเว็บคืออะไร (ปิดประเด็น W1)?

access token → memory (JS variable, หายเมื่อ refresh, โดน XSS ได้แค่สั้น) · refresh token → HttpOnly cookie + SameSite=Strict (JS อ่านไม่ได้ กัน XSS + SameSite กัน CSRF) · สมดุล XSS+CSRF ดีสุดสำหรับเว็บส่วนใหญ่

ข้อ 4: JWT vs Session — banking ควรใช้อันไหน เพราะอะไร?

Session (stateful) — เพราะ revoke ทันทีสำคัญกว่า scale (user โดนแบน/logout ต้องตัดทันที) · JWT valid จนหมดอายุ revoke ยาก · banking ยอมแลก scale เพื่อ control · API/SPA ทั่วไปใช้ JWT+refresh (scale ง่ายกว่า) · เลือกตาม priority

✅ Checkpoint

  • เข้าใจ access (สั้น) + refresh (ยาว) + flow ขอ access ใหม่
  • refresh rotation (ตรวจจับการขโมย)
  • token storage strategy (access→memory, refresh→HttpOnly cookie)
  • session vs JWT trade-off
