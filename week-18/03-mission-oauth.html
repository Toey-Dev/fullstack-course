🔑 Login with Google + Token Strategy

เพิ่ม social login ที่ปลอดภัย + จัดการ token แบบ production — ยกระดับ auth ของ library-system

ทำไม Mission นี้ยกระดับ portfolio สาย security

หลังจบ library-system จะมี "Login with Google" (OAuth/OIDC) + token strategy ระดับ production (access+refresh+rotation, storage ปลอดภัย, ป้องกัน state/CSRF) · OAuth ที่ implement ปลอดภัย (ไม่ใช่แค่ copy ให้ทำงาน) คือสิ่งที่ AppSec interview เจาะ — และคุณตอบได้เพราะเข้าใจช่องโหว่จากฝั่งโจมตี

🪜 แบ่งเป็น 4 Mission ย่อย

Mission 18.1 — OAuth App + Login Flow 90 นาที

  1. ลงทะเบียน OAuth app (Google หรือ GitHub) — ได้ client id/secret (ใน .env)
  2. ใช้ library (passport/arctic/auth.js) ทำ Authorization Code flow + PKCE
  3. callback: verify id_token (signature/iss/aud) → สร้าง/หา user ในระบบ
security: ใช้ state param (verify ตอน callback) · redirect_uri allowlist · client secret ใน env

Mission 18.2 — Access + Refresh Token 75 นาที

  1. ออก access token (สั้น 15 นาที) + refresh token (ยาว)
  2. access → response (frontend เก็บใน memory) · refresh → HttpOnly cookie (SameSite)
  3. POST /auth/refresh — ขอ access ใหม่จาก refresh
✅ เทสต์: access หมดอายุ → frontend ขอใหม่อัตโนมัติ (ไม่ต้อง login) · refresh ใน cookie อ่านจาก JS ไม่ได้

Mission 18.3 — Refresh Rotation + Revoke 60 นาที

  1. rotation: ใช้ refresh แล้วออกตัวใหม่ (เก็บ hashed ใน DB + family)
  2. ตรวจใช้ token เก่าซ้ำ → revoke ทั้ง family (จับการขโมย)
  3. logout → revoke refresh + ลบ cookie
เทสต์ security: ลองใช้ refresh เก่า (ที่ rotate แล้ว) → ถูก revoke ทั้งหมด (จับการขโมย)

Mission 18.4 — Security audit + push 45 นาที

  1. ตรวจ OAuth security: state verify, redirect_uri, id_token verify, no token in URL
  2. อัปเดต SECURITY.md: auth strategy + token storage + OAuth hardening
  3. push — README + AI Usage
✅ เทสต์: SECURITY.md อธิบาย token strategy ชัด · ลอง attack (CSRF login, token เก่า) ถูกบล็อก
