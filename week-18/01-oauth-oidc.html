OAuth 2.0 + OIDC — flow และทำไม

เข้าใจ "Login with Google" ถึงแก่น — ทั้งฝั่งใช้งานและฝั่งช่องโหว่

🔑 คีย์เวิร์ดภาพจำประจำบท

🎟 ให้สิทธิ์โดยไม่ให้รหัส = OAuth 🪪 OAuth + บอกว่าเป็นใคร = OIDC (id_token) 🎫 code → แลก token = Authorization Code flow 🔐 กันขโมย code = PKCE 🛡 state = กัน CSRF

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "OAuth = ให้สิทธิ์ โดยไม่ต้องให้รหัสผ่าน"

"Login with Google" — คุณไม่ได้ให้รหัส Google กับเว็บนั้น · OAuth ให้เว็บได้ "token" ที่ Google ออกให้ (แทนรหัส) · เว็บใช้ token เข้าถึงข้อมูลที่คุณอนุญาต โดยไม่เคยเห็นรหัสคุณ · นี่คือ "delegated authorization" — มอบสิทธิ์บางส่วนอย่างปลอดภัย

หลักการ 2: "OAuth = authorization, OIDC = authentication"

OAuth เดิมตอบ "เว็บนี้ทำอะไรแทนคุณได้บ้าง" (สิทธิ์) — ไม่ได้ตอบ "คุณเป็นใคร" · OIDC (OpenID Connect) สร้างบน OAuth เพิ่ม id_token ที่บอก identity (ชื่อ, email, sub) · "login ด้วย Google" จริง ๆ คือ OIDC (ต้องรู้ว่าเป็นใคร) · คนสับสน 2 อันนี้บ่อย — แยกได้ = เข้าใจลึก

หลักการ 3: "อย่าเขียน OAuth เอง — แต่ต้องเข้าใจว่ามันทำอะไร"

OAuth มีรายละเอียด security เยอะ (PKCE, state, token verify) — เขียนเองพลาดง่าย ใช้ library ที่ตรวจแล้ว · แต่ต้องเข้าใจ flow เพื่อ debug + รู้ว่าช่องโหว่อยู่ไหน · pentester ที่เข้าใจ OAuth flow หาช่องโหว่ (open redirect, CSRF) ได้

ส่วนที่ 1: Authorization Code Flow (+ PKCE)

ทำไมต้อง "code" ก่อน "token" (ไม่ส่ง token ตรง): code ผ่าน browser (เห็นได้) แต่ใช้ครั้งเดียว+อายุสั้น · การแลก code→token ทำหลังบ้าน (server-to-server) ที่ token ไม่ผ่าน browser → token ไม่รั่ว · PKCE เพิ่มการพิสูจน์ว่า "คนที่แลก code = คนที่เริ่ม flow" (กันขโมย code ไปแลก) — จำเป็นสำหรับ SPA/mobile

ส่วนที่ 2: OIDC — เพิ่ม Identity

// OAuth คืน: access_token (ใช้เรียก API ของ Google แทนผู้ใช้)
// OIDC เพิ่ม: id_token (JWT ที่บอกว่าผู้ใช้เป็นใคร)
// id_token decoded:
{
  "sub": "google-user-id-123",   // unique id ของผู้ใช้
  "email": "vee@gmail.com",
  "name": "Vee",
  "iss": "https://accounts.google.com",   // ใครออก token
  "aud": "our-client-id",                  // ออกให้ใคร (เรา)
  "exp": ...
}
OAuth 2.0OIDC
ตอบคำถาม"ทำอะไรแทนได้" (authz)"เป็นใคร" (authn) + authz
คืนaccess_tokenaccess_token + id_token
ใช้ทำเข้าถึง API (เช่น อ่าน calendar)login (รู้ว่าใคร)
🔐 verify id_token เสมอ: ก่อนเชื่อ id_token ต้อง verify signature (จาก Google), เช็ค iss (ออกจาก Google จริง), aud (ออกให้เราจริง ไม่ใช่เว็บอื่น), exp (ไม่หมดอายุ) · ไม่ verify = ผู้โจมตีปลอม id_token สวมรอยได้

ส่วนที่ 4: ตัวอย่างประยุกต์

🏢 งานวี: automation service ที่ดึง GA4/Meta — จริง ๆ ใช้ OAuth! (ลูกค้าอนุญาตให้เราเข้าถึงข้อมูล GA4 ผ่าน OAuth) · เข้าใจ flow ทำให้ debug การเชื่อม (token หมดอายุ → refresh), จัดการ scope (ขอสิทธิ์เท่าที่จำเป็น), เก็บ refresh token ปลอดภัย · Week 18 คือทฤษฎีเบื้องหลังการต่อ API ที่ต้อง authorize

🌏 อุตสาหกรรม: OAuth/OIDC เป็นมาตรฐาน auth ของทุกระบบ (login social, API access, SSO องค์กร) · ช่องโหว่ OAuth เป็นหัวข้อ pentest/bug bounty ยอดนิยม (จ่ายสูง) · dev ที่เข้าใจ OAuth ลึก + เห็นช่องโหว่ = AppSec ที่หายาก

📝 Quiz ท้ายบท

ข้อ 1: OAuth ต่างจาก OIDC ยังไง?

OAuth = authorization (ทำอะไรแทนได้ — access_token) · OIDC = OAuth + authentication (เป็นใคร — เพิ่ม id_token) · "Login with Google" จริง ๆ คือ OIDC (ต้องรู้ว่าใคร) · OAuth เดิมแค่ให้สิทธิ์ ไม่บอก identity

ข้อ 2: ทำไม Authorization Code flow ใช้ "code" ก่อน "token" ไม่ส่ง token ตรง?

code ผ่าน browser (เห็นได้) แต่ใช้ครั้งเดียว+อายุสั้น · แลก code→token ทำหลังบ้าน (server-to-server) ที่ token ไม่ผ่าน browser → token ไม่รั่ว · PKCE เพิ่มพิสูจน์ว่าคนแลก code = คนเริ่ม flow (กันขโมย code)

ข้อ 3 (security): state param กันอะไร ไม่มีเกิดอะไร?

กัน CSRF — state สุ่มตอนเริ่ม flow + verify ตอน callback · ไม่มี = ผู้โจมตีหลอกให้เหยื่อ login ด้วย account ผู้โจมตี (login CSRF) → เหยื่อใช้งานในบัญชีผู้โจมตีโดยไม่รู้ตัว · state ผูก request กับ session ของผู้ใช้จริง

ข้อ 4 (security): ทำไมต้อง verify id_token (iss/aud/signature)?

ไม่ verify = ผู้โจมตีปลอม id_token สวมรอยเป็นใครก็ได้ · verify: signature (Google ออกจริง), iss (จาก Google), aud (ออกให้เรา ไม่ใช่เว็บอื่น), exp (ไม่หมดอายุ) · เหมือน JWT verify (W5) แต่ token มาจาก provider ภายนอก ยิ่งต้องเช็คครบ

✅ Checkpoint

  • เข้าใจ Authorization Code flow + PKCE (code→token หลังบ้าน)
  • แยก OAuth (authz) vs OIDC (authn + id_token)
  • verify id_token (signature/iss/aud/exp)
  • รู้ช่องโหว่ OAuth (state/CSRF, open redirect, token in URL)
