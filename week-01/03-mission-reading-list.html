Week 1 · Mission ใหญ่

📚 My Reading List — ชิ้นงานปิดจบ Frontend

รวมทุกทักษะเป็นแอปจริง 1 ชิ้น: Semantic HTML · Bootstrap · Fetch · LocalStorage · Deploy

ทำไม Mission นี้สำคัญ

นี่คือ ตัววัดว่าพร้อมเข้า Backend หรือยัง — ถ้าทำจบได้โดยไม่เปิดเฉลย แปลว่า Frontend ของคุณแน่นพอแล้ว ถ้าติดตรงไหน นั่นคือจุดที่ต้องกลับไปทบ (และผมจะรู้ว่าต้องเสริมอะไรใน Week 2)

เปิดดูได้เฉพาะ "โจทย์และรูปผลลัพธ์" ใน บท 19 ของคอร์สเดิม — ห้ามดูโค้ดตัวอย่าง ให้เขียนจากสเปคด้านล่างนี้แทน

📋 สเปคงาน (เขียนแบบที่ลูกค้าจริงจะส่งให้คุณ)

แอปอ่านบทความ "My Reading List"

  • ดึงบทความ 6 ชิ้นจาก https://jsonplaceholder.typicode.com/posts?_limit=6 แสดงในพื้นที่หลัก
  • แต่ละบทความมีปุ่ม "บันทึกไว้อ่านทีหลัง" → ไปโผล่ใน sidebar ขวา และอยู่รอดข้าม refresh (localStorage)
  • ลบรายการที่บันทึกได้ทีละชิ้น (ปุ่ม X) หรือล้างทั้งหมด (มี confirm ก่อน)
  • บันทึกซ้ำชิ้นเดิมไม่ได้ — ต้องเด้งเตือน
  • Responsive: จอคอม 2 คอลัมน์ / มือถือเรียงซ้อน

โครงสร้างข้อมูลจาก API: แต่ละบทความเป็น object { id, title, body }

🪜 แบ่งเป็น 5 Mission ย่อย — ทำทีละขั้น เทสต์ทุกขั้น

หลักการทำงานแบบมืออาชีพ: ไม่เขียนรวดเดียวจบแล้วค่อยเทสต์ แต่สร้างทีละชั้นและพิสูจน์ว่าชั้นนั้นทำงานก่อนไปต่อ (นิสัยนี้จะกลายเป็น unit testing ใน Week 3)

Mission 3.1 — โครง HTML 45 นาที

1
สร้าง reading-list.html ใส่ Bootstrap 5 CDN (CSS ใน head, JS bundle ก่อนปิด body)
2
วางโครง Semantic: <header> (navbar ดำ มีชื่อแอป) → <div class="container"><div class="row"> ครอบ <main class="col-md-8"> และ <aside class="col-md-4"> → <footer>
3
ใน main: หัวข้อ + กล่องเปล่า id="articleContainer" ใส่ข้อความ "กำลังโหลด..." ไว้ก่อน
ใน aside: card หัวสีน้ำเงิน "📌 บทความที่บันทึก" + กล่อง id="savedList" + ปุ่ม id="btnClear"
✅ เทสต์ก่อนไปต่อ: เปิดไฟล์ → จอคอมเห็น 2 คอลัมน์ (8:4) → ย่อจอแคบ ๆ → ต้องเรียงซ้อนกัน

Mission 3.2 — Fetch + Render 45 นาที

1
เขียนฟังก์ชัน loadArticles() ตาม pattern 7 ขั้นจากบทที่ 2 (loading → fetch → เช็ค res.ok → json → render → catch → finally)
2
Render ด้วย .map() + template literal แล้ว join("") — ไม่ใช่ต่อ string ทีละรอบใน for loop (ท่า map เป็นมาตรฐานที่ React ใช้ คุ้นไว้ตั้งแต่ตอนนี้)
3
เนื้อหา (body) ยาวเกิน 80 ตัวอักษร → ตัดด้วย substring(0, 80) + "..."
4
ปุ่มบันทึกในแต่ละ card ฝัง data ไว้: onclick="saveArticle(${id}, '...')" (หรือใครอยากเท่กว่า: ใช้ data-attribute + event delegation — ถามผมได้)
✅ เทสต์: เปิดปกติเห็น 6 บทความ → ปิด WiFi แล้ว refresh → ต้องเห็นข้อความ error ไม่ใช่ "กำลังโหลด..." ค้าง

Mission 3.3 — บันทึกลง LocalStorage 60 นาที

1
saveArticle(id, title): อ่าน array เดิม (JSON.parse(...) ?? []) → เช็คซ้ำด้วย .some(a => a.id === id) → ถ้าซ้ำ alert เตือนแล้วจบ → ถ้าไม่ซ้ำ push {id, title} → stringify เซฟกลับ
2
เรียก renderSaved() ทุกครั้งหลังเซฟ เพื่ออัปเดต sidebar ทันที
✅ เทสต์: บันทึก 2 ชิ้น → เห็นใน sidebar → กดบันทึกชิ้นเดิมซ้ำ → ต้องเด้งเตือน ไม่เพิ่มซ้ำ

Mission 3.4 — โหลดกลับ + ลบ 45 นาที

1
renderSaved(): อ่านจาก localStorage → ว่าง = ข้อความ "ยังไม่มีบทความ" → มีของ = list รายการ + ปุ่ม X ท้ายแต่ละแถว → เรียกตอนเปิดเว็บด้วย (บรรทัดล่างสุดของ script)
2
removeArticle(id): ใช้ .filter(a => a.id !== id) เก็บเฉพาะตัวที่ไม่ถูกลบ → เซฟกลับ → render ใหม่
3
ปุ่ม btnClear: confirm("ลบทั้งหมด?") → ตกลงค่อย removeItem → render ใหม่
✅ เทสต์ครบวงจร: บันทึก 3 → refresh → อยู่ครบ 3 → ลบ 1 → refresh → เหลือ 2 → ล้างหมด → refresh → ว่าง

Mission 3.5 — Deploy ขึ้นโลกจริง 30 นาที

📦 มาตรฐาน Portfolio: repo นี้คือผลงานสมัครงานชิ้นแรก — ก่อน push ให้เปิด course/portfolio-standard.md ทำตาม checklist: README ตาม template (มี Features, Tech Stack, What I Learned, AI Usage), commit message แบบ feat:/fix:, ไม่มีโค้ดตาย, ไม่มี secret
1
เปลี่ยนชื่อไฟล์เป็น index.html (กฎของ GitHub Pages)
2
git init → git add . → git commit -m "feat: reading list app with fetch and localstorage" (สังเกต commit message แบบมืออาชีพ — บอกว่าทำอะไร ไม่ใช่ "update")
3
สร้าง repo ชื่อ reading-list บน GitHub (Public) → push ขึ้นไป
4
Settings → Pages → Deploy from branch → main → Save → รอ 1-2 นาที ได้ URL
✅ เทสต์สุดท้าย: เปิด URL จากมือถือ (คนละเครื่อง) — ใช้งานได้ครบ = ผ่าน 🎉
