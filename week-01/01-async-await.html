Async/Await ให้เข้าใจถึงแก่น

เรียนรู้วิธีที่ JavaScript จัดการกับ "งานที่ต้องรอ" — หัวใจของการต่อ API และ Database ทุกระบบ

🎯 จบบทนี้คุณจะ...

  • อธิบายได้ว่า "ทำไม" JS ต้องมี async ไม่ใช่แค่ "ใช้ยังไง"
  • เขียน async/await + try/catch ได้โดยไม่เปิด docs
  • อ่านโค้ด Promise ของคนอื่นออก (จำเป็นมากตอนเข้าทีมใหม่)
  • ตอบคำถาม interview เรื่อง Event Loop เบื้องต้นได้

🔑 คีย์เวิร์ดภาพจำประจำบท (ใช้ตอน Friday Rebuild: เห็นคำ → ดึงภาพ → ดึงโค้ด)

☕ บาริสต้าคนเดียว = single thread 🛎 สั่งทิ้งไว้ เสร็จแล้วติ๊ง! = async 🧾 ใบรับของ 3 สถานะ = Promise ⏸ "รอตรงนี้" = await 🕳 กล่องเปล่ายังไม่มีของ = ลืม await 🛟 ตาข่ายรองรับ = try/catch

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

ก่อนแตะ syntax ต้องเข้าใจ "วิธีคิด" ที่อยู่เบื้องหลังก่อน — ถ้าเข้าใจหลักการเหล่านี้ ต่อให้ลืม syntax ก็เขียนเองใหม่ได้ เพราะรู้ว่ากำลังแก้ปัญหาอะไร:

หลักการ 1: "Don't block the main thread" (อย่าขวางเส้นเลือดใหญ่)

JS มี thread เดียวที่ต้องทั้งรันโค้ดและวาดหน้าจอ — งานไหนที่ "ใช้เวลาไม่แน่นอน" (network, อ่านไฟล์, database) ห้ามให้มันยืนรอ ต้องสั่งทิ้งไว้แล้วไปทำอย่างอื่น หลักการนี้คือเหตุผลการมีอยู่ของ async ทั้งระบบ และเป็นหลักเดียวกับที่ Node.js ใช้รับ request เป็นพัน ๆ พร้อมกันได้ด้วย thread เดียว (จะเจอใน Week 2)

หลักการ 2: "Errors are values, not surprises" (ความผิดพลาดคือสิ่งที่ต้องวางแผน ไม่ใช่อุบัติเหตุ)

Dev มือใหม่เขียนเผื่อแค่ทางสำเร็จ (happy path) — dev มืออาชีพถามก่อนเสมอว่า "ถ้าพังตรงนี้ ผู้ใช้จะเห็นอะไร?" นี่คือเหตุผลที่ try/catch ไม่ใช่ของแถม แต่เป็นส่วนหนึ่งของ design ตั้งแต่บรรทัดแรก

หลักการ 3: "Readability over cleverness" (โค้ดอ่านง่ายชนะโค้ดเท่)

Callback → Promise → async/await คือวิวัฒนาการของ "ความอ่านง่าย" ล้วน ๆ ความสามารถเท่าเดิมทุกยุค แต่โค้ดที่อ่านง่ายกว่า = ทีม maintain ได้ = บั๊กน้อยกว่า บริษัทจ่ายแพงให้คนเขียนโค้ดที่ คนอื่นอ่านรู้เรื่อง ไม่ใช่โค้ดที่ฉลาดที่สุด

เมื่อไหร่ใช้ / ไม่ใช้: ใช้ async กับงาน I/O ทุกชนิด (network, ไฟล์, DB) / ไม่ต้องใช้กับงานคำนวณธรรมดา (บวกเลข, จัด array, แปลง string) — ใส่ async มั่ว ๆ ทุกฟังก์ชันคือ signal ว่าไม่เข้าใจ ซึ่ง interviewer จับได้ทันที

ส่วนที่ 1: ทำไมต้องมี Async? (เข้าใจปัญหาก่อนเข้าใจทางแก้)

1

JavaScript ทำงานทีละบรรทัด ด้วย "พนักงานคนเดียว"

ลองนึกภาพร้านกาแฟที่มี บาริสต้าแค่ 1 คน (JS เรียกว่า single thread) ลูกค้าสั่งอะไรมา เขาก็ทำทีละอย่าง เสร็จอันแรกค่อยเริ่มอันถัดไป — โค้ดปกติก็ทำงานแบบนี้:

console.log("รับออเดอร์ A");   // ทำก่อน
console.log("รับออเดอร์ B");   // ทำต่อ
console.log("รับออเดอร์ C");   // ทำสุดท้าย
2

ปัญหาเกิดเมื่อมี "งานที่ต้องรอนาน"

สมมติออเดอร์ B คือ "ขอข้อมูลจาก Server" ซึ่งใช้เวลา 3 วินาที (ข้อมูลต้องวิ่งข้ามอินเทอร์เน็ตไปกลับ) ถ้าบาริสต้า ยืนเฉย ๆ รอ 3 วินาที ลูกค้าคนอื่นทั้งร้านต้องรอหมด — บนหน้าเว็บแปลว่า ทั้งหน้าค้าง กดปุ่มไม่ได้ scroll ไม่ได้ พิมพ์ไม่ได้

นี่คือสิ่งที่เรียกว่า Blocking — งานเดียวที่ช้า ทำให้ทุกอย่างหยุดหมด เว็บที่ค้างแบบนี้ผู้ใช้ปิดทิ้งภายใน 3 วินาที
3

ทางแก้: Asynchronous = "สั่งงานทิ้งไว้ แล้วไปทำอย่างอื่นก่อน"

บาริสต้าที่ฉลาดจะ กดเครื่องชงอัตโนมัติทิ้งไว้ แล้วหันไปรับลูกค้าคนถัดไปทันที พอเครื่องชงเสร็จมีเสียง "ติ๊ง!" ค่อยกลับมาเสิร์ฟ — นี่คือ Async:

console.log("1. กดเครื่องชงกาแฟ (สั่งงานทิ้งไว้)");

setTimeout(() => {
  console.log("3. ติ๊ง! กาแฟเสร็จ กลับมาเสิร์ฟ");
}, 3000); // งานนี้ใช้เวลา 3 วิ แต่ไม่บล็อกใคร

console.log("2. รับลูกค้าคนถัดไปทันที ไม่ต้องรอ");

// ลำดับ output จริง: 1 → 2 → (รอ 3 วิ) → 3
ตัวอย่างในชีวิตจริงที่คุณเจอทุกวัน: Facebook โหลด feed เพิ่มขณะที่คุณยัง scroll ได้ · YouTube โหลดวิดีโอขณะที่คุณพิมพ์ comment ได้ · เว็บ welovefukushima ของคุณตอน AJAX filter ทำงาน ผู้ใช้ก็ยังเลื่อนหน้าได้ — ทั้งหมดคือ async

ส่วนที่ 2: วิวัฒนาการ 3 ยุค (เล่าได้ = สอบผ่าน interview)

คำถามยอดฮิตใน interview คือ "Callback, Promise, Async/Await ต่างกันยังไง" วิธีตอบที่ดีที่สุดคือเล่าเป็น "ปัญหา → ทางแก้" ทีละยุค:

ยุคที่ 1: Callback — ฝากเบอร์ไว้ เสร็จแล้วโทรกลับ

Callback คือ "ฟังก์ชันที่ฝากไว้ให้เรียกทีหลังเมื่องานเสร็จ" เหมือนสั่งของแล้วฝากเบอร์โทรไว้ ฟังดูดี แต่พอต้องทำงานต่อกันหลายขั้น (login → ดึงโปรไฟล์ → ดึงออเดอร์) โค้ดจะซ้อนลึกลงเรื่อย ๆ:

// 😱 Callback Hell — ซ้อน 3 ชั้น ดัก error 3 รอบ
loginUser("vee", "1234", (err, user) => {
  if (err) return console.error(err);          // ดักชั้น 1
  getProfile(user.id, (err, profile) => {
    if (err) return console.error(err);        // ดักชั้น 2
    getOrders(profile.id, (err, orders) => {
      if (err) return console.error(err);      // ดักชั้น 3
      console.log("ออเดอร์:", orders);
    });
  });
});

โค้ดรูปนี้มีชื่อเล่นว่า "Pyramid of Doom" — อ่านยาก แก้ยาก เพิ่มขั้นตอนใหม่ทีต้องนับปีกกาให้ดี

ยุคที่ 2: Promise — ใบรับของที่สัญญาว่า "เดี๋ยวได้แน่"

Promise คือ object ที่แทน "งานที่กำลังทำอยู่" มี 3 สถานะ:

สถานะความหมายเทียบกับร้านกาแฟ
pendingกำลังทำงานกาแฟกำลังชง
fulfilledสำเร็จ (ได้ค่า)กาแฟเสร็จ มารับได้
rejectedล้มเหลว (ได้ error)เมล็ดหมด ขอโทษครับ

จุดเปลี่ยนสำคัญ: จากซ้อนลึก → กลายเป็น ต่อคิวแบน ๆ ด้วย .then() และดัก error จุดเดียว ด้วย .catch():

loginUser("vee", "1234")
  .then(user    => getProfile(user.id))   // เสร็จแล้วส่งต่อ
  .then(profile => getOrders(profile.id)) // เสร็จแล้วส่งต่อ
  .then(orders  => console.log(orders))
  .catch(err    => console.error(err));   // พังชั้นไหนก็ตกมาที่นี่ที่เดียว

ยุคที่ 3: Async/Await — เขียนโค้ดรอ ให้หน้าตาเหมือนโค้ดปกติ

คือ "น้ำตาลเคลือบ Promise" (syntactic sugar) — ข้างใต้ยังเป็น Promise ทุกอย่าง แต่เขียนอ่านง่ายเหมือนโค้ดบนลงล่างธรรมดา:

const run = async () => {           // 1. ประกาศว่าฟังก์ชันนี้มีงานต้องรอ
  try {
    const user    = await loginUser("vee", "1234"); // 2. รอตรงนี้ (ไม่บล็อกคนอื่น)
    const profile = await getProfile(user.id);
    const orders  = await getOrders(profile.id);
    console.log(orders);
  } catch (err) {                   // 3. พังบรรทัดไหนก็กระโดดมานี่
    console.error(err);
  }
};
กฎทอง 2 ข้อ (ท่องไว้):
① คำสั่งไหนต้องรอ → ใส่ await หน้า และฟังก์ชันที่ครอบมัน ต้องมี async
② ครอบด้วย try { } catch (err) { } เพื่อดัก error รวบยอดกล่องเดียว

ส่วนที่ 3: กับดัก 3 อันที่มือใหม่ตกทุกคน

กับดัก 1: ลืม await แล้วได้ Promise {<pending>}

const data = getData();        // ❌ ลืม await
console.log(data);             // Promise {<pending>} — ไม่ใช่ข้อมูล!

const data = await getData();  // ✅ ได้ข้อมูลจริง

เมื่อไหร่ที่ console.log แล้วเจอ Promise {<pending>} ให้รู้ทันที: ลืม await ที่ไหนสักแห่ง

กับดัก 2: ฟังก์ชัน async คืนค่าเป็น Promise เสมอ

const getNumber = async () => 42;   // ดูเหมือนคืนเลข...

console.log(getNumber());           // ❌ Promise {42} ไม่ใช่ 42
console.log(await getNumber());     // ✅ 42

แปลว่า "คนเรียกฟังก์ชัน async ก็ต้อง await ด้วย" — ลูกโซ่นี้คือเหตุผลที่ใน React เราถึงเรียก async function ใน useEffect แบบพิเศษ (จะเจอใน Week 8)

กับดัก 3: await เรียงกันโดยไม่จำเป็น = ช้าฟรี

// ❌ ช้า: รองาน A เสร็จก่อน ค่อยเริ่ม B (รวม 4 วิ)
const a = await fetchA();  // 2 วิ
const b = await fetchB();  // 2 วิ — ทั้งที่ B ไม่ได้ใช้ผลของ A เลย!

// ✅ เร็ว: ยิงพร้อมกัน รอพร้อมกัน (รวม 2 วิ)
const [a, b] = await Promise.all([fetchA(), fetchB()]);

หลักคิด: ถ้างานไม่ขึ้นต่อกัน → Promise.all / ถ้างานต้องใช้ผลของกันและกัน → await เรียงตามลำดับ

ส่วนที่ 3.5: Event Loop — เครื่องยนต์ที่อยู่เบื้องหลัง (เฉลย Mission 1.2)

คำถาม interview ระดับลึก: "JS thread เดียว แล้วจัดการ async ยังไง?" คำตอบคือ Event Loop มันคือเหตุผลว่าทำไม Mission 1.2 ลำดับถึงไม่เป็น A→B→C→D→E ตรง ๆ

กลไก 3 ขั้น:

  1. โค้ด sync รันก่อนทั้งหมด ใน Call Stack — Mission 1.2: A, C (ใน f ก่อนถึง await), E
  2. await null ทำให้ที่เหลือ (console.log D) ถูกส่งเข้า Microtask Queue
  3. setTimeout B เข้า Macrotask Queue — แม้ตั้ง 0ms ก็ต้องรอ microtask หมดก่อน

เมื่อ Call Stack ว่าง Event Loop หยิบ microtask จนหมดก่อน (D) แล้วค่อยหยิบ macrotask (B) → ผลคือ A → C → E → D → B

กฎจำง่ายสำหรับ interview: "sync → Promise → setTimeout" — Promise (microtask) เสมอชนะ setTimeout (macrotask) แม้ setTimeout จะตั้ง 0ms ก็ตาม นี่คือคำถามที่กลุ่ม 100k+ ถามจริง

ส่วนที่ 3.6: Promise combinators — เครื่องมือจัดการหลายงานพร้อมกัน

นอกจาก Promise.all ยังมีอีก 3 ตัวที่ production ใช้จริง — เลือกตามสถานการณ์:

ตัวพฤติกรรมใช้เมื่อ
Promise.allรอครบทุกตัว · 1 ตัวพัง= พังหมดต้องการครบทุกอันถึงทำงานได้
Promise.allSettledรอครบทุกตัว · พังบางตัวได้ คืนผลแยกสำเร็จ/ล้มเหลว⭐ report ที่ "บางแหล่งล่มก็ยังทำต่อ"
Promise.raceใครเสร็จก่อน (สำเร็จหรือพัง) เอาคนนั้นtimeout — แข่งกับนาฬิกา
Promise.anyใครสำเร็จก่อนเอาคนนั้น (พังข้าม)มีหลาย server เอาที่ตอบเร็วสุด
// 🏢 ปรับ automation report ให้ทนทานขึ้น (production จริง):
// ถ้าใช้ Promise.all แล้ว LINE API ล่ม → report พังทั้งใบ ❌
// ใช้ allSettled → แหล่งที่ล่มข้ามไป แหล่งอื่นยังได้ ✅
const results = await Promise.allSettled([
  fetchMetaAds(id), fetchLineOA(id), fetchInstagram(id), fetchGA4(id),
]);
const ok = results.filter(r => r.status === "fulfilled").map(r => r.value);
const failed = results.filter(r => r.status === "rejected");
if (failed.length) notifyTeam(`${failed.length} แหล่งล่ม แต่ report ยังออก`);
หลักคิด production: ระบบจริงต้อง "ทนต่อความพังบางส่วน" (partial failure) — report การตลาดที่ขาด LINE ไป 1 แหล่งยังมีค่ากว่า report ที่พังทั้งใบ นี่คือความต่างของโค้ด tutorial (Promise.all) กับโค้ด production (allSettled)

ส่วนที่ 3.7: Error Handling เชิงลึก — try / catch / finally

const loadData = async () => {
  showLoading();
  try {
    const data = await fetchData();
    render(data);
  } catch (err) {
    showError(err.message);     // จัดการ error ที่ผู้ใช้เข้าใจ
    logToServer(err);           // ส่ง error จริงให้ทีม (ไม่โชว์ผู้ใช้)
  } finally {
    hideLoading();              // ⭐ ทำเสมอ ไม่ว่าสำเร็จหรือพัง
  }
};

ทำไมต้อง finally: ถ้าปิด loading ไว้ใน try อย่างเดียว เวลา error จะลืมปิด → ผู้ใช้เห็น "กำลังโหลด..." ค้างตลอดกาล finally รับประกันว่างาน "เก็บกวาด" (ปิด loading, ปิด connection, ปลด lock) ทำทุกกรณี

กับดักที่ทำระบบล่มเงียบ: async function ที่ไม่มี try/catch และไม่มีใคร await ถ้าพัง = Unhandled Promise Rejection — error หายไปเงียบ ๆ ไม่มี log ใน Node.js (Week 2+) เคสนี้ทำ process crash ได้ → ทุก async ที่อาจพัง ต้องมีคนจัดการ error เสมอ

ส่วนที่ 4: ตัวอย่างการประยุกต์ในงานจริงของคุณ

🏢 กรณีที่ 1: งาน Automation Report ที่ media-presto

ตอนนี้คุณดึงข้อมูลการตลาดจากหลายแหล่ง (Meta, LINE, IG, GA4) — ทั้ง 4 แหล่ง ไม่ขึ้นต่อกัน ดังนั้นท่าที่ถูกคือ:

const buildReport = async (clientId) => {
  try {
    // ยิงพร้อมกัน 4 แหล่ง — เร็วกว่าเรียงกัน ~4 เท่า
    const [meta, line, ig, ga4] = await Promise.all([
      fetchMetaAds(clientId),
      fetchLineOA(clientId),
      fetchInstagram(clientId),
      fetchGA4(clientId),
    ]);
    return summarize({ meta, line, ig, ga4 });
  } catch (err) {
    notifyTeam(`Report ${clientId} ล้มเหลว: ${err.message}`);
  }
};

นี่คือโค้ดที่คุณจะเขียนจริงใน Week 16 ตอน rebuild automation report เป็น TypeScript service

📚 กรณีที่ 2: ระบบยืมหนังสือใน library-system (Kcode Day 5)

การยืมหนังสือ ขึ้นต่อกันเป็นลูกโซ่ — ต้องเช็คสมาชิกก่อน แล้วค่อยเช็คหนังสือ แล้วค่อยบันทึก ดังนั้นต้อง await เรียงลำดับ (ใช้ Promise.all ไม่ได้):

const borrowBook = async (memberId, bookId) => {
  const member = await findMember(memberId);      // ขั้น 1
  if (member.fines > 0) throw new Error("มีค่าปรับค้าง");
  const book = await findBook(bookId);            // ขั้น 2 (ใช้หลังรู้ว่าสมาชิกผ่าน)
  if (!book.available) throw new Error("หนังสือถูกยืมแล้ว");
  return await createTransaction(member, book);   // ขั้น 3
};

สังเกต: ตัวอย่างสองกรณีนี้คือคำตอบของ Quiz ข้อ 3 ในชีวิตจริง — รู้ว่าเมื่อไหร่ใช้ท่าไหน

🧹 Clean Code: เขียน async แบบ production

1. async/await แทน .then chain

❌ .then ซ้อน

function load() {
  return fetchUser()
    .then(u => fetchPosts(u.id))
    .then(posts => render(posts))
    .catch(e => showError(e));
}

✅ เรียงตรง อ่านเป็นเรื่อง

async function load() {
  try {
    const u = await fetchUser();
    const posts = await fetchPosts(u.id);
    render(posts);
  } catch (e) { showError(e); }
}

2. ขนานเมื่อทำได้ — อย่า await เรียงโดยไม่จำเป็น

❌ ช้า 2 เท่า (รอทีละตัว)

const user = await getUser();
const config = await getConfig();
// 2 อันนี้ไม่เกี่ยวกันเลย!

✅ ยิงพร้อมกัน

const [user, config] = await Promise.all([
  getUser(), getConfig()
]);

3. throw Error object ไม่ใช่ string

❌ throw string

if (!ok) throw "something failed";
// catch แล้วไม่มี stack trace, .message

✅ throw Error

if (!ok) throw new Error("something failed");
// ได้ err.message, err.stack ครบ
กฎทอง: async ที่อาจพังต้องมีคนจัดการ error เสมอ · ขนานเมื่องานไม่ขึ้นต่อกัน · error message ที่โชว์ผู้ใช้ ≠ error จริงที่ log (อย่ารั่วรายละเอียดภายใน — มุม security)

ข้อ 1: ฟังก์ชัน async ที่ return 42 — คนเรียกจะได้อะไร และต้องเอาค่าออกมายังไง?

ได้ Promise ที่ resolve เป็น 42 ไม่ใช่เลข 42 ตรง ๆ ต้องใช้ await getNumber() หรือ getNumber().then(n => ...) เพื่อเอาค่าจริงออกมา — กฎคือ "async คืน Promise เสมอ ไม่ว่าจะ return อะไร"

ข้อ 2: โค้ดนี้ผิดตรงไหน: const data = fetch(url).json();

ผิด 2 จุด: ① fetch(url) คืน Promise — ต้อง await ก่อนถึงจะได้ response object ② .json() ก็คืน Promise เช่นกัน — ต้อง await อีกชั้น ที่ถูกคือ const res = await fetch(url); const data = await res.json();

ข้อ 3: await 3 ตัวที่ไม่ขึ้นต่อกันเรียงกัน ช้าเพราะอะไร แก้ยังไง?

ช้าเพราะแต่ละ await รอตัวก่อนหน้าเสร็จก่อน ทั้งที่งานไม่เกี่ยวกัน (เวลารวม = ผลบวกของทุกงาน) แก้ด้วย Promise.all([a, b, c]) ให้ทุกงานวิ่งพร้อมกัน (เวลารวม = งานที่ช้าที่สุดงานเดียว)

ข้อ 4 (Event Loop): console.log(1); Promise.resolve().then(()=>console.log(2)); setTimeout(()=>console.log(3),0); console.log(4); — ลำดับ?

1 → 4 → 2 → 3 · sync ก่อน (1, 4) → microtask/Promise (2) → macrotask/setTimeout (3) แม้ setTimeout ตั้ง 0ms ก็ยังทำหลัง Promise เสมอ เพราะ microtask queue เคลียร์หมดก่อน macrotask

ข้อ 5 (production): report ดึง 4 แหล่ง แต่ LINE API ล่มบ่อย — ใช้ Promise.all ดีไหม? ถ้าไม่ ใช้อะไร?

ไม่ดี — Promise.all ถ้า 1 ตัวพัง = report พังทั้งใบ ใช้ Promise.allSettled แทน มันรอครบทุกตัวแล้วคืนผลแยก fulfilled/rejected → แหล่งที่ล่มข้ามไป แหล่งอื่นยังได้ข้อมูล (ทนต่อ partial failure = หลัก production)

ข้อ 6: ทำไม async function ที่พังโดยไม่มีใคร catch ถึงอันตรายกว่าใน Node.js (Week 2)?

เป็น Unhandled Promise Rejection — บน browser แค่ขึ้น warning ใน console แต่ใน Node.js เคสนี้ทำ process crash ได้ (server ดับทั้งตัว) → ทุก async ฝั่ง server ต้องมี try/catch หรือ error handler เสมอ
