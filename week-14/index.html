Week 14 — CD: Build Image + Deploy

ปิดวงจร: push code → build image → deploy ขึ้น server อัตโนมัติ — ไม่ต้อง deploy มือ

🎯 เป้าหมายของสัปดาห์นี้

Week 13 ทำ CI (ตรวจโค้ด) — Week 14 ทำ CD (Continuous Deployment): หลัง test ผ่าน → build Docker image → push ขึ้น registry → deploy ขึ้น server อัตโนมัติ · push code เข้า main แล้วของขึ้น production เองภายในไม่กี่นาที · นี่คือทักษะที่แยก "เขียนโค้ดได้" จาก "ส่งของถึงผู้ใช้ได้เอง" — สิ่งที่ทำให้ resume คุณพูดว่า "deploy ได้ทั้ง pipeline"

  • CD concept: pipeline เต็ม (source→build→test→deploy) · delivery vs deployment
  • build & push Docker image ขึ้น registry (GHCR) ใน Actions
  • deploy ขึ้น server ผ่าน SSH (pull image + compose up)
  • secrets flow: SSH key, registry token (ปลอดภัย)
  • deployment strategy + rollback (กลับเวอร์ชันเก่าเมื่อพัง)
ทำไมสัปดาห์นี้คือ checkpoint เฟส 2: "push → deploy อัตโนมัติ" คือเป้าหมายของเฟส 2 ทั้งหมด · การ deploy ที่ปลอดภัย (จัดการ SSH key, secret, least privilege) คือจุดที่ pentester ของคุณได้เปรียบ

🧹 Clean Code ประจำ Week 14

1. Tag image ด้วย commit SHA ไม่ใช่แค่ latest

❌ latest อย่างเดียว (rollback ยาก)

tags: myapp:latest
# deploy พัง — ไม่รู้ version ก่อนหน้าคืออะไร
# rollback ลำบาก

✅ tag ด้วย SHA + latest

tags: |
  myapp:latest
  myapp:${{ github.sha }}
# พัง → deploy SHA เก่า = rollback ได้ทันที

2. Deploy ต้อง rollback ได้เสมอ

เก็บ image หลาย version (tag SHA) → deploy พัง pull version เก่ากลับ · ไม่มีทาง rollback = deploy แล้วภาวนา (อันตราย)

กฎทอง: tag image ด้วย SHA (rollback ได้) · deploy หลัง test เขียวเท่านั้น · SSH key ใน secrets (ไม่ commit) · deploy user สิทธิ์จำกัด (ไม่ใช่ root) · มี rollback plan ก่อน deploy ทุกครั้ง
