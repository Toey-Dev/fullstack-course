🚀 CI/CD เต็มวงจร — Deploy library-system

push code เข้า main → ของขึ้น production เอง — checkpoint สำคัญของเฟส 2

ทำไม Mission นี้คือ checkpoint เฟส 2

หลังจบ library-system จะมี pipeline เต็ม: push → test → build image → push registry → deploy ขึ้น server เอง · "push → deploy อัตโนมัติ" คือเป้าหมายของเฟส 2 ทั้งหมด · นี่คือสิ่งที่ทำให้คุณพูดใน interview ได้ว่า "ทำ CI/CD เป็นทั้งวงจร" — ทักษะที่บริษัท 100k+ ถือเป็นพื้นฐาน

ต้องมี server: ใช้ VPS ราคาถูก (DigitalOcean/Linode ~$5-6) หรือ free tier (Oracle Cloud, fly.io) — สำหรับฝึก deploy จริง

🪜 แบ่งเป็น 4 Mission ย่อย

Mission 14.1 — Build & Push ขึ้น GHCR 75 นาที

  1. เพิ่ม job build-and-push (needs: test) ใน workflow
  2. login GHCR + build-push-action · tag latest + ${{ github.sha }}
  3. permissions: packages: write (least privilege)
✅ เทสต์: push → image โผล่ใน GHCR (Packages ของ repo) มี 2 tag (latest + sha)

Mission 14.2 — เตรียม server + SSH deploy 75 นาที

  1. server: ติดตั้ง Docker, สร้าง deploy user (ไม่ใช่ root), วาง compose.prod + .env
  2. สร้าง SSH key pair → public บน server, private ใน GitHub Secrets
  3. job deploy (needs: build-and-push, if: main) — SSH pull + compose up
security: deploy user สิทธิ์จำกัด · DB port ปิดจากภายนอก (Week 12) · SSH key ใน secrets เท่านั้น

Mission 14.3 — Rollback + deploy จริง 60 นาที

  1. compose ใช้ env IMAGE_TAG (rollback ง่าย)
  2. push เข้า main → ดู pipeline ทำงานครบ → เปิด URL production เห็นของจริง
  3. ซ้อม rollback: deploy SHA เก่า → ระบบกลับเวอร์ชันก่อนหน้า
✅ เทสต์: push code → ~ไม่กี่นาที → เปิด production URL เห็นการเปลี่ยนแปลง · rollback กลับได้

Mission 14.4 — Document + push 45 นาที

  1. README: diagram CI/CD flow + URL production + วิธี rollback
  2. เพิ่ม CI/CD badge · DEPLOYMENT.md (ขั้นตอน + rollback plan)
  3. push + AI Usage
✅ เทสต์: คนอ่าน README เข้าใจ flow + รู้วิธี rollback — เหมือน runbook ทีมจริง
