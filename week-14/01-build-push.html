Build & Push Image ขึ้น Registry

แปลงโค้ดเป็น image แล้วส่งขึ้น "คลังกลาง" ที่ server ดึงไป deploy ได้

🔑 คีย์เวิร์ดภาพจำประจำบท

📦 คลังเก็บ image = registry (GHCR) 🏷 tag = ป้ายเวอร์ชัน (SHA + latest) 🤖 build บน CI = build-push-action 🚀 CD = ส่งของถึงผู้ใช้อัตโนมัติ

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Registry = คลังกลางที่ทุกที่ดึง image เดียวกัน"

build image ในเครื่อง dev แล้ว server เอาไปรันไม่ได้ตรง ๆ — ต้องผ่าน "คลังกลาง" (registry) · CI build แล้ว push ขึ้น registry, server pull จาก registry · ทุกที่ได้ image เดียวกันเป๊ะ (เครื่อง dev, CI, production รัน image ตัวเดียวกัน) · นี่คือ "build once, run anywhere"

หลักการ 2: "Build ที่เดียว (CI) ไม่ใช่ที่ server"

build บน server = server ต้องมี source + build tools + เสี่ยง build ต่างจาก dev · build บน CI แล้วส่ง image สำเร็จไป server server แค่ "รัน" image (ไม่ต้อง build) → เร็ว, สะอาด, image เดียวกันที่ test แล้ว · separation: CI สร้าง, server รัน

หลักการ 3: "Tag คือประวัติศาสตร์ — ต้อง rollback ได้"

tag image ด้วย commit SHA (ไม่ใช่แค่ latest) · ทุก deploy มี image เฉพาะของ commit นั้น · deploy พัง → ดึง image SHA ก่อนหน้ากลับ = rollback ทันที · latest อย่างเดียว = ทับของเก่า ไม่รู้จะกลับไปไหน · tag = ความสามารถในการย้อนเวลา

ส่วนที่ 1: CD Pipeline ภาพรวม

Delivery vs Deployment: Continuous Delivery = พร้อม deploy (มีคนกดอนุมัติ) · Continuous Deployment = deploy อัตโนมัติเลย · เราทำแบบ deploy main อัตโนมัติ (แต่ใส่ approval gate ได้)

ส่วนที่ 2: Registry — GHCR

GHCR (GitHub Container Registry, ghcr.io) — registry ที่ผูกกับ GitHub (ฟรี, ใช้ token ของ Actions ได้เลย):

# job build-push ใน workflow (ต่อจาก test ที่ผ่าน)
build-and-push:
  needs: test                    # ⭐ รันต่อเมื่อ test ผ่าน (Week 13)
  runs-on: ubuntu-latest
  permissions:
    contents: read
    packages: write              # สิทธิ์ push ไป GHCR (least privilege)
  steps:
    - uses: actions/checkout@v4
    - uses: docker/login-action@v3      # login GHCR
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}   # token อัตโนมัติของ Actions
    - uses: docker/build-push-action@v6
      with:
        push: true
        tags: |
          ghcr.io/${{ github.repository }}:latest
          ghcr.io/${{ github.repository }}:${{ github.sha }}   # ⭐ tag SHA
ทำไม GHCR: ผูกกับ repo, ใช้ secrets.GITHUB_TOKEN (สร้างให้อัตโนมัติทุก workflow ไม่ต้องตั้งเอง), ฟรีสำหรับ public · Docker Hub ก็ได้แต่ต้องตั้ง token เอง · permissions: packages: write = ให้สิทธิ์เท่าที่จำเป็น (least privilege)

ส่วนที่ 3: Image Tagging Strategy

tagใช้ทำอะไร
:latestเวอร์ชันล่าสุด (สะดวก แต่ทับของเก่า)
:${{ github.sha }}⭐ เวอร์ชันเฉพาะ commit — rollback ได้
:v1.2.3release version (ถ้าใช้ semantic versioning)

production deploy ควรชี้ SHA หรือ version ที่แน่นอน ไม่ใช่ latest — เพราะ latest เปลี่ยนได้ตลอด (deploy ไม่ deterministic, rollback ยาก)

ส่วนที่ 4: ตัวอย่างประยุกต์

🏢 งานวี: automation service — push code → CI build image → ขึ้น GHCR · ทีม/ลูกค้า pull image เดียวกันไปรันได้ทุกที่ · tag SHA ทำให้รู้ว่า production รัน commit ไหน (debug ง่าย: "เวอร์ชันที่พังคือ SHA นี้")

🌏 อุตสาหกรรม: build-once-push-to-registry-deploy-anywhere เป็นมาตรฐานทุกที่ · GHCR/ECR/Docker Hub เป็น registry หลัก · tag ด้วย SHA/version เป็น best practice สำหรับ rollback · least-privilege token (packages: write เท่านั้น) เป็นสิ่งที่ security ตรวจ

📝 Quiz ท้ายบท

ข้อ 1: ทำไมต้อง push image ขึ้น registry ไม่ build บน server ตรง ๆ?

registry = คลังกลาง ทุกที่ pull image เดียวกัน (dev/CI/prod รันตัวเดียวกันเป๊ะ) · build บน server = server ต้องมี source+build tools, เสี่ยง build ต่างจากที่ test · build once บน CI → server แค่รัน = เร็ว สะอาด deterministic

ข้อ 2: ทำไม tag SHA สำคัญกว่า latest อย่างเดียว?

SHA = image เฉพาะของแต่ละ commit → deploy พัง ดึง SHA ก่อนหน้ากลับ = rollback ทันที + รู้ว่า production รัน commit ไหน · latest ทับของเก่า ไม่รู้จะกลับไปไหน · production ควรชี้ SHA/version แน่นอน ไม่ใช่ latest

ข้อ 3: needs: test ใน build job ทำอะไร?

บอกว่า build-push รันต่อเมื่อ job test ผ่าน — ไม่ build/deploy โค้ดที่ test แดง · เป็น gate: test เขียว → ค่อย build → ค่อย deploy · กันของพังขึ้น production

ข้อ 4 (security): permissions: packages: write สะท้อนหลักอะไร?

Least privilege — ให้ token สิทธิ์เท่าที่จำเป็น (push package) ไม่ให้สิทธิ์เต็ม · ถ้า workflow โดน compromise ความเสียหายจำกัด · หลักเดียวกับ auth role (Week 5) แต่ระดับ CI token

✅ Checkpoint

  • เข้าใจ CD pipeline + registry (build once, run anywhere)
  • build & push image ขึ้น GHCR ใน Actions
  • tag SHA + latest (rollback ได้)
  • needs: test (deploy เฉพาะที่ผ่าน) + least-privilege permissions
