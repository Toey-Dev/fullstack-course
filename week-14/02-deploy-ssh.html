Deploy ผ่าน SSH + Strategy + Rollback

ส่ง image ขึ้น server จริง — อย่างปลอดภัยและกลับเวอร์ชันได้เมื่อพัง

🔑 คีย์เวิร์ดภาพจำประจำบท

🔑 SSH key ใน secrets ไม่ commit 📥 pull image + compose up = deploy ↩️ พัง → ดึง SHA เก่า = rollback 👤 deploy user ≠ root

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Server แค่ pull + run — ไม่ build, ไม่มี source"

deploy = SSH เข้า server → docker compose pull (ดึง image ใหม่จาก registry) → up -d (รันแทนของเก่า) · server ไม่ต้องมี source code/build tools — แค่ Docker + compose file · image ที่ test แล้วใน CI = ตัวเดียวกับที่รัน production

หลักการ 2: "Deploy ต้องกลับได้ — ไม่มี rollback = ห้าม deploy"

ทุก deploy ต้องตอบได้ว่า "ถ้าพังจะกลับยังไง" · เพราะ tag image ด้วย SHA (บทที่ 1) → rollback = compose up ด้วย SHA เก่า · deploy โดยไม่มีแผน rollback = เล่นการพนันกับ production · มืออาชีพเตรียมทางถอยก่อนเดินหน้าเสมอ

หลักการ 3: "Credential ของ deploy คือกุญแจบ้าน — ป้องกันสุดชีวิต"

SSH key ที่ deploy เข้า server = ถ้าหลุด ใครก็เข้า server ได้ · เก็บใน GitHub Secrets (ไม่ commit), ใช้ deploy user สิทธิ์จำกัด (ไม่ใช่ root), verify host key · มุม pentester: deploy pipeline คือเป้าหมายโจมตียอดนิยม (เข้าถึง production ได้)

ส่วนที่ 1: Deploy Job ผ่าน SSH

deploy:
  needs: build-and-push           # ต่อจาก image ขึ้น registry แล้ว
  runs-on: ubuntu-latest
  if: github.ref == 'refs/heads/main'    # ⭐ deploy เฉพาะ main เท่านั้น
  steps:
    - uses: appleboy/ssh-action@v1
      with:
        host: ${{ secrets.DEPLOY_HOST }}
        username: ${{ secrets.DEPLOY_USER }}      # user สิทธิ์จำกัด ไม่ใช่ root
        key: ${{ secrets.DEPLOY_SSH_KEY }}        # private key ใน secrets
        script: |
          cd /opt/library-system
          echo ${{ secrets.GITHUB_TOKEN }} | docker login ghcr.io -u ${{ github.actor }} --password-stdin
          docker compose pull                      # ดึง image ใหม่
          docker compose up -d                     # รันแทนของเก่า
          docker image prune -f                    # ลบ image เก่าที่ไม่ใช้
🔐 จุด security 4 ข้อ: ① SSH key ใน secrets (ไม่ commit) ② if: ref == main deploy เฉพาะ main (ไม่ใช่ทุก branch/PR) ③ deploy user สิทธิ์จำกัด (ไม่ใช่ root) ④ login registry ด้วย token ชั่วคราว · คุณในฐานะ pentester รู้ว่า deploy pipeline ที่หละหลวม = ทางเข้า production

ส่วนที่ 2: เตรียม Server (ครั้งเดียว)

# บน server (setup ครั้งแรก)
# 1. ติดตั้ง Docker + compose
# 2. สร้าง deploy user (ไม่ใช่ root) + เพิ่มเข้า docker group
sudo adduser deploy && sudo usermod -aG docker deploy
# 3. ใส่ public key ของ deploy ใน ~/.ssh/authorized_keys
# 4. วาง compose.prod.yml + .env (secret จริง) ที่ /opt/library-system
# 5. เปิด firewall เฉพาะ port ที่ใช้ (80/443) ปิด DB port

server มีแค่ compose file + .env — ไม่มี source code · image มาจาก registry · นี่คือ "immutable infrastructure" เบื้องต้น (server เป็นแค่ที่รัน image)

ส่วนที่ 3: Deployment Strategy

strategyทำงานdowntime
Recreateหยุดเก่า → start ใหม่มี (วินาที-นาที) — ง่ายสุด
Rollingค่อย ๆ เปลี่ยนทีละ instanceเกือบ 0 (ต้องมีหลาย instance)
Blue-Greenรันใหม่ข้าง ๆ → สลับ traffic0 + rollback ทันที (สลับกลับ)

เริ่มจาก Recreate (compose up -d ก็คือ recreate — ง่าย พอสำหรับเริ่มต้น) · zero-downtime (rolling/blue-green) ทำเมื่อ traffic เยอะจริง — อย่า over-engineer ตั้งแต่แรก

ส่วนที่ 4: Rollback

# deploy พัง → rollback ด้วย SHA เก่า (เพราะ tag SHA จากบทที่ 1)
# บน server:
docker compose down
# แก้ image tag ใน compose เป็น SHA ก่อนหน้า (หรือ env IMAGE_TAG)
IMAGE_TAG=<previous-sha> docker compose up -d
# ระบบกลับเวอร์ชันที่ทำงานได้ภายในไม่กี่วินาที
ทำให้ rollback ง่าย: ใช้ env IMAGE_TAG ใน compose (image: ghcr.io/...:${IMAGE_TAG}) · rollback = เปลี่ยน IMAGE_TAG แล้ว up -d · เก็บ image หลาย version ใน registry · ซ้อม rollback ก่อนต้องใช้จริง (ตอนวิกฤตไม่ใช่เวลาเรียน)

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: deploy automation service ขึ้น VPS — push main → CI build → SSH deploy อัตโนมัติ · rollback ได้ทันทีถ้า logic สรุปพัง · deploy user สิทธิ์จำกัด + DB port ปิด = ปลอดภัยพอ production จริง

🌏 อุตสาหกรรม: SSH deploy เหมาะกับ VPS/single server (startup, project เล็ก-กลาง) · scale ใหญ่ใช้ K8s/managed (rolling อัตโนมัติ) · rollback strategy + secret management ของ deploy เป็นสิ่งที่ทุกทีมต้องมี · pentester ที่เข้าใจ "deploy pipeline security" หายากและมีค่าในสาย DevSecOps

📝 Quiz ท้ายบท

ข้อ 1: ตอน deploy ทำไม server แค่ pull + up ไม่ build เอง?

image ถูก build + test บน CI แล้ว (ตัวเดียวกับที่จะรัน) · server แค่ pull จาก registry + รัน → เร็ว, สะอาด, ไม่ต้องมี source/build tools, deterministic (รัน image ที่ test แล้ว) · separation: CI สร้าง, server รัน

ข้อ 2: ทำไมทุก deploy ต้องมี rollback plan?

deploy อาจพัง (bug ที่ test ไม่จับ, config ผิด) · ต้องกลับเวอร์ชันที่ทำงานได้เร็ว · เพราะ tag SHA → rollback = up -d ด้วย SHA เก่า · ไม่มี rollback = deploy แล้วภาวนา = เล่นพนันกับ production

ข้อ 3 (security): จุด security ของ SSH deploy 4 ข้อคืออะไร?

① SSH key ใน secrets ไม่ commit ② deploy เฉพาะ main (if: ref==main) ไม่ใช่ทุก branch ③ deploy user สิทธิ์จำกัด ไม่ใช่ root ④ ปิด port ที่ไม่ใช้ (DB) + verify host · deploy pipeline = ทางเข้า production ต้องป้องกันสุด

ข้อ 4: Recreate vs Blue-Green ต่างกันยังไง ควรเริ่มอันไหน?

Recreate: หยุดเก่า→start ใหม่ (มี downtime สั้น, ง่ายสุด) · Blue-Green: รันใหม่ข้าง ๆ→สลับ traffic (0 downtime + rollback ทันที แต่ซับซ้อน) · เริ่มจาก Recreate (compose up -d) — พอสำหรับเริ่มต้น · blue-green เมื่อ traffic เยอะจริง อย่า over-engineer

✅ Checkpoint

  • deploy job ผ่าน SSH (pull + compose up) · deploy เฉพาะ main
  • SSH key/secret ใน GitHub Secrets · deploy user ไม่ใช่ root
  • เข้าใจ deployment strategy (เริ่ม recreate)
  • rollback ด้วย SHA tag (ซ้อมก่อนใช้จริง)
