Week 2 · บทที่ 1

Node.js + Express จากศูนย์ถึง endpoint แรก

พา JavaScript ออกจาก browser มารันบน server — และเข้าใจว่าทำไมโลกถึงเลือกทางนี้

🔑 คีย์เวิร์ดภาพจำประจำบท

🏃 JS หนีออกจาก browser = Node.js ☕ บาริสต้าคนเดิม ร้านใหญ่ขึ้น = Event Loop บน server 📋 สมุดสูตรของโปรเจกต์ = package.json 🛣 ป้ายบอกทาง = Route 🧰 ชุดเครื่องมือสำเร็จรูป = Express ❤️ ชีพจรของ server = /api/health

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Same language, both sides" — ภาษาเดียวทั้งสองฝั่ง

ก่อนปี 2009 frontend เขียน JS, backend ต้องเขียน PHP/Java/Python — สมองสลับภาษาตลอดวัน Node.js เกิดจากคำถามเดียว: "ถ้า JS รันนอก browser ได้ล่ะ?" ผลคือ dev คนเดียวเขียนได้ทั้งระบบด้วยภาษาเดียว — นี่คือเหตุผลที่ "Full Stack JS" กลายเป็นตำแหน่งที่ตลาดจ้างมากที่สุด และคือเส้นทางที่คุณกำลังเดิน

หลักการ 2: "Don't block" ภาคต่อ — เหตุผลที่ Node รับพันคนด้วย thread เดียว

จำบาริสต้าคนเดียวจาก Week 1 ได้ไหม? บน server หลักการเดิมกลายเป็นพลังมหาศาล: เมื่อ request มาขอข้อมูล database (งานรอ) Node ไม่ยืนเฝ้า — มันหันไปรับ request ถัดไปทันที เสร็จเมื่อไหร่ค่อยวนกลับมาตอบ ดังนั้น thread เดียวจึงรับผู้ใช้พร้อมกันเป็นพันได้ ตราบใดที่ไม่มีใครเขียนโค้ด blocking ขวางไว้ — ประโยคหลังคือหน้าที่ของคุณตลอดอาชีพ

หลักการ 3: "Convention over reinvention" — ใช้ของที่โลกตกลงกันแล้ว

Node มี http module ในตัว สร้าง server ได้เปล่า ๆ แต่ต้องเขียน routing, แกะ body, จัดการ error เองหมด Express คือ "ข้อตกลงร่วม" ที่คนทั้งวงการใช้ — เข้าทีมไหนก็อ่านโค้ดกันรู้เรื่อง Dev ที่ดีไม่ได้เก่งเพราะสร้างทุกอย่างเอง แต่เก่งเพราะรู้ว่าอะไรควรสร้าง อะไรควรใช้ของมาตรฐาน

เมื่อไหร่ใช้ / ไม่ใช้ Node: เหมาะกับ I/O หนัก (API, realtime, microservice) — ไม่เหมาะกับงานคำนวณหนักต่อเนื่อง (วิดีโอ, ML training) เพราะคำนวณหนัก = block บาริสต้า ทั้งร้านหยุด (คำถามนี้โผล่ใน interview บ่อยมาก)

ส่วนที่ 1: เริ่มโปรเจกต์อย่างถูกวิธี

1

สร้างบ้านของ library-system

mkdir library-system && cd library-system
npm init -y          # สร้าง package.json
npm install express  # ติดตั้ง Express
npm install -D nodemon  # -D = ใช้เฉพาะตอน dev

ทำไมแต่ละบรรทัด: package.json คือ "สมุดสูตร" ของโปรเจกต์ — บอกว่าใช้ library อะไรเวอร์ชันไหน เพื่อนร่วมทีม clone ไปแล้วพิมพ์ npm install ก็ได้ของครบเหมือนเรา / -D (devDependencies) สำคัญ: nodemon ใช้แค่ตอนพัฒนา (restart server อัตโนมัติเมื่อ save) production ไม่ต้องการมัน — นี่คือจุดที่ server เก่าใน D:\react ของคุณใส่ผิดช่อง ครั้งนี้ทำให้ถูก

2

ตั้ง script ใน package.json

"scripts": {
  "dev": "nodemon src/server.js",
  "start": "node src/server.js"
}

ทำไม: npm run dev ตอนพัฒนา (auto-restart) / npm start ตอน production — แยกสองโหมดตั้งแต่วันแรก เป็น convention ที่ทุกบริษัทใช้

ส่วนที่ 2: Express server แรก — เข้าใจทุกบรรทัด

// src/server.js
const express = require("express");  // 1
const app = express();               // 2

app.use(express.json());             // 3

app.get("/api/health", (req, res) => {          // 4
  res.json({ ok: true, time: new Date() });     // 5
});

const PORT = process.env.PORT || 3000;          // 6
app.listen(PORT, () => console.log(`Server on :${PORT}`));  // 7
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
1-2สร้าง "แอป" 1 ตัว — app คือศูนย์รวมทุก route/middleware ของระบบ
3สอนให้ Express อ่าน JSON จาก body — ถ้าไม่มีบรรทัดนี้ req.body จะเป็น undefined ตลอด (บั๊กอันดับ 1 ของมือใหม่ และ Express 5 มีในตัวแล้ว ไม่ต้องลง body-parser แยกแบบโค้ดเก่าของคุณ)
4Route = "ถ้ามีคน GET มาที่ path นี้ ให้รันฟังก์ชันนี้" — req คือซองคำขอ, res คือซองคำตอบ
5res.json() แปลง object → JSON + ตั้ง Content-Type ให้อัตโนมัติ (ดีกว่า res.send สำหรับ API)
6อ่าน PORT จาก environment ก่อน ค่อย fallback 3000 — เพราะ production (Docker/cloud) เป็นคนกำหนด port ไม่ใช่เรา hardcode (จะเจอจริง Week 11)
7เปิดร้านรอรับลูกค้า — callback ไว้ยืนยันว่าเปิดสำเร็จ
ทำไมต้องมี /api/health? ทุกระบบ production มี endpoint นี้ — load balancer, Docker, ระบบ monitor ใช้มัน "จับชีพจร" ว่า server ยังหายใจไหม ใส่ตั้งแต่วันแรก = ติดนิสัยคิดแบบ production ฟรี ๆ (Week 15 จะ monitor ตัวนี้จริง)

ส่วนที่ 2.5: กายวิภาคของ Request & Response (ต้องเห็นภาพนี้)

ทุก endpoint คือการรับ "ซองคำขอ" (req) แล้วส่ง "ซองคำตอบ" (res) กลับ — รู้จักส่วนประกอบของมันคือพื้นฐานที่ debug ทุกอย่างต้องใช้:

กับดักจริง "Cannot set headers after they are sent": เรียก res 2 ครั้งใน request เดียว (เช่นลืม return หลัง res.status(400)) — แก้ด้วยใส่ return ทุกครั้งที่ตอบ (เชื่อมกับ guard clause)

ส่วนที่ 2.6: Middleware — หัวใจของ Express (แนวคิดที่ต้องเข้าใจ)

Express ทั้งระบบคือ "ท่อส่งน้ำ" ที่ request ไหลผ่าน middleware ทีละชั้นก่อนถึงปลายทาง — เข้าใจอันนี้ = เข้าใจ Express 80%:

// middleware = ฟังก์ชันที่มี (req, res, next)
const logger = (req, res, next) => {
  console.log(`${req.method} ${req.url}`);   // ทำงานบางอย่าง
  next();          // ⭐ ส่งต่อชั้นถัดไป — ลืม next() = request ค้างตลอดกาล
};
app.use(logger);   // ใช้กับทุก request

// express.json() ที่เราใส่ ก็คือ middleware สำเร็จรูป — มันแกะ body ให้ก่อนถึง handler
ทำไมแนวคิดนี้ทรงพลัง: งานที่ต้องทำ "ทุก request" (log, ตรวจ auth, จำกัด rate, เพิ่ม CORS header) เขียนเป็น middleware ชั้นเดียว ใช้ได้ทั้งระบบ ไม่ต้อง copy ลงทุก route — JWT auth (Week 5) และ security headers (Week 17) ก็คือ middleware

📝 Quiz ท้ายบท

ข้อ 1: Node.js single thread แต่รับพันคนพร้อมกันได้ — อธิบายด้วยภาพร้านกาแฟ และบอกว่าโค้ดแบบไหนทำลายข้อดีนี้

บาริสต้าคนเดียว (thread) ไม่เคยยืนรอเครื่องชง (I/O) — สั่งงานทิ้งไว้แล้วรับลูกค้าถัดไป งาน I/O เสร็จค่อยวนกลับมาเสิร์ฟ สิ่งที่ทำลายมัน: โค้ด CPU หนัก ๆ ใน request (loop ล้านรอบ, เข้ารหัสไฟล์ใหญ่) = บาริสต้ามัวบดเมล็ดเอง ทั้งร้านหยุดรับออเดอร์

ข้อ 2: ทำไม PORT ต้องอ่านจาก process.env ก่อน ไม่ hardcode 3000 อย่างเดียว?

เพราะคนกำหนด port ตัวจริงใน production คือ environment (Docker, cloud, load balancer) ไม่ใช่โค้ดเรา — hardcode แล้ววันที่ deploy จะชนกับระบบเขา หลักการนี้เรียก config จาก environment (12-Factor App) จะเจอเต็ม ๆ ตอน Week 11-15

ข้อ 3 (security): /api/health ควรตอบอะไร และ "ไม่ควร" ตอบอะไร — คิดแบบ pentester

ควรตอบแค่ ok/เวลา/เวอร์ชันคร่าว ๆ — ไม่ควรตอบ: เวอร์ชัน dependency ละเอียด, ชื่อ database, path ภายใน, stack trace เพราะ health endpoint มักเปิด public และเป็นจุดแรกที่ผู้โจมตี (เช่นคุณในอดีต) ใช้ทำ recon เก็บข้อมูลระบบ

ข้อ 4 (middleware): logger middleware ที่ลืมเรียก next() จะเกิดอะไร?

request ค้างตลอดกาล (browser หมุนไม่จบ) เพราะ middleware ทำงานเสร็จแต่ไม่ส่งต่อชั้นถัดไป — request ไปไม่ถึง route handler ที่จะตอบ res ดังนั้นทุก middleware ที่ไม่ได้จบด้วย res ต้องเรียก next() เสมอ

ข้อ 5: error "Cannot set headers after they are sent" เกิดจากอะไร แก้ยังไง?

เรียก res 2 ครั้งใน request เดียว (res ตอบได้ครั้งเดียว) — มักเพราะลืม return หลัง res.status(400).json(...) โค้ดเลยไหลต่อไปเรียก res อีกครั้ง แก้ด้วยใส่ return ทุกครั้งที่ตอบ response
