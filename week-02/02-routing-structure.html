Week 2 · บทที่ 2

Routing, รับข้อมูล และโครงสร้างที่ไม่พังในอนาคต

ออกแบบ API ให้คนอื่นใช้รู้เรื่อง และจัดโค้ดให้ตัวเองในอีก 6 เดือนไม่ด่าตัวเอง

🔑 คีย์เวิร์ดภาพจำประจำบท

📖 GET = ขอดู · POST = ส่งเข้า 🏷 /books/:id = ป้ายชี้ของชิ้นเดียว = params 🔍 ?search= = ตัวกรองท้าย URL = query 🚧 validation = ด่านตรวจก่อนเข้าเมือง 🗂 routes/controllers/data = แยกครัว แยกหน้าร้าน แยกคลัง

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "HTTP verb คือคำกริยา — ใช้ให้ตรงความหมาย"

GET=ขอดู (ไม่เปลี่ยนข้อมูล), POST=สร้างใหม่, PUT/PATCH=แก้, DELETE=ลบ — นี่ไม่ใช่กฎตายตัวทางเทคนิค (จะ POST ทุกอย่างก็ได้) แต่เป็น ข้อตกลงร่วมของวงการ (REST) ที่ทำให้คนอื่นเดา API เราได้โดยไม่ต้องถาม — API ที่ดีคือ API ที่ "เดาถูก"

หลักการ 2: "Never trust user input" — ด่านแรกของ security ทุกระบบ

ข้อมูลที่ส่งมาจาก client = ของแปลกหน้า เชื่อไม่ได้เลย ต้อง validate ก่อนใช้เสมอ ในฐานะอดีต pentester คุณรู้ดี: ระบบส่วนใหญ่ถูกเจาะเพราะ "เชื่อ input" — SQL injection, XSS, ทุกอย่างเริ่มจากตรงนี้ วันนี้เราเริ่มจาก validation ง่าย ๆ (Week 4 จะอัปเป็น Zod, Week 17 จะมองมันเป็นเกราะ security เต็มตัว)

หลักการ 3: "Separation of Concerns" — แยกหน้าที่ ไม่ปนกัน

ไฟล์เดียวที่ทำทุกอย่าง (route + logic + ข้อมูล) เขียนเร็วตอนแรก แต่พอโต = หาอะไรไม่เจอ แก้จุดหนึ่งพังอีกจุด หลักการนี้คือสิ่งที่โค้ด welovefukushima ของคุณทำได้ดีอยู่แล้ว (แยก inc/) — เราจะเอานิสัยนั้นมาใช้ฝั่ง backend นี่คือจุดที่ทำให้คุณดู senior กว่าอายุงาน

เมื่อไหร่แยกโครง / เมื่อไหร่ไม่ต้อง: โปรเจกต์จิ๋ว 1 ไฟล์พอ — แต่พอมีเกิน ~3 endpoint หรือรู้ว่าจะโต (อย่าง library-system) ให้แยกตั้งแต่ต้น เพราะ refactor ทีหลังแพงกว่า

ส่วนที่ 1: Params vs Query — ใช้ตัวไหนเมื่อไหร่

// params (:id) — ชี้ "ทรัพยากรชิ้นเฉพาะ"
app.get("/api/books/:id", (req, res) => {
  const id = req.params.id;        // /api/books/42 → id = "42"
  // ...หาเล่ม id นี้
});

// query (?...) — "ตัวเลือก/ตัวกรอง" ของรายการ
app.get("/api/books", (req, res) => {
  const { search, limit } = req.query;  // /api/books?search=clean&limit=5
  // ...กรองตาม search, จำกัด limit
});
params :idquery ?key=
ใช้กับระบุของชิ้นเดียวที่เจาะจงกรอง/เรียง/แบ่งหน้า ของรายการ
จำเป็นไหมขาดไม่ได้ (เป็นส่วนของ path)มีหรือไม่มีก็ได้ (optional)
ตัวอย่างจริงเปิดโปรไฟล์ user คนเดียว /users/7ค้นหาสินค้า /products?cat=tech&sort=price
ทำไมแยกให้ถูก: ใช้สลับกันได้ทางเทคนิค แต่ผิด convention = ทีมงงและ recruiter เห็นว่ายังไม่เข้าใจ REST หลักจำง่าย: "ชี้ของชิ้นไหน = params / จะเอาของยังไง = query"

ส่วนที่ 2: รับข้อมูล POST + validate (ด่านแรกของ security)

app.post("/api/books", (req, res) => {
  const { title, author } = req.body;

  // ❌ ถ้าไม่ validate: ใครส่ง body ว่างมา ระบบก็บันทึกขยะ / หรือ crash
  // ✅ ด่านตรวจก่อนเข้าเมือง:
  if (!title || typeof title !== "string") {
    return res.status(400).json({ ok: false, error: "title is required" });
  }

  const newBook = { id: Date.now(), title, author: author ?? "Unknown" };
  // ...บันทึก
  res.status(201).json({ ok: true, data: newBook });  // 201 = created
});

ทำไม status code สำคัญ: 400=client ส่งผิด, 201=สร้างสำเร็จ, 404=ไม่เจอ, 500=server พัง — มันคือ "ภาษากลาง" ที่ frontend ใช้ตัดสินใจว่าจะแสดงอะไรให้ผู้ใช้ ส่ง 200 ทุกกรณี = frontend แยกไม่ออกว่าสำเร็จหรือพัง (เชื่อมกับ res.ok ที่คุณเรียนใน Week 1 พอดี — ฝั่งนู้นเช็ค ฝั่งนี้เป็นคนส่ง)

🔐 มุม pentester: error message ห้ามรั่วข้อมูลภายใน เช่น อย่าตอบ "column 'titl' does not exist in table books" — บอกโครง database ให้ผู้โจมตีฟรี ตอบแค่ "invalid input" ก็พอ (หลักการนี้คือ Quiz ของ AI Review Drill สัปดาห์นี้)

HTTP Status Code ที่ต้องรู้ (จำเป็นทุกวันในงาน API)

กลุ่มcodeความหมายใช้เมื่อ
2xx สำเร็จ200 OKสำเร็จทั่วไปGET, PUT สำเร็จ
201 Createdสร้างใหม่สำเร็จPOST สำเร็จ
4xx ผิดที่ client400 Bad Requestข้อมูลส่งมาผิดvalidation ไม่ผ่าน
401 Unauthorizedยังไม่ loginไม่มี token (Week 5)
403 Forbiddenlogin แล้วแต่ไม่มีสิทธิ์user ทั่วไปแตะของ admin
404 Not Foundไม่เจอทรัพยากรหา id ไม่เจอ
5xx ผิดที่ server500 Internal Errorโค้ดเราพังเองbug, DB ล่ม

หลักจำ: 4xx = "คุณผิด" (client), 5xx = "เราผิด" (server) · 401 = ไม่รู้ว่าคุณเป็นใคร / 403 = รู้แล้วว่าใคร แต่ห้ามเข้า

ส่วนที่ 3: แยกโครงสร้าง — ทำจริง

library-system/
├── src/
│   ├── server.js              # เปิด app + ผูก routes (บางมาก)
│   ├── routes/books.routes.js # ป้ายบอกทาง: path ไหน → controller ไหน
│   ├── controllers/books.controller.js  # logic จริง
│   └── data/books.js          # ข้อมูล (สัปดาห์นี้เป็น array/JSON)
├── package.json
└── README.md

ทำไมโครงนี้: วันที่เพิ่ม members, borrow, return (Kcode Day 4-5) คุณแค่เพิ่มไฟล์คู่ใหม่ ไม่ต้องแตะของเดิม — นี่คือ "ขยายได้โดยไม่พัง" ที่ทำให้ codebase อยู่รอดเป็นปี และเป็นโครงเดียวกับที่ Kcode สอน (handlers/routes/repositories) แค่เราเข้าใจ "ทำไม" ลึกกว่า

โค้ดจริงของการแยก 3 ไฟล์ (เห็นภาพการไหล)

// server.js — บางมาก แค่ setup + ผูก
const express = require("express");
const app = express();
app.use(express.json());
app.use("/api/books", require("./routes/books.routes"));  // ส่งงานต่อ
app.listen(3000);

// routes/books.routes.js — ป้ายบอกทาง (path → controller)
const router = require("express").Router();
const ctrl = require("../controllers/books.controller");
router.get("/", ctrl.list);          // GET /api/books
router.get("/:id", ctrl.getById);    // GET /api/books/:id
router.post("/", ctrl.create);       // POST /api/books
module.exports = router;

// controllers/books.controller.js — logic จริง
const books = require("../data/books");
exports.list = (req, res) => res.json({ ok: true, data: books });
exports.getById = (req, res) => {
  const book = books.find(b => b.id === Number(req.params.id));
  if (!book) return res.status(404).json({ ok: false, error: "not found" });
  res.json({ ok: true, data: book });
};

สังเกตการไหล: request → server.js → routes (จับคู่ path) → controller (ทำงาน) → data — แต่ละไฟล์รู้หน้าที่ตัวเอง แก้จุดเดียวไม่กระทบที่อื่น

หลักคิด layered architecture: ยุคต่อไป (Week 4-5) จะแทรกชั้น "service" (business logic) และ "repository" (คุย DB) เพิ่ม → routes → controller → service → repository → DB · แต่ละชั้นคุยกับชั้นติดกันเท่านั้น = test ง่าย เปลี่ยน DB ไม่กระทบ controller

🧹 Clean Code: เขียน Express แบบ production

1. Response รูปแบบเดียวกันทั้งระบบ (Response Contract)

❌ แต่ละ endpoint ตอบคนละแบบ

res.json(books);              // array เปล่า
res.json({ book });           // object
res.json({ error: "..." });   // คนละ key
// frontend ต้องเดาทุก endpoint

✅ contract เดียว { ok, data, error }

res.json({ ok: true, data: books });
res.json({ ok: true, data: book });
res.status(400).json({ ok: false, error: "..." });
// frontend เช็ค res.ok แบบเดียวจบ

2. Destructure req + guard clause (ทบจาก Week 1)

exports.create = (req, res) => {
  const { title, author } = req.body;        // ดึงครั้งเดียว
  if (!title) return res.status(400).json({ ok: false, error: "title required" });
  // ...เนื้อหลักอยู่ระดับบนสุด ไม่ย่อหน้าลึก
  res.status(201).json({ ok: true, data: newBook });
};

3. server.js ต้องบาง — logic อยู่ใน controller

ถ้า server.js เริ่มมี logic (if/loop/คำนวณ) = สัญญาณว่าต้องย้ายไป controller — server.js ควรมีแค่ setup + ผูก route

กฎทอง: response contract เดียวทั้งระบบ · validate + guard clause ก่อนทำงาน · status code ให้ตรงความหมาย · error ไม่รั่วข้อมูลภายใน · แยกชั้นตามหน้าที่ — ทั้งหมดนี้คือเกณฑ์ review ของ Mission สัปดาห์นี้

📝 Quiz ท้ายบท

ข้อ 1: /api/books/42 กับ /api/books?id=42 ต่างกันยังไง อันไหนถูกตามหลัก REST?

อันแรก (params) ถูก — เพราะเล่ม 42 คือ "ทรัพยากรเฉพาะชิ้น" path ควรชี้ตรงไป อันหลัง (query) เหมาะกับการกรองรายการ ไม่ใช่ชี้ของชิ้นเดียว ผิด convention REST แม้ทำงานได้

ข้อ 2: ทำไมต้องตอบ status 201 ตอน POST สำเร็จ แทนที่จะ 200 เฉย ๆ?

201 = "Created" สื่อชัดว่ามีของใหม่เกิดขึ้น ส่วน 200 = "OK" ทั่วไป — frontend และระบบอื่นอ่าน status เพื่อตัดสินใจ ความแม่นยำของ status คือคุณภาพของ API (และเป็นจุดที่ test จะเช็คใน Week 3)

ข้อ 3 (security): client ส่ง {title: "<script>alert(1)</script>"} มา — validation แค่เช็ค "มี title ไหม" พอไหม?

ไม่พอ — เช็คว่ามีค่าผ่าน แต่เนื้อหาเป็น XSS payload ถ้าเอาไปแสดงบนหน้าเว็บตรง ๆ จะโดนโจมตี validation ที่ดีต้องคิดถึง "เนื้อหาอันตราย" ไม่ใช่แค่ "มี/ไม่มี" — Week 4 (Zod) และ Week 17 (OWASP) จะลงลึกเรื่องนี้ คุณในฐานะ pentester มองออกก่อนคนอื่น = จุดแข็ง

ข้อ 4: เลือก status code — (ก) login ผิดรหัส (ข) ไม่มี token เลย (ค) user ทั่วไปจะลบของ admin (ง) หา id ไม่เจอ

(ก) 401 หรือ 400 (รหัสผิด = ยังไม่ผ่าน auth) (ข) 401 Unauthorized (ไม่รู้ว่าเป็นใคร) (ค) 403 Forbidden (รู้ว่าเป็นใครแล้ว แต่ไม่มีสิทธิ์) (ง) 404 Not Found — จำหลัก: 401=ไม่รู้จักคุณ / 403=รู้จักแต่ห้ามเข้า

ข้อ 5 (clean code): ทำไม API ควรตอบ contract เดียว { ok, data, error } ทั้งระบบ?

frontend เขียนโค้ดจัดการ response แบบเดียวใช้ได้ทุก endpoint (เช็ค res.ok + อ่าน data/error) — ถ้าแต่ละ endpoint ตอบคนละรูป frontend ต้องเขียน logic เฉพาะทุกอัน = โค้ดซ้ำ บั๊กเยอะ เป็น contract ที่ทีมตกลงกันล่วงหน้า

ข้อ 6: layered architecture (routes→controller→service→repository) ช่วยอะไรตอนเปลี่ยนจาก JSON file เป็น PostgreSQL (Week 4)?

เปลี่ยนแค่ชั้น repository (ชั้นที่คุยกับที่เก็บข้อมูล) — controller/service/routes ไม่ต้องแตะเลย เพราะแต่ละชั้นคุยกับชั้นติดกันผ่าน "สัญญา" เดียวกัน นี่คือพลังของการแยกชั้น: เปลี่ยนเทคโนโลยีข้างใต้โดยไม่กระทบข้างบน

✅ Checkpoint

  • อธิบายความต่าง params/query ได้โดยไม่เปิดดู
  • เขียน POST + validation + status code ถูกต้องเป็น
  • เข้าใจเหตุผลการแยก routes/controllers/data (ไม่ใช่แค่ทำตาม)
