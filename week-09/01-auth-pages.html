Auth Pages + Protected Routes

หน้า login/register ที่ใช้งานจริง และการป้องกันหน้าที่ทำถูกหลัก security

🔑 คีย์เวิร์ดภาพจำประจำบท

🚧 ยามที่ขอบเว็บ = middleware.ts 🔀 ไม่มีสิทธิ์ → เด้งไป login = redirect 🙈 ซ่อน UI = UX ไม่ใช่ security 🔐 กัน 3 ชั้น = middleware + UI + backend

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Defense in depth — กันหลายชั้น ไม่พึ่งชั้นเดียว"

การป้องกันหน้าใน frontend ไม่ใช่ security จริง — เป็นแค่ UX (ไม่ให้ผู้ใช้เห็นหน้าที่ใช้ไม่ได้) · คนร้ายเรียก backend API ตรง ๆ ข้าม frontend ได้ · ดังนั้นต้องกัน 3 ชั้น: ① middleware (route) ② ซ่อน UI ③ backend ตรวจ role (Week 5) — ขาดชั้น 3 = Broken Access Control (OWASP #1) ที่คุณเคยเจาะมาแล้ว

หลักการ 2: "Auth state เป็น single source of truth"

สถานะ "login อยู่ไหม / เป็นใคร / role อะไร" ควรมีที่เก็บเดียว (context/hook) แล้วทั้งแอปอ่านจากที่นั้น — ไม่ใช่แต่ละหน้าเช็ค cookie/token เอง (ซ้ำ, ลืมที่ใดที่หนึ่ง = บั๊ก) · หลักเดียวกับ Single Source of Truth ทั้งคอร์ส

หลักการ 3: "Redirect flow ต้องชัดและไม่วน"

ไม่ login เข้าหน้า protected → เด้งไป /login · login แล้วเข้า /login → เด้งไป /dashboard · token หมดอายุกลางทาง → เด้ง /login พร้อมบอกเหตุผล · ออกแบบ flow ให้ครบทุกเคส ไม่งั้นเกิด redirect วนหรือค้าง

ส่วนที่ 1: หน้า Login / Register

"use client";
import { useState } from "react";
import { useRouter } from "next/navigation";

export default function LoginPage() {
  const router = useRouter();
  const [error, setError] = useState("");
  const [loading, setLoading] = useState(false);

  async function handleSubmit(e: React.FormEvent) {
    e.preventDefault();                          // กัน reload (Week 0)
    setLoading(true); setError("");
    const form = new FormData(e.currentTarget as HTMLFormElement);
    const res = await fetch("/api/auth/login", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        email: form.get("email"), password: form.get("password"),
      }),
    });
    setLoading(false);
    if (!res.ok) return setError("อีเมลหรือรหัสผ่านไม่ถูกต้อง");  // กลาง ๆ (Week 5)
    router.push("/dashboard");                   // สำเร็จ → ไปแอป
  }
  // ...form + แสดง error + ปุ่ม disabled ตอน loading
}
checklist หน้า login: preventDefault (Week 0) · ปุ่ม disabled ตอน loading (Week 1) · error ตอบกลาง ๆ (Week 5) · validate ฝั่ง client เป็น UX แต่ backend validate จริง (Week 4) — ทุกอย่างที่เรียนมาบรรจบ

ส่วนที่ 2: Protected Routes — middleware.ts (ชั้นที่ 1)

Next.js มี middleware.ts ที่รันก่อนทุก request — ใช้กัน route ที่ต้อง login:

// middleware.ts (root ของโปรเจกต์)
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(req: NextRequest) {
  const token = req.cookies.get("token")?.value;     // HttpOnly cookie (Week 8)
  const isAuthPage = req.nextUrl.pathname.startsWith("/login");

  // ไม่มี token + เข้าหน้า protected → เด้ง login
  if (!token && !isAuthPage) {
    return NextResponse.redirect(new URL("/login", req.url));
  }
  // มี token + เข้าหน้า login → เด้งเข้าแอป
  if (token && isAuthPage) {
    return NextResponse.redirect(new URL("/dashboard", req.url));
  }
  return NextResponse.next();
}

// กำหนดว่า middleware ครอบ path ไหน
export const config = {
  matcher: ["/dashboard/:path*", "/books/:path*", "/login"],
};
🔐 จุดที่ pentester ต้องระวัง: matcher ต้องครอบ ทุก path ที่ต้องป้องกัน — ลืม path ไหน = หน้านั้นเข้าได้โดยไม่ต้อง login · และ middleware เช็คแค่ "มี token ไหม" ไม่ได้ตรวจ role/ความถูกต้องลึก → role ต้องตรวจที่ backend อีกชั้น (ชั้น 3) เสมอ

ส่วนที่ 3: ซ่อน UI ตาม role (ชั้นที่ 2 — UX)

// แสดงปุ่มลบเฉพาะ admin (UX — ไม่ใช่ security จริง)
{user?.role === "admin" && (
  <button onClick={handleDelete}>ลบหนังสือ</button>
)}
ย้ำหลักการ: นี่ซ่อนปุ่มเพื่อ UX (user ทั่วไปไม่เห็นปุ่มที่กดไม่ได้) — ไม่ใช่ security เพราะคนร้ายเรียก DELETE /api/books/:id ตรงได้ · ความปลอดภัยจริงอยู่ที่ backend requireRole("admin") (Week 5) · ทั้งสองอย่างต้องมีคู่กัน

ส่วนที่ 4: Logout + Token หมดอายุ

// logout — ลบ cookie ที่ server (route handler)
// app/api/auth/logout/route.ts
export async function POST() {
  const res = NextResponse.json({ ok: true });
  res.cookies.delete("token");      // ลบ HttpOnly cookie
  return res;
}

// token หมดอายุกลางทาง — backend ตอบ 401 → frontend เด้ง login
async function apiCall(url: string) {
  const res = await fetch(url);
  if (res.status === 401) {
    window.location.href = "/login";   // token หมด → กลับไป login
    return;
  }
  return res.json();
}

เคสที่มือใหม่ลืม: token หมดอายุระหว่างใช้งาน → ถ้าไม่จัดการ ผู้ใช้เจอ error งง ๆ · ดัก 401 แล้วเด้ง login พร้อมข้อความ = UX ดี

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: dashboard ลูกค้าหลายราย — middleware กันคนไม่ login, backend ตรวจว่า request ขอข้อมูล clientId ตรงกับ token ไหม (ownership check) · กันลูกค้า A แอบดูข้อมูล B = Broken Access Control ที่เกิดบ่อยใน multi-tenant app

🌏 อุตสาหกรรม: Broken Access Control คือ OWASP #1 (2021) — เจอบ่อยสุดใน pentest · การกัน 3 ชั้น (middleware + UI + backend) เป็นมาตรฐานที่ทีม security ตรวจใน code review · คุณทำถูกตั้งแต่แรก = จุดแข็ง

📝 Quiz ท้ายบท

ข้อ 1: ซ่อนปุ่มลบด้วย {isAdmin && ...} = ปลอดภัยพอไหม? เพราะอะไร?

ไม่พอ — เป็นแค่ UX (ซ่อนจากตา) · คนร้ายเรียก DELETE /api/books/:id ตรงผ่าน curl/Postman ข้าม UI ได้ · security จริงต้องอยู่ที่ backend (requireRole) · UI ซ่อน + backend ตรวจ = คู่กันเสมอ

ข้อ 2: middleware.ts ใน Next.js ทำหน้าที่อะไร ข้อควรระวังคืออะไร?

รันก่อนทุก request — ใช้กัน route (เช็ค token, redirect) · ข้อควรระวัง: matcher ต้องครอบทุก path ที่ต้องป้องกัน ลืม path = หน้านั้นเข้าได้ฟรี · และ middleware เช็คแค่ "มี token" ไม่ลึก → role ตรวจที่ backend

ข้อ 3 (security): การกัน 3 ชั้นคืออะไร ทำไมต้องครบ?

① middleware (กัน route ที่ frontend) ② ซ่อน UI (UX) ③ backend ตรวจ role (security จริง) · ขาดชั้น 3 = คนร้ายเรียก API ตรงได้ = Broken Access Control (OWASP #1) · frontend ป้องกันได้แค่ UX จริง ๆ ต้องที่ backend

ข้อ 4: token หมดอายุระหว่างใช้งาน ถ้าไม่จัดการเกิดอะไร แก้ยังไง?

ผู้ใช้เจอ error งง ๆ (request ล้มเหลวเงียบ ๆ) · แก้: ดัก response 401 ทุก API call → เด้งไป /login พร้อมข้อความ "เซสชันหมดอายุ" · UX ดีกว่าปล่อยให้ค้าง
