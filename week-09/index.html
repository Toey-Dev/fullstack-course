Week 9 — Auth Pages + Protected Routes + App Shell

ยกระดับ frontend จาก "หน้าเดียว" สู่ "แอปเต็มระบบ" ที่มี login, ป้องกันหน้า, และโครงแอปจริง

🎯 เป้าหมายของสัปดาห์นี้ (เริ่มเฟส 2)

เฟส 1 จบด้วย full-stack app ที่ทำงานได้ — เฟส 2 ทำให้มัน "เหมือนแอปจริงที่ใช้งานได้ทั้งระบบ" · Week 9 สร้างโครงแอป: หน้า register/login แยกจากแอปหลัก, ป้องกันหน้าที่ต้อง login, sidebar/topbar navigation, แสดงข้อมูลผู้ใช้ · นี่คือการต่อยอด library-web (Week 8) ให้เป็นระบบที่เล่าใน interview ได้ว่า "ทำแอปจริงทั้งระบบ"

  • หน้า register + login (form validation, จัดการ error, redirect)
  • ป้องกัน route ที่ต้อง login (middleware.ts + ProtectedRoute pattern)
  • แยก layout: auth pages (ไม่มี sidebar) vs app pages (มี shell)
  • App shell: sidebar + topbar + แสดงข้อมูล user จาก token
  • จัดการ auth state ทั่วแอป (logout, redirect เมื่อ token หมด)
ทำไมสัปดาห์นี้สำคัญ: "การจัดการ auth ทั้งแอป" (ป้องกันหน้า, แสดง/ซ่อนตาม role, redirect) คือสิ่งที่ทุกแอปจริงมี และเป็นจุดที่ Broken Access Control (OWASP) เกิดบ่อย — ความรู้ pentester ทำให้คุณทำถูกตั้งแต่แรก

🧹 Clean Code ประจำ Week 9

1. ป้องกัน route ที่ server ก่อน ไม่ใช่แค่ซ่อน UI

❌ ซ่อนแค่ปุ่ม (ไม่ปลอดภัย)

{isAdmin && <DeleteButton />}
// ซ่อนปุ่ม แต่ API ยังเรียกตรงได้
// = Broken Access Control

✅ กันที่ทุกชั้น

// 1. middleware.ts กัน route
// 2. ซ่อน UI (UX)
// 3. backend ตรวจ role อีกชั้น (Week 5)
// ป้องกัน 3 ชั้น

2. auth state ที่เดียว ไม่กระจาย

เก็บ/เช็คสถานะ login ที่เดียว (context/hook) แล้วใช้ทั้งแอป — ไม่ใช่แต่ละหน้าเช็ค cookie เอง (ซ้ำ ลืมง่าย)

กฎทอง: ป้องกัน route ที่ server (middleware) เป็นหลัก · UI แค่ช่วย UX · backend ตรวจซ้ำเสมอ · auth state รวมที่เดียว · redirect ชัดเจน (ไม่ login → /login, login แล้วเข้า /login → /dashboard)
