App Shell — Layout Separation + Navigation

โครงแอปจริง: หน้า login ไม่มี sidebar / หน้าในระบบมี shell — และ navigation ที่ใช้งานได้

🔑 คีย์เวิร์ดภาพจำประจำบท

📁 (folder) = route group (ไม่เป็น path) 🖼 2 layout = auth shell vs app shell 🧭 sidebar + topbar = app shell 📍 active link = รู้ว่าอยู่หน้าไหน

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Layout แยกตามบริบท — ไม่ยัดทุกหน้าในกรอบเดียว"

หน้า login/register ควรเต็มจอ ไม่มี sidebar (ผู้ใช้ยังไม่ login จะมีเมนูไปไหน?) · หน้าในระบบควรมี shell (sidebar+topbar) · Next.js route groups (auth) และ (app) ให้แต่ละกลุ่มมี layout ของตัวเอง — โดยไม่กระทบ URL

หลักการ 2: "Shell คือ DRY ของ UI — เขียนครั้งเดียวใช้ทุกหน้าในระบบ"

sidebar, topbar, user menu เขียนใน layout ของกลุ่ม app ครั้งเดียว — ทุกหน้าในระบบได้ shell เหมือนกัน ไม่ต้อง copy nav ลงทุกหน้า · เพิ่มเมนูใหม่แก้ที่เดียว · หลักเดียวกับ layout (Week 8) แต่ scope เป็นกลุ่ม

หลักการ 3: "Navigation ต้องบอกผู้ใช้ว่าอยู่ที่ไหน"

active link (เมนูที่กำลังอยู่ไฮไลต์) ไม่ใช่ของสวยงาม — เป็น UX พื้นฐานที่ผู้ใช้ต้องรู้ตำแหน่งตัวเองในแอป · แอปที่ทุกเมนูดูเหมือนกันหมด = ผู้ใช้หลง · ใช้ usePathname() เช็คว่า path ปัจจุบันตรงเมนูไหน

ส่วนที่ 1: Route Groups — แยก layout โดยไม่กระทบ URL

src/app/
├── (auth)/                  # route group — () ไม่เป็นส่วนของ URL
│   ├── layout.tsx           # layout เต็มจอ ไม่มี sidebar
│   ├── login/page.tsx       # /login (ไม่ใช่ /auth/login)
│   └── register/page.tsx    # /register
├── (app)/                   # อีกกลุ่ม
│   ├── layout.tsx           # layout มี sidebar + topbar
│   ├── dashboard/page.tsx   # /dashboard
│   └── books/page.tsx       # /books
└── layout.tsx               # root (html, body, font)

(ชื่อ) = route group — จัดกลุ่มไฟล์ + ให้ layout ร่วม แต่ไม่เป็นส่วนของ URL · ต่างจากโฟลเดอร์ปกติที่เป็น path

ส่วนที่ 2: App Shell Layout (sidebar + topbar)

// app/(app)/layout.tsx
export default function AppLayout({ children }: { children: React.ReactNode }) {
  return (
    <div className="flex h-screen">
      <aside className="w-60 bg-gray-900 text-white">
        <Sidebar />            {/* เมนู — เขียนครั้งเดียว ทุกหน้าใช้ */}
      </aside>
      <div className="flex-1 flex flex-col">
        <Topbar />             {/* user info + logout */}
        <main className="flex-1 overflow-auto p-6">{children}</main>
      </div>
    </div>
  );
}

layout นี้ครอบทุกหน้าใน (app) — sidebar/topbar เขียนครั้งเดียว · {children} = เนื้อหาแต่ละหน้ามาแทรก · นี่คือ DRY ระดับ shell

ส่วนที่ 3: Active Link — บอกว่าอยู่หน้าไหน

"use client";
import Link from "next/link";
import { usePathname } from "next/navigation";

function NavLink({ href, children }: { href: string; children: React.ReactNode }) {
  const pathname = usePathname();
  const active = pathname === href;
  return (
    <Link href={href}
      className={active ? "bg-blue-600 text-white" : "text-gray-300 hover:bg-gray-800"}>
      {children}
    </Link>
  );
}

usePathname() = path ปัจจุบัน → เทียบกับ href → ไฮไลต์เมนูที่ตรง · ผู้ใช้รู้ตำแหน่งตัวเองในแอป (UX พื้นฐาน)

ส่วนที่ 4: แสดง User Info — ไม่ค้าง mock data

// ดึงข้อมูล user จริงจาก /api/me (token แนบให้โดย proxy)
// Topbar component
async function getUser() {
  const res = await fetch("/api/me");
  return res.ok ? (await res.json()).data : null;
}
// แสดง: ชื่อ + avatar (อักษรแรกของชื่อ) + ปุ่ม logout
เกร็ด production: อย่าค้าง mock data ("John Doe") ในโค้ด — ดึงจริงจาก /api/me · avatar ง่าย ๆ ใช้อักษรแรกของชื่อในวงกลม (ไม่ต้องมีรูปจริง) · นี่คือรายละเอียดที่ทำให้แอปดู "เสร็จจริง" ไม่ใช่ prototype

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: dashboard การตลาด — sidebar เมนู (ภาพรวม, แต่ละแพลตฟอร์ม, รายงาน), topbar แสดงชื่อลูกค้า + logout · route groups แยกหน้า login (เต็มจอ) จากหน้า dashboard (มี shell) · นี่คือโครงมาตรฐานของทุก admin/dashboard app

🌏 อุตสาหกรรม: app shell (sidebar+topbar+content) เป็น layout มาตรฐานของ B2B/internal tool ทุกตัว · route groups เป็นฟีเจอร์ Next.js ที่ทีมใช้แยก layout จริง · active link + user menu = รายละเอียดที่แยกแอป production จาก demo

📝 Quiz ท้ายบท

ข้อ 1: route group (auth) ต่างจากโฟลเดอร์ auth ปกติยังไง?

(auth) = จัดกลุ่ม + ให้ layout ร่วม แต่ ไม่เป็นส่วนของ URL (login อยู่ที่ /login ไม่ใช่ /auth/login) · โฟลเดอร์ auth ปกติเป็น path (/auth/login) · route group ใช้แยก layout โดยไม่เปลี่ยน URL

ข้อ 2: ทำไมหน้า login กับหน้า dashboard ควรมี layout ต่างกัน?

หน้า login: ผู้ใช้ยังไม่ login → ไม่ควรมี sidebar/เมนู (จะไปไหนได้?) เต็มจอเรียบ · หน้า dashboard: ต้องมี shell (sidebar+topbar) นำทาง · route groups ให้แต่ละกลุ่ม layout ของตัวเอง — แยกบริบทชัด

ข้อ 3: app shell layout ช่วยเรื่อง DRY ยังไง?

sidebar/topbar เขียนใน (app)/layout.tsx ครั้งเดียว — ทุกหน้าในกลุ่ม app ได้ shell เหมือนกันอัตโนมัติ · เพิ่มเมนูใหม่/แก้ topbar ที่เดียวมีผลทุกหน้า · ไม่ต้อง copy nav ลงทุก page

ข้อ 4: ทำไม active link สำคัญ ใช้อะไรทำใน Next.js?

บอกผู้ใช้ว่าอยู่หน้าไหน (UX พื้นฐาน — แอปที่ทุกเมนูเหมือนกันหมด ผู้ใช้หลง) · ใช้ usePathname() เทียบ path ปัจจุบันกับ href แล้วใส่ class ไฮไลต์เมนูที่ตรง

✅ Checkpoint

  • app shell: sidebar + topbar + content (DRY)
  • active link ด้วย usePathname
  • แสดง user info จริงจาก /api/me (ไม่ค้าง mock)
