Tailwind + เชื่อม API + Auth

ประกอบ frontend กับ library-system API — เชื่อมทุกอย่างที่เรียนมา 8 สัปดาห์

🔑 คีย์เวิร์ดภาพจำประจำบท

🎨 class = style (Tailwind utility) 🔄 4 สถานะ = loading/success/error/empty 🚇 route handler = ตัวกลาง (proxy) กัน token รั่ว 🎫 token แนบ header = Authorization Bearer

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Utility-first — style ด้วย class เล็ก ๆ ประกอบกัน"

Tailwind คือ Bootstrap เวอร์ชัน utility ล้วน (ที่คุณรู้หลักแล้วจาก Week 0) — แทน .card สำเร็จรูป ใช้ class เล็ก ๆ ประกอบ (p-4 rounded-lg shadow bg-white) · ยืดหยุ่นกว่า ปรับแต่งได้ละเอียด ไม่ต้องเขียน CSS แยก หลักคิดเดียวกับ Bootstrap utility (mt-3, d-flex) แค่ครอบคลุมกว่า

หลักการ 2: "Frontend ไม่ควรถือความลับ — token ผ่าน server"

จาก Week 1+5: token ใน browser เสี่ยง XSS · Next.js มี "route handler" (API ฝั่ง Next) เป็นตัวกลาง — frontend คุยกับ route handler, route handler ถือ token คุยกับ backend · token ไม่โผล่ใน browser (ปลอดภัยกว่า) นี่คือ "API Proxy" pattern ที่เชื่อมความรู้ security ทั้งหมด

หลักการ 3: "4 สถานะเสมอ — frontend จริงไม่ใช่แค่ happy path"

ทบจาก Week 1: ทุกการ fetch ต้องจัดการ loading (กำลังโหลด), success (มีข้อมูล), error (พัง), empty (สำเร็จแต่ว่าง) · discriminated union (Week 7) ทำให้ออกแบบ state นี้แบบ "ลืม case ไม่ได้" · นี่คือจุดที่ทุกอย่างที่เรียนมาบรรจบกัน

ส่วนที่ 2: Fetch จาก API + 4 สถานะ

"use client";
import { useState, useEffect } from "react";

function BookList() {
  const [state, setState] = useState<FetchState<Book[]>>({ status: "loading" });

  useEffect(() => {
    fetch("/api/books")
      .then(r => r.ok ? r.json() : Promise.reject(r.status))
      .then(d => setState({ status: "success", data: d.data }))
      .catch(e => setState({ status: "error", message: String(e) }));
  }, []);

  // 4 สถานะ (discriminated union จาก Week 7)
  if (state.status === "loading") return <Spinner />;
  if (state.status === "error")   return <ErrorMsg msg={state.message} />;
  if (state.data.length === 0)    return <Empty />;        // empty
  return <Grid books={state.data} />;                      // success
}

สังเกต: นี่คือ fetch pattern (Week 1) + discriminated union (Week 7) + Next.js (Week 8) บรรจบกัน — ความรู้ทุกสัปดาห์ทำงานร่วม

ส่วนที่ 3: Login + Token + Protected (เชื่อม Week 5)

// หน้า login — ส่ง credential, รับ token
"use client";
async function handleLogin(email: string, password: string) {
  const res = await fetch("/api/auth/login", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ email, password }),
  });
  if (!res.ok) return showError("login ไม่สำเร็จ");
  // token จัดการฝั่ง server (route handler) — ดู API Proxy ด้านล่าง
  router.push("/books");
}

API Proxy (route handler) — กัน token รั่ว:

// app/api/auth/login/route.ts — ตัวกลางฝั่ง Next (server)
export async function POST(req: Request) {
  const body = await req.json();
  const res = await fetch("http://backend/api/login", {  // คุยกับ backend
    method: "POST", headers: {...}, body: JSON.stringify(body),
  });
  const { token } = await res.json();
  // เก็บ token ใน HttpOnly cookie (JS อ่านไม่ได้ — กัน XSS, จาก Week 5)
  const response = NextResponse.json({ ok: true });
  response.cookies.set("token", token, { httpOnly: true, secure: true });
  return response;
}
🔐 จุดที่ความรู้ security ทั้งหมดบรรจบ: token เก็บใน HttpOnly cookie (Week 1+5) · frontend ไม่เคยเห็น token (route handler ถือแทน) · ทุก request ที่ต้อง auth ผ่าน route handler ที่แนบ cookie ให้ — นี่คือเหตุผลที่ Week 1 เราคุยเรื่อง localStorage vs cookie ไว้ ตอนนี้ทำจริง

📝 Quiz ท้ายบท

ข้อ 1: Tailwind ต่างจาก Bootstrap ยังไง หลักการเหมือนกันไหม?

หลัก utility-first เหมือนกัน (style ด้วย class เล็ก ๆ ประกอบ) — Tailwind ครอบคลุมกว่า, ปรับแต่งละเอียดกว่า, ไม่มี component สำเร็จรูป (ประกอบเอง) · คนที่รู้ Bootstrap utility (Week 0) เรียน Tailwind ได้เร็วเพราะหลักคิดเดียวกัน

ข้อ 2: ทำไมต้องจัดการ 4 สถานะ ไม่ใช่แค่ success?

เพราะโลกจริงไม่เป็นใจ — loading (กำลังโหลด, ไม่ให้จอว่าง), error (พัง, บอกผู้ใช้), empty (สำเร็จแต่ไม่มีข้อมูล, ต่างจาก error) · จัดการแค่ success = จอค้าง/ขาวเมื่อเจอเรื่องร้าย · discriminated union (Week 7) ทำให้ลืม case ไม่ได้

ข้อ 3 (security): ทำไมเก็บ token ใน HttpOnly cookie ผ่าน route handler ดีกว่าเก็บใน localStorage ฝั่ง client?

HttpOnly cookie JS อ่านไม่ได้ → โดน XSS ก็ขโมย token ไม่ได้ (Week 1+5) · route handler (server) ถือ token แทน frontend → token ไม่เคยอยู่ใน browser ที่ script เข้าถึงได้ · ปลอดภัยกว่า localStorage ที่ทุก script อ่านได้

ข้อ 4: API Proxy (route handler) ช่วยอะไรนอกจากเก็บ token?

ซ่อน credential/API key ที่ไม่ควรอยู่ฝั่ง browser (เช่น API key ของ third-party), เลี่ยง CORS, รวมจุดเรียก backend ไว้ที่เดียว (เปลี่ยน backend URL ที่เดียว) · เป็น pattern มาตรฐานของ Next.js full-stack
