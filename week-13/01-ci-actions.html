CI + GitHub Actions พื้นฐาน

หุ่นยนต์ที่ตรวจโค้ดให้ทุกครั้ง — เข้าใจกลไกก่อนเขียน pipeline

🔑 คีย์เวิร์ดภาพจำประจำบท

🤖 ตรวจอัตโนมัติทุก push = CI 📋 สูตรงาน = workflow (yaml) 🏃 เครื่องที่รัน = runner ⚡ พังเร็ว = fail fast 🚦 เขียวค่อย merge = required check

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Integrate บ่อย หาบั๊กเร็ว — ฆ่า merge hell"

ถ้าทุกคนเขียนแยกกันนาน ๆ แล้วค่อย merge ทีเดียว = ชนกันยับ (merge hell) · CI ให้ "รวมโค้ดบ่อย + ตรวจทุกครั้ง" — push ปุ๊บ หุ่นยนต์รัน test ทันที เจอปัญหาตอนเล็ก ๆ แก้ง่าย · ยิ่ง integrate บ่อย ยิ่ง conflict น้อย

หลักการ 2: "Automate the boring — คนลืม เครื่องไม่ลืม"

"ลืมรัน test ก่อน push", "ลืม lint", "เครื่องผมผ่านแต่ของเพื่อนพัง" — ปัญหาเพราะพึ่งคน · CI ทำให้เป็นมาตรฐานเดียว รันบนเครื่องกลาง (runner สะอาด) ทุกครั้งเหมือนกัน · "works on my machine" หายไปเพราะตรวจบนเครื่องกลาง

หลักการ 3: "Quality gate — เขียวค่อยผ่าน"

CI ไม่ใช่แค่ "รัน test" — เป็น "ประตูคุณภาพ" ที่ PR ต้องผ่านก่อน merge · lint แดง/test แดง = merge ไม่ได้ · บังคับให้โค้ดที่เข้า main มีคุณภาพขั้นต่ำเสมอ · นี่คือเหตุผลที่ทีมจริงเชื่อใจ main branch ได้

ส่วนที่ 1: โครงสร้าง GitHub Actions

ไฟล์ workflow อยู่ใน .github/workflows/ci.yml — โครงมี 4 ระดับ:

ระดับคือ
Workflowไฟล์ทั้งหมด + trigger (เมื่อไหร่รัน)
Jobกลุ่มงานที่รันบน runner หนึ่ง (test, build — ขนานหรือลำดับ)
Stepคำสั่งย่อยใน job (รันตามลำดับ)
Runnerเครื่อง (VM) ที่ GitHub ให้รัน (ubuntu-latest)

ส่วนที่ 2: Workflow แรก

# .github/workflows/ci.yml
name: CI

on:                          # trigger — เมื่อไหร่รัน
  push:
    branches: [main]
  pull_request:              # รันทุก PR (สำคัญสุด — ตรวจก่อน merge)

jobs:
  test:
    runs-on: ubuntu-latest   # runner
    steps:
      - uses: actions/checkout@v4         # 1. ดึงโค้ดมา
      - uses: actions/setup-node@v4       # 2. ติดตั้ง Node
        with:
          node-version: 20
          cache: npm                       # cache deps (เร็วขึ้น)
      - run: npm ci                        # 3. install
      - run: npm run lint                  # 4. lint
      - run: npm test                      # 5. test
      - run: npm run build                 # 6. build
ส่วนความหมาย
ontrigger — push main + ทุก PR (PR สำคัญสุด: ตรวจก่อน merge)
usesเรียก action สำเร็จรูป (checkout, setup-node) — pin version @v4 ไม่ใช่ @main
runรันคำสั่ง shell (npm ...)
🔐 จุด security: uses: action@v4 — pin version เสมอ ไม่ใช้ @main (action ที่ชี้ main อาจถูกเปลี่ยนโค้ดเป็นมัลแวร์ได้ = supply chain attack) · pin tag/SHA ปลอดภัยกว่า

ส่วนที่ 3: อ่าน Log + Status

push แล้วไปแท็บ Actions ใน GitHub — เห็น workflow รัน, แต่ละ step เขียว/แดง · คลิกดู log ที่พัง · PR จะโชว์ status check (✅/❌) ก่อน merge

วิธี debug CI ที่พัง: อ่าน log step ที่แดง (มักบอกชัด) · เคสคลาสสิก: "ผ่านในเครื่องแต่ CI แดง" = มักเพราะ runner สะอาด (ไม่มีไฟล์ที่ลืม commit, env ต่าง, หรือ test พึ่งสิ่งที่มีแค่ในเครื่อง) — CI จับเจอสิ่งที่เครื่องเราซ่อน

ส่วนที่ 4: ตัวอย่างประยุกต์

🏢 งานวี: ถ้าทำ automation report เป็นทีม — CI รัน test ทุก PR กันคนแก้ logic สรุปแล้วตัวเลขเพี้ยนโดยไม่รู้ตัว · lint บังคับ style เดียวกันทั้งทีม (ไม่ทะเลาะเรื่อง format) · main branch เชื่อใจได้ว่าไม่พัง

🌏 อุตสาหกรรม: ทุกบริษัทจริงจังมี CI — PR ที่ check แดง merge ไม่ได้ · GitHub Actions เป็นตัวเลือกอันดับต้น (ฟรีสำหรับ public repo) · "เขียน CI เป็น" คือทักษะพื้นฐานที่ JD ขอ และทำให้ repo portfolio ดูมืออาชีพ (มี badge เขียว)

📝 Quiz ท้ายบท

ข้อ 1: CI แก้ปัญหา "merge hell" และ "works on my machine" ยังไง?

merge hell: integrate บ่อย + ตรวจทุก push → conflict เล็กลง เจอเร็ว · works on my machine: รันบน runner กลางสะอาด ทุกครั้งเหมือนกัน → จับสิ่งที่เครื่อง dev ซ่อน (ไฟล์ลืม commit, env ต่าง) · มาตรฐานเดียวไม่พึ่งคน

ข้อ 2: workflow/job/step/runner คืออะไร ความสัมพันธ์?

workflow = ไฟล์ทั้งหมด + trigger · job = กลุ่มงานบน runner หนึ่ง (ขนาน/ลำดับได้) · step = คำสั่งย่อยใน job (ตามลำดับ) · runner = เครื่อง VM ที่ GitHub ให้รัน · workflow → jobs → steps

ข้อ 3 (security): ทำไม uses: action@v4 ดีกว่า @main?

@main ชี้ branch ที่เปลี่ยนได้ — เจ้าของ action (หรือคนแฮก) เปลี่ยนโค้ดเป็นมัลแวร์ได้ = supply chain attack (มัลแวร์เข้า CI ที่มี secret) · pin @v4/SHA = โค้ดคงที่ ตรวจสอบได้ ปลอดภัยกว่า

ข้อ 4: "ผ่านในเครื่องแต่ CI แดง" มักเพราะอะไร?

runner สะอาด (ไม่มีของที่เครื่องเรามี): ไฟล์ที่ลืม commit, env var ที่มีแค่ในเครื่อง, dependency ที่ลงเองไม่ได้อยู่ใน package, test ที่พึ่ง state เครื่อง · CI จับเจอสิ่งที่เครื่อง dev ซ่อนไว้ = เป็นเรื่องดี (เจอก่อน production)

✅ Checkpoint

  • เข้าใจ CI (merge hell, fail fast, quality gate)
  • workflow แรก (on push/PR, checkout, setup-node, test)
  • เข้าใจ workflow/job/step/runner · pin action version
  • อ่าน log + status ใน Actions tab
  • Quiz + โน้ต notes/ci-actions.md
