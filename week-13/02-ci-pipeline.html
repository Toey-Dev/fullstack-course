Week 13 · บทที่ 2

CI Pipeline จริง — lint/test/build + Service

pipeline ที่ test ต้องมี database จริง — และเทคนิคที่ทำให้ CI เร็วและน่าเชื่อถือ

🔑 คีย์เวิร์ดภาพจำประจำบท

🗄 DB ชั่วคราวตอน test = services 💾 จำ deps = cache 🔢 หลาย Node = matrix 🔒 ค่าลับ = secrets 🚧 เขียวค่อย merge = branch protection

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Test ต้องรันในสภาพใกล้ production"

library-system test ต้องมี PostgreSQL (Week 4) — runner ไม่มี DB ติดมา · GitHub Actions services ให้รัน container DB ชั่วคราวระหว่าง test (เหมือน compose ย่อ ๆ) · test รันกับ DB จริง → มั่นใจว่า integration ทำงานจริง ไม่ใช่ mock ทุกอย่างจนไม่เจอบั๊กจริง

หลักการ 2: "CI ต้องเร็วพอที่คนจะรอ — cache คือเพื่อน"

ถ้า CI ใช้ 10 นาที คนจะข้าม (ไม่รอ) · cache node_modules ทำให้ install เร็วขึ้นมาก (ไม่ download ใหม่ทุกครั้ง) · pipeline ที่เร็ว = คนใช้จริง = คุณภาพดีขึ้น · ความเร็วคือ feature ของ CI

หลักการ 3: "CI ไร้ค่าถ้าไม่บังคับ — branch protection"

มี CI แต่ merge ได้แม้แดง = ไม่มีประโยชน์ · branch protection บังคับ "CI เขียว + review ผ่าน" ก่อน merge เข้า main · นี่คือสิ่งที่เปลี่ยน CI จาก "ของประดับ" เป็น "ประตูคุณภาพจริง"

ส่วนที่ 1: Pipeline เต็ม + PostgreSQL Service

name: CI
on:
  pull_request:
  push: { branches: [main] }

jobs:
  test:
    runs-on: ubuntu-latest
    services:                            # ⭐ DB ชั่วคราวระหว่าง test
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: library_test
        ports: ["5432:5432"]
        options: >-                       # รอ DB พร้อมก่อน test
          --health-cmd pg_isready
          --health-interval 5s
          --health-retries 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npm run lint                 # fail fast — เร็วก่อน
      - run: npm test                     # test กับ postgres service
        env:
          DATABASE_URL: postgres://postgres:testpass@localhost:5432/library_test
      - run: npm run build

services.postgres = container DB ที่ GitHub รันให้ระหว่าง job · test เชื่อมผ่าน localhost:5432 (ใน runner) · pg_isready รอจน DB พร้อม (เหมือน healthcheck Week 11)

ส่วนที่ 2: Caching + Matrix

# cache — setup-node มี cache: npm ในตัว (cache ~/.npm)
- uses: actions/setup-node@v4
  with: { node-version: 20, cache: npm }   # install เร็วขึ้นมาก

# matrix — test หลาย Node version พร้อมกัน
strategy:
  matrix:
    node: [18, 20, 22]
steps:
  - uses: actions/setup-node@v4
    with: { node-version: ${{ matrix.node }} }
# → รัน job 3 ครั้ง (Node 18, 20, 22) ขนานกัน

cache: เก็บ deps ไม่ download ใหม่ → CI เร็วขึ้นหลายเท่า · matrix: ทดสอบหลาย version พร้อมกัน (library ที่ support หลาย Node) — งานทั่วไปใช้ version เดียวพอ

ส่วนที่ 3: Secrets ใน Actions

# เก็บ secret ที่ GitHub repo: Settings → Secrets and variables → Actions
# ใช้ใน workflow:
- run: npm test
  env:
    JWT_SECRET: ${{ secrets.JWT_SECRET }}     # ดึงจาก secrets (ไม่โผล่ใน log)
🔐 มุม pentester: GitHub Secrets ถูก mask ใน log อัตโนมัติ (ถ้า echo ออกมาจะเป็น ***) · แต่ระวัง: PR จาก fork ภายนอกไม่ได้ secret (กัน PR ปลอมขโมย secret) · และอย่า log secret หรือส่งไป third-party action ที่ไม่ trust · ความรู้ supply chain security ของคุณสำคัญตรงนี้ — CI มี secret = เป้าหมายการโจมตี

ส่วนที่ 4: Branch Protection

ตั้งที่ GitHub: Settings → Branches → Add rule สำหรับ main:

  • ✅ Require status checks to pass before merging (เลือก job ci)
  • ✅ Require pull request reviews (อย่างน้อย 1 approve)
  • ✅ Require branches to be up to date
ผล: push ตรงเข้า main ไม่ได้ → ต้องทำผ่าน PR → CI ต้องเขียว + มีคน review → ค่อย merge · main branch สะอาดเสมอ · นี่คือ workflow มาตรฐานของทีมจริง (และทำให้ portfolio คุณดูเป็นมืออาชีพ)

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: CI สำหรับ automation service — test กับ DB จริง (services), secret API key ใน GitHub Secrets, branch protection กันโค้ดพังเข้า main · ทีมแก้ logic แล้ว CI ยืนยันตัวเลขยังถูกก่อน deploy

🌏 อุตสาหกรรม: services (DB ใน CI), caching, secrets, branch protection เป็นองค์ประกอบมาตรฐานของ CI ทุกที่ · การ "test กับ DB จริงใน CI" คือความต่างของ integration test ที่เชื่อถือได้ · pentester ที่เข้าใจ CI security (secret, supply chain) เป็นที่ต้องการในสาย DevSecOps

📝 Quiz ท้ายบท

ข้อ 1: ทำไม test ใน CI ต้องมี PostgreSQL service ไม่ mock หมด?

library-system มี integration test ที่ต้องคุย DB จริง (Week 4) — mock หมดจะไม่เจอบั๊กจริง (SQL ผิด, migration พัง) · services รัน container DB ชั่วคราว → test ในสภาพใกล้ production · เชื่อถือได้กว่า mock ทุกอย่าง

ข้อ 2: cache node_modules ช่วยอะไร ทำไมสำคัญ?

ไม่ download deps ใหม่ทุกครั้ง → install เร็วขึ้นหลายเท่า → CI เร็ว · CI ที่เร็วคนใช้จริง (ไม่ข้าม) → คุณภาพดีขึ้น · ความเร็วคือ feature ของ CI

ข้อ 3 (security): PR จาก fork ภายนอกได้ secret ไหม ทำไม?

ไม่ได้ — GitHub กันไว้ เพราะถ้าได้ คนส่ง PR ปลอมที่ echo ${{ secrets.X }} จะขโมย secret ได้ · PR fork รันแบบจำกัดสิทธิ์ · นี่คือเหตุผลที่ workflow ต้องระวังเรื่อง trust ของ PR ภายนอก (supply chain)

ข้อ 4: branch protection ทำให้ CI "มีความหมาย" ยังไง?

มี CI แต่ merge ได้แม้แดง = ไร้ค่า · branch protection บังคับ CI เขียว + review ผ่าน ก่อน merge เข้า main → main สะอาดเสมอ · เปลี่ยน CI จากของประดับเป็นประตูคุณภาพจริง
