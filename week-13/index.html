Week 13 — CI: GitHub Actions

ให้หุ่นยนต์ตรวจโค้ดทุกครั้งที่ push — lint, test, build อัตโนมัติก่อน merge

🎯 เป้าหมายของสัปดาห์นี้

จนถึงตอนนี้คุณรัน test/build เองในเครื่อง — Week 13 ให้ GitHub Actions ทำให้อัตโนมัติทุกครั้งที่ push/PR · lint → test → build รันบน server ของ GitHub ถ้าผ่านถึง merge ได้ ถ้าแดงห้าม merge · นี่คือ CI (Continuous Integration) ที่ทำให้โค้ดในทีมไม่พังเงียบ ๆ และเป็นทักษะที่ทุกบริษัท 100k+ บังคับ

  • เข้าใจ CI: ทำไม, merge hell, fail fast
  • GitHub Actions: workflow, job, step, trigger, runner
  • สร้าง pipeline: lint → test → build (พร้อม PostgreSQL service สำหรับ test)
  • caching (เร็วขึ้น) · matrix (ทดสอบหลาย Node version)
  • branch protection — บังคับ CI ผ่านก่อน merge · secrets ใน Actions
ทำไมสัปดาห์นี้เปลี่ยนระดับคุณ: CI/CD คือเส้นแบ่งระหว่าง "เขียนโค้ดเป็น" กับ "ทำงานในทีมจริงเป็น" · PR ที่มี CI เขียวคือมาตรฐานของทุกทีมจริงจัง — และเป็น badge ที่ทำให้ repo คุณดูมืออาชีพทันที

🧹 Clean Code ประจำ Week 13

1. CI ต้อง fail fast — เรียง step ให้เร็วก่อน

❌ build ก่อน lint (ช้า)

- run: npm run build   # ช้า รันก่อน
- run: npm run lint    # ถ้า lint พัง รู้ทีหลัง = เสียเวลา build

✅ เร็ว/ถูกก่อน

- run: npm run lint    # เร็ว — พังก่อนไม่ต้องรอ
- run: npm test        # กลาง
- run: npm run build   # ช้าสุด ทำท้าย

2. ไม่ commit secret — ใช้ GitHub Secrets

JWT_SECRET, DB password ใน workflow → ใช้ ${{ secrets.X }} (เก็บใน repo settings) ไม่เขียนค่าตรงใน yaml

กฎทอง: fail fast (เรียง step เร็ว→ช้า) · CI ต้องผ่านก่อน merge (branch protection) · secret ใน GitHub Secrets · cache deps (เร็วขึ้น) · workflow อ่านง่าย (ตั้งชื่อ step ชัด)
