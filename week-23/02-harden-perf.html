2 · ตัวอย่างช่องโหว่ที่มักเจอใน capstone — และวิธีแก้

ช่องโหว่ยอดฮิตที่ dev มักพลาด (คุณจับได้เพราะพื้นเพ pentester):

❌ IDOR — ลืม ownership check

// เช็คแค่ว่า login + role ถูก
router.patch('/findings/:id', auth,
  requireRole('lead'), (req,res) => {
  // ❌ lead ของทีม A แก้ finding ทีม B ได้!
  update(req.params.id, req.body);
});

✅ เช็ก ownership/scope

router.patch('/findings/:id', auth,
  requireRole('lead'), async (req,res) => {
  const f = await getFinding(req.params.id);
  if (f.project_id !== req.user.project_id)
    return res.status(404).json({error:'not found'});
  update(req.params.id, req.body);   // ✅
});
หลัก: auth (คุณเป็นใคร) ≠ authorization (คุณทำสิ่งนี้กับข้อมูลชิ้นนี้ได้ไหม) · IDOR เกิดเพราะเช็คแค่ role ไม่เช็คว่าเป็นเจ้าของ resource · คืน 404 ไม่ใช่ 403 เพื่อไม่ leak ว่ามี resource อยู่

3 · Performance Pass — apply Week 21 กับ capstone

ก่อนโชว์ ให้ profile แล้วแก้คอขวด (measure ก่อนเสมอ — W21):

  • Index — EXPLAIN query หลัก (list/filter findings, dashboard) → เพิ่ม index ที่ขาด
  • N+1 — dashboard/list ยิง query ซ้ำต่อแถวไหม → JOIN
  • Pagination — list findings มี pagination (cursor) ไม่โหลดหมด
  • Cache — dashboard (อ่านบ่อย เปลี่ยนช้า) → cache + invalidate ตอน finding เปลี่ยน
  • Queue — export PDF (ช้า) → BullMQ ไม่ block request
เก็บตัวเลข: ใส่ใน docs/PERFORMANCE.md (before/after) · "dashboard 850ms → 45ms (cache + index)" คือประโยคทองใน interview และ resume bullet · พิสูจน์ว่าคุณ optimize เป็นและวัดผลได้

6 · Quiz

Q1. 🔐 IDOR คืออะไร และต่างจาก "ลืมใส่ auth" ยังไง?
เฉลยIDOR = ผู้ใช้ที่ login + มี role ถูกต้องแล้ว แต่เข้าถึง resource ของคนอื่นได้ (เช่น lead ทีม A แก้ของทีม B) · ต่างจากลืม auth ตรงที่ auth ผ่านแล้วแต่ขาด ownership/scope check · แก้: เทียบเจ้าของ resource ฝั่ง server
Q2. 🔐 ทำไมตัวอย่างคืน 404 แทน 403 เมื่อ resource ไม่ใช่ของผู้ใช้?
เฉลย403 บอกว่า "resource นี้มีอยู่แต่คุณห้ามดู" = leak การมีอยู่ของข้อมูล (enumeration) · 404 บอกแค่ "ไม่เจอ" ไม่เปิดเผยว่ามี · ลด info disclosure
Q3. ก่อน optimize performance ควรทำอะไรก่อนเสมอ (หลัก W21)?
เฉลยmeasure/profile ก่อน — EXPLAIN, จับ query count, วัด response time หาคอขวดจริง · อย่าเดา · optimize ตรงที่ช้าจริง แล้ววัด after เทียบ — ไม่ใช่ optimize สุ่ม
Q4. ทำไม case study สำคัญพอ ๆ กับตัวโค้ด?
เฉลยrecruiter/interviewer ไม่อ่านโค้ดทุกบรรทัด · case study เล่า problem→approach→trade-off→result ใน 2 นาที ทำให้เห็นว่าคุณคิดยังไง ไม่ใช่แค่เขียนได้ · เป็นบทซ้อม "tell me about a project"

7 · 🧹 Clean Code: ส่งมอบให้คนอื่นอ่านรู้เรื่อง

❌ repo ที่ไม่มีใครเริ่มได้

README ว่าง/แค่ชื่อ · ไม่บอกวิธีรัน · ไม่มี .env.example · ไม่มี screenshot · ต้องอ่านโค้ดถึงจะเข้าใจ

✅ repo พร้อมส่งมอบ

README: 1-line วิธีรัน · architecture · screenshot · live URL · .env.example · case study link · CI badge เขียว

เคล็ด: repo ที่ดีคือ "clone แล้วรันได้ใน 5 นาที" · มี .env.example, docker compose up ขึ้นเลย, README มี screenshot + live demo · tag v1.0 เมื่อเสร็จ (recruiter เห็น release = เสร็จจริง) · git history สะอาดด้วย conventional commits
