Week 23 · บทที่ 1

✅ ทำ Feature ให้ครบ + ยกระดับ Test

จาก "เขียนเสร็จ" สู่ "เสร็จจริง" — Definition of Done + test ที่มีความหมาย

1 · Definition of Done — "เสร็จ" คืออะไรกันแน่

Definition of Done (DoD) = เกณฑ์ที่ feature ต้องผ่านทั้งหมดก่อนเรียกว่าเสร็จ · ทีมมืออาชีพมี DoD เป็นข้อตกลงร่วม ป้องกัน "เสร็จแบบครึ่ง ๆ" ที่กลับมาหลอกหลอน

DoD ของ feature ใน capstone คุณ ควรเป็น:

  • ทำงานครบ happy path (เคสปกติ)
  • จัดการ edge case + error case (input ผิด, ไม่มีสิทธิ์, ไม่เจอข้อมูล)
  • มี test ครอบคลุม (unit logic สำคัญ + integration endpoint)
  • validate input (Zod) + auth/RBAC ครบ
  • UI มี loading / empty / error state
  • deploy แล้วใช้งานบน production ได้
  • doc/README อัปเดต
ทำไมสำคัญกับ interview: เมื่อ interviewer ถาม "เล่า project หน่อย" แล้วคุณพูดถึง DoD + edge case ที่จัดการ = คุณคิดแบบ engineer ที่ส่งของขึ้น production จริง ไม่ใช่นักเรียนที่ทำให้ demo ผ่าน

2 · ทำไม coverage % ไม่ใช่เป้าหมาย — test ที่มีความหมายต่างหาก

มือใหม่มักไล่ตาม "coverage 100%" · แต่ 100% ที่เทสต์แต่ getter/setter ไร้ค่า ส่วน logic สำคัญที่พังกลับไม่ได้เทสต์ · เป้าหมายจริงคือ เทสต์สิ่งที่ถ้าพังแล้วเจ็บ

❌ ไล่ % — เทสต์ผิดจุด

เทสต์ getter, constant, mapping ตรง ๆ ให้ % สวย · แต่ business logic (คำนวณ CVSS, RBAC, status transition) ไม่ถูกแตะ

✅ เทสต์จุดเสี่ยง

เทสต์ logic ที่ซับซ้อน/สำคัญ: RBAC (ใครทำอะไรได้), validation, status transition, edge case ที่เคยพัง · % ตามมาเอง

⚠️ counter-example: ทีมที่บังคับ "coverage ต้อง 90%" มักได้ test ขยะที่เขียนเอาผ่านเกณฑ์ — assert น้อย mock มั่ว · coverage เป็นสัญญาณว่าตรงไหนยังไม่แตะ ไม่ใช่เป้าหมายในตัวเอง · ดูคุณภาพ test ไม่ใช่ตัวเลข

3 · Test Pyramid — เทสต์ระดับไหน เท่าไหร่

กรอบจัดสัดส่วน test ที่คุ้มค่าสุด (apply กับ capstone):

ระดับเทสต์อะไรVulnTracker
Unit (เยอะ)logic ฟังก์ชันเดี่ยว เร็วมากคำนวณ severity, validate, RBAC helper
Integration (กลาง)endpoint + DB จริง (Supertest)POST /findings → 201, client → 403
E2E (น้อย)flow ผู้ใช้ทั้งเส้นlogin → สร้าง finding → เห็นใน dashboard

เน้น unit + integration (คุ้มสุด) · E2E แค่ flow สำคัญ 2-3 เส้น (ช้า/เปราะ)

4 · ตัวอย่าง — เทสต์ครบ happy + edge + error

feature "เปลี่ยน status ของ finding" — เทสต์ให้ครบมิติ ไม่ใช่แค่ happy path:

describe('PATCH /findings/:id/status', () => {
  // ✅ happy
  it('lead เปลี่ยน open→fixed ได้', async () => {
    const res = await request(app).patch(`/findings/${id}/status`)
      .set('Authorization', leadToken).send({ status: 'fixed' });
    expect(res.status).toBe(200);
  });

  // ⚠️ edge — transition ที่ไม่ถูกต้อง
  it('ห้ามข้าม open→verified (ต้อง fixed ก่อน)', async () => {
    const res = await request(app).patch(`/findings/${id}/status`)
      .set('Authorization', leadToken).send({ status: 'verified' });
    expect(res.status).toBe(422);          // business rule
  });

  // 🔒 error — สิทธิ์ไม่พอ (RBAC)
  it('client เปลี่ยน status ไม่ได้ (403)', async () => {
    const res = await request(app).patch(`/findings/${id}/status`)
      .set('Authorization', clientToken).send({ status: 'fixed' });
    expect(res.status).toBe(403);
  });

  // 🔒 error — ไม่ login (401) / ไม่เจอ (404) / input ผิด (400)
  it('input ผิด → 400', async () => { /* status: 'banana' → Zod reject */ });
});

มิติที่ต้องครบ: happy (ปกติ), edge (business rule/ขอบเขต), error (401/403/404/400/422) · เทสต์ error case คือที่ที่ bug ซ่อนอยู่จริง

6 · Quiz

Q1. ทำไม coverage 100% อาจยังเป็น test ที่แย่?
เฉลยเพราะอาจเทสต์โค้ดไร้ความเสี่ยง (getter/constant) ให้ % สวย แต่ logic สำคัญที่ซับซ้อนกลับ assert น้อย/mock มั่ว · coverage วัด "รันถึง" ไม่ได้วัด "เทสต์ถูกต้อง" · ดูคุณภาพ ไม่ใช่ตัวเลข
Q2. ทำไมควรมี unit เยอะ E2E น้อย (test pyramid)?
เฉลยunit เร็ว/ถูก/ชี้จุดพังตรง · E2E ช้า/เปราะ/แพงดูแล · เน้น unit+integration ให้ครอบคลุม logic แล้วใช้ E2E แค่ flow สำคัญ 2-3 เส้น — ได้ความมั่นใจคุ้มเวลา
Q3. 🔐 ในตัวอย่าง status — ทำไม "client → 403" ถึงเป็น test สำคัญที่ห้ามขาด?
เฉลยเพราะมันเทสต์ access control (OWASP A01) — ป้องกัน privilege escalation · ถ้า RBAC พังเงียบ ๆ client แก้ข้อมูลได้ = ช่องโหว่ร้ายแรง · test นี้คือ regression guard ว่าสิทธิ์ไม่หลุดเมื่อแก้โค้ดภายหลัง
Q4. 3 มิติที่ทุก feature ควรเทสต์คืออะไร?
เฉลยHappy (เคสปกติทำงาน) · Edge (ขอบเขต/business rule เช่นห้ามข้าม status) · Error (401/403/404/400/422 — input ผิด/ไม่มีสิทธิ์/ไม่เจอ) · bug ส่วนใหญ่ซ่อนใน edge+error

7 · 🧹 Clean Code: test ที่อ่านเป็นเอกสาร

❌ test ชื่อกำกวม assert เยอะมั่ว

it('test1', async () => {
  // setup 50 บรรทัด, assert 10 อย่าง
  // พังแล้วไม่รู้พังเพราะอะไร
});

✅ ชื่อบอกพฤติกรรม · AAA

it('client เปลี่ยน status ไม่ได้ (403)', () => {
  // Arrange → Act → Assert
  // 1 test = 1 พฤติกรรม
});

เคล็ด: ชื่อ test = สเปกที่อ่านได้ ("ใคร ทำอะไร ได้ผลอะไร") · ใช้ AAA (Arrange-Act-Assert) · 1 test เทสต์ 1 พฤติกรรม · ใช้ factory/helper สร้าง test data ลด setup ซ้ำ · test suite ที่ดี = เอกสารที่มีชีวิตว่าระบบทำอะไรได้บ้าง
