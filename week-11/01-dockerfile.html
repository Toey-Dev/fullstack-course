Dockerfile + Multi-stage Build

เขียน "สูตร" แพ็ค app ลง container — ให้รันเหมือนกันทุกที่

🔑 คีย์เวิร์ดภาพจำประจำบท

📜 สูตรแพ็ค app = Dockerfile 🧱 แต่ละบรรทัด = layer (cache ได้) 🏗 build แยก run = multi-stage (เล็ก) 🚫 อย่าก๊อปขยะ = .dockerignore

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Dockerfile = สูตรที่ทำซ้ำได้เป๊ะ"

Dockerfile บอกทีละขั้นว่า "สร้าง environment ของ app ยังไง" (base image, copy code, install, build, รัน) · ใครรันก็ได้ image เหมือนกัน · นี่คือ "infrastructure as code" (Week 4) ในระดับ app — env ไม่ขึ้นกับเครื่อง

หลักการ 2: "Layer caching — เรียงคำสั่งให้ใช้ cache คุ้ม"

แต่ละบรรทัดใน Dockerfile = layer ที่ cache ได้ · ถ้าบรรทัดบนไม่เปลี่ยน Docker ใช้ cache (build เร็ว) · เคล็ดลับ: copy package.json + install ก่อน copy source — เพราะ deps เปลี่ยนน้อยกว่า code (แก้ code ไม่ต้อง install ใหม่) · เรียงผิด = build ช้าทุกครั้ง

หลักการ 3: "Multi-stage — image production ควรมีแค่ที่ต้องรัน"

ตอน build ต้องการ dev deps, source, build tools — แต่ตอน รันต้องการแค่ผลลัพธ์ (dist) + prod deps · multi-stage: stage แรก build, stage สอง copy เฉพาะผลลัพธ์มา · image เล็กลงหลายเท่า = deploy เร็ว, ผิวการโจมตีน้อย (security)

ส่วนที่ 1: Dockerfile พื้นฐาน — ทีละ instruction

FROM node:20-slim              # 1. base image (Node 20 รุ่นเล็ก)
WORKDIR /app                   # 2. โฟลเดอร์ทำงานใน container
COPY package*.json ./          # 3. copy แค่ package ก่อน (cache!)
RUN npm ci                     # 4. install (ใช้ cache ถ้า package ไม่เปลี่ยน)
COPY . .                       # 5. copy source ที่เหลือ
RUN npm run build              # 6. build TS → JS (dist/)
EXPOSE 3000                    # 7. บอกว่า app ฟัง port ไหน
CMD ["node", "dist/server.js"] # 8. คำสั่งรันตอน start container
instructionทำอะไร (และทำไม)
FROMเลือก base image — slim/alpine เล็กกว่า, pin เวอร์ชัน (ไม่ใช้ latest)
WORKDIRตั้งโฟลเดอร์ทำงาน (คำสั่งถัดไปรันที่นี่)
3-4 ก่อน 5⭐ copy package + install ก่อน copy code — layer caching (แก้ code ไม่ต้อง install ใหม่)
npm ciinstall จาก lockfile แบบ clean (เร็ว + reproducible กว่า npm install)
EXPOSEเอกสาร (บอกว่าฟัง port ไหน) — ไม่เปิด port จริง (compose/-p ทำ)
CMDคำสั่งรันตอน start (ต่างจาก RUN ที่รันตอน build)

ส่วนที่ 2: Multi-stage Build

# ===== Stage 1: build =====
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci                     # ติดตั้งครบ (รวม dev deps สำหรับ build)
COPY . .
RUN npm run build              # TS → dist/

# ===== Stage 2: run =====
FROM node:20-slim AS run
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev          # prod deps เท่านั้น (ไม่เอา dev)
COPY --from=build /app/dist ./dist   # copy แค่ผลลัพธ์จาก stage build
EXPOSE 3000
CMD ["node", "dist/server.js"]

ผล: image สุดท้ายมีแค่ Node slim + prod deps + dist — เล็กกว่า single-stage หลายเท่า · deploy เร็ว, ช่องโหว่น้อย (ไม่มี build tools/source ติดไป)

ส่วนที่ 3: .dockerignore

# .dockerignore (เหมือน .gitignore แต่สำหรับ docker build)
node_modules
.env
.git
dist
*.md
🔐 มุม pentester: ถ้าไม่มี .dockerignore แล้ว COPY . . จะก๊อป .env (secret), .git (history มี secret เก่า) เข้า image — ใครได้ image ไปก็ขุด secret ได้ · .dockerignore กันเรื่องนี้ + ทำ build เร็วขึ้น (ไม่ก๊อป node_modules ใหญ่ ๆ)

ส่วนที่ 4: build + run

docker build -t library-api .          # build จาก Dockerfile → image ชื่อ library-api
docker run -p 3000:3000 library-api    # รัน container (map port)
docker images                          # ดูขนาด image (เทียบ single vs multi-stage)

ลองเทียบขนาด: single-stage อาจ ~1GB+ · multi-stage slim อาจ ~150-200MB — ต่างกันชัดเจน

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: automation report service (Week 16) — Dockerfile multi-stage ทำให้ deploy ขึ้น cloud ได้ image เล็ก เร็ว · ลูกค้า/ทีม clone แล้ว build ได้เหมือนกัน ไม่ต้องไล่ตั้งค่า Node เวอร์ชัน

🌏 อุตสาหกรรม: ทุก app production deploy เป็น container · multi-stage + slim image เป็นมาตรฐาน (เล็ก = ถูก = ปลอดภัย) · "เขียน Dockerfile ที่ optimal ได้" เป็นสิ่งที่ JD DevOps/Backend ขอ และ pentester มอง image security (ไม่รัน root, ไม่มี secret) เป็นจุดเด่นของคุณ

📝 Quiz ท้ายบท

ข้อ 1: ทำไม copy package.json + npm ci ควรอยู่ก่อน copy source code?

Layer caching — deps เปลี่ยนน้อยกว่า code · ถ้า copy package ก่อน install แล้วค่อย copy code: แก้ code ไม่ทำให้ install ใหม่ (ใช้ cache layer install) build เร็ว · ถ้า copy . . ก่อน install: แก้ code นิดเดียว install ใหม่ทุกครั้ง ช้า

ข้อ 2: multi-stage build ทำให้ image เล็กลงยังไง?

stage build มี dev deps + source + build tools (ใหญ่) แต่ทิ้งหลัง build · stage run copy เฉพาะผลลัพธ์ (dist) + prod deps มา image เล็ก slim · image สุดท้ายไม่มีของที่ใช้แค่ตอน build → เล็กกว่าหลายเท่า, ปลอดภัยกว่า (ผิวโจมตีน้อย)

ข้อ 3 (security): ไม่มี .dockerignore แล้ว COPY . . อันตรายยังไง?

ก๊อป .env (secret), .git (history มี secret เก่า), node_modules (ใหญ่) เข้า image · ใครได้ image ไปขุด secret ได้ + image ใหญ่ช้า · .dockerignore กัน (เหมือน .gitignore) — ต้องมีเสมอ

ข้อ 4: npm ci ต่างจาก npm install ยังไง ทำไมใช้ใน Docker?

npm ci ติดตั้งจาก lockfile แบบ clean (ลบ node_modules ก่อน, ไม่แก้ lockfile) → reproducible เป๊ะและเร็วกว่า · เหมาะกับ CI/Docker ที่ต้องการ build ซ้ำได้เหมือนกันทุกครั้ง · npm install อาจอัปเดต lockfile (ไม่ deterministic)

✅ Checkpoint

  • เขียน Dockerfile (เข้าใจทุก instruction)
  • multi-stage build (build + run แยก) · image เล็ก
  • .dockerignore (กัน secret + ทำเร็ว)
  • build + run + เทียบขนาด image
