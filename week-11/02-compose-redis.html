Compose หลาย Service + Redis Caching

ต่อ app + db + redis เป็นระบบเดียว รันด้วยคำสั่งเดียว — และเร่งความเร็วด้วย cache

🔑 คีย์เวิร์ดภาพจำประจำบท

🍱 หลาย service คำสั่งเดียว = compose 📛 เรียกกันด้วยชื่อ = service name = hostname ⏳ รอ db ก่อน = depends_on ⚡ จำคำตอบไว้ = Redis cache

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "ทั้งระบบเป็นไฟล์เดียว — clone แล้วรันได้"

app จริงไม่ได้มีแค่ตัวเอง — ต้องมี database, cache, อาจมี queue · compose รวมทุก service ในไฟล์เดียว เพื่อนร่วมทีม clone → docker compose up → ได้ระบบครบเหมือนคุณ ไม่ต้องไล่ลง Postgres/Redis เอง · นี่คือ onboarding ที่เร็วที่สุด

หลักการ 2: "Service คุยกันด้วยชื่อ ไม่ใช่ localhost"

ใน compose แต่ละ service มี "ชื่อ" ที่ใช้เป็น hostname — app เชื่อม db ด้วย postgres://db:5432 (ชื่อ service "db") ไม่ใช่ localhost · เพราะแต่ละ container คือ "เครื่อง" แยกกัน localhost ของ app ไม่ใช่ของ db · Docker จัด network ให้คุยกันด้วยชื่อ

หลักการ 3: "Cache — จำคำตอบที่ถามบ่อย ลดภาระ DB"

query ที่เรียกบ่อยและไม่ค่อยเปลี่ยน (รายการหนังสือ, dashboard stats) — เก็บผลใน Redis (เร็วมาก, in-memory) ครั้งถัดไปดึงจาก cache ไม่ต้อง query DB · ลดภาระ DB, ตอบเร็วขึ้น · แต่ต้องจัดการ "cache เก่า" (invalidation) ให้ดี

ส่วนที่ 1: docker-compose หลาย service

services:
  app:
    build: .                       # build จาก Dockerfile ในโฟลเดอร์นี้
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://postgres:secret@db:5432/library  # "db" = ชื่อ service!
      REDIS_URL: redis://redis:6379
    depends_on:
      - db                         # รอ db เริ่มก่อน
      - redis

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: library
    volumes:
      - db_data:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine          # cache (in-memory)

volumes:
  db_data:
จุดที่มือใหม่งง: app เชื่อม DB ด้วย db:5432 ไม่ใช่ localhost:5432 — เพราะใน compose แต่ละ service เป็นเครื่องแยก · "db" คือ hostname ที่ Docker resolve ให้ · ใช้ localhost = app หา DB ไม่เจอ (localhost คือตัว app เอง)

ส่วนที่ 2: depends_on + healthcheck

app:
  depends_on:
    db:
      condition: service_healthy   # รอจน db "พร้อมจริง" ไม่ใช่แค่ "เริ่มแล้ว"
db:
  image: postgres:16
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U postgres"]
    interval: 5s
    retries: 5
กับดักคลาสสิก: depends_on เปล่า ๆ รอแค่ "container เริ่ม" ไม่ใช่ "DB พร้อมรับ connection" — app อาจต่อ DB ก่อน DB พร้อม → พัง · ใช้ condition: service_healthy + healthcheck รอจนพร้อมจริง (production ต้องมี)

ส่วนที่ 3: Redis Caching — pattern จริง

npm install ioredis

// cache pattern: เช็ค cache ก่อน → ไม่มีค่อย query DB → เก็บ cache
async function getBooks() {
  const cached = await redis.get("books:all");
  if (cached) return JSON.parse(cached);        // 1. เจอใน cache → ตอบเลย (เร็ว)

  const books = await pool.query("SELECT * FROM books");   // 2. ไม่เจอ → query DB
  await redis.set("books:all", JSON.stringify(books.rows), "EX", 60);  // 3. เก็บ 60 วิ
  return books.rows;
}

// ⚠️ invalidation: เมื่อข้อมูลเปลี่ยน ต้องลบ cache
async function createBook(data) {
  await pool.query("INSERT ...");
  await redis.del("books:all");                 // ลบ cache เก่า → ครั้งหน้าดึงใหม่
}
หลักคิดสำคัญ — "There are only two hard things: cache invalidation and naming": cache ที่ลืม invalidate = ผู้ใช้เห็นข้อมูลเก่า (เพิ่มหนังสือแล้วไม่โผล่) · กฎ: ทุกที่ที่ "เขียน" ข้อมูล ต้องลบ cache ที่เกี่ยวข้อง · เริ่มจาก cache แบบมี TTL (หมดอายุเอง เช่น 60 วิ) ปลอดภัยกว่า + เพิ่ม invalidation ตอน mutate

ส่วนที่ 4: ตัวอย่างประยุกต์

🏢 งานวี: dashboard การตลาด — ข้อมูลสรุปที่ดึงจาก API ภายนอก (ช้า, มี rate limit) เก็บ cache ใน Redis (เช่น 5 นาที) · ลูกค้าเปิด dashboard บ่อย ๆ ไม่ต้องยิง API ทุกครั้ง — เร็วขึ้นมาก + ไม่ชน rate limit · นี่คือ use case จริงของคุณ

🌏 อุตสาหกรรม: Redis เป็น cache มาตรฐานของทุกระบบ scale (session, cache, queue, rate limit) · compose หลาย service เป็นวิธี dev + test ระบบหลายส่วนในเครื่องเดียว · cache invalidation เป็นทักษะที่ interviewer ถามจริง (เพราะยาก)

📝 Quiz ท้ายบท

ข้อ 1: app เชื่อม DB ใน compose ทำไมใช้ db:5432 ไม่ใช่ localhost:5432?

แต่ละ service ใน compose เป็น container แยก (เหมือนคนละเครื่อง) · localhost ของ app = ตัว app เอง ไม่ใช่ db · Docker ให้ service คุยกันด้วย "ชื่อ service" เป็น hostname (db, redis) — Docker resolve ให้

ข้อ 2: depends_on เปล่า ๆ ไม่พอ ทำไม ต้องเพิ่มอะไร?

depends_on เปล่า รอแค่ "container เริ่ม" ไม่ใช่ "DB พร้อมรับ connection" · app อาจต่อ DB ก่อนพร้อม → พัง · เพิ่ม condition: service_healthy + healthcheck (pg_isready) รอจน DB พร้อมจริง

ข้อ 3: cache pattern 3 ขั้นคืออะไร?

① เช็ค cache — เจอ → ตอบเลย (เร็ว) ② ไม่เจอ → query DB ③ เก็บผลลง cache (มี TTL) · ครั้งถัดไปเจอใน cache ลดภาระ DB · เริ่มจาก TTL (หมดอายุเอง) ปลอดภัยกว่า

ข้อ 4: "cache invalidation" คืออะไร ทำไมยาก?

การลบ/อัปเดต cache เมื่อข้อมูลจริงเปลี่ยน — ลืม invalidate = ผู้ใช้เห็นข้อมูลเก่า (เพิ่มของแล้วไม่โผล่) · ยากเพราะต้องรู้ว่า "เขียนตรงไหน กระทบ cache ไหนบ้าง" ครบทุกจุด · กฎ: ทุกที่ที่เขียนข้อมูล ลบ cache ที่เกี่ยวข้อง + ใช้ TTL เป็นตาข่าย
