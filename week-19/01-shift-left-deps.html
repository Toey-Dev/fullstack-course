Week 19 · บทที่ 1

Shift Left + Dependency & Secret Scanning

หาช่องโหว่ตั้งแต่ต้นทาง — dependency ที่มีช่องโหว่ และ secret ที่เผลอ commit

🔑 คีย์เวิร์ดภาพจำประจำบท

⬅️ ตรวจตั้งแต่ต้น = shift left 📦 lib มีช่องโหว่ = npm audit / Dependabot 🔑 secret หลุด = gitleaks 🪝 ตรวจก่อน commit = pre-commit hook

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Shift Left — ยิ่งหาเร็ว ยิ่งถูก"

เจอช่องโหว่ตอน production = แพง (ผู้ใช้โดน, ต้อง hotfix, อาจข้อมูลรั่ว) · เจอตอน PR = ถูก (แก้ก่อน merge) · เจอตอนพิมพ์ = ถูกสุด · "shift left" = ขยับการตรวจไปต้นทาง (ซ้ายของ timeline) · automated scan ใน CI ทำให้ตรวจทุก PR โดยไม่พึ่งคนจำ · นี่คือหัวใจ DevSecOps

หลักการ 2: "โค้ดเราอาจปลอดภัย แต่ dependency ไม่ — A06"

app สมัยใหม่พึ่ง dependency หลายร้อยตัว (node_modules) · แต่ละตัวอาจมีช่องโหว่ (CVE) ที่เรารับมาโดยไม่รู้ · npm audit/Dependabot สแกน dependency เทียบฐานข้อมูลช่องโหว่ → เตือน + เสนอเวอร์ชันที่แก้แล้ว · คุณเขียน code ดีแค่ไหน ถ้า lib รั่วก็โดน (supply chain)

หลักการ 3: "Secret ที่ commit แล้ว = หลุดถาวร"

เผลอ commit .env/key → push → อยู่ใน git history ตลอด (แม้ลบ commit ทีหลัง) · bot สแกน public repo เจอใน <1 นาที (W4) · secret scanning (gitleaks) จับก่อน commit/push · ถ้าหลุดแล้ว = ต้อง rotate (เปลี่ยน secret) ไม่ใช่แค่ลบ commit

ส่วนที่ 1: Shift Left — ตรวจที่ไหนคุ้มสุด

ส่วนที่ 2: Dependency Scanning

# npm audit — สแกน dependency เทียบ CVE database
npm audit                          # ดูช่องโหว่
npm audit --audit-level=high       # เฉพาะ high+ (ใช้ใน CI gate)
npm audit fix                      # แก้อัตโนมัติ (อัปเดตเวอร์ชันที่ปลอดภัย)

# ใน GitHub Actions
- run: npm audit --audit-level=high   # fail ถ้ามี high/critical

Dependabot (ฟรีใน GitHub) — สแกน dependency อัตโนมัติ + เปิด PR อัปเดตให้:

# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: npm
    directory: "/"
    schedule: { interval: weekly }
วิธีจัดการ: Critical/High → แก้ก่อน (audit fix / อัปเดต major ถ้าจำเป็น) · ถ้าแก้ไม่ได้ทันที → ประเมินว่า "ช่องโหว่นี้ exploitable ใน context เราไหม" (มุม pentester) + บันทึก · อย่าเพิกเฉย Critical

ส่วนที่ 3: Secret Scanning (gitleaks)

# gitleaks — สแกนหา secret ที่เผลอ commit (รวม git history)
gitleaks detect --source .         # สแกนทั้ง history

# ใน GitHub Actions
- uses: gitleaks/gitleaks-action@v2
  env: { GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }} }

gitleaks หา pattern ของ secret (API key, token, private key, password) ในโค้ด+history · GitHub เองก็มี secret scanning ในตัว (push protection):

🔐 ถ้า secret หลุดแล้ว (สำคัญมาก): ลบ commit ไม่พอ — secret ยังอยู่ใน git history + อาจมีคน clone ไปแล้ว · ต้อง rotate ทันที (สร้าง key/token ใหม่ ยกเลิกอันเก่า) · นี่คือสิ่งที่คุณเคยเจอตอน pentest (เจอ key ใน repo เก่า) — รู้ว่าต้อง rotate ไม่ใช่แค่ลบ

ส่วนที่ 4: Pre-commit Hooks (ตรวจก่อน commit)

npm install -D husky lint-staged

# .husky/pre-commit — รันก่อน commit ทุกครั้ง
npx lint-staged              # lint + format เฉพาะไฟล์ที่แก้
gitleaks protect --staged    # สแกน secret ก่อน commit

pre-commit hook = ตรวจตั้งแต่ "ก่อน commit" (ซ้ายสุดของ shift left) — จับ secret/lint ก่อนเข้า git ด้วยซ้ำ · เร็วและถูกสุด · เป็นด่านแรกก่อน CI

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: automation service ที่มี API key ลูกค้า — gitleaks กัน key หลุด (pre-commit + CI), npm audit กัน dependency ที่ดึงข้อมูลรั่ว · Dependabot อัปเดต lib ที่มีช่องโหว่อัตโนมัติ · ระบบที่จัดการข้อมูลลูกค้าต้องมี security scanning เป็นพื้นฐาน

🌏 อุตสาหกรรม: dependency + secret scanning เป็น baseline DevSecOps ทุกที่ · supply chain attack (lib ถูกฝังมัลแวร์) เพิ่มขึ้นมาก → การ scan dependency จำเป็น · "ตั้ง security scanning ใน CI เป็น" คือทักษะที่ JD DevSecOps ขอตรง ๆ

📝 Quiz ท้ายบท

ข้อ 1: "shift left" คืออะไร ทำไมถูกกว่า?

ขยับการตรวจ security ไปต้นทาง (ซ้ายของ timeline: pre-commit/PR แทน production) · ยิ่งเจอเร็วยิ่งถูก — เจอตอน PR แก้ง่าย, เจอตอน production แพง (hotfix, ผู้ใช้โดน), โดนเจาะแพงสุด · automated scan ใน CI = ตรวจทุก PR ไม่พึ่งคนจำ

ข้อ 2: โค้ดเราปลอดภัยแต่ทำไมยังต้อง npm audit?

app พึ่ง dependency หลายร้อยตัว แต่ละตัวอาจมี CVE ที่เรารับมาโดยไม่รู้ (A06 OWASP / supply chain) · npm audit สแกนเทียบ CVE database เตือน + เสนอเวอร์ชันที่แก้แล้ว · เขียน code ดีแค่ไหน lib รั่วก็โดน

ข้อ 3 (security): เผลอ commit API key แล้วลบ commit — พอไหม?

ไม่พอ — secret ยังอยู่ใน git history (แม้ลบ commit ล่าสุด) + อาจมีคน clone/bot สแกนไปแล้ว · ต้อง rotate (สร้าง key ใหม่ ยกเลิกเก่า) ทันที · ลบ commit ไม่ทำให้ key ที่หลุดปลอดภัย · gitleaks/pre-commit ป้องกันตั้งแต่แรก

ข้อ 4: scanner เตือน false positive ควรทำยังไง (ไม่ใช่ปิด scan)?

ประเมินว่าจริงไหม (มุม pentester) → ถ้า false positive จริง ignore เฉพาะจุดนั้น + comment เหตุผล (audit trail) · ไม่ปิด scan ทั้งหมด (จะพลาด true positive อื่น) · ทักษะ "อ่านผล scanner แล้วตัดสินถูก" คือคุณค่าของ AppSec

✅ Checkpoint

  • เข้าใจ shift left (ตรวจต้นทางถูกกว่า)
  • npm audit + Dependabot (dependency scanning)
  • gitleaks (secret scanning) + เข้าใจต้อง rotate ถ้าหลุด
  • pre-commit hooks (ด่านแรก)
  • Quiz + โน้ต notes/shift-left.md
