SAST (Semgrep) + Security Pipeline

ให้เครื่องอ่านโค้ดหา pattern อันตราย — และรวมทุก scan เป็น security gate

🔑 คีย์เวิร์ดภาพจำประจำบท

🔬 อ่านโค้ดหา pattern อันตราย = SAST 🏃 ทดสอบ app ที่รันอยู่ = DAST 🛡 Semgrep = SAST ที่เขียน rule เองได้ 🚧 security gate = scan ไม่ผ่าน = ไม่ merge

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "SAST = อ่านโค้ด (ไม่ต้องรัน) หา pattern อันตราย"

SAST (Static Application Security Testing) วิเคราะห์ source code หา pattern ที่มักเป็นช่องโหว่ — SQL ต่อ string, eval(input), secret hardcode, dangerouslySetInnerHTML · ไม่ต้องรัน app · เร็ว ใส่ใน CI ได้ · เหมือนมี security reviewer อ่านทุก PR · Semgrep เป็นตัวนิยม (เขียน rule เองได้)

หลักการ 2: "SAST vs DAST — มองคนละมุม ใช้คู่กัน"

SAST อ่านโค้ด (รู้โครงสร้างภายใน แต่ไม่รู้ runtime behavior) · DAST (Dynamic) ทดสอบ app ที่รันอยู่ (เหมือนผู้โจมตียิงจริง — ZAP, Burp ที่คุณเคยใช้!) แต่ไม่เห็นโค้ด · ใช้คู่กัน: SAST จับตอน build, DAST จับตอนรัน · คุณคุ้น DAST จาก pentest — SAST เป็นมุมเสริม

หลักการ 3: "Security gate — scan ไม่ผ่าน = หยุด ไม่ merge"

scan แล้วเตือนแต่ merge ได้ = ไร้ค่า (เหมือน CI ที่ไม่ branch protect) · ทำให้ security scan เป็น "required check" — High/Critical = fail = merge ไม่ได้ · บังคับให้ช่องโหว่ถูกแก้ก่อนเข้า main · นี่คือสิ่งที่ทำให้ DevSecOps มีฟัน ไม่ใช่แค่รายงาน

ส่วนที่ 1: SAST vs DAST

SAST (static)DAST (dynamic)
ทำงานอ่าน source code (ไม่รัน)ยิง request ใส่ app ที่รันอยู่
เห็นโครงสร้างภายใน, ทุก pathพฤติกรรมจริง runtime
จับได้ดีinjection pattern, secret, unsafe APIauth bypass, config จริง, runtime XSS
เครื่องมือSemgrep, CodeQLOWASP ZAP, Burp (คุณเคยใช้!)
เมื่อตอน build/CI (เร็ว)ตอน app รัน (staging)

SAST + DAST เสริมกัน — SAST จับ "โค้ดเขียนอันตราย", DAST จับ "ระบบที่รันมีช่องโหว่" · Week 19 เน้น SAST (ใน CI), DAST คุณคุ้นจาก pentest (W20 จะใช้)

ส่วนที่ 2: Semgrep

# Semgrep — SAST ที่เขียน rule เองได้ (มี ruleset สำเร็จรูป)
semgrep --config auto .             # ใช้ rule อัตโนมัติ (OWASP, security)

# ใน GitHub Actions
- uses: semgrep/semgrep-action@v1
  with:
    config: p/javascript p/owasp-top-ten   # ruleset

Semgrep เจอ pattern เช่น:

// Semgrep เตือน: SQL injection
db.query("SELECT * FROM users WHERE id = " + userId);   // ❌ ต่อ string

// Semgrep เตือน: hardcoded secret
const apiKey = "sk-1234567890abcdef";                   // ❌ secret ในโค้ด

// Semgrep เตือน: XSS
element.innerHTML = userInput;                          // ❌ unsafe
เขียน custom rule: Semgrep ให้เขียน rule เฉพาะทีม (เช่น "ห้ามใช้ฟังก์ชัน X", "ทุก route ต้องมี auth middleware") — บังคับ secure pattern ของทีมอัตโนมัติ · มุม AppSec: คุณเขียน rule จากช่องโหว่ที่เคยเจอ → กันไม่ให้เกิดซ้ำทั้งทีม

ส่วนที่ 3: รวมเป็น Security Pipeline

# .github/workflows/security.yml
name: Security
on: [pull_request]
jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm audit --audit-level=high          # dependency
      - uses: gitleaks/gitleaks-action@v2          # secret
      - uses: semgrep/semgrep-action@v1            # SAST
        with: { config: p/owasp-top-ten }
      # ทั้ง 3 ต้องผ่าน → PR ถึง merge ได้ (branch protection W13)

ส่วนที่ 4: จัดการ False Positive

// Semgrep: ignore เฉพาะบรรทัด + เหตุผล (ไม่ปิดทั้ง rule)
// nosemgrep: rule-id — เหตุผลว่าทำไมปลอดภัยในเคสนี้
const query = buildSafeQuery(input);   // nosemgrep: sql-injection (ใช้ query builder ที่ escape แล้ว)

// npm audit: ถ้าแก้ไม่ได้ทันที — บันทึก + ประเมิน exploitability
หลักการจัดการ false positive: ① ประเมินว่าจริงไหม (true/false positive) ② ถ้า false — ignore เฉพาะจุด + comment · ③ ไม่ปิดทั้ง rule (จะพลาด true positive อื่น) ④ มี audit trail (ทำไม ignore) · มุม AppSec: ความสามารถแยก signal จาก noise = คุณค่าหลัก ไม่ใช่แค่รัน scanner

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: security pipeline ให้ automation service — Semgrep จับ pattern ต่อ SQL string/hardcode key, gitleaks กัน API key หลุด, npm audit กัน lib รั่ว · ทุก PR ผ่าน security gate ก่อน merge · custom Semgrep rule บังคับ "ทุก fetch ต้อง validate URL" (กัน SSRF จาก W17)

🌏 อุตสาหกรรม: SAST (Semgrep/CodeQL) + dependency + secret scanning ใน CI เป็น DevSecOps มาตรฐาน · security gate คือสิ่งที่บังคับคุณภาพ · "ตั้ง + tune security pipeline + จัดการ false positive เป็น" คือทักษะหลักของ AppSec/DevSecOps engineer — สาย B ของคุณ

📝 Quiz ท้ายบท

ข้อ 1: SAST vs DAST ต่างกันยังไง ใช้คู่กันได้ยังไง?

SAST อ่าน source code (ไม่รัน) — จับ pattern อันตราย (injection, secret) เห็นทุก path · DAST ยิง app ที่รันอยู่ (ZAP/Burp) — จับช่องโหว่ runtime (auth bypass, config จริง) แต่ไม่เห็นโค้ด · คู่กัน: SAST ตอน build, DAST ตอนรัน — ครอบคลุมทั้งสองมุม

ข้อ 2: Semgrep จับช่องโหว่แบบไหนได้ ตัวอย่าง?

pattern อันตรายในโค้ด: SQL ต่อ string (injection), hardcoded secret, innerHTML=input (XSS), eval(input) · + เขียน custom rule บังคับ secure pattern ของทีม (เช่น ทุก route ต้องมี auth) · เป็น "security reviewer อัตโนมัติ" ทุก PR

ข้อ 3: ทำไม security scan ต้องเป็น "gate" (required check)?

scan แล้วเตือนแต่ merge ได้ = ไร้ค่า (คนเพิกเฉย) · gate: High/Critical = fail = merge ไม่ได้ (branch protection W13) → บังคับแก้ช่องโหว่ก่อนเข้า main · ทำให้ DevSecOps "มีฟัน" ไม่ใช่แค่รายงานที่ไม่มีใครอ่าน

ข้อ 4: scanner false positive จัดการยังไงให้ถูกหลัก?

① ประเมิน true/false ② false → ignore เฉพาะจุด (nosemgrep + comment เหตุผล) ③ ไม่ปิดทั้ง rule (พลาด true positive อื่น) ④ audit trail · ความสามารถแยก signal/noise = คุณค่า AppSec ไม่ใช่แค่รัน scanner เปล่า

✅ Checkpoint

  • เข้าใจ SAST vs DAST (ใช้คู่กัน)
  • Semgrep ใน CI (ruleset + custom rule)
  • security pipeline รวม (audit+gitleaks+semgrep) เป็น gate
  • จัดการ false positive รายจุด (ไม่ปิดทั้งหมด)
