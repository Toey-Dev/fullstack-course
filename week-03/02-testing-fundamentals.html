Testing Fundamentals — ทำไมต้อง test + Vitest

ทักษะที่ dev ส่วนใหญ่ในตลาดไทยไม่มี — และเป็นสิ่งที่ resume คุณสัญญาไว้

🔑 คีย์เวิร์ดภาพจำประจำบท

🛡 ตาข่ายนิรภัย = test ให้ refactor กล้า 🔺 ฐานกว้างยอดแคบ = test pyramid 📋 เตรียม-ทำ-ตรวจ = AAA 🔴🟢 แดงก่อนเขียว = TDD 📊 coverage = วัดว่าทดสอบครอบแค่ไหน

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Test เป็นตาข่ายนิรภัย ไม่ใช่ภาระ"

มือใหม่คิดว่า test = งานเพิ่ม เสียเวลา — มืออาชีพรู้ว่า test คือสิ่งที่ทำให้ กล้าแก้โค้ด ไม่มี test = แก้อะไรนิดเดียวก็กลัวพังที่อื่น ต้องไล่กดเทสต์มือทุกหน้า / มี test = แก้แล้วรัน 1 คำสั่งรู้ทันทีว่าพังตรงไหน ยิ่งโปรเจกต์โต test ยิ่งมีค่า — library-system ที่จะโตอีก 20 สัปดาห์ต้องมี test ตั้งแต่ตอนนี้

หลักการ 2: "Test พฤติกรรม ไม่ใช่ implementation"

test ที่ดีถามว่า "ใส่ x เข้าไป ได้ y ออกมาไหม" (พฤติกรรม) ไม่ใช่ "ฟังก์ชันเรียก method นี้กี่ครั้ง" (ภายใน) — เพราะถ้า test ผูกกับ implementation พอ refactor (เปลี่ยนวิธีข้างใน ผลเหมือนเดิม) test จะพังทั้งที่โค้ดถูก = test ที่เปราะ ไร้ค่า

หลักการ 3: "Test คือเอกสารที่รันได้และไม่มีวันล้าสมัย"

comment โกหกได้ (โค้ดเปลี่ยนแต่ลืมแก้ comment) — แต่ test โกหกไม่ได้ ถ้าพฤติกรรมเปลี่ยน test แดงทันที dev ใหม่เข้าทีมอ่าน test แล้วเข้าใจว่าระบบควรทำงานยังไง เร็วกว่าอ่าน implementation

เชื่อมกับ resume: คุณเขียนว่า "Conducted software testing including test case creation and API testing" — Week 3 ทำให้คุณพูดได้เต็มปากว่าทำ automated test เป็น ไม่ใช่แค่กดทดสอบมือ (ซึ่งคนละระดับในสายตา interviewer)

ส่วนที่ 1: Test Pyramid — ทดสอบอะไร แค่ไหน

ระดับทดสอบอะไรเครื่องมือ (คอร์สนี้)
Unitฟังก์ชันเดียวโดด ๆ (เช่น addVat, validateEmail)Vitest (บทนี้)
Integration/APIหลายส่วนทำงานร่วม (route + controller + data)Supertest (Mission)
E2Eทั้งระบบเหมือนผู้ใช้จริง (เปิด browser คลิก)Playwright (เฟสหลัง)

หลักคิด: เขียน unit เยอะ ๆ (เร็ว รันทุกครั้งที่ save) · API test ครอบ flow สำคัญ · E2E แค่ flow critical (login, จ่ายเงิน) เพราะช้าและเปราะ

ส่วนที่ 2: เขียน Unit Test แรกด้วย Vitest

npm install -D vitest      # ติดตั้ง (devDependency)

// package.json
"scripts": { "test": "vitest" }

สมมติมีฟังก์ชันใน src/utils/vat.js:

// src/utils/vat.js
export const addVat = (price) => {
  if (typeof price !== "number" || price < 0) throw new Error("ราคาไม่ถูกต้อง");
  return Math.round(price * 1.07);
};

เขียน test ใน src/utils/vat.test.js (ลงท้าย .test.js Vitest หาเจอเอง):

import { describe, it, expect } from "vitest";
import { addVat } from "./vat.js";

describe("addVat", () => {
  it("บวก VAT 7% แล้วปัดเศษ", () => {
    expect(addVat(100)).toBe(107);     // Arrange+Act+Assert ในบรรทัดเดียว
  });

  it("ปัดเศษถูกต้อง", () => {
    expect(addVat(99)).toBe(106);      // 99*1.07 = 105.93 → 106
  });

  it("โยน error เมื่อราคาติดลบ", () => {
    expect(() => addVat(-50)).toThrow("ราคาไม่ถูกต้อง");
  });

  it("โยน error เมื่อไม่ใช่ตัวเลข", () => {
    expect(() => addVat("100")).toThrow();
  });
});

คำศัพท์ที่ต้องรู้: describe = กลุ่ม test · it/test = 1 เคส · expect(x).toBe(y) = ตรวจว่า x เท่ากับ y · .toThrow() = ตรวจว่าโยน error

matcherใช้ตรวจ
toBe(x)ค่าเท่ากันเป๊ะ (number, string, boolean)
toEqual(obj)object/array เนื้อในเท่ากัน
toThrow()โยน error
toBeTruthy() / toBeNull()เช็คความจริง/null
toContain(x)array/string มี x

ส่วนที่ 3: คิดเคสให้ครบ — "ทดสอบอะไรบ้าง"

มือใหม่เขียน test แค่ "ทางสำเร็จ" — มืออาชีพคิดครบ 3 กลุ่ม (นี่คือสิ่งที่ AI มักลืม → AI Review Drill สัปดาห์นี้):

กลุ่มตัวอย่าง (addVat)
✅ Happy pathราคาปกติ → ได้ผลถูก
⚠️ Edge case (ขอบ)ราคา 0, ทศนิยมที่ต้องปัด, เลขใหญ่มาก
❌ Error case (พัง)ติดลบ, ไม่ใช่ตัวเลข, undefined, null
หลักคิด: ถามตัวเอง "ผู้ใช้/ระบบอื่นจะส่งอะไรแปลก ๆ มาได้บ้าง" — ทุกอย่างที่นึกออกควรมี test คุณในฐานะอดีต pentester ได้เปรียบ เพราะคิดเคส "ป่วน" เป็นนิสัยอยู่แล้ว

ส่วนที่ 4: TDD เบื้องต้น — แดงก่อนเขียว

Test-Driven Development = เขียน test ก่อนเขียนโค้ดจริง วน 3 จังหวะ:

  1. 🔴 Red — เขียน test ที่ยังต้องพัง (เพราะยังไม่มีโค้ด)
  2. 🟢 Green — เขียนโค้ดน้อยที่สุดให้ test ผ่าน
  3. 🔵 Refactor — ปรับโค้ดให้สวย (test เป็นตาข่ายกันพัง)

ไม่ต้องทำ TDD ทุกครั้งในชีวิตจริง — แต่เข้าใจวงจรนี้ช่วยให้คิด "ผลลัพธ์ที่ต้องการ" ก่อนเขียนโค้ด และ interviewer ถามบ่อย

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: ฟังก์ชันแปลงข้อมูล GA4/Meta ใน automation report — แต่ละตัวควรมี unit test (ใส่ข้อมูลตัวอย่าง → เช็คว่าสรุปถูก) พอแก้ logic แล้วรัน test รู้ทันทีว่าตัวเลขยังถูกไหม ไม่ต้องไล่เช็คมือทุกลูกค้า

🌏 อุตสาหกรรม: ทุกบริษัทที่จ่าย 100k+ บังคับ test — PR ที่ไม่มี test ไม่ผ่าน review, CI (Week 13) รัน test อัตโนมัติทุก push ถ้าแดง = merge ไม่ได้ test คือ "ใบเบิกทาง" เข้าทีมจริงจัง

📝 Quiz ท้ายบท

ข้อ 1: ทำไม unit test ควรมีเยอะกว่า E2E test (test pyramid)?

unit test เร็ว (มิลลิวินาที) ถูก เขียนง่าย รันได้ทุกครั้งที่ save · E2E ช้า (เปิด browser จริง) เปราะ (UI เปลี่ยนนิดเดียวพัง) แพงในการดูแล → เขียน unit เยอะครอบ logic ละเอียด, E2E แค่ flow สำคัญสุด (login, จ่ายเงิน)

ข้อ 2: "test พฤติกรรม ไม่ใช่ implementation" หมายความว่าอะไร ทำไมสำคัญ?

ทดสอบ "ใส่ x ได้ y" (ผลลัพธ์) ไม่ใช่ "เรียก method ภายในกี่ครั้ง" — เพราะถ้าผูกกับ implementation พอ refactor (เปลี่ยนวิธีข้างใน ผลเหมือนเดิม) test จะพังทั้งที่โค้ดถูก test ที่ดีต้องรอด refactor = ทดสอบสัญญา (input→output) ไม่ใช่วิธีการ

ข้อ 3: เขียน test ให้ addVat ควรมีกี่กลุ่มเคส ยกตัวอย่าง

3 กลุ่ม: happy (ราคาปกติ→ถูก), edge (0, ทศนิยมปัดเศษ, เลขใหญ่), error (ติดลบ, ไม่ใช่ตัวเลข, null/undefined) — เขียนแค่ happy path = test ที่ดูเหมือนผ่านแต่ระบบพังจริงตอนเจอ input แปลก

ข้อ 4: TDD 3 จังหวะคืออะไร และ "Red" ก่อนมีประโยชน์ยังไง?

🔴 Red (เขียน test ที่พังก่อน) → 🟢 Green (เขียนโค้ดให้ผ่าน) → 🔵 Refactor (ปรับสวย) · Red ก่อนพิสูจน์ว่า test "ทำงานจริง" (ถ้าเขียนโค้ดก่อนแล้ว test เขียว อาจเป็นเพราะ test ไม่เคยเช็คอะไรเลย) + บังคับคิดผลลัพธ์ที่ต้องการก่อนเขียน

✅ Checkpoint

  • ติดตั้ง Vitest + เขียน unit test แรกผ่าน (addVat หรือฟังก์ชันของตัวเอง)
  • test ครบ 3 กลุ่ม (happy/edge/error) อย่างน้อย 1 ฟังก์ชัน
  • อธิบาย test pyramid + "test พฤติกรรม" ได้
