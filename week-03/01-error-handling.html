Week 3 · บทที่ 1

Error Handling + 404 — ทำ API ให้ทนทาน

เปลี่ยน API จาก "ทำงานได้ตอนทุกอย่างปกติ" เป็น "ทำงานได้แม้เจอเรื่องร้าย"

🔑 คีย์เวิร์ดภาพจำประจำบท

🕳 ตกท่อ = error handler 4 args 🗺 หลงทาง = 404 handler 📮 next(err) = ส่งต่อให้คนเก็บ error 🔇 ไม่รั่วข้อมูล = error message ปลอดภัย 🧹 ที่เดียวจบ = centralized

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "DRY error handling" — อย่าซ้ำตัวเอง

ถ้าทุก route เขียน try/catch + ตอบ error เองหมด = โค้ดเดิมซ้ำ 50 จุด วันหน้าอยากเปลี่ยนรูปแบบ error ต้องแก้ 50 ที่ Express ออกแบบให้มี "จุดรวมจัดการ error" จุดเดียว (centralized) — error ทุกที่ส่งมารวมที่นี่ จัดการแบบเดียวจบ

หลักการ 2: "Fail loud internally, fail quiet externally"

error จริง (stack trace, ชื่อ DB, query ที่พัง) ต้อง log ละเอียดให้ทีมเห็น เพื่อ debug — แต่ที่ตอบกลับผู้ใช้ต้อง คลุมเครือ ("เกิดข้อผิดพลาด ลองใหม่") เพราะรายละเอียดภายในคือของขวัญให้ผู้โจมตี (คุณในฐานะ pentester เคยใช้ error message รั่ว ๆ เก็บข้อมูลระบบมาแล้ว — ตอนนี้อยู่ฝั่งปิด)

หลักการ 3: "Errors are part of the contract" — error คือส่วนหนึ่งของ API

API ที่ดีไม่ได้บอกแค่ "สำเร็จหน้าตาแบบไหน" แต่บอกด้วยว่า "พังหน้าตาแบบไหน" (status + รูปแบบ error) frontend (Week 1 ของคุณ) ต้องพึ่ง contract นี้ในการแสดงผล — error ที่ไม่เป็นระบบ = frontend จัดการไม่ได้

ส่วนที่ 1: ปัญหา — try/catch ซ้ำทุก route

// ❌ ทุก route ทำซ้ำแบบนี้ — เยอะ ซ้ำ แก้ยาก
app.get("/api/books/:id", (req, res) => {
  try {
    const book = findBook(req.params.id);
    if (!book) return res.status(404).json({ ok: false, error: "not found" });
    res.json({ ok: true, data: book });
  } catch (err) {
    console.error(err);
    res.status(500).json({ ok: false, error: "server error" });  // ← ซ้ำทุก route
  }
});

ลองนึกภาพมี 20 endpoint — block catch ข้างล่างนี้ถูก copy 20 รอบ วันหน้าอยากเพิ่ม log แบบใหม่ = แก้ 20 ที่ ลืมที่เดียว = บั๊ก

ส่วนที่ 2: ทางแก้ — Centralized Error Handler

Express มี middleware พิเศษที่มี 4 พารามิเตอร์ (err, req, res, next) — Express รู้ว่านี่คือ "คนเก็บ error" โดยอัตโนมัติ:

// 1. route โยน error (ไม่ต้อง catch เอง)
app.get("/api/books/:id", (req, res, next) => {
  const book = findBook(req.params.id);
  if (!book) {
    const err = new Error("Book not found");
    err.status = 404;              // แปะ status ที่อยากตอบ
    return next(err);              // ⭐ ส่งต่อให้ error handler
  }
  res.json({ ok: true, data: book });
});

// 2. Error Handler — วางท้ายสุด หลังทุก route (4 args!)
app.use((err, req, res, next) => {
  console.error(err.stack);                          // log จริง (ทีมเห็น)
  const status = err.status || 500;
  res.status(status).json({
    ok: false,
    error: status === 500 ? "Internal error" : err.message,  // 500 ไม่รั่วรายละเอียด
  });
});
กฎตำแหน่ง: error handler ต้องวาง ท้ายสุด หลังทุก route และต้องมีครบ 4 args — ถ้าเขียน 3 args Express จะนึกว่าเป็น middleware ธรรมดา ไม่ใช่ error handler (บั๊กที่หาเจอยาก)

ส่วนที่ 3: async error — กับดักที่ต่างจาก sync

Express 4 ไม่จับ error จาก async function อัตโนมัติ — ถ้า await พังใน route ต้อง catch แล้ว next เอง:

// ❌ async พัง = Unhandled rejection (Express 4 ไม่ส่งไป error handler)
app.get("/api/books", async (req, res) => {
  const books = await db.getBooks();   // ถ้าพังตรงนี้...
  res.json(books);                     // ...handler ข้างล่างไม่ถูกเรียก
});

// ✅ จับแล้ว next(err)
app.get("/api/books", async (req, res, next) => {
  try {
    const books = await db.getBooks();
    res.json({ ok: true, data: books });
  } catch (err) { next(err); }         // ส่งต่อให้ centralized handler
});

เกร็ด production: ทีมจริงมักใช้ตัวห่อ (wrapper) asyncHandler หรือ Express 5 (จับ async ให้อัตโนมัติ) เพื่อไม่ต้องเขียน try/catch ทุก route — แต่เข้าใจกลไกพื้นฐานก่อนค่อยใช้ของลัด

// 🎁 ของลัดที่ทีมใช้: asyncHandler ห่อ route ให้ catch อัตโนมัติ
const asyncHandler = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);

app.get("/api/books", asyncHandler(async (req, res) => {
  const books = await db.getBooks();   // พังเมื่อไหร่ catch → next เองอัตโนมัติ
  res.json({ ok: true, data: books });
}));

ส่วนที่ 4: 404 Handler — เมื่อไม่มี route ไหนรับ

// วางก่อน error handler — จับทุก request ที่ไม่ตรง route ไหนเลย
app.use((req, res) => {
  res.status(404).json({ ok: false, error: `ไม่พบ ${req.method} ${req.url}` });
});

ถ้าไม่มี 404 handler — request ที่หลง (เช่น /api/bookssss) จะได้ HTML error หน้าตาแปลก ๆ ของ Express ซึ่ง frontend parse ไม่ได้ (คาดหวัง JSON) → ใส่ 404 handler ทำให้ API ตอบ JSON สม่ำเสมอแม้ path ผิด

ลำดับที่ถูกต้องใน server.js: ① middleware (json, logger) → ② routes → ③ 404 handler → ④ error handler (ท้ายสุด) — ลำดับนี้สำคัญมาก สลับแล้วพัง

ส่วนที่ 5: ตัวอย่างประยุกต์ในงานจริง

🏢 งานวี — automation report: เวลา API ดึง GA4 แล้ว token หมดอายุ (พังกลางทาง) — centralized error handler จับได้ที่เดียว ตอบ frontend ว่า "เชื่อมต่อ GA4 ไม่สำเร็จ" (ไม่รั่วว่า token อะไรหมด) พร้อม log error จริงให้คุณ debug — นี่คือความต่างของ tool ภายในที่ใช้ได้จริงกับ script ที่พังเงียบ

🌏 อุตสาหกรรม: ทุก API ระดับ production มี error handler กลาง + ระบบ log (Sentry — Week 15) — เพราะเมื่อมี request ล้านครั้ง/วัน คุณต้องรู้ว่าพังกี่ครั้ง พังที่ไหน โดยไม่ให้ผู้ใช้เห็นรายละเอียดภายใน

🧹 Clean Code: error handling แบบ production

❌ ตอบ error ดิบ

catch (err) {
  res.status(500).json({ error: err.message });
  // err.message อาจมี: "ECONNREFUSED 10.0.0.5:5432"
  // → บอก IP database ให้ผู้โจมตี!
}

✅ log จริง + ตอบปลอดภัย

catch (err) {
  logger.error(err);   // รายละเอียดเต็มให้ทีม
  res.status(500).json({
    ok: false,
    error: "เกิดข้อผิดพลาด กรุณาลองใหม่",  // user เห็นแค่นี้
  });
}
กฎทอง: error handler เดียวทั้งระบบ · log ภายในละเอียด ตอบ user คลุมเครือ · 500 ห้ามรั่วรายละเอียด · ลำดับ middleware ถูกต้อง (404 ก่อน error handler ท้ายสุด)

📝 Quiz ท้ายบท

ข้อ 1: error handler ของ Express ต่างจาก middleware ธรรมดายังไง — Express รู้ได้ไง?

นับจากจำนวน argument — middleware ธรรมดา 3 args (req,res,next) / error handler 4 args (err,req,res,next) เขียน 4 args Express จะส่ง error มาที่นี่อัตโนมัติ ถ้าเผลอเขียน 3 args จะกลายเป็น middleware ธรรมดา error ไม่เข้า (บั๊กหายาก)

ข้อ 2: ทำไม error handler ต้องวางท้ายสุดของ server.js?

เพราะ Express ทำงาน middleware ตามลำดับที่เขียน — error handler ต้องอยู่หลังทุก route เพื่อ "ดักของที่ตกลงมา" จากข้างบนทั้งหมด ถ้าวางบนสุด route ที่อยู่ข้างล่างจะส่ง error มาไม่ถึง

ข้อ 3 (security): route ดึง DB แล้วพัง ตอบ { error: err.message } ตรง ๆ อันตรายยังไง?

err.message ของ DB มักมีรายละเอียดภายใน เช่น ชื่อ table/column, IP database, SQL ที่พัง — ผู้โจมตีใช้ข้อมูลนี้ทำ recon วางแผนเจาะต่อ (เช่น รู้ว่ามี table users ก็ลอง SQL injection) ตอบแค่ "Internal error" + log จริงไว้ฝั่ง server

ข้อ 4: async route ใน Express 4 พังแล้ว error handler ไม่ทำงาน เพราะอะไร แก้ยังไง?

Express 4 ไม่จับ rejection จาก async function อัตโนมัติ → error หลุดเป็น Unhandled rejection ไม่ถึง handler แก้: try/catch แล้ว next(err) เอง หรือใช้ wrapper asyncHandler หรืออัปเป็น Express 5 (จับ async ให้)

✅ Checkpoint

  • ใส่ centralized error handler + 404 handler ใน library-system (ลำดับถูก)
  • แปลง route ที่มี try/catch ซ้ำ ให้ใช้ next(err)
  • error 500 ไม่รั่วรายละเอียดภายใน
