ไฮไลต์ของคอร์ส — เอา library-system มาเจาะเอง แล้วเขียนรายงาน attack→fix · จุดแข็งที่สุดของคุณ

🎯 เป้าหมายของสัปดาห์นี้ (ปิดเฟส 3 — สัปดาห์ที่คุณรอ)

20 สัปดาห์ที่ผ่านมาคุณ "สร้างกำแพง" — Week 20 กลับสู่ตัวตนเดิม: เป็นผู้โจมตี เจาะ library-system ของตัวเอง · ทำ pentest เต็มรูปแบบตาม methodology ที่คุณถนัด (recon → exploit → report) แล้วเขียน รายงาน attack→fix อย่างน้อย 5 ช่องโหว่ · นี่คือ portfolio piece ที่ทรงพลังที่สุดในตลาด — แทบไม่มี dev คนไหนมี "pentest report ของระบบที่ตัวเองสร้าง"

  • Pentest methodology: recon → vulnerability assessment → exploit → report
  • ใช้เครื่องมือที่คุณถนัด (Burp/ZAP) + manual testing
  • หาช่องโหว่จริงในระบบตัวเอง (ตั้งใจเปิดบางช่องเพื่อสาธิต)
  • เขียน pentest report มืออาชีพ (finding, severity, PoC, remediation)
  • แก้ทุกช่องโหว่ + verify (attack → fix → re-test)
ทำไมนี่คือ portfolio ที่ดีที่สุดของคุณ: "สร้างระบบ + เจาะระบบตัวเอง + แก้ + เขียนรายงาน" = แสดงครบทั้ง dev และ security ในชิ้นเดียว · นี่คือเรื่องเล่า interview ที่ AppSec/Product Security/pentest อยากได้ที่สุด และพิสูจน์ว่าคุณ "เข้าใจวงจรเต็ม"

🧹 หลักการประจำ Week 20 — Ethical & Professional

1. Pentest เฉพาะระบบที่ได้รับอนุญาต

⚖️ กฎเหล็ก: pentest ระบบตัวเองเท่านั้น (library-system ที่คุณสร้าง บน localhost/staging ของคุณ) · เจาะระบบคนอื่นโดยไม่ได้รับอนุญาต = ผิดกฎหมาย (พ.ร.บ.คอมพิวเตอร์) · นี่คือจรรยาบรรณที่ pentester มืออาชีพยึด — คุณรู้ดีจากกองทัพ

2. Report ต้อง actionable — ไม่ใช่แค่ "พบช่องโหว่"

finding ที่ดีบอก: ช่องโหว่อะไร, รุนแรงแค่ไหน (severity), พิสูจน์ยังไง (PoC), กระทบอะไร (impact), แก้ยังไง (remediation) · report ที่ dev อ่านแล้วแก้ได้ทันที

กฎทอง: pentest เฉพาะระบบตัวเอง · report ทุก finding มี severity+PoC+remediation · attack→fix→re-test (verify ว่าแก้จริง) · เขียนให้ทั้ง dev (technical) และผู้บริหาร (executive summary) อ่านได้
