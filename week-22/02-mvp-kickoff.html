Week 22 · บทที่ 2

🚀 MVP Scoping + Project Kickoff

ตัดให้เหลือสิ่งที่สำคัญสุด แล้วสร้าง vertical slice แรกให้ทำงานได้จริง

1 · MVP คืออะไร — และทำไมต้อง scope

MVP (Minimum Viable Product) = เวอร์ชันเล็กสุดที่ยังใช้งานได้จริงและพิสูจน์คุณค่าหลัก · ไม่ใช่ครึ่ง ๆ กลาง ๆ — แต่คือ "feature หลักที่ทำงานครบ" โดยตัด nice-to-have ออกไปก่อน

สำหรับ capstone 2 สัปดาห์ คุณต้อง scope ให้ดี ไม่งั้นทำไม่เสร็จ · หลักคือแยก 3 กอง:

กองหมายถึงVulnTracker
MVP (ทำก่อน)ขาดไม่ได้ ระบบไร้ความหมายถ้าไม่มีauth + CRUD finding + RBAC + dashboard นับเลข
Should-haveดีมากแต่รอได้audit log, export PDF, filter ละเอียด
Nice (ตัดก่อน)หรู ไม่จำเป็นตอนนี้email noti, theme, comment, real-time
กฎ scope: ทำ MVP ให้ เสร็จและดี ก่อน แล้วค่อยไต่ขึ้น should-have · capstone ที่ทำ MVP เสร็จ-polished ดีกว่าทำ 10 feature ครึ่ง ๆ ที่ไม่มีอันไหนใช้ได้จริง

2 · ทำไม vertical slice ก่อน — ไม่ใช่ layer by layer

มี 2 วิธีสร้างระบบ:

❌ Horizontal (layer by layer)

ทำ DB ทุกตารางก่อน → ทำ API ทุก endpoint → ทำ UI ทุกหน้า · เพิ่งเห็นระบบทำงานจริงตอนท้ายสุด · integration พังตอนสาย

✅ Vertical slice (feature by feature)

"สร้าง finding" ครบทั้ง stack (UI→API→DB→test→deploy) ให้ทำงาน end-to-end ก่อน → แล้วเพิ่ม feature ถัดไป

⚠️ gotcha: vertical slice แรกต้องผ่านทุกชั้นรวม deploy — ถ้าทำ "สร้าง finding" แล้ว deploy ขึ้น Vercel/Railway ได้เลย คุณพิสูจน์ว่า pipeline ทั้งหมดทำงาน ก่อนจะลงทุนกับ feature อื่น (เจอปัญหา env/CORS/migration ตั้งแต่ feature แรก ไม่ใช่ feature ที่ 10)

4 · ตัวอย่าง — vertical slice "สร้าง finding"

slice แรกครบทุกชั้น (ย่อ) — auth ไว้ก่อนแล้ว เพิ่ม feature นี้:

// 1) DB migration
CREATE TABLE findings (
  id SERIAL PRIMARY KEY, title TEXT NOT NULL,
  severity TEXT NOT NULL, status TEXT DEFAULT 'open',
  created_by INT REFERENCES users(id), created_at TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_findings_status ON findings(status);   -- จาก W21

// 2) Zod schema (validate input — W18)
const CreateFinding = z.object({
  title: z.string().min(3).max(200),
  severity: z.enum(['low','medium','high','critical']),
});

// 3) Route + RBAC (server เช็ค role เสมอ — threat model)
router.post('/findings', auth, requireRole('pentester','lead'), async (req, res) => {
  const data = CreateFinding.parse(req.body);             // throw → 400
  const f = await findingService.create(data, req.user.id);
  await audit(req.user.id, 'create', 'finding', f.id);   // repudiation defense
  res.status(201).json(f);
});

// 4) Test (W6) — ต้องผ่านใน CI
it('creates finding when pentester', async () => {
  const res = await request(app).post('/findings')
    .set('Authorization', pentesterToken)
    .send({ title: 'SQLi in login', severity: 'high' });
  expect(res.status).toBe(201);
});
it('rejects client (403)', async () => { /* RBAC */ });

slice เดียวนี้แตะ: migration+index (W21), Zod (W18), JWT+RBAC (W15/threat), audit (threat), test (W6) — ครบ stack + ปลอดภัย

5 · ภาพรวม + คำช่วยจำ

ลำดับ kickoff:

Scope (3 กอง) → Scaffold + CI เขียว → Vertical slice แรก (ครบชั้น) → Deploy → feature ถัดไป

"MVP ก่อน · vertical ไม่ horizontal · apply ทุกอย่างตั้งแต่ commit แรก · deploy ตั้งแต่ feature แรก"

6 · Quiz

Q1. "MVP" ไม่ได้แปลว่าทำครึ่ง ๆ — แล้วแปลว่าอะไร?
เฉลยเวอร์ชันเล็กสุดที่ยังใช้งานได้จริงและเสร็จสมบูรณ์ · ทำ feature หลักให้ครบและดี โดยตัด nice-to-have ออก — ไม่ใช่ทุก feature ทำครึ่งเดียว
Q2. ทำไม vertical slice แรกควรรวม deploy ด้วย ไม่ใช่ deploy ตอนจบ?
เฉลยเพื่อพิสูจน์ทั้ง pipeline (build/env/migration/CORS) ตั้งแต่ feature แรก · เจอปัญหา deploy ตอนยังเล็ก ง่ายแก้ · ไม่ใช่มาเจอตอนจบที่ทุกอย่างพันกัน
Q3. 🔐 ทำไม "apply security ตั้งแต่ scaffold" ดีกว่าเพิ่มทีหลัง?
เฉลยsecurity ที่ bolt-on ทีหลังมักมีรู (ลืม endpoint, ลืม validate บางจุด) · ทำตั้งแต่ต้น = ทุก feature ใหม่อยู่บนฐานที่ปลอดภัยอยู่แล้ว (helmet/rate-limit/Zod/RBAC เป็น default) · shift-left (W19)
Q4. ถ้าเหลือเวลาน้อย ควรเลือกทำ should-have หรือ polish MVP?
เฉลยpolish MVP · capstone ที่ core เสร็จ-ขัดเงา (test ครบ, README ดี, deploy ได้, ไม่มี bug) สร้างความประทับใจมากกว่ากองฟีเจอร์ที่ทำงานไม่ครบ — interviewer ดู depth ไม่ใช่ count

7 · 🧹 Clean Code: เริ่มโปรเจกต์ให้สะอาดตั้งแต่วันแรก

❌ "เดี๋ยวค่อยจัด"

ไม่มี lint/test/CI ตอนเริ่ม · โค้ดมั่วสะสม · พอจะเพิ่ม test ทีหลัง = rework เยอะ · README ว่างเปล่า

✅ ฐานสะอาดตั้งแต่ commit 1

lint+test+CI+Docker พร้อมก่อน feature แรก · ทุก PR ต้องเขียว · README มีวิธีรัน+architecture+screenshot ตั้งแต่ต้น

เคล็ด: commit แรกที่ดี = "chore: scaffold with TS, lint, test, docker, CI" + CI เขียว · ใช้ conventional commits (feat/fix/chore) · branch ต่อ feature · ทำให้ git history ของ capstone ดูเป็นมืออาชีพ (recruiter ดู commit history จริง ๆ)
