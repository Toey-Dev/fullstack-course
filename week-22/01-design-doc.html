Week 22 · บทที่ 1

📐 System Design Document + Threat Model

ออกแบบระบบให้เสร็จบนกระดาษก่อนเขียนโค้ดบรรทัดแรก

1 · Design Document คืออะไร และทำไมต้องมี

Design Document (design doc) คือเอกสารสั้น ๆ (1-3 หน้า) ที่อธิบายว่าจะสร้างอะไร และจะสร้างอย่างไร ก่อนลงมือเขียนโค้ด · เป็นมาตรฐานของบริษัทใหญ่ทุกที่ (Google เรียก design doc, Amazon เรียก 6-pager) · มันบังคับให้คุณ "คิดให้จบ" ก่อนเสียเวลาเขียนโค้ดผิดทาง

มันตอบคำถามสำคัญก่อนที่จะสาย:

  • Problem — แก้ปัญหาอะไร ใครใช้
  • Requirements — ต้องทำอะไรได้ (functional) + ต้องเร็ว/ปลอดภัย/scale แค่ไหน (non-functional)
  • Architecture — ระบบมีกี่ส่วน คุยกันยังไง
  • Data model — เก็บข้อมูลอะไร ความสัมพันธ์
  • API design — endpoint อะไรบ้าง
  • Threat model — มีช่องโหว่อะไร ป้องกันยังไง (จุดแข็งคุณ!)
  • Trade-offs — เลือกอะไร ไม่เลือกอะไร เพราะอะไร
ทำไมสำคัญกับเป้าหมาย senior: mid-level เขียนโค้ดตามที่บอก · senior ออกแบบระบบเองได้และอธิบาย trade-off ได้ · design doc คือหลักฐานที่จับต้องได้ของทักษะนี้ — และเป็น artifact ที่ทำให้ portfolio/GitHub ของคุณดูคนละระดับ

2 · ทำไมต้อง design ก่อน — และจะเกิดอะไรถ้าไม่ทำ

เหตุผลที่ไม่ใช่แค่ "ทำตามฟอร์ม":

① หาจุดพังก่อนเขียน — แก้ปัญหาบนกระดาษถูกกว่าแก้ในโค้ด 100 เท่า · เช่น เพิ่งรู้ตอน design ว่า "expiring link" ต้องเก็บ token + expiry + เช็คทุกครั้ง — ดีกว่ามารู้ตอนเขียนไปครึ่งทาง · ② scope ชัด — รู้ว่าอะไรทำอะไรไม่ทำ ไม่หลงทาง · ③ คุยกับคนอื่นได้ — ใน interview/ทีม คุณอธิบายระบบจาก doc ได้ใน 5 นาที

❌ ไม่มี design — เขียนเลย

เขียน auth ก่อน → เขียน upload → เพิ่งรู้ว่า data model ผิด → rework ทั้งหมด · 2 สัปดาห์หายไป · architecture ปะผุ

✅ design 2 ชม. ก่อน

เห็นภาพรวม → รู้ลำดับที่ถูก → data model นิ่งตั้งแต่ต้น → เขียนราบรื่น · 2 ชม. ประหยัด 2 สัปดาห์

⚠️ counter-example (over-design ก็พังเหมือนกัน): design ทุก edge case 2 สัปดาห์ไม่เขียนสักบรรทัด = analysis paralysis · design doc ที่ดี = พอเห็นโครง (1-3 หน้า) แล้วเริ่มเลย ปรับระหว่างทาง · doc คือเข็มทิศ ไม่ใช่พิมพ์เขียวที่ห้ามแก้

3 · ตัวอย่างจริง — design doc ของ VulnTracker

มาดู design doc ตัวอย่างของ VulnTracker (capstone option ที่ตรงงานคุณ) — สั้น กระชับ แต่ครบ:

① Problem & Users

ทีม pentest บันทึก finding กระจัดกระจาย (Excel/Notion) → ติดตามสถานะยาก, ทำ report ซ้ำ · User: pentester (สร้าง finding), team lead (มอบหมาย+ดู dashboard), client (ดู report read-only)

② Requirements

FunctionalNon-functional
สร้าง/แก้ finding (title, severity, desc, CVSS)auth ทุก endpoint (RBAC 3 role)
มอบหมาย + เปลี่ยน status (open→fixed→verified)audit log ทุก action ที่เปลี่ยนข้อมูล
dashboard (count by severity/status)response < 300ms (มี index + pagination)
export report (PDF/markdown)input validation (Zod) + OWASP Top 10

③ Architecture (SVG)

④ Data Model (หลัก)

users(id, email, password_hash, role)        -- role: pentester|lead|client
findings(id, title, severity, cvss, status,  -- severity: low..critical
         description, created_by, assigned_to, project_id, created_at)
audit_log(id, user_id, action, entity, entity_id, timestamp)  -- ใครทำอะไร

⑤ API Design

POST   /api/findings        สร้าง finding         (pentester+)
GET    /api/findings        list + filter + page  (ทุก role, scope ตาม role)
PATCH  /api/findings/:id    เปลี่ยน status/assign  (lead+ / audit log)
GET    /api/dashboard       สรุป by severity      (cache 60s)
POST   /api/reports/:id     export PDF (→ queue)  (lead+)

นี่คือ design doc จริง ๆ ที่ทุกบริษัทใช้ — สั้น แต่พอให้ใครก็เริ่มเขียนได้ และอธิบาย interviewer ได้

4 · Threat Model — จุดแข็ง ex-pentester ของคุณ

Threat model = ระบุภัยคุกคามอย่างเป็นระบบก่อนสร้าง · ใช้กรอบ STRIDE (มาตรฐาน Microsoft) · นี่คือสิ่งที่ dev ทั่วไปไม่ทำ แต่คุณทำได้เพราะพื้นเพ pentester — ใส่ใน design doc แล้วคุณโดดเด่นทันที

STRIDEภัยVulnTracker ป้องกันยังไง
Spoofingปลอมตัวเป็นคนอื่นJWT + bcrypt · ไม่มี endpoint ไหนไม่ auth
Tamperingแก้ข้อมูลที่ไม่ควรRBAC (client แก้ไม่ได้) · Zod validate · audit log
Repudiationปฏิเสธว่าไม่ได้ทำaudit_log บันทึกทุก action ที่เปลี่ยนข้อมูล
Info Disclosureเห็นข้อมูลที่ไม่ควรscope by role+project · ไม่ leak ใน error · HTTPS
DoSถล่มจนล่มrate-limit · pagination · query มี index
Elevationยกระดับสิทธิ์เช็ค role ฝั่ง server เสมอ (ไม่เชื่อ client) · IDOR check
🎯 จุดขายของคุณ: ใน interview เมื่อคุณพูดถึง capstone แล้วโชว์ threat model แบบ STRIDE + "ผมทำ IDOR check ตรงนี้เพราะเคยเจอตอน pentest" — คุณคุยคนละภาษากับ junior ทันที · นี่คือสะพานจาก pentester → AI-Security Engineer

6 · Quiz

Q1. ทำไม design doc ที่ดีควรสั้น (1-3 หน้า) ไม่ใช่ละเอียดทุก edge case?
เฉลยเพราะ over-design = analysis paralysis (ไม่ได้เริ่มเขียนสักที) · doc คือเข็มทิศพอเห็นโครงแล้วเริ่ม ปรับระหว่างทาง — ไม่ใช่พิมพ์เขียวที่สมบูรณ์แบบตั้งแต่แรก
Q2. ความต่างของ functional vs non-functional requirement?
เฉลยFunctional = ระบบทำอะไรได้ (สร้าง finding, export report) · Non-functional = ทำได้ดีแค่ไหน (เร็ว <300ms, ปลอดภัย RBAC, scale) · ทั้งคู่ต้องระบุก่อนสร้าง
Q3. 🔐 ในกรอบ STRIDE — "client เปลี่ยน status ของ finding ได้ทั้งที่ควรเป็น read-only" คือภัยตัวไหน และป้องกันยังไง?
เฉลยTampering (+Elevation of Privilege) · ป้องกัน: เช็ค role ฝั่ง server เสมอ ไม่เชื่อ client · RBAC middleware · บันทึก audit log · นี่คือ broken access control (OWASP A01)
Q4. ทำไม "vertical slice" ดีกว่าทำ backend ให้เสร็จก่อนแล้วค่อยทำ frontend?
เฉลยvertical slice = 1 feature ครบ stack (UI→API→DB→deploy) ทำให้เห็นทั้งระบบทำงาน early, เจอปัญหา integration เร็ว · ทำ backend หมดก่อน = เพิ่งรู้ปัญหาตอนต่อ frontend (สาย)

7 · 🧹 Clean Code: design doc ที่ maintain ได้

design doc เป็นเอกสารมีชีวิต ไม่ใช่เขียนทิ้ง:

❌ doc ตายตั้งแต่วันแรก

เขียนครั้งเดียวตอนเริ่ม · code เปลี่ยนแต่ doc ไม่อัปเดต · 1 เดือนผ่านไป doc โกหก → ไม่มีใครเชื่อ

✅ doc อยู่ใน repo + อัปเดต

เก็บใน docs/DESIGN.md · เปลี่ยน architecture สำคัญ → อัปเดต doc ใน PR เดียวกัน · มี "Decisions" log สั้น ๆ

เคล็ด: ใช้ docs/DESIGN.md + docs/DECISIONS.md (ADR — บันทึกว่าทำไมเลือก X ไม่เลือก Y) · interviewer ที่เห็น ADR ใน repo คุณ = "คนนี้คิดเป็นระบบ" · เก็บ threat model ใน docs/THREAT-MODEL.md แยกต่างหาก
