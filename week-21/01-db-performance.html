Database Performance — Index, N+1, Pagination

3 เรื่องที่ทำให้ระบบช้าหรือเร็ว — และที่ interview ถามบ่อยสุด

🔑 คีย์เวิร์ดภาพจำประจำบท

📑 สารบัญหนังสือ = index 🔁 1 query กลายเป็น N+1 = N+1 problem 📄 offset = นับใหม่ทุกหน้า (ช้าที่ scale) 🎯 cursor = ชี้จุดต่อ (เร็ว) 🔬 EXPLAIN = ดูว่า DB ทำงานยังไง

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Index = สารบัญ — หาเร็วโดยไม่ต้องอ่านทั้งเล่ม"

ไม่มี index → DB อ่านทุกแถวเพื่อหา (full table scan) — ช้ามากเมื่อข้อมูลเยอะ · index = โครงสร้างที่ช่วยหาเร็ว (เหมือนสารบัญหนังสือ) · แต่ index มีต้นทุน (กินที่ + เขียนช้าลง เพราะต้อง update index) → index เฉพาะ column ที่ search/join/sort บ่อย ไม่ใช่ทุก column

หลักการ 2: "N+1 — bug performance ที่ซ่อนอยู่ในโค้ดสวย ๆ"

ดึง 100 หนังสือ (1 query) แล้ว loop ดึง author ของแต่ละเล่ม (100 query) = 101 queries · โค้ดดูสะอาดแต่ DB ตายเพราะยิงร้อยครั้ง · นี่คือ bug performance อันดับ 1 ของ ORM/app · แก้ด้วย JOIN (1 query) หรือ batch loading · ต้องรู้จัก "ดู query count"

หลักการ 3: "Pagination — offset ดูง่ายแต่พังที่ scale"

หน้า 1000 ด้วย OFFSET 10000 → DB ต้องนับข้าม 10000 แถวก่อน (ช้าขึ้นเรื่อย ๆ ตามหน้า) · cursor-based ("เอาต่อจาก id นี้") เร็วคงที่ทุกหน้า · offset โอเคสำหรับข้อมูลน้อย/หน้าแรก ๆ · scale ใหญ่/infinite scroll ใช้ cursor · เลือกตาม use case

ส่วนที่ 1: Database Index

-- query นี้ช้าถ้าไม่มี index บน email (DB อ่านทุกแถว)
SELECT * FROM users WHERE email = 'vee@x.com';

-- สร้าง index → หาเร็วขึ้นมาก
CREATE INDEX idx_users_email ON users(email);

-- composite index (หลาย column ที่ query ด้วยกัน)
CREATE INDEX idx_borrows_user_status ON borrows(user_id, returned_at);

-- ดูว่า DB ใช้ index ไหม
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'vee@x.com';
-- เห็น "Index Scan" = ใช้ index (ดี) / "Seq Scan" = อ่านทั้งตาราง (ช้า)
ควร indexไม่ควร index
column ที่ WHERE/JOIN/ORDER BY บ่อยcolumn ที่ไม่ค่อย query
foreign key (user_id ใน borrows)column ที่ค่าซ้ำเยอะมาก (boolean)
column ที่ค่าหลากหลาย (email, id)ตารางที่ write หนักมาก (index ทำ write ช้า)
EXPLAIN คือเครื่องมือหลัก: ก่อนเดาว่า query ช้าเพราะอะไร — รัน EXPLAIN ANALYZE ดูว่า DB ทำงานยังไง (ใช้ index ไหม, อ่านกี่แถว) · "Seq Scan" บนตารางใหญ่ = สัญญาณต้อง index · measure ก่อน optimize

ส่วนที่ 2: N+1 Problem

// ❌ N+1 — โค้ดดูสะอาดแต่ยิง 101 queries
const books = await db.query("SELECT * FROM books");        // 1 query
for (const book of books) {
  book.author = await db.query(                              // ×100 queries!
    "SELECT * FROM authors WHERE id = $1", [book.author_id]
  );
}

// ✅ JOIN — 1 query ได้ครบ
const books = await db.query(`
  SELECT b.*, a.name AS author_name
  FROM books b JOIN authors a ON b.author_id = a.id
`);
วิธีจับ N+1: log จำนวน query ต่อ request (เห็น 100+ queries = สงสัย) · ORM (Prisma) มี include/eager loading แก้ให้ · หรือ DataLoader (batch) · N+1 ซ่อนในโค้ดสวย — ต้องวัด query count ถึงเจอ

ส่วนที่ 3: Pagination — Offset vs Cursor

-- ❌ offset — หน้าลึก ๆ ช้า (DB นับข้าม 10000 แถวก่อน)
SELECT * FROM books ORDER BY id LIMIT 20 OFFSET 10000;

-- ✅ cursor — "เอาต่อจาก id ล่าสุด" เร็วคงที่ทุกหน้า
SELECT * FROM books WHERE id > 10000 ORDER BY id LIMIT 20;
OffsetCursor
ทำงานนับข้าม N แถวชี้จุดต่อ (WHERE id > last)
ความเร็วช้าลงตามหน้า (ลึก = ช้า)เร็วคงที่ทุกหน้า
ข้อดีไปหน้าไหนก็ได้ (jump to page)เร็ว, ไม่ skip/ซ้ำตอนข้อมูลเปลี่ยน
เหมาะกับข้อมูลน้อย, ตารางมีเลขหน้าinfinite scroll, scale ใหญ่, feed

offset ปัญหาอีกอย่าง: ถ้าข้อมูลถูกเพิ่ม/ลบระหว่างเลื่อนหน้า → เห็นซ้ำ/ข้าม · cursor ไม่มีปัญหานี้ (ชี้ที่ค่าจริง) · Facebook/IG feed ใช้ cursor

ส่วนที่ 4: ตัวอย่างประยุกต์

🏢 งานวี: dashboard ที่ดึงข้อมูลลูกค้าเยอะ — index บน clientId/date (query เร็ว), แก้ N+1 ตอนดึงข้อมูลหลายแพลตฟอร์มต่อลูกค้า, cursor pagination สำหรับ list ยาว · automation report ที่ query ข้อมูลย้อนหลัง — index ทำให้เร็วขึ้นมาก

📝 Quiz ท้ายบท

ข้อ 1: index ช่วยอะไร ทำไมไม่ index ทุก column?

index = สารบัญ ช่วยหาเร็วโดยไม่ต้อง scan ทั้งตาราง · แต่มีต้นทุน: กินที่ + ทำ write ช้าลง (ต้อง update index ทุกครั้งที่เขียน) · → index เฉพาะ column ที่ WHERE/JOIN/ORDER BY บ่อย · index มั่ว = เปลือง + write ช้า

ข้อ 2: N+1 problem คืออะไร แก้ยังไง?

ดึง list (1 query) แล้ว loop ดึง related ของแต่ละ item (N queries) = N+1 queries · โค้ดดูสะอาดแต่ DB ยิงร้อยครั้ง ช้ามาก · แก้: JOIN (1 query ได้ครบ) หรือ eager loading (Prisma include) / DataLoader (batch) · จับด้วยการวัด query count

ข้อ 3: ทำไม OFFSET pagination ช้าที่หน้าลึก ๆ? cursor แก้ยังไง?

OFFSET 10000 = DB ต้องนับข้าม 10000 แถวก่อนถึงจะได้ผล → ช้าขึ้นตามหน้า · cursor (WHERE id > last LIMIT 20) ชี้จุดต่อตรง ๆ → เร็วคงที่ทุกหน้า + ไม่ซ้ำ/ข้ามตอนข้อมูลเปลี่ยน · feed (IG/FB) ใช้ cursor

ข้อ 4: ทำไมต้อง EXPLAIN ก่อน optimize query?

EXPLAIN ANALYZE บอกว่า DB ทำงานยังไงจริง (ใช้ index ไหม, อ่านกี่แถว, ช้าตรงไหน) — measure ก่อน optimize · เดาว่าช้าเพราะอะไรแล้วแก้มั่ว = อาจ optimize ผิดจุด · "Seq Scan" บนตารางใหญ่ = สัญญาณต้อง index

  • จับ + แก้ N+1 (JOIN/eager loading)
  • เลือก offset vs cursor pagination ถูก use case
  • measure ก่อน optimize
