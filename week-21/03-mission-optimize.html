Week 21 · Mission ใหญ่

⚡ Optimize library-system + วัดผล

ทำให้ระบบเร็วและ scale ได้ — พร้อมตัวเลข before/after พิสูจน์

ทำไม Mission นี้ทำให้คุณดู senior

หลังจบ library-system จะมี optimization จริง (index, แก้ N+1, pagination, cache, queue) + ตัวเลข before/after · "optimize เป็น + วัดผลได้ + อธิบาย trade-off" คือสิ่งที่ system design interview (Agoda) วัด · และตัวเลขปรับปรุงคือเรื่องเล่าที่ดีใน resume

measure ก่อน/หลังเสมอ: ทุก optimization ต้องมีตัวเลขพิสูจน์ (query time, request time) — ไม่ใช่ "น่าจะเร็วขึ้น"

🪜 แบ่งเป็น 4 Mission ย่อย

Mission 21.1 — Index + EXPLAIN 60 นาที

  1. seed ข้อมูลเยอะ ๆ (10k+ books) เพื่อเห็นผล performance
  2. EXPLAIN ANALYZE query หลัก — หา Seq Scan ที่ช้า
  3. เพิ่ม index (email, foreign key, search column) → EXPLAIN ซ้ำ (Index Scan)
✅ เทสต์: บันทึก query time before/after index — เร็วขึ้นชัดเจน

Mission 21.2 — แก้ N+1 + Pagination 75 นาที

  1. หา N+1 (log query count — endpoint ที่ยิงเยอะ) → แก้ด้วย JOIN
  2. เพิ่ม pagination (cursor-based) ให้ list books/borrows
  3. วัด query count + response time before/after
เทสต์: endpoint ที่เคยยิง 100+ queries → เหลือไม่กี่ queries · pagination ทำงานถูก (ไม่ซ้ำ/ข้าม)

Mission 21.3 — Cache + Invalidation 60 นาที

  1. cache endpoint ที่อ่านบ่อย (books list, dashboard) — cache-aside + TTL
  2. invalidation: ลบ cache ตอน create/update/delete (ครบทุก key ที่กระทบ)
  3. ทดสอบ: เพิ่มหนังสือ → โผล่ทันที (invalidate ถูก) ไม่รอ TTL
✅ เทสต์: cache hit เร็วขึ้น · update แล้วข้อมูลไม่ค้าง (invalidate ทำงาน)

Mission 21.4 — Queue + วัดผล + push 75 นาที

  1. BullMQ: งานช้า (ส่ง email ยืนยันยืม / สร้าง report) → เข้า queue + worker
  2. request ตอบทันที (ไม่รอ email) + retry config
  3. เขียน PERFORMANCE.md: ตาราง before/after ทุก optimization (query time, query count, response time) + push
✅ เทสต์: request ที่เคยรอ email → ตอบทันที · PERFORMANCE.md มีตัวเลขพิสูจน์
