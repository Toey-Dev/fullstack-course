Caching Strategies + Queue (BullMQ)

2 เครื่องมือ scaling — จำคำตอบ (cache) และทำงานหนักเบื้องหลัง (queue)

🔑 คีย์เวิร์ดภาพจำประจำบท

💾 จำคำตอบ = cache-aside 🗑 ลบ cache เก่า = invalidation (ยากสุด) 📮 งานหนักทำทีหลัง = queue 👷 worker = คนทำงานใน queue ⚡ ไม่ block request = async architecture

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Cache — แลก consistency กับ speed"

cache เก็บคำตอบที่ถามบ่อย → ตอบเร็ว ลดภาระ DB · แต่แลกมาด้วย "ข้อมูลอาจเก่า" (cache ไม่ตรง DB ชั่วคราว) · ไม่มี free lunch — ต้องจัดการ invalidation (ลบ cache เมื่อข้อมูลเปลี่ยน) · เลือกว่าข้อมูลไหน cache ได้ (ทนเก่าได้) ข้อมูลไหนต้องสด

หลักการ 2: "งานหนัก/ช้า อย่าให้ผู้ใช้รอ — ทำเบื้องหลัง (queue)"

ส่ง email, สร้าง report, ประมวลผลรูป = ช้า (วินาที-นาที) · ถ้าทำใน request ผู้ใช้รอนาน (block) · queue: รับงานเข้าคิว → ตอบผู้ใช้ทันที ("กำลังดำเนินการ") → worker ทำเบื้องหลัง · ผู้ใช้ไม่รอ + ถ้าพังมี retry · นี่คือ async architecture ที่ระบบ scale ใช้

หลักการ 3: "Trade-off everywhere — senior อธิบายได้ว่าแลกอะไร"

cache (เร็ว แต่ข้อมูลอาจเก่า) · queue (ไม่ block แต่ซับซ้อนขึ้น, eventual ไม่ instant) · ทุกเทคนิค scaling มีต้นทุน · senior ไม่ใช่คน "ใส่ cache/queue ทุกที่" แต่ "รู้ว่าเมื่อไหร่ควรใช้ + อธิบาย trade-off" · นี่คือสิ่งที่ system design interview วัด

ส่วนที่ 1: Caching Strategies (ต่อจาก W11)

strategyทำงานเหมาะกับ
Cache-aside (ใช้บ่อยสุด)เช็ค cache → ไม่มี query DB → เก็บ cacheread หนัก (จาก W11)
Write-throughเขียน cache + DB พร้อมกันdata ต้องสดใน cache เสมอ
Write-behindเขียน cache ก่อน → DB ทีหลังwrite หนักมาก (ซับซ้อน)
// cache-aside + invalidation (ทบ+เจาะลึกจาก W11)
async function getBook(id) {
  const cached = await redis.get(`book:${id}`);
  if (cached) return JSON.parse(cached);              // hit
  const book = await db.findById(id);                 // miss → DB
  await redis.set(`book:${id}`, JSON.stringify(book), "EX", 300);
  return book;
}
async function updateBook(id, data) {
  await db.update(id, data);
  await redis.del(`book:${id}`);                       // ⭐ invalidate
  await redis.del("books:all");                        // ลบ list cache ด้วย
}
Cache invalidation — "หนึ่งในสองเรื่องยากที่สุด": ลืม invalidate = ผู้ใช้เห็นข้อมูลเก่า · ยากเพราะต้องรู้ว่า "เขียนตรงไหน กระทบ cache key ไหนบ้าง" (update book กระทบ book:id + books:all + อาจ dashboard cache) · กลยุทธ์: TTL เป็นตาข่าย (เก่าได้ไม่เกิน X) + invalidate ตอน write · cache layer (browser→CDN→app→DB) แต่ละชั้นมี trade-off

ส่วนที่ 2: Queue — ทำไมต้องมี

เมื่อไหร่ใช้ queue: งานช้า (email, report, รูป/วิดีโอ), งานที่ retry ได้ (ส่ง notification), งานที่ไม่ต้องตอบทันที, งานที่ peak load (รับเข้าคิว ค่อย ๆ ทำ ไม่ล้น) · เชื่อม W16: automation report ควรเข้า queue (ไม่ block)

ส่วนที่ 3: BullMQ (Queue บน Redis)

npm install bullmq            # ใช้ Redis (จาก W11) เก็บคิว

// producer — เพิ่มงานเข้าคิว (ใน request)
import { Queue } from "bullmq";
const emailQueue = new Queue("email");
app.post("/api/borrow", async (req, res) => {
  await borrowBook(...);                          // งานหลัก (เร็ว)
  await emailQueue.add("borrow-confirm", { userId, bookId });  // งานช้าเข้าคิว
  res.json({ ok: true });                         // ตอบทันที ไม่รอ email
});

// worker — ทำงานเบื้องหลัง (แยก process)
import { Worker } from "bullmq";
new Worker("email", async (job) => {
  await sendEmail(job.data.userId, ...);          // ช้าได้ ผู้ใช้ไม่รอ
}, { /* retry, concurrency config */ });
BullMQ ให้ประโยชน์
retry + backoffงานพังลองใหม่อัตโนมัติ (W16)
concurrencyจำกัดทำพร้อมกันกี่งาน
scheduled/repeatcron job (report รายเดือน W16)
persist (Redis)restart ไม่หาย (resume ได้)

BullMQ ดีกว่า node-cron (W16) สำหรับงานสำคัญ — persist, retry, แยก worker, scale ได้ · เชื่อม: automation report ใช้ BullMQ จะ robust กว่า

ส่วนที่ 4: ตัวอย่างประยุกต์

🏢 งานวี: automation report — เข้า BullMQ queue (request trigger → ตอบทันที → worker ดึง+สรุป+ส่ง email เบื้องหลัง + retry ถ้า API ล่ม) · cache ผล GA4/Meta (5 นาที) ลด API call · dashboard ลูกค้าเปิดบ่อยใช้ cache · นี่คือ architecture จริงของ automation service

🌏 อุตสาหกรรม + interview: cache + queue เป็นหัวใจของ system design (ทุก scale system มี) · "ออกแบบ X ที่รับล้านผู้ใช้" = คำตอบมักมี cache (ลด DB) + queue (async งานหนัก) + index + sharding · Agoda interview เจาะเรื่องนี้ — ตอบได้ว่า "ใช้อะไร เพราะอะไร แลกอะไร" = senior

📝 Quiz ท้ายบท

ข้อ 1: cache แลกอะไรกับอะไร? ข้อมูลแบบไหน cache ได้/ไม่ได้?

แลก consistency (ข้อมูลอาจเก่าชั่วคราว) กับ speed (เร็ว + ลดภาระ DB) · cache ได้: ข้อมูลที่ทนเก่าได้ (รายการสินค้า, สถิติเมื่อวาน) · ไม่ควร cache: ข้อมูลที่ต้องสดเป๊ะ (ยอดเงินคงเหลือ realtime) · ต้องมี invalidation เสมอ

ข้อ 2: ทำไม cache invalidation ยาก?

ต้องรู้ว่า "เขียนตรงไหน กระทบ cache key ไหนบ้าง" ครบ — update book กระทบ book:id + books:all + dashboard cache · ลืมจุดไหน = ผู้ใช้เห็นข้อมูลเก่า · กลยุทธ์: TTL เป็นตาข่าย + invalidate ตอน write ทุกจุด · เป็น "หนึ่งในสองเรื่องยากที่สุดใน CS"

ข้อ 3: เมื่อไหร่ควรใช้ queue? ยกตัวอย่าง

งานช้า/หนัก ที่ไม่ต้องตอบทันที: ส่ง email, สร้าง report, ประมวลผลรูป/วิดีโอ, notification · queue: รับเข้าคิว → ตอบผู้ใช้ทันที → worker ทำเบื้องหลัง + retry · ผู้ใช้ไม่รอ (ไม่ block request) · งานเร็ว (CRUD) ทำใน request ปกติ

ข้อ 4: BullMQ ดีกว่า node-cron (W16) ยังไงสำหรับงานสำคัญ?

persist ใน Redis (restart ไม่หาย, resume ได้), retry+backoff (งานพังลองใหม่), concurrency control, แยก worker (scale) · node-cron ง่ายแต่ใน process เดียว restart = งานหาย · งานสำคัญ/เยอะ → BullMQ
