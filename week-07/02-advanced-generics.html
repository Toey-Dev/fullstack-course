Generic Constraints + Mapped Types + Enums

เครื่องมือ TS ขั้นสูงที่ทำให้เขียนโค้ด reusable และ type-safe พร้อมกัน

🔑 คีย์เวิร์ดภาพจำประจำบท

🔒 generic ที่มีเงื่อนไข = constraints (extends) 🗺 แปลงทุก key = mapped types 📇 dictionary แบบ type = Record 📋 ชุดค่าคงที่ = enum vs union

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Reusable + type-safe ไปด้วยกันได้"

generic constraints ทำให้เขียนฟังก์ชัน/class ที่ "ใช้ได้หลาย type แต่ยังรู้ว่า type นั้นมีอะไร" — เช่น repository ที่ทำงานกับ entity ไหนก็ได้ แต่ทุก entity ต้องมี id · ได้ทั้ง reuse และ safety ไม่ต้องเลือก

หลักการ 2: "Derive อย่าซ้ำ — type ใหม่ควรมาจาก type เดิม"

mapped types + utility types ทำให้สร้าง type ใหม่จากที่มี (แทนเขียนซ้ำ) · แก้ต้นทางที่เดียว ทุก type ที่ derive ตามหมด · นี่คือ DRY ระดับ type — codebase ใหญ่อยู่ได้เพราะหลักนี้

หลักการ 3: "เลือกเครื่องมือให้พอดี — อย่า over-engineer type"

TS ทำ type magic ได้ลึกมาก (conditional types, recursive) แต่ 90% ของงานจริงใช้แค่ basics + utility + discriminated union · type ที่ซับซ้อนจนทีมอ่านไม่ออก = หนี้ เหมือนโค้ดที่ฉลาดเกิน · เลือกความเรียบง่ายก่อนเสมอ

ส่วนที่ 1: Generic Constraints — generic ที่มีเงื่อนไข

// ❌ generic ล้วน — ไม่รู้ว่า T มี id ไหม
function getId<T>(item: T) {
  return item.id;   // ❌ TS เตือน: T อาจไม่มี id
}

// ✅ constraint: T ต้อง "extends" (มีอย่างน้อย) { id: number }
function getId<T extends { id: number }>(item: T) {
  return item.id;   // ✅ TS มั่นใจว่า T มี id
}
getId({ id: 1, title: "x" });   // ✅
getId({ title: "x" });          // ❌ ไม่มี id

extends ใน generic = "T ต้องมีอย่างน้อยรูปร่างนี้" — ใช้สร้าง utility/repository ที่ทำงานกับ type ไหนก็ได้ที่ตรงเงื่อนไข

ส่วนที่ 2: Generic Repository Pattern (ใช้จริงใน Mission)

// repository ที่ทำงานกับ entity ไหนก็ได้ (Book, Member, ...) ที่มี id
class Repository<T extends { id: number }> {
  constructor(private table: string) {}

  async findById(id: number): Promise<T | null> {
    const res = await pool.query(
      `SELECT * FROM ${this.table} WHERE id = $1`, [id]
    );
    return (res.rows[0] as T) ?? null;
  }
  async findAll(): Promise<T[]> {
    const res = await pool.query(`SELECT * FROM ${this.table}`);
    return res.rows as T[];
  }
}

// ใช้ — type-safe ต่อ entity
const bookRepo = new Repository<Book>("books");
const memberRepo = new Repository<Member>("members");
const book = await bookRepo.findById(1);   // book: Book | null
ทำไมทรงพลัง: เขียน repository ครั้งเดียว ใช้ได้ทุก entity โดยยังรู้ type — ไม่ต้อง copy BookRepo, MemberRepo · นี่คือ pattern ที่ทีมจริงใช้ลด boilerplate · หมายเหตุ: table name ต่อ string ตรงนี้ปลอดภัยเพราะเราคุมเอง (ไม่ใช่ input ผู้ใช้) — ค่า id ยังใช้ $1

ส่วนที่ 3: Mapped Types + Record

// Record<K, V> — object ที่ key เป็น K ค่าเป็น V (dictionary แบบ type)
type BookStock = Record<string, number>;   // { [title]: จำนวน }
const stock: BookStock = { "Clean Code": 5, "Refactoring": 2 };

// Record กับ literal union — บังคับครบทุก key
type RolePermission = Record<"admin" | "user", string[]>;
const perms: RolePermission = {
  admin: ["read", "write", "delete"],
  user: ["read"],
  // ลืม key ไหน TS เตือน
};

mapped types (เบื้องหลัง Partial/Pick) — "วน key ของ type แล้วแปลง":

// นี่คือวิธีที่ Partial ทำงานข้างใน (รู้ไว้พออ่านโค้ดออก)
type MyPartial<T> = { [K in keyof T]?: T[K] };   // ทุก key เป็น optional

งานทั่วไปใช้ Record บ่อย, ใช้ utility (Partial/Pick) ที่ build จาก mapped types · เขียน mapped type เองตอนทำ library/util ขั้นสูง

ส่วนที่ 4: Enums vs Union — เลือกให้ถูก

// enum — มีตัวตนตอน runtime (object จริง)
enum Role { Admin = "admin", User = "user" }
Role.Admin;   // "admin"

// literal union — เป็นแค่ type (หายตอน compile, เบากว่า)
type Role = "admin" | "user";
enumliteral union
runtimeมี object จริง (กิน bundle)ไม่มี (หายตอน compile)
iterate ค่าได้ได้ (Object.values)ไม่ได้ตรง ๆ
คำแนะนำยุคใหม่ใช้เมื่อต้อง iterate/map ค่า⭐ default — เบา ง่าย พอสำหรับ status/role ส่วนใหญ่

เทรนด์ TS ยุคใหม่เอนไป literal union (+ as const) มากกว่า enum เพราะเบาและตรงไปตรงมา · enum ยังดีเมื่อต้องวนค่าทั้งหมด

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: generic Repository ใช้กับทุกตารางใน library-system (books, members, borrows) — เขียนครั้งเดียว · Record<Platform, Metrics> เก็บผลแต่ละแพลตฟอร์มของ automation report แบบ type-safe (ลืมแพลตฟอร์มไหน TS เตือน)

🌏 อุตสาหกรรม: generic repository/service เป็น pattern มาตรฐานลด boilerplate · Record ใช้ทุกที่ที่มี dictionary · การเลือก enum vs union ถูกถามใน code review จริง — เข้าใจ trade-off = ดู mature

📝 Quiz ท้ายบท

ข้อ 1: <T extends { id: number }> แก้ปัญหาอะไรของ generic ล้วน?

generic ล้วน <T> ไม่รู้ว่า T มี property อะไร (ใช้ item.id ไม่ได้) · constraint extends { id: number } บอกว่า "T ต้องมีอย่างน้อย id" → ใช้ item.id ได้อย่าง type-safe และยังรับ type ไหนก็ได้ที่มี id

ข้อ 2: Record<"admin"|"user", string[]> ให้ประโยชน์อะไรเหนือ object ธรรมดา?

บังคับว่าต้องมี key ครบทุกค่าใน union (admin และ user) — ลืม key ไหน TS เตือน · และค่าต้องเป็น string[] ทุกอัน · object ธรรมดาไม่บังคับ → อาจลืม role ไหนแล้วพังตอนรัน

ข้อ 3: enum vs literal union — ยุคใหม่นิยมอันไหน เพราะอะไร?

literal union (default) — เบากว่า (หายตอน compile ไม่กิน bundle), ตรงไปตรงมา, พอสำหรับ status/role ส่วนใหญ่ · enum ใช้เมื่อต้อง iterate ค่าทั้งหมดตอน runtime (Object.values) · เลือกตามว่าต้องใช้ค่าตอน runtime ไหม

ข้อ 4: TS ทำ type ซับซ้อนได้ลึกมาก — ทำไมไม่ควร over-engineer?

type ที่ซับซ้อนจนทีมอ่านไม่ออก = หนี้ (เหมือนโค้ดฉลาดเกิน) · 90% ของงานใช้แค่ basics + utility + discriminated union · เลือกความเรียบง่ายที่ทีม maintain ได้ ก่อน type magic — readability over cleverness (หลักเดิมจากทั้งคอร์ส)

✅ Checkpoint

  • ใช้ generic constraints (extends) ได้
  • เขียน generic Repository หรือ utility ที่ reusable + type-safe
  • ใช้ Record + เข้าใจ mapped types เบื้องต้น
  • เลือก enum vs union ได้ตามสถานการณ์
