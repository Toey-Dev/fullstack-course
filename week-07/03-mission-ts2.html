หลังจบ Mission นี้ library-system จะมี type ที่ "ออกแบบให้บั๊กบางอย่างเป็นไปไม่ได้" — ApiResponse แบบ discriminated, generic repository, 0 any · นี่คือระดับโค้ดที่เล่าใน interview ได้ว่า "ใช้ TS เป็นเครื่องมือออกแบบ ไม่ใช่แค่กันพิมพ์ผิด" + test (Week 3) ยืนยันทุกอย่างยังทำงาน = portfolio piece ที่สมบูรณ์

หมายเหตุ: Mission นี้ "ขัดของที่มี" ไม่ใช่สร้างใหม่ — ต่อยอด library-system ที่แปลง TS แล้วจาก Week 6

🪜 แบ่งเป็น 4 Mission ย่อย

Mission 7.1 — typed ApiResponse (discriminated) 60 นาที

  1. สร้าง type ApiResponse<T> = { ok: true; data: T } | { ok: false; error: string }
  2. ทุก controller คืน response ตาม type นี้ (success มี data, error มี error)
  3. helper ok(data) / fail(error) สร้าง response แบบ type-safe
✅ เทสต์: ลองสร้าง { ok: true, error: "x" } → TS ต้องเตือน (สถานะผิดเขียนไม่ได้)

Mission 7.2 — Generic Repository 75 นาที

  1. Repository<T extends { id: number }> — findAll, findById, create
  2. ใช้กับ books (และเตรียมไว้สำหรับ members/borrows)
  3. type ผลจาก pg query ให้ตรง (จัดการ null จาก findById)
คิดแบบ pentester: table name มาจากเราคุมเอง (ปลอดภัย) แต่ทุกค่า id/input ยังต้องใช้ $1 — อย่าให้ generic ทำให้เผลอต่อ string ค่า input

Mission 7.3 — กวาด any + type guard 60 นาที

  1. ไล่หา any ทุกตัว → แทนด้วย type จริง / unknown + type guard
  2. จุดรับข้อมูลภายนอก (JSON.parse, req ที่ไม่ผ่าน Zod) → ใช้ type guard เช็คจริง
  3. error handling: type err: unknown ใน catch → narrow ก่อนใช้ (err instanceof Error)
✅ เทสต์: search "any" ในโปรเจกต์ = 0 (หรือมี comment เหตุผลทุกตัวที่เหลือ)

Mission 7.4 — test เขียว + push + ปิดเฟส 1 45 นาที

  1. npm run build ผ่าน strict · npm test เขียว
  2. README สรุปสมบูรณ์: Tech Stack (Node/Express/TS/PostgreSQL/Zod/JWT/Vitest), Features, Setup, What I Learned, AI Usage
  3. push — นี่คือ portfolio piece หลักของเฟส 1 · pin ไว้บน GitHub
✅ เทสต์: เปิด repo จากสายตา recruiter — README ครบ, โค้ด type แข็งแรง, test เขียว, commit history เล่าเรื่องได้
