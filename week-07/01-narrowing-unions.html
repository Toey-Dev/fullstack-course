Narrowing + Type Guards + Discriminated Unions

ใช้ type เป็นเครื่องมือออกแบบ — ทำให้สถานะผิด ๆ เกิดขึ้นไม่ได้เลย

🔑 คีย์เวิร์ดภาพจำประจำบท

🔬 บีบ type ให้แคบลง = narrowing 🕵️ ฟังก์ชันยืนยัน type = type guard 🏷 field บอกชนิด = discriminant (ok, type, status) 🚫 สถานะผิดเกิดไม่ได้ = make impossible states impossible

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Make impossible states impossible"

แทนที่จะเขียนโค้ดเช็คว่า "สถานะนี้ถูกไหม" ตลอด — ออกแบบ type ให้ "สถานะผิดเขียนไม่ได้ตั้งแต่แรก" เช่น API response: ถ้า success ต้องมี data, ถ้า fail ต้องมี error — ไม่มีทางเกิด "success แต่ไม่มี data" หรือ "success และมี error พร้อมกัน" type ที่ดีคือ type ที่ "บั๊กบางอย่างเป็นไปไม่ได้"

หลักการ 2: "TS narrow ตาม control flow — ใช้ให้เป็น"

TS ฉลาดพอตาม logic: ในกิ่ง if (typeof x === "string") มันรู้ว่า x เป็น string แน่ — นี่เรียก narrowing · เข้าใจมันทำให้เขียน TS ไหลลื่น ไม่ต้อง as (ซึ่งคือการโกหก compiler)

หลักการ 3: "Assertion (as) คือทางออกสุดท้าย ไม่ใช่ทางลัด"

data as Book = บอก TS "เชื่อฉัน นี่คือ Book" โดยไม่ได้เช็คจริง — ถ้าผิด พังตอนรัน (เสียประโยชน์ TS) ใช้ type guard (เช็คจริง) แทน · as ใช้เฉพาะตอนรู้มากกว่า TS จริง ๆ (เช่น DOM element ที่ TS เดาไม่ได้) และควรมี comment

ส่วนที่ 1: Type Narrowing — TS รู้ type แคบลงตาม logic

function format(value: string | number) {
  // ตรงนี้ value เป็น string | number (ยังกว้าง)
  if (typeof value === "string") {
    return value.toUpperCase();   // ✅ TS รู้: ในกิ่งนี้ value = string
  }
  return value.toFixed(2);        // ✅ TS รู้: เหลือแค่ number
}

ส่วนที่ 2: Custom Type Guard — ฟังก์ชันที่ยืนยัน type

// type predicate: "value is Book" บอก TS ว่าถ้า return true = เป็น Book
function isBook(value: unknown): value is Book {
  return (
    typeof value === "object" && value !== null &&
    "title" in value && "id" in value
  );
}

// ใช้งาน
function process(data: unknown) {
  if (isBook(data)) {
    data.title;   // ✅ TS มั่นใจว่า Book — auto-complete ให้
  }
}
เมื่อไหร่ใช้: ตอนรับข้อมูลที่ TS ไม่รู้ type (API response, JSON.parse, input ภายนอก) — guard เช็คจริงแล้วยืนยัน type ให้ ปลอดภัยกว่า as · ใช้คู่กับ unknown (type ที่บังคับต้องเช็คก่อนใช้)

ส่วนที่ 3: Discriminated Union — หัวใจของบทนี้

type ที่มี "field บอกชนิด" (discriminant) ทำให้ TS narrow ได้แม่นยำ และออกแบบให้สถานะผิดเกิดไม่ได้:

// แต่ละสถานะมี field ที่ต้องมีคู่กัน — ผิดไม่ได้
type ApiResponse<T> =
  | { ok: true;  data: T }          // success ต้องมี data
  | { ok: false; error: string };   // fail ต้องมี error

function handle(res: ApiResponse<Book>) {
  if (res.ok) {
    res.data;      // ✅ TS รู้: ต้องมี data (ไม่มี error)
    // res.error;  // ❌ TS เตือน: ไม่มี error ในกิ่งนี้
  } else {
    res.error;     // ✅ TS รู้: ต้องมี error (ไม่มี data)
  }
}

เทียบกับแบบหลวม { ok: boolean; data?: T; error?: string } — แบบนั้นเกิด { ok: true, error: "x" } (ขัดแย้ง) หรือ { ok: true } (ไม่มี data) ได้ → โค้ดต้องเช็คตลอด discriminated union ตัดปัญหานี้ที่ระดับ type

ส่วนที่ 4: ใช้จริง — state ของการโหลดข้อมูล

// state ที่ frontend (Week 1) ต้องจัดการ — เป็น discriminated union ได้สวย
type FetchState<T> =
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; message: string };

function render(state: FetchState<Book[]>) {
  switch (state.status) {
    case "loading": return showSpinner();
    case "success": return showList(state.data);    // TS รู้ว่ามี data
    case "error":   return showError(state.message); // TS รู้ว่ามี message
  }
  // ถ้าลืม case ไหน TS เตือน (exhaustiveness)
}

นี่คือ 4 สถานะ UI (loading/success/error/empty) จาก Week 1 ที่ตอนนี้ออกแบบเป็น type ที่ "ลืม case ไม่ได้" — TS บังคับจัดการครบ

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: ผลการดึง API การตลาดแต่ละแหล่ง — type Result = {ok:true, metrics} | {ok:false, reason} บังคับให้โค้ดสรุป report จัดการทั้งเคสสำเร็จและล้มเหลว ลืมไม่ได้ (ตรงกับ Promise.allSettled ที่เรียน Week 1)

🌏 อุตสาหกรรม: discriminated union คือ pattern หลักของ Redux action, state machine, API response ใน codebase TS ทุกที่ · "make impossible states impossible" เป็นปรัชญาที่ทีมระดับสูงใช้ออกแบบ — เห็นในโค้ด = สัญญาณ senior

📝 Quiz ท้ายบท

ข้อ 1: "make impossible states impossible" หมายความว่าอะไร ยกตัวอย่าง

ออกแบบ type ให้สถานะผิด/ขัดแย้งเขียนไม่ได้ตั้งแต่แรก แทนเช็คใน runtime · เช่น API response แบบ discriminated union: {ok:true, data} | {ok:false, error} — เกิด "ok:true แต่ไม่มี data" หรือ "ok:true และมี error" ไม่ได้เลย

ข้อ 2: ทำไม type guard ดีกว่า data as Book?

type guard เช็คจริง (runtime) แล้วยืนยัน type · as แค่บอก TS "เชื่อฉัน" โดยไม่เช็ค — ถ้า data ไม่ใช่ Book จริง พังตอนรัน (เสียประโยชน์ TS) · ใช้ guard กับข้อมูลภายนอก (API, JSON.parse) ที่ไม่รู้ type จริง

ข้อ 3: discriminated union กับ { ok: boolean; data?; error? } ต่างกันยังไง อันไหนดีกว่า?

แบบ optional เกิดสถานะขัดแย้งได้ ({ok:true, error}) → โค้ดต้องเช็คตลอด · discriminated union ผูก field กับ discriminant (ok:true→data, ok:false→error) สถานะผิดเขียนไม่ได้ + TS narrow ให้อัตโนมัติ → ดีกว่าชัดเจน

ข้อ 4: switch บน discriminated union แล้วลืม case — TS ช่วยยังไง?

exhaustiveness check — ถ้าจัดการครบทุก case TS โอเค, ถ้าลืม case (เพิ่ม status ใหม่แต่ลืมจัดการ) TS เตือน (โดยเฉพาะถ้าใส่ default: const _: never = state) — กันบั๊ก "ลืมจัดการสถานะใหม่"

  • ออกแบบ discriminated union (ApiResponse, FetchState)
