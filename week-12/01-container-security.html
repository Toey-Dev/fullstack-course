Week 12 · บทที่ 1

Container Security + Hardening

ทำให้ container ปลอดภัย — หัวข้อที่อดีต pentester อย่างคุณได้เปรียบที่สุด

🔑 คีย์เวิร์ดภาพจำประจำบท

👑 root ใน container = อันตราย 👤 USER node = สิทธิ์จำกัด 🪶 image เล็ก = ผิวโจมตีน้อย 🔍 scan = หา CVE ก่อน deploy 🔑 secret runtime ≠ build-time

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Least privilege — container ก็ต้องสิทธิ์น้อยที่สุด"

default container รันด้วย root · ถ้าแอปโดนเจาะ (RCE) ผู้โจมตีได้ root ใน container — และถ้า container escape ได้ก็คุม host · รัน non-root (USER node) จำกัดความเสียหาย: โดนเจาะก็ทำได้แค่สิทธิ์ user · หลัก least privilege เดียวกับ auth (Week 5/9) แต่ระดับ infra

หลักการ 2: "ผิวการโจมตีน้อย = ปลอดภัยกว่า (minimal image)"

image ที่มี shell, package manager, build tools เยอะ = เครื่องมือให้ผู้โจมตีใช้ต่อ · minimal image (slim/distroless) มีแค่สิ่งที่ app ต้องใช้ → ผู้โจมตีเจาะเข้าไปก็ไม่มีเครื่องมือ · นี่คือ "attack surface reduction" ที่คุณคุ้นจาก pentest

หลักการ 3: "Shift left — หาช่องโหว่ก่อน deploy ไม่ใช่หลังโดน"

scan image หา CVE (ช่องโหว่ที่รู้แล้ว) ตอน build ก่อนขึ้น production · ถูกกว่าและปลอดภัยกว่ารอให้โดนเจาะแล้วค่อยแก้ · นี่คือ DevSecOps: ฝัง security เข้า pipeline (จะทำใน CI Week 13) แทนตรวจทีหลัง

ส่วนที่ 1: Non-root User

# stage run — รัน non-root
FROM node:20-slim AS run
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY --from=build /app/dist ./dist
USER node                          # ⭐ node image มี user "node" ให้แล้ว
CMD ["node", "dist/server.js"]
🔐 มุม pentester: คุณเคยเจาะระบบที่ container รัน root แล้ว escape ไปคุม host ได้ — ตอนนี้รู้วิธีปิด · ถ้าต้อง custom user: RUN addgroup app && adduser -S -G app app แล้ว USER app · ระวัง: ไฟล์ที่ COPY มาต้องให้ user ใหม่อ่านได้ (chown ถ้าจำเป็น)

ส่วนที่ 2: Minimal Image — เลือก base ให้เล็ก

base imageขนาด (โดยประมาณ)มีอะไร
node:20~1GBเต็ม (มี build tools, shell, ฯลฯ)
node:20-slim~200MBเล็กลง (Debian slim)
node:20-alpine~130MBเล็กสุด (Alpine Linux) ⚠️ บาง native module มีปัญหา
distroless~120MBไม่มี shell เลย (ปลอดภัยสุด แต่ debug ยาก)

เลือก: slim เป็น default ปลอดภัย · alpine เล็กสุดแต่ระวัง native deps · distroless สำหรับ production ที่เน้น security สูงสุด (ไม่มี shell ให้ผู้โจมตีใช้)

ส่วนที่ 3: Image Scanning — หา CVE

# docker scout (built-in ใน Docker Desktop ใหม่)
docker scout cves library-api

# หรือ trivy (นิยมในวงการ security)
trivy image library-api

ผลที่ได้: list ช่องโหว่ (CVE) ใน image — เวอร์ชัน base image / dependency ที่มีช่องโหว่ที่รู้แล้ว · พร้อมระดับความรุนแรง (Critical/High/...):

วิธีแก้ CVE ที่เจอ: ① อัปเดต base image เป็นเวอร์ชันใหม่ (node:20.x ล่าสุด) ② อัปเดต dependency (npm audit fix) ③ ถ้าแก้ไม่ได้ตอนนี้ — ประเมินว่า exploitable จริงไหม (context) · Critical/High ควรแก้ก่อน deploy · นี่คือ workflow ที่ DevSecOps ทำจริงทุกวัน
🔐 จุดแข็งของคุณ: dev ทั่วไปเห็น list CVE แล้วงง — คุณในฐานะ pentester อ่านได้ว่าตัวไหน exploitable จริง ตัวไหน false alarm (context) · ความสามารถ "ประเมินความเสี่ยงจริง" คือสิ่งที่ทำให้ AppSec engineer มีค่ากว่า scanner

ส่วนที่ 4: Secret Management — build vs runtime

# ❌ ผิด — secret ใน Dockerfile (ติดใน image layer ถาวร!)
ENV JWT_SECRET=mysecret123       # ใครได้ image ก็ docker history เห็น
ARG DB_PASSWORD=secret           # ARG ก็ยังเห็นใน history

# ✅ ถูก — secret ตอน runtime (ไม่อยู่ใน image)
# ส่งผ่าน compose env / docker run -e / secret manager
# Dockerfile ไม่รู้ค่า secret เลย
🔐 มุม pentester: docker history <image> โชว์ทุก layer รวม ENV/ARG — secret ที่ใส่ตอน build ติดอยู่ใน image ถาวร แม้ลบ ENV ทีหลัง (อยู่ใน layer เก่า) · ใครได้ image (เช่น registry รั่ว) ขุดเจอ · กฎ: secret ส่งตอน run เท่านั้น (env/docker secret/vault) Dockerfile ไม่ควรรู้ค่า secret ใด ๆ

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: automation report service deploy เป็น container — non-root + scan + secret runtime ทำให้ปลอดภัยพอขึ้น production ได้ · ถ้า credential ของ GA4/Meta (มูลค่าสูง) อยู่ใน secret runtime ไม่ใช่ image = ปลอดภัยจริง

🌏 อุตสาหกรรม: container security เป็นหัวข้อหลักของ DevSecOps — non-root, image scanning, secret management เป็น checklist ที่ทีม security ตรวจทุก image ก่อน production · เครื่องมือ (Trivy, Snyk, docker scout) อยู่ใน CI ของบริษัทใหญ่ · คุณทำเป็น = พร้อมสาย B (Product Security) จริง

📝 Quiz ท้ายบท

ข้อ 1: ทำไมรัน container ด้วย root อันตราย แก้ยังไง?

ถ้าแอปโดนเจาะ (RCE) ผู้โจมตีได้ root ใน container + ถ้า escape ได้คุม host · แก้: USER node (non-root) — โดนเจาะก็ได้แค่สิทธิ์ user จำกัด · หลัก least privilege ระดับ infra

ข้อ 2: minimal image (distroless/slim) ปลอดภัยกว่ายังไง?

มีแค่สิ่งที่ app ต้องใช้ ไม่มี shell/build tools/package manager — ผู้โจมตีเจาะเข้าไปก็ไม่มีเครื่องมือใช้ต่อ · "attack surface reduction" + image เล็ก deploy เร็ว · distroless ไม่มี shell เลย = ปลอดภัยสุด

ข้อ 3 (security): ใส่ ENV JWT_SECRET=x ใน Dockerfile อันตรายยังไง?

secret ติดอยู่ใน image layer ถาวร — docker history โชว์ได้ แม้ลบ ENV ทีหลัง (อยู่ใน layer เก่า) · ใครได้ image (registry รั่ว) ขุดเจอ · กฎ: secret ส่งตอน runtime (env/secret manager) เท่านั้น Dockerfile ไม่รู้ค่า

ข้อ 4: image scan เจอ CVE Critical ควรทำยังไง (workflow)?

① อัปเดต base image เป็นเวอร์ชันล่าสุด ② npm audit fix (dependency) ③ ถ้าแก้ไม่ได้ — ประเมิน exploitable จริงไหมตาม context · Critical/High แก้ก่อน deploy · "shift left" หาก่อนขึ้น production ถูกกว่าโดนแล้วแก้

✅ Checkpoint

  • Dockerfile รัน non-root (USER node)
  • เลือก minimal image (slim/distroless) เหมาะสม
  • scan image (docker scout/trivy) + เข้าใจ CVE
  • secret runtime ไม่ฝัง image
