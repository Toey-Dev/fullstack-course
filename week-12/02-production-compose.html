Production Compose + Dockerize Frontend

ทำให้ระบบทนทานพอ production — และแพ็ค frontend ลง container ด้วย

🔑 คีย์เวิร์ดภาพจำประจำบท

❤️ ตรวจชีพจร = healthcheck 🔄 ล้มแล้วลุก = restart policy 🚧 จำกัดโควต้า = resource limits 📦 standalone = Next.js image เล็ก

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Production ต้อง self-healing — ล้มแล้วต้องลุกเอง"

ตอน dev container ล้มก็ start เอง · production ต้องมี restart policy (ล้ม = start ใหม่อัตโนมัติ) + healthcheck (ระบบรู้ว่า container "ป่วย" แม้ยังไม่ตาย เช่นค้าง) → restart/ดึงออกจาก traffic · ระบบที่ดูแลตัวเองได้ = uptime สูง

หลักการ 2: "Resource limit — กัน 1 service ทำพังทั้งเครื่อง"

ไม่จำกัด resource → service ที่ memory leak กิน RAM หมด → ทุก service บนเครื่องตาย (noisy neighbor) · ตั้ง limit (CPU/RAM) ให้แต่ละ service → service พังจำกัดความเสียหายแค่ตัวมัน · production มาตรฐาน

หลักการ 3: "แยก config dev กับ prod — ไม่ใช้ไฟล์เดียว"

dev ต้องการ hot reload, expose port debug · prod ต้องการ restart, limit, secret · แยกด้วย compose.yml (base) + compose.prod.yml (override) · run prod ด้วย -f ทั้งสอง · ไม่เอา config dev (เช่น expose DB port) ขึ้น prod (= ช่องโหว่)

ส่วนที่ 1: Healthcheck + Restart + Limits

services:
  app:
    build: .
    restart: unless-stopped          # ล้ม = start ใหม่ (ยกเว้นสั่งหยุดเอง)
    healthcheck:
      test: ["CMD", "node", "-e", "fetch('http://localhost:3000/api/health').then(r=>process.exit(r.ok?0:1))"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s              # รอ app warm up ก่อนเริ่มเช็ค
    deploy:
      resources:
        limits:
          cpus: "0.5"                # ใช้ได้ครึ่ง core
          memory: 512M               # ใช้ได้ 512MB

healthcheck ใช้ /api/health (ที่เราทำตั้งแต่ Week 2!) — ระบบ orchestrator (compose/k8s) ใช้รู้ว่า container พร้อมรับ traffic ไหม · restart + limit = ทนทาน + ไม่ล่มลามกัน

ส่วนที่ 2: แยก dev / prod compose

# compose.yml (base — ใช้ร่วม)
services:
  app:
    build: .
    environment:
      DATABASE_URL: ${DATABASE_URL}

# compose.override.yml (dev — auto โหลดตอน dev)
services:
  app:
    volumes: [".:/app"]          # hot reload
    ports: ["3000:3000"]

# compose.prod.yml (prod — override ด้วย -f)
services:
  app:
    restart: unless-stopped
    deploy: { resources: { limits: { memory: 512M } } }

# รัน prod:
docker compose -f compose.yml -f compose.prod.yml up -d
🔐 จุด security: ใน prod อย่า expose port ของ db/redis ออกนอก (dev อาจ expose เพื่อ debug ด้วย DBeaver) — prod ให้ app คุยกับ db ผ่าน network ภายในเท่านั้น ไม่เปิด 5432 ออกโลกภายนอก (ไม่งั้นใครก็ลองต่อ DB ได้)

ส่วนที่ 3: Dockerize Next.js Frontend

// next.config.js — เปิด standalone output (image เล็ก)
module.exports = { output: "standalone" };
# Dockerfile.frontend (multi-stage)
FROM node:20-slim AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build                  # Next build → .next/standalone

FROM node:20-slim AS run
WORKDIR /app
COPY --from=build /app/.next/standalone ./
COPY --from=build /app/.next/static ./.next/static
COPY --from=build /app/public ./public
USER node
EXPOSE 3000
CMD ["node", "server.js"]          # standalone มี server.js ให้

output: "standalone" = Next.js แพ็คเฉพาะไฟล์ที่ต้องรัน (ไม่เอา node_modules ทั้งก้อน) → image เล็กมาก · เพิ่ม frontend เข้า compose เป็น service ที่ 4

ส่วนที่ 4: ตัวอย่างประยุกต์

🏢 งานวี: ถ้า deploy full-stack (frontend + backend) ของลูกค้า — compose prod ที่มี healthcheck/restart ทำให้ระบบ uptime สูง (ล่มกลางคืน start เอง) · resource limit กันไม่ให้ลูกค้ารายหนึ่งกิน resource ของรายอื่น (ถ้า shared host)

🌏 อุตสาหกรรม: healthcheck + restart + resource limit เป็นพื้นฐานก่อนขึ้น orchestrator จริง (Kubernetes) · แยก dev/prod config + ไม่ expose internal service เป็นหลัก security ที่ทีมตรวจ · standalone output เป็นวิธี dockerize Next.js มาตรฐาน

📝 Quiz ท้ายบท

ข้อ 1: restart policy + healthcheck ทำให้ระบบ "self-healing" ยังไง?

restart: ล้ม → start ใหม่อัตโนมัติ · healthcheck: รู้ว่า container "ป่วย" (ค้าง แม้ยังไม่ตาย) → restart/ดึงออกจาก traffic · รวมกัน = ระบบดูแลตัวเอง uptime สูงโดยไม่ต้องคนเฝ้า · healthcheck ใช้ /api/health ที่ทำไว้ตั้งแต่ Week 2

ข้อ 2: ไม่ตั้ง resource limit แล้ว 1 service memory leak เกิดอะไร?

service นั้นกิน RAM เรื่อย ๆ จนหมด → ทุก service บนเครื่องเดียวกันตายตาม (noisy neighbor) · ตั้ง limit (memory/cpu) → service พังจำกัดความเสียหายแค่ตัวมัน ไม่ลามทั้งเครื่อง

ข้อ 3 (security): ทำไม prod ไม่ควร expose port ของ db/redis ออกนอก?

expose = ใครก็ลองต่อ DB/redis จากภายนอกได้ (brute-force, เจาะ) · prod ให้ app คุย db ผ่าน network ภายใน compose เท่านั้น ไม่เปิด 5432/6379 ออกโลก · dev อาจ expose เพื่อ debug แต่ prod ต้องปิด (แยก compose dev/prod)

ข้อ 4: ทำไมแยก compose dev กับ prod ดีกว่าไฟล์เดียว?

dev ต้องการ hot reload + expose port debug · prod ต้องการ restart + limit + ปิด port internal + secret · ความต้องการต่างกัน · ไฟล์เดียว = เผลอเอา config dev (expose DB) ขึ้น prod = ช่องโหว่ · แยก base + override ชัดเจน ปลอดภัย

✅ Checkpoint

  • healthcheck (/api/health) + restart policy + resource limits
  • แยก compose dev/prod + ไม่ expose internal service ใน prod
  • dockerize Next.js (standalone output)
