Week 12 · Mission ใหญ่

🛡 Harden + Production-ize Docker

ทำให้ container ปลอดภัยและพร้อม production — พร้อมเขียน hardening report (ซ้อมงาน AppSec)

ทำไม Mission นี้คือ portfolio สาย security

หลังจบ คุณจะมี container ที่ non-root, scan ผ่าน, prod-ready (healthcheck/limit/restart) + hardening report ที่เขียนเองว่า "ทำให้ปลอดภัยขึ้นยังไง แก้ CVE อะไร" — นี่คือเอกสารที่ตำแหน่ง DevSecOps/AppSec อยากเห็น และเป็นการซ้อมเขียนรายงานแบบ pentest ที่คุณถนัด แต่อยู่ฝั่ง "สร้างกำแพง"

🪜 แบ่งเป็น 4 Mission ย่อย

Mission 12.1 — Harden Dockerfile 60 นาที

  1. เพิ่ม USER node (non-root) ใน stage run · ตรวจว่าไฟล์ permission ถูก
  2. ใช้ minimal image (slim หรือ distroless ถ้าทำได้)
  3. ตรวจว่าไม่มี secret ใน Dockerfile (ENV/ARG) — secret runtime เท่านั้น
✅ เทสต์: docker run แล้ว whoami ใน container = node (ไม่ใช่ root)

Mission 12.2 — Scan + แก้ CVE 60 นาที

  1. scan image (docker scout cves หรือ trivy)
  2. แก้ Critical/High: อัปเดต base image + npm audit fix
  3. scan ซ้ำ — บันทึก "ก่อน/หลัง" (จำนวน CVE ลดลง)
มุม pentester: CVE ที่แก้ไม่ได้ทันที — ประเมิน exploitable จริงไหม + บันทึกเหตุผลในรายงาน

Mission 12.3 — Production compose + frontend 75 นาที

  1. compose.prod: healthcheck (/api/health) + restart + resource limits ทุก service
  2. prod ไม่ expose port db/redis ออกนอก (เฉพาะ network ภายใน)
  3. dockerize Next.js frontend (standalone) + เพิ่มเป็น service ที่ 4
✅ เทสต์: prod compose up → healthcheck เขียว · db port ไม่เปิดออกนอก · frontend + backend คุยกันได้

Mission 12.4 — Hardening Report + push 45 นาที

  1. เขียน SECURITY.md: สิ่งที่ทำเพื่อ harden (non-root, minimal, scan, secret, no expose) + CVE ก่อน/หลัง + เหตุผล
  2. push — README + SECURITY.md + AI Usage
  3. นี่คือเอกสารที่ใส่ portfolio สาย security ได้
✅ เทสต์: SECURITY.md อ่านแล้วเข้าใจว่า "ทำไมระบบนี้ปลอดภัย" — เหมือนรายงานที่ลูกค้า/ทีม security อ่าน
