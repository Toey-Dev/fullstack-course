Week 5 — JWT Authentication + Protect Routes

สัปดาห์ที่จุดแข็ง pentester ของคุณเปล่งประกาย — สร้างระบบ login ที่ปลอดภัยจริง

🎯 เป้าหมายของสัปดาห์นี้

จนถึงตอนนี้ใคร ๆ ก็เรียก API ของคุณได้ — Week 5 ใส่ "ยาม" ให้ระบบ: ต้อง login ก่อนถึงทำบางอย่างได้ คุณจะสร้าง register/login ที่ เก็บรหัสผ่านอย่างปลอดภัย (hash ไม่ใช่ plaintext), ออก JWT token, และทำ middleware ป้องกัน route — พร้อมตอบคำถาม "token เก็บที่ไหน" ที่ค้างจาก Week 1

  • เข้าใจความต่าง Authentication (คุณเป็นใคร) vs Authorization (คุณทำอะไรได้)
  • hash password ด้วย bcrypt — ไม่เก็บ plaintext เด็ดขาด
  • เข้าใจ JWT ถึงแก่น: 3 ส่วน, signed ≠ encrypted, เก็บที่ไหนปลอดภัย
  • ทำ auth middleware ป้องกัน route + role-based (admin vs user)
  • ยืม-คืนหนังสือด้วย transaction (สำเร็จทั้งคู่ หรือล้มทั้งคู่)
  • เขียน test ครอบ auth flow (register/login/protected/403)
ทำไมสัปดาห์นี้คือจุดขายของคุณ: auth คือจุดที่ระบบส่วนใหญ่ถูกเจาะ — ความรู้ pentester ทำให้คุณเห็นช่องโหว่ที่ dev ทั่วไปมองข้าม (timing attack, token leak, weak hash) Week 5 คือ portfolio piece ที่เล่าใน interview สาย security ได้ทรงพลังที่สุด

🧹 Clean Code ประจำ Week 5

1. ไม่เก็บ password เป็น plaintext เด็ดขาด

❌ หายนะ (รั่วแล้วจบ)

INSERT INTO users (email, password)
VALUES ($1, $2)  // password ดิบ!
// DB รั่ว = รหัสทุกคนหลุด

✅ hash ก่อนเก็บเสมอ

const hash = await bcrypt.hash(password, 10);
INSERT INTO users (email, password)
VALUES ($1, $2)  // เก็บ hash
// DB รั่ว = ได้แค่ hash ถอดกลับไม่ได้

2. ไม่ส่ง password (แม้ hash) กลับใน response

❌ ส่ง user ทั้ง object

res.json({ ok: true, data: user });
// มี password hash ติดไปด้วย

✅ ตัด field ลับออก

const { password, ...safe } = user;
res.json({ ok: true, data: safe });
กฎทอง: hash password เสมอ (bcrypt) · ไม่ส่ง password/hash กลับ client · JWT secret อยู่ใน .env · error login ตอบกลาง ๆ ("email หรือรหัสผ่านไม่ถูกต้อง") ไม่บอกว่าผิดตรงไหน (กัน enumeration)
