Week 5 · บทที่ 1

Authentication + JWT — login ที่ปลอดภัยจริง

สร้างระบบยืนยันตัวตนที่ผ่านสายตา pentester — และเข้าใจ JWT ถึงแก่น ไม่ใช่แค่ copy มาใช้

🔑 คีย์เวิร์ดภาพจำประจำบท

🆔 คุณเป็นใคร = Authentication 🚪 คุณทำอะไรได้ = Authorization 🔨 บดทางเดียว = hash (bcrypt) 🎫 บัตรผ่านมีลายเซ็น = JWT ✍️ signed ≠ encrypted (อ่านได้ แต่ปลอมไม่ได้)

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Authentication ≠ Authorization" (2 คำที่คนสับสน)

Authentication (AuthN) = "พิสูจน์ว่าคุณเป็นใคร" (login ด้วย email+password) · Authorization (AuthZ) = "คุณมีสิทธิ์ทำอะไรบ้าง" (admin ลบได้ user ลบไม่ได้) · ทำ AuthN ก่อนเสมอ แล้วค่อย AuthZ — เทียบสนามบิน: เช็คพาสปอร์ต (คุณเป็นใคร) → เช็คตั๋ว (คุณขึ้นเครื่องไหนได้)

หลักการ 2: "Password เป็นความลับที่แม้แต่เราก็ไม่ควรรู้"

ระบบที่ดี เก็บ password ไม่ได้ (เก็บแค่ hash) — แม้ DB รั่ว แม้ admin ทุจริต ก็เอา password จริงไปไม่ได้ hash คือ "บดทางเดียว" — แปลง password → ตัวเลขยาว ๆ ที่ ถอดกลับไม่ได้ ตอน login เอา password ที่กรอกมา hash ใหม่แล้วเทียบ hash ที่เก็บ ตรงกัน = ผ่าน

หลักการ 3: "Stateless = server ไม่ต้องจำว่าใคร login"

JWT ทำให้ server ไม่ต้องเก็บ session ในหน่วยความจำ — ทุกอย่างที่ต้องรู้ (ใคร, role อะไร) อยู่ใน token เอง server แค่ตรวจ "ลายเซ็น" ว่าจริงไหม ข้อดี: scale ได้ (server หลายตัวไม่ต้องแชร์ session) ข้อเสีย: ยกเลิก token กลางคันยาก (จะคุยใน "ข้อจำกัด")

ส่วนที่ 1: Hash Password ด้วย bcrypt

npm install bcrypt

// ตอน register — hash ก่อนเก็บ
const hash = await bcrypt.hash(password, 10);   // 10 = ความแรง (cost)
// เก็บ hash ลง DB (ไม่ใช่ password)

// ตอน login — เทียบ password ที่กรอก กับ hash ที่เก็บ
const ok = await bcrypt.compare(inputPassword, user.password);
if (!ok) return res.status(401).json({ ok: false, error: "อีเมลหรือรหัสผ่านไม่ถูกต้อง" });
ทำไม bcrypt ไม่ใช่ MD5/SHA: bcrypt ออกแบบมาให้ "ช้าโดยตั้งใจ" (cost factor) — ทำให้ผู้โจมตี brute-force ช้ามาก · MD5/SHA เร็วเกินไป crack ได้เป็นพันล้านครั้ง/วินาที · bcrypt ยังใส่ "salt" อัตโนมัติ (สุ่มต่อท้ายก่อน hash) ทำให้ password เดียวกัน 2 คน ได้ hash ต่างกัน — กัน rainbow table
🔐 มุม pentester ของคุณ: คุณเคยเจอระบบที่เก็บ password เป็น MD5/plaintext ตอน pentest มาแล้ว — ตอนนี้รู้แล้วว่า "ทำไมมันพัง" และ "ทำให้ถูกยังไง" นี่คือความรู้ที่ทำให้คุณ review โค้ดคนอื่นได้คมกว่า dev ทั่วไป

ส่วนที่ 2: JWT — เข้าใจถึงแก่น (ไม่ใช่แค่ copy)

JWT (JSON Web Token) = "บัตรผ่านที่มีลายเซ็น" มี 3 ส่วนคั่นด้วยจุด xxxxx.yyyyy.zzzzz:

npm install jsonwebtoken

// ตอน login สำเร็จ — ออก token
const token = jwt.sign(
  { userId: user.id, role: user.role },   // payload (ข้อมูลที่อยากแนบ)
  process.env.JWT_SECRET,                  // secret (อยู่ใน .env!)
  { expiresIn: "1h" }                      // หมดอายุ
);
res.json({ ok: true, token });

// ตอนตรวจ token (ใน middleware)
const decoded = jwt.verify(token, process.env.JWT_SECRET);  // ปลอม/หมดอายุ = throw
// decoded = { userId: 7, role: "admin", iat, exp }
🔐 จุดที่ pentester ต้องรู้ (และ dev ทั่วไปพลาด):
① payload อ่านได้ — เป็นแค่ base64 ใครเอา token ไปแปะ jwt.io ก็เห็นข้อมูลข้างใน → ห้ามใส่ password/ข้อมูลลับใน payload
② signature กันแค่ "ปลอม" ไม่ได้กัน "อ่าน" — JWT คือ signed ไม่ใช่ encrypted
③ JWT_SECRET หลุด = ปลอม token ได้ทั้งระบบ → อยู่ใน .env เสมอ ยาว ๆ สุ่ม ๆ

ส่วนที่ 3: Token เก็บที่ไหน — เฉลยคำถามค้างจาก Week 1

จำได้ไหม Week 1 เราคุยว่า localStorage โดน XSS ขโมยได้ — ตอนนี้ token มาจริงแล้ว ต้องตัดสินใจ:

เก็บที่ข้อดีความเสี่ยง
localStorageง่าย ใช้กับ API ข้าม domain ง่าย🔴 XSS อ่านได้ (script อ่าน localStorage)
HttpOnly Cookie🟢 JS อ่านไม่ได้ → XSS ขโมยไม่ได้ต้องระวัง CSRF (Week 17)

คำตอบที่ตรงไปตรงมา: ไม่มีที่ไหนสมบูรณ์แบบ — เลือกตาม "ภัยที่กลัวกว่า": ระบบที่มีเนื้อหาผู้ใช้เยอะ (เสี่ยง XSS) → HttpOnly Cookie · SPA ที่คุม XSS ได้ดี → localStorage ก็ใช้ได้ถ้า token อายุสั้น สิ่งที่ผิดเสมอ: เก็บ token อายุยาว (เป็นเดือน) ใน localStorage โดยไม่มี refresh — โดน XSS ทีเดียวเสียยาว

ความรู้ที่ทำให้คุณดู senior: ตอบ interview ว่า "ขึ้นกับ threat model" + อธิบาย trade-off ได้ = คะแนนเต็ม · ตอบว่า "localStorage เสมอ" หรือ "Cookie เสมอ" = ยังไม่เข้าใจลึก

ส่วนที่ 4: Register / Login Flow เต็ม

🔐 จุด security ที่ต้องระวังตอน login: ถ้า email ไม่มี ตอบ "ไม่พบอีเมล" / ถ้ารหัสผิด ตอบ "รหัสผิด" — ผู้โจมตีใช้แยกได้ว่า email ไหน "มีในระบบ" (user enumeration) → ตอบกลาง ๆ เหมือนกัน "อีเมลหรือรหัสผ่านไม่ถูกต้อง" ทั้ง 2 เคส

ส่วนที่ 5: ข้อจำกัดของ JWT (เล่าได้ = ดู senior)

JWT ไม่ใช่ยาวิเศษ — interviewer ชอบถามข้อเสีย:

  • ยกเลิกกลางคันยาก — token ที่ออกไปแล้ว valid จนหมดอายุ ต่อให้ user โดนแบนก็ยังใช้ได้จนหมดเวลา (ทางแก้: token อายุสั้น + refresh token, หรือ blacklist)
  • payload โตทำให้ทุก request หนักขึ้น — token แนบไปทุก request อย่าใส่ข้อมูลเยอะ
  • secret หลุด = จบ — ปลอม token ได้ทั้งระบบ

ทางเลือก: session-based (server เก็บ session, ยกเลิกง่าย แต่ scale ยากกว่า) — เลือกตามระบบ ไม่มีถูกผิดเด็ดขาด

ส่วนที่ 6: ตัวอย่างประยุกต์

🏢 งานวี: ถ้าทำ dashboard ให้ลูกค้าหลายรายดูสถิติ — แต่ละลูกค้าต้อง login เห็นแค่ข้อมูลตัวเอง (authZ) JWT แนบ clientId ใน payload, middleware เช็คว่า request ขอข้อมูล clientId ตรงกับใน token ไหม — กันลูกค้า A แอบดูข้อมูลลูกค้า B

🌏 อุตสาหกรรม: JWT เป็นมาตรฐานของ API สมัยใหม่ (microservices ส่ง token ต่อกันได้) · แต่ fintech/banking มักใช้ session + token อายุสั้นมาก เพราะ "ยกเลิกได้ทันที" สำคัญกว่า scale · เข้าใจ trade-off = ผ่าน interview สาย security

📝 Quiz ท้ายบท

ข้อ 1: Authentication กับ Authorization ต่างกันยังไง ทำอันไหนก่อน?

Authentication = "คุณเป็นใคร" (login) · Authorization = "คุณทำอะไรได้" (สิทธิ์) · ทำ AuthN ก่อนเสมอ แล้วค่อย AuthZ เทียบสนามบิน: เช็คพาสปอร์ต (เป็นใคร) ก่อน เช็คตั๋ว (ขึ้นเครื่องไหนได้)

ข้อ 2: ทำไม bcrypt ดีกว่า MD5/SHA สำหรับ password?

① bcrypt "ช้าโดยตั้งใจ" (cost factor) ทำให้ brute-force ช้ามาก ส่วน MD5/SHA เร็วเกินไป crack ได้พันล้าน/วินาที ② bcrypt ใส่ salt อัตโนมัติ → password เดียวกัน 2 คนได้ hash ต่างกัน กัน rainbow table

ข้อ 3 (security): เพื่อนบอก "JWT เข้ารหัสข้อมูลแล้ว ใส่ password ใน payload ได้" — ผิดยังไง?

ผิด! JWT คือ signed ไม่ใช่ encrypted — payload เป็นแค่ base64 ใครเอา token ไปแปะ jwt.io ก็อ่านได้หมด signature กันแค่ "ปลอม" ไม่ได้กัน "อ่าน" → ห้ามใส่ password/ข้อมูลลับใน payload เด็ดขาด

ข้อ 4 (security): login ตอบ "ไม่พบอีเมล" กับ "รหัสผ่านผิด" แยกกัน อันตรายยังไง?

User Enumeration — ผู้โจมตีลองหลาย email ดูว่าอันไหนตอบ "รหัสผิด" (= email นี้มีในระบบ) เก็บรายชื่อ user จริงไปโจมตีต่อ แก้: ตอบกลาง ๆ เหมือนกันทั้ง 2 เคส "อีเมลหรือรหัสผ่านไม่ถูกต้อง"

ข้อ 5: ข้อเสียของ JWT ที่ interviewer ชอบถามคืออะไร?

ยกเลิก token กลางคันยาก — token valid จนหมดอายุ ต่อให้ user โดนแบนก็ยังใช้ได้ · ทางแก้: token อายุสั้น + refresh token หรือ blacklist · ตอบได้ = แสดงว่าเข้าใจ trade-off ไม่ได้ท่องมา

✅ Checkpoint

  • hash password ด้วย bcrypt + เทียบด้วย compare
  • ออก/ตรวจ JWT ด้วย jsonwebtoken (secret ใน .env)
  • อธิบายได้ว่า JWT signed≠encrypted + token เก็บที่ไหนปลอดภัย
  • login error ตอบกลาง ๆ (กัน enumeration)
