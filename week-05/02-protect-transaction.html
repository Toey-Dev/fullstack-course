Week 5 · บทที่ 2

Protect Routes + Authorization + Transaction

ใส่ยามให้ route · แยกสิทธิ์ admin/user · และทำให้ "ยืม-คืน" สำเร็จทั้งคู่หรือล้มทั้งคู่

🔑 คีย์เวิร์ดภาพจำประจำบท

💂 ยามหน้าประตู = auth middleware 🪪 req.user = ตัวตนที่ยามแปะให้ 401 = ไม่รู้จักคุณ · 403 = รู้จักแต่ห้ามเข้า ⚛️ ทั้งหมดหรือไม่เลย = transaction (atomic)

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Auth เป็น middleware — ยามชั้นเดียว คุมหลายประตู"

แทนที่จะเช็ค token ในทุก route (ซ้ำ ลืมง่าย) — ทำเป็น middleware ชั้นเดียว (จาก Week 2) วางหน้า route ที่ต้องป้องกัน ยามตรวจ token, ถ้าผ่านแปะ req.user ให้ แล้วส่งต่อ · ถ้าไม่ผ่านปฏิเสธทันที — route ข้างในไม่ต้องห่วงเรื่อง auth เลย

หลักการ 2: "Least privilege — ให้สิทธิ์น้อยที่สุดเท่าที่จำเป็น"

user ทั่วไปไม่ควรลบหนังสือได้ · ทุก action ต้องถามว่า "ใครควรทำได้บ้าง" แล้วกันคนอื่นออก default ควรเป็น "ห้าม" แล้วเปิดสิทธิ์เฉพาะที่จำเป็น — ไม่ใช่ "เปิดหมด" แล้วค่อยปิด (พลาดที่เดียว = รั่ว)

หลักการ 3: "Atomicity — บางงานต้องสำเร็จทั้งหมด หรือไม่เกิดเลย"

ยืมหนังสือ = (1) สร้าง record การยืม + (2) เปลี่ยนสถานะหนังสือเป็น "ถูกยืม" — ถ้าทำข้อ 1 สำเร็จแต่ข้อ 2 พัง (ไฟดับ) = ข้อมูลเพี้ยน (มีบันทึกยืมแต่หนังสือยังว่าง) · transaction บังคับว่า "ทำครบทุกข้อ หรือ rollback ทั้งหมด" — ไม่มีสภาพครึ่ง ๆ กลาง ๆ

ส่วนที่ 1: Auth Middleware — ยามตรวจ token

// src/middlewares/auth.js
const jwt = require("jsonwebtoken");

const auth = (req, res, next) => {
  const header = req.headers.authorization;          // "Bearer xxxxx"
  if (!header?.startsWith("Bearer ")) {
    return res.status(401).json({ ok: false, error: "ไม่ได้เข้าสู่ระบบ" });
  }
  const token = header.split(" ")[1];
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded;        // ⭐ แปะตัวตนให้ route ข้างในใช้ต่อ
    next();                    // ผ่านยาม → ไป route
  } catch {
    return res.status(401).json({ ok: false, error: "token ไม่ถูกต้องหรือหมดอายุ" });
  }
};
module.exports = auth;
// ใช้: วาง middleware หน้า route ที่ต้องล็อกอิน
router.get("/me", auth, usersController.me);             // ต้อง login
router.post("/borrow", auth, borrowController.create);   // ต้อง login
router.get("/books", booksController.list);              // เปิด public (ไม่ใส่ auth)
Bearer token คืออะไร: client ส่ง token มาใน header Authorization: Bearer xxxxx — "Bearer" = "ผู้ถือ" (ใครถือ token นี้ = คนนั้น) มาตรฐาน OAuth · frontend (Week 1) จะแนบ header นี้ทุก request ที่ต้อง auth

ส่วนที่ 2: Authorization — แยกสิทธิ์ตาม role

// middleware เช็ค role (ใช้ต่อจาก auth)
const requireRole = (...roles) => (req, res, next) => {
  if (!roles.includes(req.user.role)) {
    return res.status(403).json({ ok: false, error: "ไม่มีสิทธิ์ทำรายการนี้" });
  }
  next();
};

// ใช้: ต้อง login (auth) + เป็น admin (requireRole)
router.delete("/books/:id", auth, requireRole("admin"), booksController.remove);
401 vs 403 (ทบจาก Week 2): 401 = ยังไม่ได้ login (ไม่รู้จักคุณ) · 403 = login แล้วแต่ไม่มีสิทธิ์ (รู้จักแล้วแต่ห้ามเข้า) — สับสน 2 อันนี้ = สัญญาณว่ายังไม่เข้าใจ auth

ส่วนที่ 3: Transaction — ยืม-คืนแบบ atomic

การยืมหนังสือมีหลายขั้นที่ต้อง "สำเร็จพร้อมกัน" — ใช้ transaction บังคับ:

// borrow ด้วย transaction (pg)
const client = await pool.connect();
try {
  await client.query("BEGIN");                          // เริ่ม transaction

  const book = await client.query(
    "SELECT * FROM books WHERE id = $1 FOR UPDATE",      // ล็อกแถวกัน race
    [bookId]
  );
  if (!book.rows[0]?.available) throw new Error("หนังสือถูกยืมแล้ว");

  await client.query(
    "INSERT INTO borrows (user_id, book_id) VALUES ($1, $2)", [userId, bookId]
  );
  await client.query(
    "UPDATE books SET available = false WHERE id = $1", [bookId]
  );

  await client.query("COMMIT");                          // สำเร็จทั้งหมด → ยืนยัน
} catch (err) {
  await client.query("ROLLBACK");                        // พังข้อใดข้อหนึ่ง → ย้อนทั้งหมด
  throw err;
} finally {
  client.release();                                      // คืน connection เข้า pool
}
ทำไม transaction สำคัญ: ถ้าไม่มี — INSERT borrow สำเร็จ แต่ UPDATE available พัง (ไฟดับ/error) = ข้อมูลเพี้ยน (มีบันทึกยืม แต่หนังสือยังโชว์ว่าง คนอื่นยืมซ้ำได้) · COMMIT/ROLLBACK บังคับ "ทั้งหมดหรือไม่เลย"
FOR UPDATE = ล็อกแถวระหว่าง transaction กัน 2 คนยืมเล่มเดียวพร้อมกัน (race condition)

ส่วนที่ 4: ทดสอบ auth (เชื่อม Week 3)

// test ที่ต้องมี — ทั้งเคสผ่านและโดนปฏิเสธ
describe("Protected routes", () => {
  it("คืน 401 เมื่อไม่ส่ง token", async () => {
    const res = await request(app).get("/api/me");
    expect(res.status).toBe(401);
  });
  it("คืน 200 + ข้อมูล เมื่อ token ถูกต้อง", async () => {
    const res = await request(app).get("/api/me")
      .set("Authorization", `Bearer ${validToken}`);
    expect(res.status).toBe(200);
  });
  it("คืน 403 เมื่อ user ทั่วไปลบหนังสือ", async () => {
    const res = await request(app).delete("/api/books/1")
      .set("Authorization", `Bearer ${userToken}`);   // ไม่ใช่ admin
    expect(res.status).toBe(403);
  });
});

test เคส "โดนปฏิเสธ" (401/403) สำคัญพอ ๆ กับเคสผ่าน — พิสูจน์ว่ายามทำงานจริง ไม่ใช่ปล่อยผ่านหมด (มุม pentester: ทดสอบว่ากำแพงกันได้จริง)

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: dashboard ลูกค้าหลายราย — auth middleware เช็ค login + requireRole/ownership เช็คว่า request ขอข้อมูล clientId ตรงกับใน token ไหม กันลูกค้าแอบดูข้อมูลกัน (Broken Access Control = OWASP #1 ปี 2021)

🌏 อุตสาหกรรม: transaction คือหัวใจของ fintech — โอนเงิน = หักบัญชี A + เพิ่มบัญชี B ต้อง atomic (หักแล้วเพิ่มพัง = เงินหาย) · middleware-based auth เป็นมาตรฐานทุก API · Broken Access Control คือช่องโหว่ที่ pentester เจอบ่อยสุด — คุณได้เปรียบ

📝 Quiz ท้ายบท

ข้อ 1: ทำไมทำ auth เป็น middleware ดีกว่าเช็ค token ในทุก route?

เขียนครั้งเดียว ใช้ได้ทุก route ที่ต้องป้องกัน (DRY) · ลืมยากกว่า (วาง middleware หน้า route ชัดเจน) · route ข้างในไม่ต้องห่วง auth · แก้ logic auth ที่เดียวมีผลทุก route — เป็นหลัก Separation of Concerns เดียวกับทั้งคอร์ส

ข้อ 2: ผู้ใช้ที่ login แล้วแต่ไม่ใช่ admin พยายามลบหนังสือ — ตอบ status อะไร ทำไมไม่ใช่ 401?

403 Forbidden — เพราะ "รู้แล้วว่าเป็นใคร (login ผ่าน) แต่ไม่มีสิทธิ์" · 401 ใช้เมื่อ "ไม่รู้ว่าเป็นใคร" (ยังไม่ login) user คนนี้ login แล้ว = ผ่าน authN แต่ไม่ผ่าน authZ → 403

ข้อ 3: ยืมหนังสือไม่ใช้ transaction แล้ว INSERT สำเร็จ UPDATE พัง — เกิดอะไร?

ข้อมูลเพี้ยน — มีบันทึกการยืม (borrows) แต่หนังสือยังโชว์ available=true → คนอื่นยืมเล่มเดียวกันซ้ำได้ · transaction (BEGIN/COMMIT/ROLLBACK) บังคับ "ทั้งหมดหรือไม่เลย" พังข้อใด ROLLBACK ย้อนทั้งหมด ไม่เหลือสภาพครึ่ง ๆ

ข้อ 4 (security): ทำไม test เคส "โดนปฏิเสธ" (401/403) สำคัญพอ ๆ กับเคสผ่าน?

เพราะมันพิสูจน์ว่า "กำแพงกันได้จริง" — ระบบที่ test แค่เคสผ่าน อาจปล่อยทุกคนเข้าได้โดยไม่รู้ตัว (ยามไม่ทำงานแต่ test เขียว) มุม pentester: ต้องทดสอบว่ากันคนที่ "ไม่ควรเข้า" ได้จริง ไม่ใช่แค่ปล่อยคนที่ควรเข้า

✅ Checkpoint

  • auth middleware ตรวจ token + แปะ req.user
  • requireRole แยกสิทธิ์ admin/user (401 vs 403 ถูก)
  • borrow/return ใช้ transaction (atomic + FOR UPDATE)
  • test ครอบเคสผ่าน + โดนปฏิเสธ (401/403)
