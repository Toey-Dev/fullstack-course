Structured Logging ด้วย pino

เปลี่ยน console.log มั่ว ๆ เป็น log ที่ค้นได้ ตามรอยได้ ในระบบจริง

🔑 คีย์เวิร์ดภาพจำประจำบท

📋 log เป็น JSON ค้นได้ = structured 🎚 error/warn/info/debug = log levels 🧵 ตามรอย request เดียว = correlation ID 🙈 ไม่ log ความลับ = redact

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "Log เพื่อ 'ค้นหา' ไม่ใช่ 'อ่าน'"

console.log string สวยสำหรับคนอ่านทีละบรรทัด — แต่ production มี log ล้านบรรทัด/วัน คุณต้อง "ค้น" (filter userId นี้, action นี้, error ช่วงนี้) · structured log (JSON) ทำให้ log system (Datadog/Loki/CloudWatch) ค้น/filter/alert ได้ · log ที่ค้นไม่ได้ = log ที่ไร้ค่าตอนวิกฤต

หลักการ 2: "Correlation ID — ตามรอย 1 request ข้ามหลายจุด"

1 request ผ่านหลาย log (เข้า → query DB → error) · ถ้าทุก log มี "requestId" เดียวกัน คุณ filter เห็น lifecycle ทั้งหมดของ request นั้น · ไม่มี = log ปนกันมั่ว แยกไม่ออกว่าอันไหนของ request ไหน · นี่คือสิ่งที่ทำให้ debug production เป็นไปได้

หลักการ 3: "Log level — แยกเสียงดังจากเสียงสำคัญ"

error (พังจริง) / warn (น่าสงสัย) / info (เหตุการณ์สำคัญ) / debug (รายละเอียด dev) · production เปิด info ขึ้นไป (debug ปิด — ไม่งั้น log ท่วม) · level ทำให้ filter "เอาแค่ error" ตอนวิกฤตได้ · console.log ไม่มี level = ทุกอย่างเสียงดังเท่ากัน

ส่วนที่ 1: pino — structured logger

npm install pino pino-http

// src/logger.ts
import pino from "pino";
export const logger = pino({
  level: process.env.LOG_LEVEL ?? "info",
  redact: ["req.headers.authorization", "password", "*.token"],  // ⭐ ไม่ log secret
});

// ใช้งาน — ส่ง object (context) + ข้อความ
logger.info({ userId: 7, bookId: 3, action: "borrow" }, "book borrowed");
// output: {"level":30,"time":...,"userId":7,"bookId":3,"action":"borrow","msg":"book borrowed"}
logger.error({ err }, "failed to borrow");
logger.warn({ attempts: 5 }, "too many login attempts");

pino เร็วมาก (ออกแบบมาเพื่อ production), log เป็น JSON · field แรก = context (object), field สอง = ข้อความ · redact ปกปิด field อ่อนไหวอัตโนมัติ

ส่วนที่ 2: Request Logging + Correlation ID

import pinoHttp from "pino-http";
import { randomUUID } from "crypto";

// middleware — log ทุก request + แปะ requestId
app.use(pinoHttp({
  logger,
  genReqId: (req) => req.headers["x-request-id"] ?? randomUUID(),  // correlation ID
}));

// ในแต่ละ route ใช้ req.log (มี requestId ติดอยู่แล้ว)
app.post("/api/borrow", (req, res) => {
  req.log.info({ bookId: req.body.bookId }, "borrow requested");
  // ทุก log ของ request นี้มี reqId เดียวกัน → filter ตามรอยได้
});

ส่วนที่ 3: 🔐 ไม่ log Secret/PII

มุม pentester — log leak คือช่องโหว่จริง: log ไปอยู่ใน log system ที่อาจเข้าถึงได้กว้างกว่า DB (ทีม ops, third-party log service) · ถ้า log มี password, token, credit card, เลขบัตรประชาชน → ข้อมูลรั่วผ่าน log · คุณเคยเจอระบบที่ log token แล้วใช้ขโมย session มาแล้วตอน pentest
ห้าม logเหตุผล
password (แม้ hash)ความลับ
JWT/token/API keyใช้สวมรอยได้
credit card, เลขบัตร ปชช.PII — กฎหมาย (PDPA/GDPR)
request body เต็ม (อาจมี secret)เลือก field ที่ปลอดภัย

pino redact ปกปิดให้อัตโนมัติ (แทนด้วย [Redacted]) · แต่ดีกว่าคือ "เลือก log เฉพาะ field ที่ปลอดภัย" ตั้งแต่ต้น (userId แทน user ทั้ง object)

ส่วนที่ 4: Log ใน Docker/Production

# container log ออก stdout — Docker/orchestrator เก็บให้
docker compose logs -f app
# production: ส่ง stdout ไป log aggregator (Loki/Datadog/CloudWatch)
# pino-pretty สำหรับ dev เท่านั้น (production เก็บ JSON ดิบ)
หลักคิด (12-Factor): app ไม่ควรเขียน log ลงไฟล์เอง — เขียนออก stdout แล้วให้ platform (Docker/cloud) จัดการเก็บ/ส่งต่อ · แยกหน้าที่: app สร้าง log, platform จัดการ log · ทำให้ scale/ย้าย platform ง่าย

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: automation service — log แต่ละ report job (clientId, แพลตฟอร์ม, สำเร็จ/ล้มเหลว) แบบ structured · เมื่อ report ลูกค้าผิด คุณ filter clientId นั้นเห็นว่าแหล่งไหนล่ม · requestId ตามรอย job เดียวข้าม API หลายตัว · และไม่ log API key ของ GA4/Meta (redact) — ป้องกัน credential รั่วผ่าน log

🌏 อุตสาหกรรม: structured logging + correlation ID เป็นมาตรฐานของ production ทุกที่ (debug distributed system ไม่ได้ถ้าไม่มี) · log security (ไม่รั่ว PII/secret) เป็นข้อบังคับ PDPA/GDPR ที่ทีม security ตรวจ · pentester ที่มอง log leak เป็นจุดเด่นในสาย AppSec

📝 Quiz ท้ายบท

ข้อ 1: structured log (JSON) ดีกว่า console.log string ยังไงใน production?

ค้น/filter/alert ได้ (log system อ่าน JSON) — production มี log ล้านบรรทัด ต้องค้น (userId นี้, action นี้, error ช่วงนี้) · string ค้นยาก, ไม่มี level/timestamp/context มาตรฐาน · log ที่ค้นไม่ได้ = ไร้ค่าตอนวิกฤต

ข้อ 2: correlation ID (requestId) ช่วย debug ยังไง?

1 request ผ่านหลาย log — ถ้าทุก log มี requestId เดียวกัน filter เห็น lifecycle ครบ (เข้า→query→error→response) · ไม่มี = log ปนกัน แยกไม่ออกว่าอันไหนของ request ไหน · จำเป็นมากใน distributed system

ข้อ 3 (security): ทำไม log password/token เป็นช่องโหว่ แม้ DB ปลอดภัย?

log ไปอยู่ใน log system ที่เข้าถึงได้กว้างกว่า DB (ทีม ops, third-party service) · token ใน log = ใครเห็นก็สวมรอยได้ · password/PII = ผิด PDPA/GDPR · redact หรือเลือก log เฉพาะ field ปลอดภัย (userId ไม่ใช่ user ทั้ง object)

ข้อ 4: ทำไม app ควร log ออก stdout ไม่เขียนไฟล์เอง?

12-Factor — แยกหน้าที่: app สร้าง log (stdout), platform (Docker/cloud) จัดการเก็บ/ส่งต่อ (aggregator) · ทำให้ scale/ย้าย platform ง่าย, ไม่ต้องจัดการ log rotation เอง · container log → stdout เป็นมาตรฐาน

✅ Checkpoint

  • pino structured logging (JSON + context)
  • request logging + correlation ID (pino-http)
  • log levels เหมาะสม · redact secret
  • เข้าใจว่าทำไมไม่ log PII/secret + log ออก stdout
