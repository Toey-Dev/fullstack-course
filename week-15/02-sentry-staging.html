Week 15 · บทที่ 2

Error Tracking (Sentry) + Staging Environment

รู้ทันทีเมื่อ production พัง — และมีที่ทดสอบก่อนของจริง

🔑 คีย์เวิร์ดภาพจำประจำบท

🚨 พังแล้วเด้งเตือน = Sentry 🔬 stack trace + context = รู้ว่าพังตรงไหน 🎭 ที่ซ้อมก่อนจริง = staging 🏷 รู้ว่า version ไหนพัง = release tracking

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "อย่ารอผู้ใช้บอกว่าพัง — ระบบต้องบอกคุณก่อน"

ถ้ารู้ว่าพังตอนผู้ใช้ร้องเรียน = สายไป (ผู้ใช้เจอ error ไปแล้วหลายคน) · error tracking (Sentry) เด้งแจ้งทันทีที่มี error พร้อม stack trace + ผู้ใช้ที่เจอ + เกิดบ่อยแค่ไหน · คุณแก้ก่อนผู้ใช้ส่วนใหญ่จะเจอ · proactive ไม่ใช่ reactive

หลักการ 2: "Error ต้องมี context — ไม่ใช่แค่ 'something failed'"

error ที่มีประโยชน์บอก: เกิดที่ไหน (stack trace), ใครเจอ (user), ตอนทำอะไร (request), version ไหน (release) · Sentry รวมทั้งหมด → debug ได้โดยไม่ต้องเดา · ต่างจาก log "error" เปล่า ๆ ที่ต้องไล่หาเอง

หลักการ 3: "Staging — ซ้อมใหญ่ก่อนแสดงจริง"

staging = สำเนาของ production (เหมือนกันทุกอย่าง แต่ไม่ใช่ของจริง) · deploy ขึ้น staging ก่อน → ทดสอบ → ค่อยขึ้น production · จับบั๊กที่ test ไม่เจอ (integration กับของจริง, config) ก่อนผู้ใช้เจอ · "ไม่ทดสอบบน staging = ผู้ใช้เป็น tester"

ส่วนที่ 1: Sentry Setup

npm install @sentry/node

// src/instrument.ts (โหลดก่อนสุด)
import * as Sentry from "@sentry/node";
Sentry.init({
  dsn: process.env.SENTRY_DSN,            // จาก Sentry project (ใน .env)
  environment: process.env.NODE_ENV,      // แยก staging/production
  release: process.env.GIT_SHA,           // ⭐ รู้ว่า version ไหนพัง
  tracesSampleRate: 0.1,                  // sample performance (ไม่เก็บทุก request)
});

// Express — error handler ส่ง error ไป Sentry (ต่อจาก Week 3)
Sentry.setupExpressErrorHandler(app);    // หลัง routes ก่อน error handler เรา

Sentry จับ unhandled error อัตโนมัติ + ส่ง stack trace, request, user, release ไป dashboard · เด้ง email/Slack เมื่อมี error ใหม่

ส่วนที่ 2: Capture แบบมี Context

// จับ error เองพร้อม context (เมื่อต้องการรายละเอียดเพิ่ม)
try {
  await borrowBook(userId, bookId);
} catch (err) {
  Sentry.captureException(err, {
    tags: { feature: "borrow" },
    extra: { bookId },              // context ช่วย debug
    user: { id: userId },           // ใครเจอ (ไม่ใส่ PII เกินจำเป็น)
  });
  throw err;   // โยนต่อให้ error handler ตอบ user
}
🔐 มุม pentester: Sentry ก็คือ log system — ไม่ส่ง secret/PII เข้า Sentry · ระวัง: Sentry เก็บ request body/headers อัตโนมัติ อาจมี token/password → ตั้ง beforeSend ลบ field อ่อนไหวก่อนส่ง · user context ใส่แค่ id (ไม่ใส่ email/ชื่อเต็มถ้าไม่จำเป็น) · PDPA/GDPR ครอบคลุม Sentry ด้วย

ส่วนที่ 3: Staging vs Production Environment

StagingProduction
คือสำเนา prod (ทดสอบ)ของจริง ผู้ใช้ใช้
ข้อมูลปลอม/copy (ไม่ใช่ของลูกค้าจริง)จริง
deploy เมื่อทุก merge mainหลัง staging ผ่าน (อาจ manual approve)
ใช้ทำทดสอบก่อนของจริง—
# แยกด้วย environment variable + compose/branch ต่างกัน
# staging: NODE_ENV=staging, DB แยก, domain staging.app.com
# production: NODE_ENV=production, DB จริง, domain app.com
# CI/CD: merge main → deploy staging → (approve) → deploy production
เริ่มเล็ก: ไม่ต้องมี staging แยกเครื่องตั้งแต่แรก — เริ่มจาก compose staging บนเครื่องเดียว/branch แยก · สิ่งสำคัญคือ "มีที่ทดสอบก่อน production" ไม่ใช่ infrastructure หรู

ส่วนที่ 4: Monitoring + Uptime (เบื้องต้น)

  • Uptime monitor (UptimeRobot ฟรี) — ping /api/health (Week 2) ทุกนาที เด้งเตือนถ้าล่ม
  • Sentry alerts — error ใหม่/error เกิน threshold → Slack/email
  • Docker healthcheck (Week 12) — restart container ที่ป่วยอัตโนมัติ

3 ชั้นนี้ทำให้รู้ปัญหาเร็ว: healthcheck (restart เอง) → uptime (รู้ว่าล่ม) → Sentry (รู้ว่าพังตรงไหน) · เริ่มจากฟรี tier พอ

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: automation service — Sentry เด้งเตือนทันทีถ้า report job พัง (พร้อม clientId, แพลตฟอร์มที่ล่ม) · staging ทดสอบ logic สรุปใหม่กับข้อมูลจริงก่อนขึ้น production · uptime monitor ping /health · ลูกค้าไม่ต้องเป็นคนบอกว่าระบบล่ม

🌏 อุตสาหกรรม: Sentry/error tracking + staging + monitoring เป็นมาตรฐานของทุก production · release tracking ทำให้รู้ว่า "deploy ไหนทำให้ error พุ่ง" (rollback ได้ทันด้วย Week 14) · log/Sentry security (ไม่รั่ว PII) เป็นข้อบังคับ compliance

📝 Quiz ท้ายบท

ข้อ 1: ทำไม error tracking (Sentry) ดีกว่ารอผู้ใช้บอกว่าพัง?

เด้งเตือนทันทีที่มี error (proactive) พร้อม stack trace + user + ความถี่ → แก้ก่อนผู้ใช้ส่วนใหญ่เจอ · รอผู้ใช้บอก = สายไป (เจอ error หลายคนแล้ว) + ไม่มีรายละเอียดพอ debug

ข้อ 2: release tracking ใน Sentry ช่วยอะไร เชื่อมกับ Week 14 ยังไง?

รู้ว่า error เกิดใน version (release/SHA) ไหน → "deploy นี้ทำให้ error พุ่ง" → rollback version นั้น (Week 14 tag SHA) · เชื่อม deploy กับ error: เห็นว่า commit ไหนทำพัง แก้/กลับได้ตรงจุด

ข้อ 3 (security): ส่ง error เข้า Sentry มีความเสี่ยงอะไร แก้ยังไง?

Sentry เก็บ request body/headers อัตโนมัติ → อาจมี token/password/PII รั่วเข้า Sentry (ระบบภายนอก) · แก้: beforeSend ลบ field อ่อนไหวก่อนส่ง, user context ใส่แค่ id, ไม่ส่ง secret · PDPA/GDPR ครอบ Sentry ด้วย

ข้อ 4: ทำไมต้องมี staging ไม่ deploy ขึ้น production ตรง ๆ?

staging = สำเนา prod ทดสอบก่อน — จับบั๊กที่ test ไม่เจอ (integration จริง, config, env) ก่อนผู้ใช้เจอ · deploy prod ตรง = ผู้ใช้เป็น tester (เจอบั๊กแทนเรา) · เริ่มเล็กได้ (compose staging) สำคัญคือ "มีที่ทดสอบก่อน"
