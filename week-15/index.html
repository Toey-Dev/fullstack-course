deploy ขึ้น production แล้ว (Week 14) — แต่ถ้าพังกลางดึก คุณรู้ได้ยังไง? ผู้ใช้คนไหนเจอ error? · Week 15 เพิ่ม "ตา" ให้ระบบ: structured logging (pino) ที่ค้นได้, error tracking (Sentry) ที่แจ้งเตือนทันทีพร้อม stack trace, และ staging environment สำหรับทดสอบก่อนขึ้น production · นี่คือ observability ที่แยกระบบ "เล่น ๆ" จาก "production จริง"

  • Structured logging (pino) — log เป็น JSON ค้นได้ ไม่ใช่ console.log มั่ว
  • log levels + request logging + correlation ID (ตามรอย request เดียวได้)
  • Error tracking (Sentry) — รู้ทันทีเมื่อพัง + stack trace + ผู้ใช้ที่เจอ
  • Staging vs Production environment — ทดสอบก่อนของจริง
  • 🔐 ไม่ log secret/PII (มุม pentester) + monitoring/uptime เบื้องต้น

🧹 Clean Code ประจำ Week 15

1. Structured log ไม่ใช่ string มั่ว

❌ console.log string (ค้นไม่ได้)

console.log("user " + id + " borrowed " + bookId);
// ค้นหา/filter ใน production ลำบาก
// ไม่รู้ level, timestamp, context

✅ structured (JSON ค้นได้)

logger.info({ userId: id, bookId, action: "borrow" },
  "book borrowed");
// {"level":"info","userId":7,"bookId":3,...}
// filter ด้วย userId/action ได้ใน log system

2. ไม่ log secret/PII

❌ log ข้อมูลอ่อนไหว

logger.info({ user });  // มี password hash, email เต็ม
logger.info("token: " + jwt);  // 💀

✅ redact / เลือก field

logger.info({ userId: user.id });  // แค่ id
// pino redact: paths: ["password","token"]
กฎทอง: structured log (JSON) เสมอ · ใส่ context (userId, requestId) · ไม่ log secret/PII/password/token · log level เหมาะสม (error/warn/info/debug) · error ส่ง Sentry, log ปกติไป stdout
🤖 AI Code Review Drill #14 (ศุกร์ 30 นาที): ขอให้ AI เขียน logging ให้ login endpoint แล้วหาว่า log อะไรที่ไม่ควร log (password? token? email เต็ม? credit card? — PII/secret ที่รั่วผ่าน log) — log leak คือช่องโหว่ที่ pentester เจอบ่อย (log ไปอยู่ใน log system ที่อาจเข้าถึงได้กว้าง) จดลง notes/ai-review-log.md
