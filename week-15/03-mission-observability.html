Week 15 · Mission ใหญ่

👁 เพิ่ม Observability ให้ library-system

ทำให้ระบบ "มองเห็นได้" — logging, error tracking, staging · ปิดเฟส 2 สมบูรณ์

ทำไม Mission นี้ปิดเฟส 2 ได้สมบูรณ์

หลังจบ library-system จะ "ดูแลได้จริง": รู้ว่าเกิดอะไรขึ้น (structured log), รู้ทันทีเมื่อพัง (Sentry), มีที่ทดสอบก่อนของจริง (staging), และ log ไม่รั่ว secret (มุม pentester) · นี่คือสิ่งที่ทำให้ library-system เป็น "production-grade application" เต็มตัว — portfolio piece ที่เล่า interview ได้ครบทั้ง dev + ops + security

🪜 แบ่งเป็น 4 Mission ย่อย

Mission 15.1 — Structured Logging (pino) 75 นาที

  1. setup pino logger + pino-http (request logging + correlation ID)
  2. แทน console.log ทั้งหมดด้วย structured log (context + level เหมาะสม)
  3. redact secret (authorization, password, token) — ไม่ log ความลับ
✅ เทสต์: log เป็น JSON มี requestId · filter requestId เห็น lifecycle · ไม่มี secret ใน log

Mission 15.2 — Error Tracking (Sentry) 60 นาที

  1. Sentry init + setupExpressErrorHandler + release (GIT_SHA)
  2. beforeSend กรอง secret/PII ก่อนส่ง
  3. ทดสอบ: ทำให้ error เกิด → เห็นใน Sentry dashboard (พร้อม context)
security: ตรวจว่า request body/headers ที่ส่งเข้า Sentry ไม่มี token/password (beforeSend)

Mission 15.3 — Staging Environment 60 นาที

  1. แยก env staging (NODE_ENV, DB แยก, SENTRY environment)
  2. CI/CD: merge main → deploy staging → (approve) → production (ต่อจาก Week 14)
  3. uptime monitor (UptimeRobot) ping /api/health
✅ เทสต์: deploy ขึ้น staging ก่อน · Sentry แยก staging/prod · uptime เด้งถ้า /health ล่ม

Mission 15.4 — Audit log security + push 45 นาที

  1. ไล่ตรวจ log ทั้งระบบ: ไม่มี password/token/PII หลุด (เขียนใน SECURITY.md)
  2. README: observability section (log, Sentry, monitoring) + อัปเดต architecture
  3. push + AI Usage
✅ เทสต์: review log จริง — ไม่มี secret · SECURITY.md ระบุการป้องกัน log leak
