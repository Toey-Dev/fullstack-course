OWASP Top 10 — มุมมองผู้สร้างกำแพง

คุณรู้จัก OWASP จากฝั่งโจมตี — บทนี้พลิกเป็นฝั่งป้องกัน ทีละข้อ

🔑 คีย์เวิร์ดภาพจำประจำบท

🚪 A01 ไม่มีสิทธิ์แต่เข้าได้ = Broken Access Control 🔐 A02 crypto/secret พลาด 💉 A03 injection (SQL/XSS) ⚙️ A05 misconfig 🌐 A10 SSRF

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "คิดแบบผู้โจมตี เพื่อป้องกันแบบผู้สร้าง"

OWASP Top 10 = 10 ช่องโหว่ที่พบบ่อยสุดในเว็บ · คุณเคยใช้มันเจาะ — ตอนนี้ใช้มันเป็น "checklist ป้องกัน" · ข้อได้เปรียบ: dev ที่ไม่เคย pentest ป้องกันแบบท่องจำ (ใส่ helmet เพราะเขาบอก) · คุณป้องกันเพราะ "รู้ว่าถ้าไม่ทำจะโดนยังไง" → ตรงจุดกว่า

หลักการ 2: "Security ไม่ใช่ feature เพิ่มทีหลัง — เป็นส่วนหนึ่งของทุกบรรทัด"

ช่องโหว่ส่วนใหญ่เกิดจากการ "ลืมคิด" ตอนเขียน (ลืม validate, ลืมเช็คสิทธิ์, ลืม escape) · secure coding = คิดเรื่อง security ขณะเขียนทุก feature ไม่ใช่ "เขียนเสร็จค่อยมา patch" · นี่คือ mindset ที่แยก secure dev จาก dev ทั่วไป

หลักการ 3: "ส่วนใหญ่ของ OWASP คือสิ่งที่เรียนมาแล้ว — แค่รวมเป็นระบบ"

access control (W5/9), injection (W4), crypto (W5), auth (W5), logging (W15) — เราทำมาทั้งคอร์ส · Week 17 แค่ "รวมเป็น checklist เดียว" + เติมที่ขาด (SSRF, misconfig) · เห็นภาพรวมว่าทุกอย่างที่เรียนคือ defense in depth

ส่วนที่ 2: A03 Injection — เจาะลึก XSS (เพิ่มจาก SQLi)

SQLi เรียนแล้ว (W4 parameterized) — เพิ่ม XSS (เจาะ frontend):

// ❌ Stored XSS — เก็บ input แล้ว render เป็น HTML
el.innerHTML = comment;   // comment = "<img src=x onerror=alert(document.cookie)>"
// → script รันในหน้าทุกคนที่เปิด

// ✅ Output encoding — render เป็นข้อความ ไม่ใช่ HTML
el.textContent = comment;            // browser escape ให้ (W0)
// React/Next.js escape อัตโนมัติเมื่อใส่ {comment} (ปลอดภัยโดย default)
// อันตราย: dangerouslySetInnerHTML — ใช้กับข้อมูลผู้ใช้ = เปิดประตู XSS
ชนิด injectionป้องกัน
SQL injectionparameterized query ($1) — ไม่ต่อ string (W4)
XSSoutput encoding (textContent / React escape) + CSP (บทที่ 2)
Command injectionไม่ส่ง input เข้า shell / ใช้ execFile ไม่ใช่ exec
มุม pentester ของคุณ: หลัก injection ทุกชนิดเหมือนกัน — "อย่าให้ input ผู้ใช้กลายเป็น 'คำสั่ง/โค้ด'" · SQL: แยกคำสั่งจากข้อมูล · XSS: แยกข้อความจาก HTML · คุณเข้าใจ root cause จากการเจาะ → ป้องกันได้ทุกรูปแบบ

ส่วนที่ 3: A10 SSRF — ของใหม่ที่สำคัญกับ Week 16

SSRF (Server-Side Request Forgery) = หลอกให้ server ยิง request ไปที่ผู้โจมตีกำหนด — อันตรายกับ service ที่ fetch URL จาก input:

// ❌ SSRF — รับ URL จากผู้ใช้แล้ว fetch ตรง
app.post("/fetch", async (req) => {
  const data = await fetch(req.body.url);   // url = "http://169.254.169.254/..." (cloud metadata!)
  // → ผู้โจมตีอ่าน credential ของ cloud / ยิง service ภายใน
});

// ✅ allowlist + validate
const ALLOWED = ["api.meta.com", "analytics.google.com"];
const url = new URL(req.body.url);
if (!ALLOWED.includes(url.hostname)) throw new Error("URL ไม่อนุญาต");
// + บล็อก private IP (127.x, 169.254.x, 10.x, 192.168.x)
โยงกับ Week 16: automation service ที่ fetch API ภายนอก — ถ้ารับ URL/endpoint จาก config ผู้ใช้ ต้อง allowlist (เฉพาะ domain ที่รู้จัก) + บล็อก internal IP · ไม่งั้นผู้โจมตีหลอกให้ยิง cloud metadata/service ภายในได้ · SSRF เป็นช่องโหว่ที่โตขึ้นมากในยุค cloud

ส่วนที่ 4: A04 Insecure Design + A05 Misconfig

A04 — Threat Modeling เบื้องต้น: ก่อนสร้าง feature ถามว่า "ผู้โจมตีจะใช้มันยังไง" · เช่น feature reset password — คิดเผื่อ: enumeration (รู้ว่า email ไหนมี), token เดาได้ไหม, link หมดอายุไหม · ออกแบบ security ตั้งแต่ design ไม่ใช่ patch ทีหลัง

A05 — Misconfiguration ที่พบบ่อย:

  • error message รั่วข้อมูลภายใน (W3 — ตอบ 500 กลาง ๆ)
  • security headers ขาด (บทที่ 2 — helmet)
  • default credential / debug mode เปิดใน production
  • directory listing / endpoint ที่ลืมปิด (เช่น /debug)

A04/A05 คือ "ลืมคิด" — checklist + threat model ช่วยจับก่อนขึ้น production

ส่วนที่ 5: ตัวอย่างประยุกต์

🏢 งานวี: automation service — A10 SSRF (allowlist API domain), A02 (key ใน env), A09 (ไม่ log key), A06 (scan dependency) · dashboard ลูกค้า — A01 (ownership: ลูกค้าเห็นแค่ของตัวเอง), A03 XSS (escape ข้อมูลที่แสดง) · คุณ map OWASP กับงานจริงได้เพราะเคยเจาะ

🌏 อุตสาหกรรม: OWASP Top 10 เป็นมาตรฐานที่ทุกทีม security ใช้เป็น checklist · "secure coding ตาม OWASP" เป็นข้อบังคับใน code review ของบริษัทที่ใส่ใจ · dev ที่เข้าใจ OWASP ลึก (จากการเจาะจริง) = ผู้สมัคร AppSec/Product Security ที่บริษัทตามหา

📝 Quiz ท้ายบท

ข้อ 1: หลัก "ป้องกัน injection" ทุกชนิด (SQL/XSS/command) เหมือนกันยังไง?

"อย่าให้ input ผู้ใช้กลายเป็นคำสั่ง/โค้ด" — SQL: แยกคำสั่งจากข้อมูล (parameterized $1) · XSS: แยกข้อความจาก HTML (encode/textContent) · command: ไม่ส่ง input เข้า shell · root cause เดียว: ปนข้อมูลกับคำสั่ง → แยกออกจากกัน

ข้อ 2: SSRF คืออะไร อันตรายยังไงกับ service ที่ fetch URL?

หลอกให้ server ยิง request ไปที่ผู้โจมตีกำหนด · ถ้ารับ URL จาก input แล้ว fetch ตรง — ผู้โจมตีใส่ 169.254.169.254 (cloud metadata) อ่าน credential cloud / ยิง service ภายในได้ · ป้องกัน: allowlist domain + บล็อก private IP

ข้อ 3: ทำไม 8/10 ของ OWASP เราทำมาแล้วในคอร์ส แต่ยังต้องมี Week 17?

เพื่อ "รวมเป็นระบบ/checklist เดียว" + เติมที่ขาด (A04 design, A05 misconfig, A10 SSRF) · ทำให้เห็นภาพรวมว่าทุกอย่างที่เรียน (access control, injection, crypto, auth, logging) คือ defense in depth ที่ทำงานร่วมกัน ไม่ใช่เรื่องแยก ๆ

ข้อ 4: A04 Insecure Design ป้องกันด้วยอะไร ต่างจาก A05 ยังไง?

A04: threat modeling — คิด "ผู้โจมตีจะใช้ feature นี้ยังไง" ตั้งแต่ออกแบบ (ก่อนเขียน) · A05: misconfiguration — ตั้งค่าหละหลวม (error รั่ว, header ขาด, debug เปิด) แก้ที่ config/deployment · A04 = ปัญหาที่ design, A05 = ปัญหาที่ setup
