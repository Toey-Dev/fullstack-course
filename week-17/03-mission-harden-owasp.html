Week 17 · Mission ใหญ่

🛡 Harden library-system + Defense Report

ปิดช่องโหว่ตาม OWASP + เขียนรายงานป้องกัน — portfolio สาย security ที่ทรงพลัง

ทำไม Mission นี้คือ portfolio สาย B (AppSec)

หลังจบ library-system จะ hardened ตาม OWASP + มี OWASP Defense Report ที่คุณเขียนเองว่า "ปิดช่องโหว่อะไร ยังไง" · นี่คือเอกสารที่ตำแหน่ง Product Security/AppSec อยากเห็น และพิสูจน์ว่าคุณ "เข้าใจทั้งโจมตีและป้องกัน" — โปรไฟล์ที่หายากในไทย

🪜 แบ่งเป็น 4 Mission ย่อย

Mission 17.1 — Security Middleware 60 นาที

  1. helmet (security headers) ครอบทั้งแอป
  2. rate limiting: ทั่วไป + /login,/register เข้ม (counter ใน Redis ถ้าทำได้)
  3. CORS allowlist (เฉพาะ frontend domain ไม่ใช่ *)
✅ เทสต์: ดู response headers (helmet ใส่แล้ว) · ยิง /login รัว ๆ → โดน rate limit · CORS จาก origin แปลก → block

Mission 17.2 — ปิดช่องตาม OWASP 75 นาที

  1. map OWASP Top 10 กับ library-system — list ว่าข้อไหนปิดแล้ว/ยัง
  2. ปิดที่ยังขาด: CSP (XSS), error ไม่รั่ว (W3), validate ทุก input (Zod W4)
  3. ถ้ามี endpoint ที่ fetch URL → allowlist (SSRF A10)
คิดแบบ pentester: ลองโจมตีระบบตัวเองตาม OWASP (XSS ใน field, ยิง /login รัว, เรียก API ข้าม origin) → ปิดที่เจอ

Mission 17.3 — OWASP Defense Report 60 นาที

  1. เขียน SECURITY.md (ต่อจาก W12) — ตาราง OWASP Top 10: แต่ละข้อ "ป้องกันยังไง + โค้ดที่ไหน"
  2. ระบุ: ช่องที่เคยเปิด → วิธีปิด (เหมือนรายงาน pentest แต่ฝั่ง fix)
  3. เขียนสไตล์ที่ทีม security/recruiter อ่านเข้าใจ
✅ เทสต์: SECURITY.md อ่านแล้วเห็นว่าระบบป้องกัน OWASP ครบ — เป็น case study สาย security

Mission 17.4 — ทดสอบ + push 45 นาที

  1. เขียน test ยืนยัน security (rate limit คืน 429, CORS block, headers มี)
  2. ทดสอบมือ: ลอง attack แล้วถูกบล็อก
  3. push — README + SECURITY.md (OWASP) + AI Usage
✅ เทสต์: test security เขียว · attack ที่ลองถูกบล็อกจริง
