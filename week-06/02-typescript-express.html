TS ในงานจริง — Generics + Express + Zod

จาก type พื้นฐาน สู่การใช้ TS กับ Express และ Zod ที่คุณเขียนมาแล้ว

🔑 คีย์เวิร์ดภาพจำประจำบท

📦 กล่องที่ใส่ type อะไรก็ได้ = Generic <T> 🔧 แปลง type สำเร็จรูป = Utility types 🎫 Request<params,res,body> = typed Express 🪄 z.infer = type จาก Zod ฟรี 🔒 strict = TS เข้มสุด

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "DRY กับ type ด้วย — อย่าเขียน type ซ้ำ"

ถ้าเขียน type Book ไว้ใน 5 ที่ วันหน้าเพิ่ม field ต้องแก้ 5 ที่ ลืม 1 = ไม่ตรง · สร้าง type ที่เดียว แล้ว import ไปใช้ หรือ "แปลง" จาก type เดิม (utility types) — type ก็ต้อง DRY เหมือนโค้ด

หลักการ 2: "Single source of truth — schema กับ type ต้องตรงกันเสมอ"

คุณมี Zod schema (Week 4) ที่บอกว่าข้อมูลถูกต้องหน้าตายังไง · ถ้าเขียน TS type แยกอีกอัน = 2 ที่ต้อง sync กัน (พลาดง่าย) Zod infer ดึง type ออกจาก schema ให้ฟรี — แก้ schema ที่เดียว type ตามอัตโนมัติ · นี่คือพลังของการรวม validation+type

หลักการ 3: "Strict mode คือมาตรฐาน ไม่ใช่ option"

TS แบบหลวม (ไม่ strict) ปล่อย null/undefined ผ่านได้ = เสียประโยชน์ครึ่งหนึ่ง · strict mode บังคับจัดการ null, ห้าม implicit any — เข้มขึ้นแต่ปลอดภัยจริง · ทุกโปรเจกต์ production เปิด strict เป็น default

ส่วนที่ 1: Generics — "กล่องที่ใส่ type อะไรก็ได้"

// ❌ ไม่มี generic — เขียนซ้ำต่อ type
function firstString(arr: string[]): string { return arr[0]; }
function firstNumber(arr: number[]): number { return arr[0]; }

// ✅ generic <T> — T = type อะไรก็ได้ ที่ส่งมาตอนเรียก
function first<T>(arr: T[]): T {
  return arr[0];
}
first([1, 2, 3]);        // T = number → คืน number
first(["a", "b"]);       // T = string → คืน string

คุณใช้ generic มาแล้วโดยไม่รู้ตัว — Array<Book>, Promise<Book> คือ generic (Array ที่ใส่ Book, Promise ที่ resolve เป็น Book):

const books: Array<Book> = [];          // = Book[]
async function getBook(): Promise<Book> { ... }   // async คืน Promise ของ Book
เมื่อไหร่เขียน generic เอง: ตอนเขียนฟังก์ชัน/utility ที่ "ทำงานกับ type ไหนก็ได้" (เช่น API response wrapper) — งานทั่วไปใช้ generic ของคนอื่น (Array, Promise) มากกว่าเขียนเอง · เข้าใจไว้พออ่านโค้ดคนอื่นออก

ส่วนที่ 2: Utility Types — แปลง type สำเร็จรูป

TS มี "เครื่องมือแปลง type" ที่ใช้บ่อยมากในงานจริง — ไม่ต้องเขียน type ใหม่:

interface Book { id: number; title: string; author: string; available: boolean; }

Partial<Book>     // ทุก field เป็น optional → ใช้ตอน update (ส่งมาบางอัน)
Pick<Book, "title" | "author">   // เอาแค่บาง field
Omit<Book, "id">  // เอาทุก field ยกเว้น id → ใช้ตอน create (ยังไม่มี id)
Required<Book>    // ทุก field บังคับมี
Readonly<Book>    // แก้ไม่ได้
Utilityใช้จริงเมื่อ
Omit<Book, "id">type ของ "book ที่จะสร้าง" (ยังไม่มี id, DB gen ให้)
Partial<Book>type ของ "update" (ส่งมาแก้แค่บาง field)
Pick<User, "id"|"email">type ของ "user ที่ส่งกลับ client" (ตัด password)

utility types คือเหตุผลที่ "เขียน type ครั้งเดียว ใช้ได้หลายรูป" — Book → BookInput (Omit id) → BookUpdate (Partial) ทั้งหมดมาจาก type เดียว

ส่วนที่ 3: z.infer — type จาก Zod ฟรี (เชื่อม Week 4)

จุดที่ TS + Zod เข้าคู่กันสวยที่สุด — เขียน schema ครั้งเดียว ได้ทั้ง validation และ type:

import { z } from "zod";

// schema เดียว (จาก Week 4) — เป็นทั้ง validation และต้นทางของ type
const BookSchema = z.object({
  title: z.string().min(1),
  author: z.string().optional(),
  available: z.boolean().default(true),
});

// ดึง type ออกมาฟรี — แก้ schema ที่เดียว type ตามอัตโนมัติ
type BookInput = z.infer<typeof BookSchema>;
// = { title: string; author?: string; available: boolean }

function createBook(data: BookInput) { ... }   // ใช้ type ที่ infer มา
ทำไมนี่คือ best practice: ไม่มี type Book แยกที่ต้อง sync กับ schema — schema คือ single source of truth ทั้ง runtime (validate) และ compile-time (type) · นี่คือ pattern ที่ทีม TS ทั่วโลกใช้ และเป็นสิ่งที่ทำให้ Week 4 (Zod) + Week 6 (TS) ต่อกันลงตัว

ส่วนที่ 4: Type Express req/res

import { Request, Response } from "express";

// typed handler — บอก type ของ params/body
app.get("/api/books/:id", (req: Request, res: Response) => {
  const id = req.params.id;     // string เสมอ (จาก URL)
  res.json({ ok: true });
});

// type req.body ให้ตรงกับ Zod (generic ของ Request)
import { z } from "zod";
type BookInput = z.infer<typeof BookSchema>;

app.post("/api/books", (req: Request<{}, {}, BookInput>, res: Response) => {
  const { title, author } = req.body;   // TS รู้ type แล้ว auto-complete ให้
});

Request generic: Request<Params, ResBody, ReqBody> — ใส่ type body ให้ req.body มี type ถูกต้อง · ในงานจริงมักทำ typed middleware ห่อให้สวยกว่านี้

ส่วนที่ 5: Setup TS + tsconfig strict

npm install -D typescript tsx @types/node @types/express
npx tsc --init                # สร้าง tsconfig.json

// tsconfig.json — ค่าที่สำคัญ
{
  "compilerOptions": {
    "strict": true,            // ⭐ เปิดเข้มสุด (บังคับจัดการ null, ห้าม implicit any)
    "target": "ES2022",
    "module": "commonjs",      // หรือ esnext ตาม setup
    "outDir": "./dist",
    "rootDir": "./src"
  }
}

// package.json scripts
"dev": "tsx watch src/server.ts",    // tsx = รัน TS ตรง ๆ ตอน dev (เร็ว)
"build": "tsc",                       // compile เป็น JS ใน dist/
"start": "node dist/server.js"
strict ทำให้เจอ error เยอะตอนแรก — นั่นคือเรื่องดี: error เหล่านั้นคือบั๊กที่ JS ซ่อนไว้ เช่น book อาจเป็น undefined (หาไม่เจอ) ที่ JS ปล่อยผ่านจนพังตอนรัน · strict บังคับให้จัดการตั้งแต่ต้น · อย่าปิด strict เพื่อหนี error

ส่วนที่ 6: ตัวอย่างประยุกต์

🏢 งานวี: automation report — Zod schema ของ config ลูกค้า → infer type → ใช้ทั้ง validate request และ type ในโค้ดสรุป แก้ schema ที่เดียว ทั้ง runtime และ type ตามหมด · ไม่มีทาง config type ไม่ตรงกับที่ validate

🌏 อุตสาหกรรม: Zod + z.infer เป็น pattern มาตรฐานของ API ยุคใหม่ (tRPC, Next.js API ก็ใช้) · utility types (Omit/Partial/Pick) อยู่ในทุก codebase TS · เข้าใจ 2 อย่างนี้ = อ่านโค้ด production ทีมใหญ่ออก

📝 Quiz ท้ายบท

ข้อ 1: generic <T> แก้ปัญหาอะไร ยกตัวอย่างที่คุณใช้อยู่แล้ว

เขียนฟังก์ชัน/type ที่ทำงานกับ "type ไหนก็ได้" โดยไม่เสีย type safety (ไม่ต้องใช้ any) — เช่น first<T>(arr: T[]): T · ที่ใช้อยู่แล้ว: Array<Book> (= Book[]), Promise<Book> (async คืน Promise ของ Book)

ข้อ 2: อยาก type "book ที่จะสร้าง" (ยังไม่มี id) จาก interface Book — ใช้ utility อะไร?

Omit<Book, "id"> — เอาทุก field ยกเว้น id (เพราะ DB generate id ให้ตอน insert) · ดีกว่าเขียน interface ใหม่ เพราะถ้า Book เพิ่ม field, BookInput ตามอัตโนมัติ

ข้อ 3: ทำไม z.infer ดีกว่าเขียน type Book แยกจาก Zod schema?

schema เป็น single source of truth — แก้ schema ที่เดียว type ตามอัตโนมัติ · ถ้าเขียน type แยก = 2 ที่ต้อง sync เอง พลาดง่าย (validate กับ type ไม่ตรง = บั๊กที่หายากมาก) · z.infer รวม runtime validation + compile-time type เป็นหนึ่ง

ข้อ 4: strict mode ทำให้เจอ error เยอะตอนแปลง — ทำไมไม่ควรปิดมันหนี?

error เหล่านั้นคือบั๊กที่ JS ซ่อนอยู่จริง (เช่น ค่าอาจเป็น undefined ที่ไม่ได้จัดการ) · ปิด strict = หนีปัญหา ไม่ได้แก้ บั๊กยังอยู่ รอพังตอน production · จัดการให้ครบตอนนี้ = ปลอดภัยจริงและได้ประโยชน์ TS เต็มที่

✅ Checkpoint

  • เข้าใจ generic (อ่านโค้ดที่ใช้ <T> ออก)
  • ใช้ utility types (Omit/Partial/Pick) ได้
  • ใช้ z.infer ดึง type จาก Zod schema
  • setup tsconfig strict + tsx/build script
