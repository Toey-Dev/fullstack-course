TypeScript Basics — ทำไม + types พื้นฐาน

เข้าใจว่า TypeScript แก้ปัญหาอะไร ก่อนเรียน syntax — ไม่งั้นจะรู้สึกว่ามันแค่ "จุกจิก"

🔑 คีย์เวิร์ดภาพจำประจำบท

🔍 ตรวจก่อนรัน = compile-time check 📋 พิมพ์เขียวของข้อมูล = type/interface 🚪 อย่างใดอย่างหนึ่ง = union (|) 🤖 TS เดาเอง = type inference ⚠️ any = ปิดสวิตช์ TS

🧠 แนวคิด & หลักการของ Dev ประจำบทนี้

หลักการ 1: "เจอ error ตอนพิมพ์ ดีกว่าตอนผู้ใช้เจอ"

JavaScript เงียบเวลาผิด type — user.nmae (พิมพ์ผิด) รันได้ แต่ได้ undefined แล้วพังที่อื่นทีหลัง หาสาเหตุยาก TypeScript ตรวจตอน "compile" (ก่อนรัน) — พิมพ์ผิดปุ๊บ editor ขีดแดงทันที · นี่คือการย้าย bug จาก "production ตอนกลางคืน" มา "ตอนพิมพ์ตอนกลางวัน" ยิ่งระบบใหญ่ ยิ่งคุ้ม — library-system ที่จะโตอีกหลายสัปดาห์ต้องการความปลอดภัยนี้

หลักการ 2: "Type คือเอกสารที่บังคับให้ตรง"

function borrow(memberId, bookId) — JS ไม่บอกว่ารับอะไร (string? number? object?) TS: function borrow(memberId: number, bookId: number) — อ่านปุ๊บรู้ทันที และ บังคับว่าส่งผิด type ไม่ได้ type = comment ที่ไม่มีวันโกหก (เพราะถ้าโค้ดเปลี่ยน type ไม่ตรง TS เตือน)

หลักการ 3: "TS = JS + ชั้นตรวจสอบ (ลบออกตอนรัน)"

TypeScript ไม่ใช่ภาษาใหม่ทั้งหมด — มันคือ JavaScript + ระบบ type · ตอน build type ถูก "ลบทิ้ง" เหลือ JS ธรรมดารัน แปลว่าทุกอย่างที่คุณรู้จาก 5 สัปดาห์ที่ผ่านมา (async, array methods, Express) ใช้ได้หมด แค่เพิ่ม "ป้ายบอก type"

ส่วนที่ 1: ดูปัญหาที่ TS แก้ (ก่อนดู syntax)

// JavaScript — เงียบ พังตอนรัน
function getPrice(book) {
  return book.price * 1.07;
}
getPrice({ titel: "Clean Code" });   // พิมพ์ผิด title + ไม่มี price
// → undefined * 1.07 = NaN → พังเงียบ ๆ หาสาเหตุยาก

// TypeScript — เตือนตอนพิมพ์
function getPrice(book: { title: string; price: number }) {
  return book.price * 1.07;
}
getPrice({ titel: "Clean Code" });
// ❌ editor ขีดแดงทันที: "titel ไม่มีในtype, ขาด price"

นี่คือคุณค่าหลักของ TS — ไม่ใช่ "เขียนยากขึ้น" แต่ "ผิดยากขึ้น"

ส่วนที่ 2: Types พื้นฐาน

// primitive
let name: string = "Vee";
let age: number = 29;
let active: boolean = true;

// array
let scores: number[] = [80, 90];
let names: string[] = ["a", "b"];

// object
let user: { id: number; email: string } = { id: 1, email: "v@x.com" };

// function — type ของ param และ return
function add(a: number, b: number): number {
  return a + b;
}

// optional (?) และ default
function greet(name: string, title?: string) {  // title มีหรือไม่มีก็ได้
  return title ? `${title} ${name}` : name;
}

ส่วนที่ 3: Type Inference — อย่า annotate เกิน

TS ฉลาดพอเดา type เองจากค่า — ใส่ type ซ้ำที่มันเดาได้ = รก:

let name = "Vee";          // TS รู้เองว่า string (ไม่ต้องเขียน : string)
let nums = [1, 2, 3];      // TS รู้เองว่า number[]
const double = (n: number) => n * 2;   // param ต้องบอก, return TS เดาเองว่า number
กฎเมื่อไหร่ต้องใส่ type: ใส่ที่ "ขอบ" — input ฟังก์ชัน (TS เดา param ไม่ได้), return ของ API (เพื่อความชัด), ตัวแปรที่ยังไม่มีค่า (let book: Book) · ปล่อย infer ที่เหลือ — มือใหม่ชอบใส่ type ทุกที่ (รก) มืออาชีพใส่เท่าที่จำเป็น

ส่วนที่ 4: interface vs type — สร้าง "พิมพ์เขียว" ของข้อมูล

// interface — นิยมสำหรับ "รูปร่าง object"
interface Book {
  id: number;
  title: string;
  author?: string;       // optional
  available: boolean;
}

// type — ยืดหยุ่นกว่า (ทำ union, primitive ได้)
type Status = "available" | "borrowed" | "lost";   // literal union
type ID = number | string;                         // union

// ใช้งาน
const book: Book = { id: 1, title: "Clean Code", available: true };
const status: Status = "available";   // ใส่ค่าอื่น = TS เตือน
interfacetype
ใช้กับรูปร่าง object เป็นหลักได้ทุกอย่าง (union, primitive, object)
ขยาย (extends)extends ได้สวยใช้ & (intersection)
คำแนะนำobject/model → interfaceunion/alias → type

ในทางปฏิบัติ: ใช้อันไหนก็ได้สำหรับ object · ทีมส่วนใหญ่เลือกอันใดอันหนึ่งให้สม่ำเสมอ · จำง่าย: object→interface, union→type

ส่วนที่ 5: Union & Literal — "อย่างใดอย่างหนึ่ง"

// union — ได้หลาย type
function printId(id: number | string) {
  // ต้อง "narrow" ก่อนใช้ method เฉพาะ type
  if (typeof id === "string") return id.toUpperCase();  // ตรงนี้ TS รู้ว่า string
  return id.toFixed(0);                                  // ตรงนี้ TS รู้ว่า number
}

// literal union — จำกัดค่าที่เป็นไปได้ (ดีมากสำหรับ status/role)
type Role = "admin" | "user";
function setRole(role: Role) { ... }
setRole("admin");   // ✅
setRole("manager"); // ❌ TS เตือน: ไม่ใช่ค่าที่อนุญาต
literal union ทรงพลังกว่าที่คิด: แทน status: string (รับอะไรก็ได้ รวมค่าผิด) ใช้ status: "available" | "borrowed" — TS กันค่าผิดตั้งแต่พิมพ์ + editor auto-complete ให้ · ใช้กับ role, status, type ทุกที่

ส่วนที่ 6: ตัวอย่างประยุกต์

🏢 งานวี: automation report รับ config จากหลายแหล่ง — type Platform = "meta" | "line" | "ig" | "ga4" กันพิมพ์ชื่อแพลตฟอร์มผิด (typo ที่ JS จะปล่อยผ่านจน report เพี้ยน) · และ type ของข้อมูลที่ดึงมา ทำให้ refactor logic สรุปได้ปลอดภัย

🌏 อุตสาหกรรม: codebase ใหญ่ (Agoda แสนไฟล์) อยู่ไม่ได้ถ้าไม่มี TS — เพราะแก้จุดหนึ่งต้องรู้ว่ากระทบที่ไหน TS บอกได้ทันที (เปลี่ยน type → ทุกที่ที่ใช้ผิดขีดแดง) · นี่คือเหตุผลที่ TS = requirement ของบริษัทใหญ่ทุกที่

📝 Quiz ท้ายบท

ข้อ 1: คุณค่าหลักของ TypeScript คืออะไร (ไม่ใช่ "เขียนยากขึ้น")?

เจอ error ตอน compile (ก่อนรัน) แทนตอน runtime (ตอนผู้ใช้เจอ) — ย้าย bug จาก production มาตอนพิมพ์ · บวก: type = เอกสารที่บังคับให้ตรง + ทำให้ refactor codebase ใหญ่ปลอดภัย (เปลี่ยน type → ที่ผิดขีดแดงหมด)

ข้อ 2: เมื่อไหร่ควรใส่ type เอง เมื่อไหร่ปล่อย infer?

ใส่ที่ "ขอบ" — param ฟังก์ชัน (TS เดาไม่ได้), return API (ความชัด), ตัวแปรที่ยังไม่มีค่า · ปล่อย infer ตัวแปรที่มีค่าชัด (let x = 5 ไม่ต้องเขียน : number) · ใส่เกิน = รก, ใส่น้อยไป = เสียประโยชน์

ข้อ 3: ทำไม status: "available" | "borrowed" ดีกว่า status: string?

literal union จำกัดค่าได้แค่ที่อนุญาต — กันค่าผิด (typo, สถานะที่ไม่มีจริง) ตั้งแต่พิมพ์ + editor auto-complete ให้ · string รับอะไรก็ได้ รวมค่าผิด = เสียประโยชน์ TS

ข้อ 4: ใช้ any แก้ปัญหา type error — ทำไมเป็นนิสัยที่ไม่ดี?

any = "ปิดสวิตช์ TS" ตรงนั้น — TS เลิกตรวจ พิมพ์ผิด/เรียก method ที่ไม่มีก็ไม่เตือน กลับไปเป็น JS เงียบ ๆ ถ้าไม่รู้ type จริงใช้ unknown (บังคับ narrow ก่อนใช้) ปลอดภัยกว่า · any ใน codebase = จุดที่ bug ซ่อนได้

✅ Checkpoint

  • อธิบายได้ว่า TS แก้ปัญหาอะไร (compile-time vs runtime)
  • เขียน type พื้นฐาน + interface + union/literal เป็น
  • รู้ว่าเมื่อไหร่ใส่ type / เมื่อไหร่ปล่อย infer · เลี่ยง any
